### SCHEMA-NORM-01 — Surrogate FK for `construct_rule_triples` (perf)

**Problem**: `_pg_ripple.construct_rules` uses `name TEXT PRIMARY KEY`. Every row in `construct_rule_triples` (one row per derived triple per rule) stores the full rule name string as a TEXT foreign key. With millions of derived triples this wastes ~20–30 bytes per provenance row and causes text comparison in every retraction join.

**Deliverable**:
1. Add `id BIGINT GENERATED ALWAYS AS IDENTITY` to `_pg_ripple.construct_rules`; add `UNIQUE` constraint on `name`.
2. Replace `construct_rule_triples.rule_name TEXT` with `rule_id BIGINT NOT NULL REFERENCES _pg_ripple.construct_rules(id)`.
3. Update all Rust code in `src/construct_rules/` to look up `id` by name once and pass `rule_id` to insert/delete paths.
4. Migration script rewrites provenance rows via a JOIN on name.

**Affected files**: `src/construct_rules/delta.rs`, `src/construct_rules/retract.rs`, `src/construct_rules/mod.rs`, migration SQL.

### SCHEMA-NORM-02 — Surrogate FK for Datalog rule sets (perf)

**Problem**: `_pg_ripple.rule_sets` uses `name TEXT PRIMARY KEY`. The `rules.rule_set TEXT` column stores the set name in every rule row. `predicates.rule_set TEXT` and `rule_firing_log.rule_set TEXT` repeat it further. Every `WHERE rule_set = $1` query does a text scan; index lookups on TEXT are larger than on BIGINT.

**Deliverable**:
1. Add `id BIGINT GENERATED ALWAYS AS IDENTITY` to `_pg_ripple.rule_sets`; add `UNIQUE` on `name`.
2. Replace `rules.rule_set TEXT` with `rule_set_id BIGINT NOT NULL REFERENCES _pg_ripple.rule_sets(id)`.
3. Replace `predicates.rule_set TEXT` with `predicates.rule_set_id BIGINT REFERENCES _pg_ripple.rule_sets(id)`.
4. Replace `rule_firing_log.rule_set TEXT` with `rule_firing_log.rule_set_id BIGINT REFERENCES _pg_ripple.rule_sets(id)`.
5. Update all Rust code in `src/datalog/` to resolve name → id once per API call.

**Affected files**: `src/datalog/compiler.rs`, `src/datalog/explain.rs`, `src/datalog/dred.rs`, `src/lib.rs`, migration SQL.

### SCHEMA-NORM-03 — `rule_firing_log.rule_id TEXT` → `BIGINT` (perf)

**Problem**: `rule_firing_log.rule_id` is `TEXT` but `_pg_ripple.rules.id` is `BIGINT`. The firing log records one row per derived triple per rule firing. Storing and comparing a text rule identifier wastes space and prevents a direct FK constraint.

**Deliverable**:
1. Change `rule_firing_log.rule_id` to `BIGINT NOT NULL REFERENCES _pg_ripple.rules(id)`.
2. Update `explain_inference()` return type to return `rule_id BIGINT` (or join to `rules` and return `rule_set_id` + `rule_text` for human display).
3. Migration: if existing rows store a text representation of the integer id, cast in-place; otherwise clear the log (it is an ephemeral audit table).

**Affected files**: `src/datalog/explain.rs`, migration SQL.

### SCHEMA-NORM-04 — Drop redundant `construct_rules.target_graph TEXT` (perf)

**Problem**: `_pg_ripple.construct_rules` stores both `target_graph TEXT` (raw IRI string) and `target_graph_id BIGINT` (dictionary-encoded). The text column is never used for joins — only the BIGINT is passed to the SQL generator. It wastes ~50–200 bytes per rule row and can drift out of sync.

**Deliverable**:
1. Drop `target_graph TEXT` from `_pg_ripple.construct_rules`.
2. Update any Rust code that reads `target_graph` (display/debug) to decode via `dictionary::decode(target_graph_id)` instead.
3. Update `pg_ripple.register_construct_rule()` to stop inserting into the column.

**Affected files**: `src/construct_rules/mod.rs`, `src/construct_rules/catalog.rs`, migration SQL.

### SCHEMA-NORM-05 — `construct_rules.source_graphs TEXT[]` → `BIGINT[]` (perf)

**Problem**: `construct_rules.source_graphs` stores source graph IRIs as a text array. The hot `on_graph_delete()` path does `WHERE source_graphs @> ARRAY[$1]::text[]` — a text array containment check. Dictionary-encoding the array to `source_graph_ids BIGINT[]` enables integer containment (`@>`) which uses a faster operator class and eliminates string allocation.

**Deliverable**:
1. Add `source_graph_ids BIGINT[]` column to `_pg_ripple.construct_rules`.
2. On `register_construct_rule()`, populate `source_graph_ids` by encoding each IRI via `dictionary::encode()`.
3. Change `on_graph_delete(graph_iri)` to encode the IRI once and query `WHERE source_graph_ids @> ARRAY[$1::bigint]`.
4. Keep `source_graphs TEXT[]` as a nullable display column (or drop it after a migration grace period).

**Affected files**: `src/construct_rules/mod.rs`, `src/construct_rules/scheduler.rs`, migration SQL.

### SCHEMA-NORM-06 — `tenants.graph_iri TEXT` → `graph_id BIGINT` (consistency + perf)

**Problem**: `_pg_ripple.tenants.graph_iri TEXT` stores a raw IRI string while the entire rest of the system identifies graphs by dictionary-encoded `BIGINT`. Queries joining tenants to VP tables or named-graph catalogs must encode/decode on the fly.

**Deliverable**:
1. Add `graph_id BIGINT` column to `_pg_ripple.tenants`, populated by `dictionary::encode(graph_iri, KIND_IRI)`.
2. Add `UNIQUE` constraint on `graph_id`.
3. Keep `graph_iri TEXT` as a human-readable display column; add a generated or trigger-maintained constraint that they stay in sync.
4. Update `grant_graph()` / `revoke_graph()` to operate on `graph_id`.

**Affected files**: `src/admin/tenant.rs` (or equivalent), migration SQL.

### SCHEMA-NORM-07 — `federation_health.url TEXT` → `endpoint_id BIGINT FK` (perf)

**Problem**: `_pg_ripple.federation_health` is written after every `SERVICE` call — potentially the highest-insert-rate catalog table in the system. Each row stores the full URL string (often 50–150 bytes) instead of a FK to `federation_endpoints`. The composite index `(url, probed_at DESC)` used by `is_endpoint_healthy()` packs far fewer entries per page than an equivalent integer index.

**Deliverable**:
1. Add `endpoint_id BIGINT NOT NULL REFERENCES _pg_ripple.federation_endpoints(id)` to `federation_health` (requires SCHEMA-NORM-08 to add `id` to `federation_endpoints` first).
2. Drop `url TEXT` from `federation_health`; replace index with `(endpoint_id, probed_at DESC)`.
3. Update `is_endpoint_healthy()` and all probe-log writers in `src/sparql/execute.rs` (federation path) to resolve URL → id once and store the integer.

**Affected files**: `src/sparql/execute.rs`, `src/stats/`, migration SQL.

### SCHEMA-NORM-08 — `federation_cache.(url, query_hash)` PK normalization (perf)

**Problem**: The federation result cache primary key is `(url TEXT, query_hash TEXT)`. `url` is repeated across every cached result for the same endpoint. `query_hash` stores the XXH3-128 fingerprint as a 32-char hex string; the raw bytes are already 16 bytes, wasting another 16 bytes per row.

**Deliverable**:
1. Add `id BIGINT GENERATED ALWAYS AS IDENTITY` to `federation_endpoints`; add `UNIQUE` on `url`.
2. Replace `federation_cache.url TEXT` with `endpoint_id BIGINT NOT NULL REFERENCES _pg_ripple.federation_endpoints(id)`.
3. Change `federation_cache.query_hash TEXT` to `query_hash BYTEA NOT NULL` (store the raw 16-byte XXH3 hash, not hex-encoded).
4. Update the Rust lookup path to encode the hash as bytes before querying.

**Affected files**: `src/sparql/execute.rs`, migration SQL.

### SCHEMA-NORM-09 — `shape_hints.hint_type TEXT` → `SMALLINT` (perf)

**Problem**: `_pg_ripple.shape_hints` has composite PK `(predicate_id BIGINT, hint_type TEXT)`. `hint_type` takes exactly two values (`'max_count_1'`, `'min_count_1'`). Storing and comparing a string in a primary key that is consulted during every SPARQL query plan check adds needless overhead to index lookups.

**Deliverable**:
1. Replace `hint_type TEXT` with `hint_type SMALLINT NOT NULL` using constants `1 = max_count_1`, `2 = min_count_1`.
2. Add a `CHECK (hint_type IN (1, 2))` constraint.
3. Update all Rust code in `src/shacl/` that reads/writes `shape_hints` to use the integer constants.
4. Expose human-readable names via a simple helper or a view.

**Affected files**: `src/shacl/`, migration SQL.

### SCHEMA-NORM-10 — `embeddings.model TEXT` in composite PK → `model_id SMALLINT` (perf)

**Problem**: `_pg_ripple.embeddings` PK is `(entity_id BIGINT, model TEXT)`. At tens of millions of rows (one per entity per model), the repeated model string adds 8–20 bytes per row to the heap and index. With multiple embedding models, each row is denormalized.

**Deliverable**:
1. Create `_pg_ripple.embedding_models (id SMALLINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY, name TEXT NOT NULL UNIQUE)`.
2. Replace `embeddings.model TEXT` with `model_id SMALLINT NOT NULL REFERENCES _pg_ripple.embedding_models(id)`.
3. Migrate the `kge_embeddings` table the same way if it has a parallel `model TEXT` column.
4. Update all Rust and SQL code that reads/writes `embeddings` to resolve model name → id once.

**Affected files**: `src/export/`, `src/sparql/`, migration SQL.

### SCHEMA-NORM-11 — `inferred_schema.(class_iri, property_iri)` → dictionary-encoded BIGINT (consistency + perf)

**Affected files**: `src/stats/` (or wherever `schema_summary` is implemented), migration SQL.
