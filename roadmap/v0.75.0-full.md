### JOURNAL-DATALOG-01 — Wire mutation journal into Datalog/R2RML/CDC (CF-D, HF-C)

**Problem**: `grep -rn "mutation_journal::" src/datalog/ src/r2rml*.rs src/cdc*.rs` returns zero matches. Datalog materialization, R2RML materialization, and CDC ingest all write triples that never fire CONSTRUCT writeback rules. Metrics and provenance accounting are also bypassed.

**Deliverable**:
1. Audit each subsystem's write surface: identify whether writes go through `batch_insert_encoded` (which records per-graph journal entries) or via direct SQL INSERT.
2. For Datalog: if `batch_insert_encoded` is used, add `mutation_journal::flush()` at the end of every `materialize_*` function in `src/datalog/`. If direct SQL, route through `batch_insert_encoded` first.
3. Same pattern for `src/r2rml*.rs` and `src/cdc*.rs`.
4. Regression test for each: insert via Datalog rule → assert CONSTRUCT writeback fires; insert via R2RML → same; insert via CDC → same.

**Affected files**: `src/datalog/compiler.rs` (or whichever file contains materialize functions), `src/r2rml*.rs`, `src/cdc*.rs`, `src/cdc_bridge_api.rs`, `tests/pg_regress/sql/`.

### FLUSH-DEFER-01 — Per-statement flush deferral for `dict_api.rs` (MF-F)

**Problem**: `dict_api::insert_triple` and `dict_api::delete_triple` each call `mutation_journal::flush()` after every single quad. A loop of N inserts fires CWB N times — the original Assessment 10 CF-2 quadratic cost, now the only path that flushes correctly.

**Deliverable**:
1. Move the flush to a per-statement boundary: use an executor-end hook or a thread-local "pending flush" flag that is checked at statement boundary.
2. Ensure the single-triple API still flushes correctly (just not per-call).
3. Regression test: insert 100 triples via `pg_ripple.insert_triple()` in a DO block with a CWB rule; assert CWB fires exactly once (not 100 times).

**Affected files**: `src/dict_api.rs`, `src/lib.rs` (executor-end hook), `tests/pg_regress/sql/`.
