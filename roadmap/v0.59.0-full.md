## Shard-Pruning Design

### Detection in the algebra optimizer

After `spargebra` parses the SPARQL query and `sparopt` applies initial
rewrites, a new `CitusShardPruner` pass walks the algebra tree and detects
`TriplePattern` nodes where the subject position is a bound IRI or literal
(not a variable). For each such node it:

1. Looks up the subject IRI in the dictionary (`encode(iri)` → `i64`).
2. Computes `shard_id = (hash(subject_id) & 0x7FFF_FFFF) % shard_count`
   using the same algorithm Citus uses internally (32-bit positive modulo).
3. Replaces the logical VP delta table reference in the generated SQL with
   `_pg_ripple.vp_{pred_id}_delta_{shard_id}` — the physical shard table.

The shard count and distribution are read once from `pg_dist_shard` at
query-plan time and cached in the per-session SPARQL plan cache.

### Fallback

If the subject is not bound (variable) or the dictionary lookup fails
(unknown IRI), the pruner falls back to the existing full-fanout path.
This is safe and preserves correct query semantics.

### Interaction with FILTER

When a bound term is inside a FILTER (e.g. `FILTER(?s = <iri>)`) rather
than a direct triple pattern subject, the pruner does NOT apply — the
SQL generator would need to see the filter value at plan time. A future
release may extend this to cover FILTER-bound subjects.

## Rebalance NOTIFY Protocol

```
citus_rebalance() flow:
  pg_notify('pg_ripple.merge_start', '{"context":"rebalance","pid":PID}')
  pg_advisory_lock(0x5052_5000)          -- fence: wait for merge workers
  citus_rebalance_start()                -- Citus async job
  pg_advisory_unlock(0x5052_5000)
  pg_notify('pg_ripple.merge_end', '{"context":"rebalance","pid":PID}')
```

pg-trickle v0.34.0 processes these NOTIFY signals via the existing
`handle_vp_promoted` / scheduler wiring and uses them as a hint to
suspend per-worker slot polling until the rebalance completes.

## EXPLAIN Output (Citus section)

```sql
SELECT pg_ripple.explain_sparql(
  'SELECT ?p ?o WHERE { <http://example.org/Alice> ?p ?o }',
  citus := true
);
```

Returns additional JSON section:
```json
{
  "citus": {
    "pruned_to_shard": 7,
    "worker": "worker1:5432",
    "full_fanout_avoided": true,
    "estimated_rows_per_shard": 142
  }
}
```

## Migration: 0.58.0 → 0.59.0

No schema changes. Shard-pruning activates automatically when
`pg_ripple.citus_sharding_enabled = on` and Citus is detected.
