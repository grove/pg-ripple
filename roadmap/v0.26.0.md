# v0.26.0 — GraphRAG Integration

**Effort estimate: 4–6 person-weeks**

**Goal:** Microsoft GraphRAG is an open-source system that uses large language models to extract a knowledge graph from documents and answer complex questions far better than standard vector-search RAG. This release makes pg_ripple a drop-in backend for GraphRAG: entities and relationships are stored as RDF triples with full SPARQL queryability, Datalog reasoning derives relationships the LLM may have missed, and SHACL shapes reject malformed extractions before they corrupt the graph.

---

## GraphRAG RDF Ontology

**A ready-made vocabulary maps GraphRAG's data model to RDF.**

GraphRAG's output consists of entities, relationships, text units, communities, and community reports. This release defines an RDF vocabulary (using the `gr:` prefix) that represents all of these concepts and their properties in standard linked-data terms. The ontology file is loadable directly with `pg_ripple.load_turtle_file()` and the `gr:` prefix is pre-registered for compact use in SPARQL queries.

---

## BYOG Parquet Export

**Export the enriched knowledge graph back to Parquet for GraphRAG's community-detection step.**

GraphRAG's "Bring Your Own Graph" (BYOG) feature accepts pre-built entity and relationship tables as Parquet files. Three export functions — `pg_ripple.export_graphrag_entities()`, `export_graphrag_relationships()`, and `export_graphrag_text_units()` — query the knowledge graph via SPARQL and write Parquet files in exactly the column schema that GraphRAG expects. A Python CLI script (`scripts/graphrag_export.py`) provides the same capability for managed PostgreSQL environments where file-path functions are restricted.

---

## SHACL Shapes for Data Quality

**LLM extraction errors are caught by SHACL before they corrupt the graph.**

Large language models occasionally extract malformed entities: missing required fields, wrong data types, relationships to non-existent entities. The GraphRAG SHACL shapes enforce: every entity must have a title and type; relationships must have valid source and target entities; relationship weights must be between 0 and 1. When these shapes are active, bad LLM output is rejected with a structured error rather than silently polluting the knowledge graph.

---

## Datalog Enrichment Rules

**Implicit relationships that the LLM missed can be derived automatically.**

Four Datalog rules enrich the knowledge graph beyond what was explicitly extracted. `gr:coworker` derives coworker relationships between any two people who both have relationships to the same organisation. `gr:collaborates` derives collaboration when two entities appear together in the same text unit. `gr:indirectReport` derives the full management chain transitively. Running `pg_ripple.infer('graphrag_enrichment')` materialises all of these derived relationships, improving the quality of GraphRAG's community detection without needing to re-run the LLM extraction.

---

## End-to-End Example

**A complete worked example shows the full integration from load to export.**

The `examples/graphrag_byog.sql` file walks through the complete workflow: create a named graph, load sample entities and relationships as Turtle, run Datalog enrichment, validate with SHACL, query the enriched graph via SPARQL, and export to Parquet. A commented `settings.yaml` snippet shows how to point GraphRAG's `index` command at the exported files to run community detection over the enriched knowledge graph.
