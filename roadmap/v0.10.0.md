# v0.10.0 — Datalog Reasoning Engine

**Effort estimate: 8–12 person-weeks**

**Goal:** This is the largest and most ambitious release to date. It adds a complete Datalog reasoning engine that can derive new facts automatically from user-defined logical rules. Combined with built-in RDFS and OWL-RL rule sets, pg_ripple can now close a knowledge graph under standard ontological reasoning — deriving implicit facts that were not explicitly stored, while keeping everything inside the PostgreSQL database.

---

## Defining and Loading Rules

**Write logical rules in a simple syntax; pg_ripple derives their consequences.**

Datalog rules are logical implications: "if X and Y are true, then Z is true". For example, "if Alice manages Bob and Bob manages Carol, then Alice transitively manages Carol". You write these rules in a straightforward text format, load them with `pg_ripple.load_rules()`, and then call `pg_ripple.infer()` to materialize all their consequences. The derived facts are stored as `source = 1` (inferred) triples in the same VP tables as explicitly asserted facts, so SPARQL queries automatically see both.

---

## Built-in RDFS and OWL-RL Rule Sets

**Standard ontological reasoning comes pre-packaged.**

Rather than writing rules for common ontological patterns from scratch, `pg_ripple.load_rules_builtin('rdfs')` and `load_rules_builtin('owl-rl')` load the standard rule sets for RDFS property and class hierarchies and for OWL 2 RL reasoning. These cover the most common patterns in enterprise and linked-data knowledge graphs: subclass and subproperty inheritance, domain and range entailment, property chains, and symmetric/transitive/inverse properties.

---

## Named Rule Sets

**Multiple independent rule sets can coexist in the same database.**

Rules are grouped into named sets. You can have an `'rdfs'` set for standard RDFS rules, an `'owl-rl'` set for OWL reasoning, and custom domain-specific rule sets alongside each other. Each set can be independently loaded, enabled, disabled, and dropped. The `infer('rdfs')` call only runs the RDFS rules; `infer('all')` runs every enabled rule set in stratification order.

---

## Constraint Rules

**Rules that detect violations rather than deriving new facts.**

Beyond deriving new facts, Datalog rules can also express integrity constraints: conditions that should never be true. If a constraint rule fires, it means a data quality violation has been detected — for instance, two entities claimed to be the same (via `owl:sameAs`) but having logically inconsistent properties. Constraint violations are collected and surfaced through the same violation-reporting mechanism as SHACL.

---

## SPARQL On-Demand Datalog

**SPARQL queries can trigger inference inline, without pre-materialising everything.**

For predicates that do not have too many derived facts, pg_ripple can embed the Datalog derivation as an inline SQL CTE inside the SPARQL query itself, rather than pre-materialising all derived facts. This "on-demand" mode reduces storage overhead and avoids the need to re-run full inference after every data change — the derivation happens at query time, fresh for the current state of the graph.

---

## Goal Query API

**Ask for only the derived facts that answer a specific question.**

The `infer_goal()` function takes a rule set and a specific query goal (for instance, "what type is entity X?") and derives only the facts needed to answer that question, rather than computing the full closure of all rules. For large ontologies where full materialization would take minutes, goal-directed inference can answer targeted questions in milliseconds.
