# v0.27.0 — Vector + SPARQL Hybrid: Foundation

**Effort estimate: 5–7 person-weeks**

**Goal:** AI applications increasingly combine semantic graph queries with vector similarity search — finding entities that are "semantically close" to a concept, then refining with precise graph constraints. This release adds the foundation: vector embedding storage, k-nearest-neighbour search, and a SPARQL extension function that brings similarity into graph queries. The integration is optional: pg_ripple works fully without pgvector.

---

## Embedding Storage with HNSW Index

**Each entity in the knowledge graph can have a vector embedding.**

A new table `_pg_ripple.embeddings` stores one row per (entity, model) pair: the entity's dictionary ID, the embedding model name, and the embedding vector. By default the vector column uses pgvector's `vector(1536)` type and is indexed with an HNSW (Hierarchical Navigable Small World) index — the state-of-the-art approximate nearest-neighbour algorithm that provides the best recall-per-millisecond trade-off for most embedding workloads. Half-precision and binary quantization options reduce storage by 50% and 96% respectively for very large graphs where exact recall is less critical.

---

## Batch Embedding via OpenAI-Compatible APIs

**Load embeddings for all entities in one batch call to your embedding API.**

The `pg_ripple.embed_entities()` function queries the knowledge graph for all entities (using their `rdfs:label` as the text to embed), batches them, and calls any OpenAI-compatible embedding API (OpenAI itself, Ollama, vLLM, or any self-hosted model). Embeddings are stored with an upsert so that re-running the function updates stale embeddings and adds new ones. The embedding API URL and model are configured via GUC parameters.

---

## k-NN Similarity Search

**Find the entities most semantically similar to a query phrase.**

The `pg_ripple.similar_entities(query_text, k)` function encodes the query text using the configured embedding API, then uses pgvector's HNSW index to find the `k` nearest entities by cosine similarity. Results include the entity IRI and distance score. This function powers all downstream AI retrieval features: hybrid search, RAG pipelines, and entity alignment.

---

## `pg:similar()` SPARQL Extension Function

**Use vector similarity directly inside a SPARQL query.**

The SPARQL extension function `pg:similar(?entity, "query text", k)` returns the cosine similarity score between an entity's stored embedding and the query text. It can appear in BIND expressions and FILTER clauses. This means SPARQL queries can simultaneously apply precise graph constraints ("is this entity a drug that has no contraindications with aspirin?") and similarity ranking — with both components evaluated inside the same PostgreSQL query plan.

---

## Pre-computed Embedding Support

**Bring your own embeddings from external pipelines.**

Not every embedding needs to come from a live API call. The `pg_ripple.store_embedding(entity_iri, embedding_array)` function accepts a pre-computed embedding (as a PostgreSQL float array) and stores it directly. This is useful for embeddings computed by Knowledge Graph Embedding models (TransE, RotatE, ComplEx) which capture relational structure rather than text semantics, and for embeddings computed in batch by an external ML pipeline.
