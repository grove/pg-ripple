# v0.21.0 — SPARQL Built-in Functions & Query Correctness

**Effort estimate: 4–6 person-weeks**

**Goal:** About forty SPARQL 1.1 built-in functions were silently ignored in earlier releases — queries that used them returned wrong results without any error. This release implements all of them and fixes three known correctness bugs in query result handling. Unsupported expressions now produce named errors rather than silent failures.

---

## Forty Missing SPARQL Functions

**String manipulation, type testing, date arithmetic, and more now work correctly.**

SPARQL 1.1 defines a comprehensive library of built-in functions: string operations (`UCASE`, `LCASE`, `STRLEN`, `SUBSTR`, `CONCAT`), conditional evaluation (`IF`, `COALESCE`, `BOUND`), type testing (`isIRI`, `isBlank`, `isLiteral`, `DATATYPE`, `LANG`), numeric functions (`ROUND`, `CEIL`, `FLOOR`, `ABS`), date functions (`NOW`, `YEAR`, `MONTH`, `DAY`), and hash functions (`MD5`, `SHA1`, `SHA256`). Before this release, queries using these functions would either silently return wrong results (treating unknown functions as NULL) or produce query errors with unhelpful messages. All of them now work correctly.

---

## NULL Sort Order Fix

**SPARQL defines a specific sort order for NULL/unbound values; this release enforces it.**

The SPARQL 1.1 specification requires that unbound variables (NULL equivalents) sort after all bound values when ORDER BY is used. PostgreSQL's default NULL sorting puts NULLs last for ascending order, which happened to match, but the treatment was not always consistent. The sort order is now explicitly enforced for all ORDER BY clauses to match the SPARQL specification.

---

## Property Path Reflexive Row Fix

**Zero-length paths no longer generate phantom "self" rows.**

The zero-or-more path operator (`p*`) should return, among its results, the starting entity matched with itself — representing the zero-length path. In some cases the engine was generating multiple copies of this reflexive row when it should appear exactly once. The fix ensures that the reflexive case is deduplicated correctly.

---

## Named Errors for Unsupported Expressions

**Unknown functions now produce a clear error code, not silent wrong results.**

When a SPARQL query uses a function that is not supported, the engine previously treated the function call as producing NULL and continued. This meant a query expecting a string comparison might silently match nothing (instead of everything), producing data quality issues that were hard to diagnose. From this release onwards, an unrecognised built-in function or an unsupported expression raises a named `PT3xx` error code that clearly identifies what was not supported.
