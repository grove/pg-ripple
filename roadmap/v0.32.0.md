# v0.32.0 — Well-Founded Semantics & Tabling

**Effort estimate: 5–7 person-weeks**

**Goal:** Two advanced Datalog capabilities that handle scenarios the earlier engine could not. Well-founded semantics gracefully handles rule programs where negation creates mutual dependencies — instead of rejecting the program or looping forever, it returns a third answer: "unknown". Tabling caches sub-query results so that the same derivation requested multiple times (within one query or across many) is only computed once.

---

## Well-Founded Semantics

**Programs with mutually recursive negation get a principled partial answer instead of an error.**

Standard stratified Datalog (from v0.10.0) requires that if rule A uses the negation of predicate B, then B must be fully computed in an earlier stratum before A runs. Programs where A negates B and B negates A are "unstratifiable" and are rejected. Well-founded semantics (published by Van Gelder, Ross, and Schlipf in 1991, and used in production systems like XSB Prolog) solves this: it computes an alternating fixpoint that assigns facts as either "certainly true", "certainly false", or "unknown" (neither provably true nor provably false). The `pg_ripple.infer_wfs()` function runs the well-founded fixpoint and returns facts with a `certainty` annotation.

---

## Safe Convergence for Stratifiable Programs

**`infer_wfs()` produces identical results to `infer()` for programs without cyclic negation.**

For rule programs that can be stratified (the common case), well-founded semantics produces the same answer as standard stratified evaluation, just via a different algorithm. This means you can use `infer_wfs()` as a safe drop-in for `infer()` on programs you are unsure about — if the program is stratifiable, you get the same answer; if it has cyclic negation, you get the partial well-founded answer instead of an error.

---

## Tabling / Memoization

**Recurring sub-queries are computed once and their results are reused.**

In complex Datalog programs and multi-step SPARQL queries, the same sub-goal often appears multiple times: "find all ancestors of X" might be needed by three different rules in the same inference run, or by multiple SPARQL queries in the same session. Tabling stores the result of each sub-goal computation in a session-scoped cache keyed on the normalised goal pattern. When the same goal is requested again, the cached result is returned immediately — no re-computation. The cache expires after a configurable TTL and is cleared whenever the underlying triples change (via CDC hook) or when rules are dropped.

---

## Tabling Statistics

**Monitor cache performance with a built-in statistics function.**

The `pg_ripple.tabling_stats()` function returns a table showing each cached goal, how many times it has been served from cache (hits), how long the original computation took, and when it was cached. This makes it straightforward to identify which sub-goals benefit most from tabling and to tune the TTL for your workload.
