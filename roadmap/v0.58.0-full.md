## v0.58.0 — Temporal Queries, SPARQL-DL & Horizontal Sharding

**Theme**: Temporal RDF queries (`point_in_time`, valid_from/valid_to), SPARQL-DL axiom-level query routing, Citus horizontal sharding of VP tables, PROV-O automatic graph provenance, and final pre-v1.0.0 integration testing.

> **In plain language:** This release completes the major feature roadmap before the v1.0.0 production hardening milestone. Time-travel queries let analysts ask "what did the knowledge graph look like on 2024-01-01?" without storing separate snapshots. SPARQL-DL lets OWL reasoner queries appear inline in SPARQL. Citus sharding breaks the single-server I/O ceiling by distributing VP tables across multiple Citus worker nodes. PROV-O provenance turns pg_ripple into an auditable, lineage-aware knowledge store.
>
> **Effort estimate: 6–9 person-weeks**

### Deliverables

#### Feature L-1.3 — Temporal RDF Queries (Major)

- [ ] New SQL function `pg_ripple.point_in_time(ts TIMESTAMPTZ) RETURNS VOID` — sets a session-local variable `_pg_ripple.pit_threshold BIGINT` equal to the maximum statement ID inserted before `ts` by consulting `_pg_ripple.statement_id_timeline (sid BIGINT, inserted_at TIMESTAMPTZ)` (new table)
- [ ] `_pg_ripple.statement_id_timeline`: populated by an `AFTER INSERT` trigger on VP delta tables that records `(nextval('statement_id_seq'), now())`
- [ ] When `_pg_ripple.pit_threshold` is set, the SQL generator appends `AND i <= $1` to every VP table scan predicate
- [ ] `pg_ripple.clear_point_in_time() RETURNS VOID` — clears the session variable
- [ ] `pg_ripple.point_in_time_info() RETURNS TABLE(threshold_sid BIGINT, approximate_ts TIMESTAMPTZ)` SRF
- [ ] Valid-from / valid-to push-down: if a BGP contains `?triple schema:validFrom ?start . ?triple schema:validThrough ?end . FILTER(?ts >= ?start && ?ts <= ?end)`, the SQL generator rewrites this as a range-predicate join on the reified triple's statement ID
- [ ] pg_regress test: `tests/pg_regress/sql/temporal_rdf.sql` — insert 3 triples at different times; call `point_in_time(t2)`; verify only triples 1 and 2 appear

#### Feature L-1.4 — SPARQL-DL (Major)

- [ ] New module `src/sparql/sparqldl.rs`: detects BGPs that use OWL vocabulary axiom patterns (`owl:subClassOf`, `owl:equivalentClass`, `owl:disjointWith`, `owl:objectProperty`, `owl:inverseOf`, `owl:someValuesFrom`) as predicates where both subject and object are unbound or one-bound
- [ ] Routes such BGPs to `src/datalog/owl_index.rs` (in-memory T-Box index built at inference time) rather than emitting VP table SQL
- [ ] Mixed queries (SPARQL-DL + standard BGPs) resolved via lateral join: the DL result set is treated as a `VALUES` clause injected into the SQL plan
- [ ] pg_regress test: `tests/pg_regress/sql/sparql_dl.sql` — verify "all subclasses of :Mammal" query returns correct results from the T-Box without touching VP tables

#### Feature L-5.4 — Citus Horizontal Sharding (Major)

- [ ] New GUC `pg_ripple.citus_sharding_enabled` (bool, default off); guard with runtime check for Citus availability; raise PT536 if Citus is absent
- [ ] When enabled, `promote_vp_table()` calls `SELECT create_distributed_table('_pg_ripple.vp_{id}', 's')` after creating the VP table; the `s` (subject hash) column becomes the distribution column
- [ ] `merge_worker` uses `run_command_on_shards()` to execute merge operations per shard in parallel
- [ ] `pg_ripple.citus_rebalance() RETURNS BIGINT` — wraps `citus_rebalance_start()` with progress reporting
- [ ] `docs/src/operations/citus-sharding.md`

#### Feature L-8.4 — PROV-O Graph Provenance (Medium-Major)

- [ ] New GUC `pg_ripple.prov_enabled` (bool, default off); when on, all ingest and inference operations emit provenance triples
- [ ] `_pg_ripple.prov` named graph IRI: `urn:pg_ripple:prov`
- [ ] For `load_turtle()` / `load_rdf_xml()` / `load_ntriples()`: insert one `prov:wasDerivedFrom` triple per source file and one `prov:wasGeneratedBy` triple pointing to a `prov:Activity` blank node with `prov:startedAtTime`, `prov:endedAtTime`, `prov:used`
- [ ] For Datalog inference strata: insert `prov:wasInferredBy` triples linking inferred statement IDs to the rule that generated them
- [ ] `pg_ripple.prov_stats() RETURNS TABLE(source TEXT, triple_count BIGINT, last_activity TIMESTAMPTZ)` SRF
- [ ] pg_regress test: load a Turtle file; verify provenance triples exist in `urn:pg_ripple:prov`

### Migration Script

`sql/pg_ripple--0.57.0--0.58.0.sql`:
- Creates `_pg_ripple.statement_id_timeline (sid BIGINT, inserted_at TIMESTAMPTZ)` table with a BRIN index on `inserted_at`
- Creates `_pg_ripple.prov_catalog (source TEXT PRIMARY KEY, activity_iri TEXT, triple_count BIGINT, last_updated TIMESTAMPTZ)` summary table (used by `prov_stats()`)
- Installs the `AFTER INSERT` timeline trigger on VP delta tables (via a DO block that iterates `_pg_ripple.predicates`)

### Exit Criteria

- `point_in_time()` test passes: time-filtered SPARQL returns only triples inserted before the threshold.
- SPARQL-DL test returns correct subclass hierarchy without touching VP tables.
- Citus sharding test: VP table is distributed; SPARQL query fan-out to 2 workers returns correct results.
- PROV-O test: provenance triples appear in `urn:pg_ripple:prov` after `load_turtle()`.
- `tests/integration/v1_readiness/` suite passes cleanly 3 consecutive times in CI.
- All CI gates previously set to `continue-on-error: true` are flipped to blocking.
