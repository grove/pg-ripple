## v0.58.0 — Temporal Queries, SPARQL-DL & Horizontal Sharding

**Theme**: Temporal RDF queries (`point_in_time`, valid_from/valid_to), SPARQL-DL axiom-level query routing, Citus horizontal sharding of VP tables, PROV-O automatic graph provenance, and final pre-v1.0.0 integration testing.

> **In plain language:** This release completes the major feature roadmap before the v1.0.0 production hardening milestone. Time-travel queries let analysts ask "what did the knowledge graph look like on 2024-01-01?" without storing separate snapshots. SPARQL-DL lets OWL reasoner queries appear inline in SPARQL. Citus sharding breaks the single-server I/O ceiling by distributing VP tables across multiple Citus worker nodes. PROV-O provenance turns pg_ripple into an auditable, lineage-aware knowledge store.
>
> **Effort estimate: 6–9 person-weeks**

#### Feature L-5.4 — Citus Horizontal Sharding (Major)

- [ ] New GUC `pg_ripple.citus_sharding_enabled` (bool, default off); guard with runtime check for Citus availability; raise PT536 if Citus is absent
- [ ] When enabled, `promote_vp_table()` calls `SELECT create_distributed_table('_pg_ripple.vp_{id}', 's')` after creating the VP table; the `s` (subject hash) column becomes the distribution column
- [ ] `merge_worker` uses `run_command_on_shards()` to execute merge operations per shard in parallel
- [ ] `pg_ripple.citus_rebalance() RETURNS BIGINT` — wraps `citus_rebalance_start()` with progress reporting
- [ ] `docs/src/operations/citus-sharding.md`
- [ ] **CDC compatibility — REPLICA IDENTITY**: after `create_distributed_table()`, execute `ALTER TABLE _pg_ripple.vp_{id}_delta REPLICA IDENTITY FULL` so that pg-trickle's per-worker logical replication slots receive full before/after row images for VP delta writes
- [ ] **CDC compatibility — merge fencing**: `merge_worker` emits `pg_notify('pg_ripple.merge_start', json_build_object('predicate_id', id, 'pid', pg_backend_pid())::text)` before swapping `vp_{id}_main`, and `pg_notify('pg_ripple.merge_end', ...)` after; pg-trickle can subscribe to these channels to hold its replication frontier during the DDL swap, preventing slot loss
- [ ] **CDC compatibility — VP promotion notifications**: `promote_vp_table()` calls `pg_notify('pg_ripple.vp_promoted', json_build_object('table', '_pg_ripple.vp_' || id, 'predicate_iri', iri)::text)` so pg-trickle's stable-naming foundation (v0.32.0) can discover newly promoted VP tables without polling `_pg_ripple.predicates`
- [ ] **CDC compatibility — co-location GUC**: new GUC `pg_ripple.citus_trickle_compat` (bool, default off); when on, `create_distributed_table()` is called with `colocate_with => 'none'` and the GUC documentation warns that any pg-trickle stream table that joins VP data must also be distributed on an `s BIGINT`-equivalent column to avoid coordinator re-routing
- [ ] **Citus reference tables — dictionary and predicates catalog**: when `citus_sharding_enabled` is on, promote `_pg_ripple.dictionary` and `_pg_ripple.predicates` to Citus reference tables via `SELECT create_reference_table('_pg_ripple.dictionary')` and `SELECT create_reference_table('_pg_ripple.predicates')`; without this, every VP table join to the dictionary is re-routed through the coordinator, defeating sharding entirely
- [ ] **`vp_rare` co-location exception**: `_pg_ripple.vp_rare (p, s, o, g, i, source)` cannot be straightforwardly distributed by `s` alone because its primary selectivity column is `p`; when `citus_sharding_enabled` is on, promote it to a reference table (`create_reference_table`) with a note in the docs that it is always coordinator-resident; auto-promotion to a dedicated VP table (and thus to a distributed table) is unaffected
- [ ] **Distributed sequence handling for `statement_id_seq`**: Citus does not support distributed sequences natively; when `citus_sharding_enabled` is on, replace per-row `nextval('statement_id_seq')` in VP delta table `DEFAULT` clauses with a coordinator-side sequence allocation block (fetch N values in bulk via `SELECT setval(...)` at load time); document the trade-off in `docs/src/operations/citus-sharding.md`
- [ ] **`pg_notify` worker propagation guard**: `pg_notify()` fired inside a Citus worker function does not propagate to the coordinator's LISTEN clients; ensure all `pg_notify('pg_ripple.merge_start', ...)`, `pg_notify('pg_ripple.merge_end', ...)`, and `pg_notify('pg_ripple.vp_promoted', ...)` calls are executed on the **coordinator** only (i.e., inside the `merge_worker` background process which runs on the coordinator, not inside `run_command_on_shards` lambdas); add an assertion in CI that notifications arrive when fired inside a multi-node Compose cluster
- [ ] **WAL DDL minimization**: pg-trickle v0.33.0 polls each Citus worker via `pg_logical_slot_get_changes()` and will see DDL WAL records (RELATION, TRUNCATE, TYPE messages) produced by the merge swap (DROP TABLE + RENAME) on worker shards; replace the `DROP TABLE … ; CREATE TABLE … AS SELECT` merge pattern with `TRUNCATE vp_{id}_main; INSERT INTO vp_{id}_main SELECT …` inside a transaction — TRUNCATE appears in the logical WAL as a `TRUNCATE` message (handled by most decoders) rather than a catalog-modification DDL event, significantly reducing decoder surprise; document the remaining DDL events pg-trickle's decoder must tolerate
- [ ] **Shard-name reverse-mapping in `vp_promoted` notification**: expand the `pg_notify('pg_ripple.vp_promoted', ...)` payload to include `shard_count INT` and `shard_table_prefix TEXT` (e.g., `_pg_ripple.vp_42_delta_`) alongside the logical table name; pg-trickle v0.33.0 needs this to enumerate shard names when creating per-worker logical replication slots, because the worker WAL stream exposes shard tables, not logical parent names
- [ ] **Merge fencing timeout and crash safety**: add GUC `pg_ripple.merge_fence_timeout_ms` (int, default 30000); wrap the merge fencing in a Rust struct implementing `Drop` so that `pg_notify('pg_ripple.merge_end', ...)` is emitted even on panic or error; if `merge_end` is not emitted within the timeout, pg-trickle must log a warning and resume slot polling autonomously — document this recovery contract in `docs/src/operations/citus-sharding.md`
- [ ] **`pg_ripple.enable_citus_sharding()` retrofit function**: add `pg_ripple.enable_citus_sharding() RETURNS TABLE(table_name TEXT, status TEXT)` — iterates `_pg_ripple.predicates`, calls `create_distributed_table` on each VP table, promotes `_pg_ripple.dictionary`, `_pg_ripple.predicates`, and `_pg_ripple.vp_rare` to reference tables, and returns a row per table with success/skip/error status; idempotent (safe to run on a fresh install or when Citus sharding is already partially enabled); required for retrofitting an existing pg_ripple deployment when Citus is installed later
- [ ] **Row identity contract for pg-trickle distributed stream tables over VP data**: document in `docs/src/operations/citus-sharding.md` that when pg-trickle materializes a distributed stream table from a VP-derived view, the natural row identity is `(s BIGINT, predicate_hash BIGINT, g BIGINT)` — the triple's dictionary-encoded subject, predicate, and graph — and that the pg-trickle `DELETE WHERE row_id IN (…)` pattern must be configured with this composite key; include a worked SQL example showing `CREATE STREAM TABLE … DISTRIBUTE BY s ROW IDENTITY (s, predicate_hash, g)`

### Migration Script

`sql/pg_ripple--0.57.0--0.58.0.sql`:
- Creates `_pg_ripple.statement_id_timeline (sid BIGINT, inserted_at TIMESTAMPTZ)` table with a BRIN index on `inserted_at`
- Creates `_pg_ripple.prov_catalog (source TEXT PRIMARY KEY, activity_iri TEXT, triple_count BIGINT, last_updated TIMESTAMPTZ)` summary table (used by `prov_stats()`)
- Installs the `AFTER INSERT` timeline trigger on VP delta tables (via a DO block that iterates `_pg_ripple.predicates`)

### Exit Criteria

- `point_in_time()` test passes: time-filtered SPARQL returns only triples inserted before the threshold.
- SPARQL-DL test returns correct subclass hierarchy without touching VP tables.
- Citus sharding test: VP table is distributed; SPARQL query fan-out to 2 workers returns correct results.
- `_pg_ripple.dictionary` and `_pg_ripple.predicates` are reference tables; verify with `SELECT logicalrelid, partmethod FROM pg_dist_partition WHERE logicalrelid::text LIKE '_pg_ripple.%'` showing `partmethod = 'n'` for both.
- `vp_rare` is a reference table; verify no coordinator fan-out on `vp_rare` queries under `EXPLAIN (VERBOSE)`.
- `statement_id_seq` bulk allocation is in use; verify no per-row `nextval()` calls appear on worker nodes under `pg_stat_statements`.
- All `pg_notify` calls arrive at a LISTEN client on the coordinator when triggered inside a 2-worker Compose cluster (CI assertion).
- Merge swap uses TRUNCATE+INSERT pattern; verify via `pg_logical_slot_peek_changes()` on a worker that no `TYPE` or catalog-modification messages appear in the WAL stream after a merge cycle.
- `pg_ripple.vp_promoted` payload includes `shard_count` and `shard_table_prefix`; verify the joint test can enumerate all shard tables from the notification alone without querying `pg_dist_shard`.
- `pg_ripple.merge_fence_timeout_ms` GUC exists; verify that killing `merge_worker` mid-merge and waiting for timeout causes zero slot-loss errors in pg-trickle logs.
- `pg_ripple.enable_citus_sharding()` is idempotent: run it twice on the same installation and verify the second run returns all rows with status `skip`.
- Row identity documentation exists in `docs/src/operations/citus-sharding.md` and the worked example query executes without errors against a 2-worker Citus cluster.
- PROV-O test: provenance triples appear in `urn:pg_ripple:prov` after `load_turtle()`.
- `tests/integration/v1_readiness/` suite passes cleanly 3 consecutive times in CI.
- All CI gates previously set to `continue-on-error: true` are flipped to blocking.
- All seven pre-implementation coordination checklist items (C-1 through C-7) are marked resolved.
