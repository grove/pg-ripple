# v0.16.0 — SPARQL Federation

**Effort estimate: 4–6 person-weeks**

**Goal:** Knowledge does not all live in one place. The SPARQL 1.1 Federation specification defines the `SERVICE` keyword so that a query can reach out to remote SPARQL endpoints and combine their results with local data in a single query. This release implements federation — with security controls that prevent the database from being used to probe internal network addresses.

---

## The SERVICE Keyword

**A single SPARQL query can pull data from multiple remote endpoints.**

The `SERVICE <endpoint-url> { … }` clause in a SPARQL query delegates that part of the query to a remote SPARQL endpoint, receives the results, and joins them with the local query. For example: "find all researchers in our local graph who have publications listed in the remote DBpedia endpoint". The remote service does its own work and returns SPARQL results; pg_ripple joins those results with whatever else is in the local graph. The entire operation is transparent to the application.

---

## Parallel Execution of Multiple SERVICE Clauses

**Independent SERVICE calls run concurrently, not sequentially.**

A query with two or more independent SERVICE clauses — where neither depends on the result of the other — will dispatch all of them in parallel and wait for all to complete before continuing. This is significantly faster than the naive sequential approach for queries that need to gather data from three or four different knowledge sources simultaneously.

---

## Endpoint Allowlist (SSRF Protection)

**Only pre-approved endpoints can be reached via SERVICE.**

A `pg_ripple.register_endpoint()` function maintains an allowlist of permitted federation target URLs. Any SERVICE clause pointing to an unregistered URL is rejected with a clear error. This prevents an attacker from using pg_ripple's HTTP client capability to probe internal services (the server-side request forgery vulnerability), which is a critical security requirement for any database that can initiate outbound network connections.

---

## Bind-Join Optimisation

**The engine passes known variable values to the remote service to reduce data transfer.**

When part of the query has already narrowed down the set of candidate values (for instance, only ten entity IRIs match the local graph pattern), the bind-join optimisation sends those ten values to the remote SERVICE endpoint as a filter, rather than asking the remote endpoint to return all its data and letting the local join filter afterwards. This can reduce network traffic and remote server load by orders of magnitude.

---

## Configurable Timeouts and Error Handling

**Slow or unavailable endpoints do not crash the whole query.**

Each SERVICE call is subject to a configurable timeout (`pg_ripple.federation_timeout_ms`). If the remote endpoint is slow to respond or unreachable, the query raises a typed error rather than waiting indefinitely. The error handling mode is configurable: `warning` mode returns empty results for the failed SERVICE clause and continues, while `error` mode aborts the entire query.
