# v0.1.0 — Foundation

**Effort estimate: 3–5 person-weeks**

**Goal:** The very first release of pg_ripple lays the essential groundwork: install the extension into a PostgreSQL database and start storing and retrieving simple facts expressed as triples (subject → predicate → object). There is no SPARQL query language yet — just the fundamental ability to insert, find, and delete individual triples, with an internal dictionary that keeps everything compact and fast.

---

## Storing Facts as Triples

**Every fact is represented as a subject, predicate, and object.**

The core idea behind pg_ripple is that any piece of knowledge can be expressed as a triple: "Alice knows Bob", "the temperature is 21°C", "London is a city". This release provides the first SQL functions — `insert_triple`, `delete_triple`, and `find_triples` — to store and retrieve these triples inside a PostgreSQL database. It is deliberately minimal: no querying language, no reasoning, just the ability to put facts in and get facts out.

---

## The Internal Dictionary

**Every string is automatically converted to a compact number for storage.**

Behind the scenes, pg_ripple never stores raw text in its data tables. Every IRI (web address), blank node, or literal value is first looked up in an internal dictionary table, which assigns each unique string a 64-bit integer. The data tables then store only integers. This means joins, comparisons, and index lookups are all working with small fixed-size numbers rather than variable-length strings — the single most important performance decision in the architecture. The encoding uses a high-speed hash algorithm (XXH3-128) and a small in-memory cache to minimise repeated database round-trips.

---

## Statement Identifiers

**Every stored triple gets a unique, monotonically increasing identifier.**

From the very first release, each triple is stamped with a globally unique statement identifier (SID) drawn from a shared PostgreSQL sequence. This turns out to be important later for incremental maintenance, provenance tracking, and the HTAP storage split — but establishing it now means the column is there in every VP table from day one, with no disruptive migration later.

---

## Vertical Partitioning

**Each distinct predicate gets its own table.**

Rather than a single enormous triples table, pg_ripple stores one table per unique predicate — a design called vertical partitioning (VP). This keeps each table narrow and highly cache-friendly: a query asking "who does Alice know?" only reads the `vp_knows` table, not a full scan of all facts. Each VP table has two B-tree indexes — one leading with the subject, one leading with the object — so lookups in either direction are equally fast.

---

## No Shared Preload Required

**The extension installs with a simple `CREATE EXTENSION` command.**

At this stage, pg_ripple does not need to be listed in the server's `shared_preload_libraries` configuration. Installing it is as simple as any other PostgreSQL extension. This changes in v0.6.0 when a shared-memory cache and background worker are introduced, but for the foundation release the experience is intentionally friction-free.
