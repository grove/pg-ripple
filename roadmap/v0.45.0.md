# v0.45.0 — SHACL Completion, Datalog Robustness & Crash Recovery

**Effort estimate: 6–8 person-weeks**

**Goal:** The last two SHACL Core constraints that were missing (`sh:equals` and `sh:disjoint`) are implemented, completing the SHACL Core feature set. SHACL violation messages are upgraded from internal IDs to human-readable IRIs. Crash-recovery scenarios that were identified in the v0.37.0 audit but deprioritised are now tested systematically. Parallel Datalog stratum evaluation gets correct transaction rollback coordination.

---

## `sh:equals` and `sh:disjoint`

**The final two SHACL Core constraints are implemented, completing the Core feature set.**

`sh:equals` validates that the set of values for one property is exactly equal to the set of values for another — for example, verifying that every `foaf:name` value of a person appears identically as their `rdfs:label`. `sh:disjoint` validates the opposite: that the two property's value sets share no common value — for example, that no IRI appears as both a `person:spouse` and a `person:parent`. These are the last two constraints in the SHACL Core specification that were unimplemented. With this release, pg_ripple implements the complete SHACL Core constraint set.

---

## Human-Readable IRIs in Violation Messages

**SHACL violations now say "ex:Alice" instead of "-3847291847329".**

Before this release, SHACL violation reports returned focus nodes and value nodes as raw 64-bit integers — the internal dictionary encoding. While technically correct, these reports were impossible for human operators to use directly. This release adds a post-processing step to the SHACL validation pipeline that decodes all focus node, value node, and constraint component IDs back to their IRI or literal forms before returning the violation table. The violation message column is also regularised to follow a template: "Focus node {iri}: value {value} violates {constraint} constraint."

## Parallel-Strata Rollback Coordination

**If any parallel Datalog worker fails mid-stratum, all workers roll back cleanly.**

The parallel stratum evaluator introduced in v0.35.0 dispatched independent rule groups to background workers. If a worker failed partway through (due to a constraint violation, out-of-memory, or a bug), previously the main process would mark the stratum complete anyway — leaving some derived facts committed and others not. This release adds rollback coordination: if any worker reports failure, the main process sends a rollback signal to all workers and the entire stratum is rolled back atomically. A pg_regress test verifies this with an injected failure at a known iteration.

---

## SHACL Async Pipeline Load Test

**The async validation pipeline sustains 10,000 writes per minute for 5 minutes without backlog growth.**

The SHACL async validation pipeline (introduced in v0.8.0) processes pending shapes asynchronously to avoid blocking writes. Under high write load, the pipeline can fall behind and accumulate a growing queue. This release's load test verifies that at 10,000 writes per minute, the queue depth stabilises — it does not grow unboundedly — for a 5-minute sustained period. A new monitoring function exposes queue depth as a metric for external alerting.

---

## Crash-Recovery Tests

**Two critical crash scenarios are now tested automatically: mid-inference kill and mid-promotion kill.**

Two crash scenarios identified in the v0.37.0 audit were deferred as "test-only" work. This release implements them. The first test kills the PostgreSQL backend mid-way through a Datalog fixpoint iteration and verifies that on restart, the knowledge graph is consistent (no partially-applied derived facts). The second test kills the server during a rare-predicate promotion (the operation that moves a predicate from the consolidated `vp_rare` table to its own VP table) and verifies that the promotion either completes fully or rolls back fully on restart, with no orphaned tables.
