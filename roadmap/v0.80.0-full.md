## FLUSH-02-01 — SPARQL Update Mutation Journal Flush

### Current state

`src/sparql/execute.rs` implements `sparql_update()` and `execute_delete_insert()`. Neither calls `mutation_journal::flush()` before returning. CONSTRUCT writeback rules that depend on updated data are therefore never triggered by SPARQL UPDATE statements, making the entire CONSTRUCT writeback subsystem a no-op for the primary mutation path.

`grep -rn "mutation_journal" src/sparql/execute.rs` returns 0 matches (verified 2026-04-30).

### What changes

1. Add `use crate::storage::mutation_journal;` import at the top of `src/sparql/execute.rs`.
2. At the successful-return point of `sparql_update()`, call `mutation_journal::flush(None)?;`.
3. At the successful-return point of `execute_delete_insert()`, call `mutation_journal::flush(None)?;`.
4. Add a `pg_test` in `tests/sparql_update_cwb.rs` that: installs a `CONSTRUCT { ?s ex:derived ?o } WHERE { ?s ex:base ?o }` rule, inserts a base triple via `sparql_update`, and asserts the derived triple is present.

### Deliverables

| ID | Deliverable |
|----|-------------|
| FLUSH-02-01a | `mutation_journal::flush()` call in `sparql_update()` |
| FLUSH-02-01b | `mutation_journal::flush()` call in `execute_delete_insert()` |
| FLUSH-02-01c | Regression test: INSERT → CONSTRUCT writeback |
| FLUSH-02-01d | Regression test: DELETE WHERE → CONSTRUCT retraction |

## JOURNAL-R2RML-01 — R2RML and CDC Mutation Journal Wiring

### Current state

`src/r2rml.rs`, `src/cdc.rs`, and `src/cdc_bridge_api.rs` contain zero calls to `mutation_journal::record_write()` or `mutation_journal::flush()`. Bulk loads via R2RML and incremental CDC ingestion silently bypass CONSTRUCT writeback.

### What changes

1. In `src/r2rml.rs`: after each batch VP insert, call `mutation_journal::record_write(predicate_id, triple_count);`. Call `mutation_journal::flush(None)?;` at the end of each mapping execution.
2. In `src/cdc.rs`: at each `apply_cdc_event()` triple insert, call `mutation_journal::record_write(predicate_id, 1);`. Call `mutation_journal::flush(None)?;` after each CDC batch commit.
3. In `src/cdc_bridge_api.rs`: delegate flushing to the `src/cdc.rs` path (no duplicate flush calls).
4. Add regression tests confirming that CONSTRUCT writeback fires after R2RML load and after CDC event ingestion.

### Deliverables

| ID | Deliverable |
|----|-------------|
| JOURNAL-R2RML-01a | `record_write` + `flush` in `src/r2rml.rs` |
| JOURNAL-R2RML-01b | `record_write` + `flush` in `src/cdc.rs` |
| JOURNAL-R2RML-01c | Delegation audit in `src/cdc_bridge_api.rs` |
| JOURNAL-R2RML-01d | Regression: R2RML load triggers CONSTRUCT writeback |
| JOURNAL-R2RML-01e | Regression: CDC event triggers CONSTRUCT writeback |

## PROPPATH-CYCLE-01 — Property-Path Cycle Clause Fix

### Current state

`src/sparql/property_path.rs` generates `WITH RECURSIVE … CYCLE o SET cycle_flag USING path_state` for property path queries. PostgreSQL's `CYCLE` clause tracks visited values of the listed columns. Listing only `o` means the CTE detects revisiting the same *object* vertex but does not detect a path that revisits the same *(subject, object) edge*, so directed graphs with parallel edges silently diverge.

### What changes

Change the CYCLE clause in the recursive CTE template from:
```sql
CYCLE o SET cycle_flag TO true DEFAULT false USING path_state
```
to:
```sql
CYCLE s, o SET cycle_flag TO true DEFAULT false USING path_state
```

PostgreSQL 18 supports multi-column `CYCLE` tracking; the change is backward-compatible with PG18.

Add regression tests with:
- A directed graph containing a parallel edge (same s and o, different predicates routed through the path)
- A graph with a self-loop (`s = o`)
- A graph that forms a cycle through 3 nodes

### Deliverables

| ID | Deliverable |
|----|-------------|
| PROPPATH-CYCLE-01a | CYCLE clause updated to `CYCLE s, o` |
| PROPPATH-CYCLE-01b | Regression: parallel-edge directed graph |
| PROPPATH-CYCLE-01c | Regression: self-loop detection |
| PROPPATH-CYCLE-01d | Regression: 3-node cycle detection |

## CACHE-RLS-01 — Plan Cache RLS Role Key

### Current state

`src/sparql/plan_cache.rs` keys each cached plan on the SPARQL query string (normalized). If two PostgreSQL roles with different RLS policies execute identical SPARQL queries, the second role receives a plan compiled under the first role's RLS context.

### What changes

Extend the cache key struct to include `current_user()` (or `SESSION_USER` if impersonation is in use) and the current value of `pg_ripple.inference_mode`. Existing cached entries become stale; the LRU eviction handles this naturally.

| CACHE-RLS-01b | Test: two users with different RLS receive different plans |

## SQL-INJ-01 — SQL Injection in `src/views.rs`

### Current state

`src/views.rs` constructs catalog INSERT statements using `format!()` with string interpolation of view name and rule IDs. An attacker who can supply a crafted view name or rule expression may inject arbitrary SQL into the `_pg_ripple.construct_views` catalog table.

### What changes

Replace all `Spi::run(format!("INSERT INTO …", escaped_name, …))` calls with `Spi::run_with_args("INSERT INTO … ($1, $2, …)", &[…])` using typed OID parameters. The function `quote_identifier()` from `pgrx` must still be used for any SQL identifiers that cannot be parameterised (e.g. table names), but all *values* must go through bind parameters.

### Deliverables

| ID | Deliverable |
|----|-------------|
| SQL-INJ-01a | All catalog INSERTs in `src/views.rs` use `Spi::run_with_args()` |
| SQL-INJ-01b | Security test: crafted view name containing SQL special characters is safely handled |

## SQL-INJ-02 — SQL Injection in `src/sparql/embedding.rs`

### Current state

`src/sparql/embedding.rs` constructs a filter clause via:
```rust
format!("AND e.model = '{}'", model_tag.replace('\'', "''"))
```
Manual quote escaping is error-prone and does not protect against all injection vectors (e.g. null bytes, Unicode escapes). `model_tag` is user-supplied through the SPARQL function call.

### What changes

Replace the interpolated filter with a parameterised predicate:
```rust
Spi::run_with_args("SELECT … WHERE … AND e.model = $1", &[model_tag.into_datum()])
```

### Deliverables

| ID | Deliverable |
|----|-------------|
| SQL-INJ-02a | Parameterised query in `src/sparql/embedding.rs` |
| SQL-INJ-02b | Security test: `model_tag` with embedded single quote and null byte |

## SSRF-RFC1918-01 — Full RFC-1918 SSRF Blocklist

### Current state

`is_blocked_host()` in `src/sparql/federation.rs` checks `127.0.0.1` and `::1` but does not block `172.16.x.x`, `192.168.x.x`, `169.254.x.x` (link-local), `10.x.x.x` (only prefix checked, not the full /8), or IPv6 Unique Local addresses (`fc00::/7`).

### What changes

Add CIDR-range checks using `std::net::IpAddr` prefix matching (no external dependency):

```rust
fn is_rfc1918(addr: IpAddr) -> bool {
    match addr {
        IpAddr::V4(v4) => {
            let octets = v4.octets();
            octets[0] == 10
                || (octets[0] == 172 && (16..=31).contains(&octets[1]))
                || (octets[0] == 192 && octets[1] == 168)
                || (octets[0] == 169 && octets[1] == 254) // link-local
                || octets[0] == 127
        }
        IpAddr::V6(v6) => {
            v6.is_loopback()
                || (v6.segments()[0] & 0xfe00) == 0xfc00 // fc00::/7
        }
    }
}
```

Also add DNS-rebinding protection: resolve the host once and check each resolved address against `is_rfc1918()` before opening the connection.

### Deliverables

| ID | Deliverable |
|----|-------------|
| SSRF-RFC1918-01a | Full RFC-1918 blocklist in `is_blocked_host()` |
| SSRF-RFC1918-01b | IPv6 Unique Local and loopback blocking |
| SSRF-RFC1918-01c | DNS rebinding protection (resolve-then-check) |
| SSRF-RFC1918-01d | Unit tests for all blocked ranges |

## HTTP-ERR-01 — HTTP Error Response Standardisation

### Current state

`pg_ripple_http` returns HTTP 200 with a plain-text error body for SPARQL syntax errors, and HTTP 500 with mixed plain-text/JSON bodies for some server errors. The W3C SPARQL 1.1 Protocol specification recommends HTTP 400 for malformed queries and HTTP 500 for server faults.

### What changes

1. Define a JSON error response type in `pg_ripple_http/src/common.rs`:
   ```rust
   #[derive(Serialize)]
   struct ErrorResponse { code: &'static str, message: String }
   ```
2. Update all HTTP handler error arms in `pg_ripple_http/src/routing.rs` to return `Json(ErrorResponse { … })` with the appropriate status code.
3. Map SPARQL parse errors (error code PT001) to HTTP 400.
4. Map internal server errors (error code PT500) to HTTP 500.
5. Set `Content-Type: application/json` on all error responses.

### Deliverables

| ID | Deliverable |
|----|-------------|
| HTTP-ERR-01a | `ErrorResponse` type in `common.rs` |
| HTTP-ERR-01b | All routing.rs error arms use `Json(ErrorResponse)` |
| HTTP-ERR-01c | SPARQL parse errors return HTTP 400 |
| HTTP-ERR-01d | Integration test: POST malformed SPARQL → HTTP 400 JSON body |

## EXPLORER-AUTH-01 — Graph Explorer Authentication

### Current state

`pg_ripple_http/src/routing/admin_handlers.rs` exposes a `GET /explorer` endpoint that serves a browser-based SPARQL CONSTRUCT visualiser. The handler function `explorer_page()` does not call `check_auth()`. The `/explorer` route is registered in `build_router()` without any authentication middleware.

Any HTTP client with network access to `pg_ripple_http` — including unauthenticated users — can render the full RDF graph, execute SPARQL CONSTRUCT queries, and explore the schema without credentials.

### What changes

Add `check_auth()` at the top of `explorer_page()`:

```rust
pub(crate) async fn explorer_page(
    State(state): State<AppState>,
    headers: HeaderMap,
) -> Response {
    if let Err(resp) = check_auth(&state, &headers) {
        return resp;
    }
    // … existing HTML response …
}
```

Update the route signature in `build_router()` to pass `State` and `HeaderMap` extractor parameters.

### Deliverables

| ID | Deliverable |
|----|-------------|
| EXPLORER-AUTH-01a | `check_auth()` added to `explorer_page()` handler |
| EXPLORER-AUTH-01b | Integration test: unauthenticated GET /explorer → HTTP 401 |
| EXPLORER-AUTH-01c | Integration test: authenticated GET /explorer → HTTP 200 HTML |
