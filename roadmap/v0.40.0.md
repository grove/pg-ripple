# v0.40.0 — Streaming Results, Explain & Observability

**Effort estimate: 9–11 person-weeks**

**Goal:** Three long-requested improvements arrive together. Large SPARQL queries can now stream results instead of materialising everything in memory. A greatly enhanced `explain_sparql()` function shows algebra, generated SQL, and execution plan in one call, with IRI-decoded output rather than raw integers. Two important query correctness bugs are fixed: OPTIONAL and property paths inside GRAPH patterns were silently broken for all predicates. OpenTelemetry spans connect pg_ripple activity to external APM tools.

---

## Streaming SPARQL Cursor API

**Large results are delivered in pages — no more out-of-memory on big CONSTRUCT queries.**

The `pg_ripple.sparql_cursor(query)` function returns a `SETOF RECORD` set-returning function that pages through results 1,024 rows at a time with batched dictionary decoding. Companion functions `sparql_cursor_turtle()` and `sparql_cursor_jsonld()` emit the results line by line in Turtle and JSON-LD formats respectively. On the HTTP service, a `POST /sparql/stream` endpoint wires these into a chunked-transfer-encoded response, so clients can start processing the first triples of a billion-triple CONSTRUCT before the last ones are even read from the database.

---

## Enhanced `explain_sparql()`

**See the full picture of how your SPARQL query is being executed.**

The enhanced `explain_sparql()` function returns a rich JSONB document with four key sections: the SPARQL algebra tree (showing how the query was parsed and optimised, with predicate IRIs decoded instead of raw integers), the generated SQL (again with IRIs decoded), the PostgreSQL `EXPLAIN (ANALYZE, BUFFERS)` plan for that SQL, and metadata about whether the plan was served from cache. When called with `analyze := true`, actual row counts and timing appear in the plan, making it straightforward to identify which join is doing the most work.

## Bug Fix: OPTIONAL and Property Paths Inside GRAPH Patterns

**These combinations were silently producing wrong results for all predicates.**

Two serious correctness bugs are fixed. The first: an OPTIONAL clause inside a GRAPH pattern was generating SQL that referenced a column that does not exist, causing queries to fail with a cryptic column error or silently return no results. The second: property path patterns inside GRAPH patterns had the same flaw. Both failures affected all predicates — not just rare ones — and were discovered through the W3C SPARQL 1.1 test suite integration. The fix propagates the named-graph filter into each VP table scan rather than applying it as a post-hoc condition on the assembled join result.

---

## Resource Governors

**Queries that would return more rows than configured are stopped before they exhaust memory.**

Three new GUC parameters limit the maximum rows that can be returned by a SPARQL query, derived by Datalog inference, or exported by an export function. When a limit is hit, the overflow action (configurable as `warn` or `error`) either emits a warning and returns the truncated result, or raises an error and cancels the operation. These limits protect shared database resources from runaway queries without requiring per-query LIMIT clauses.

---

## OpenTelemetry Tracing

**pg_ripple activity now emits spans to external APM and observability platforms.**

When `pg_ripple.tracing_enabled = on`, the extension emits OpenTelemetry spans for SPARQL parse, translate, and execute phases; for each merge cycle per predicate; for federation SERVICE calls; and for Datalog inference per stratum. The spans can be exported to stdout (for development) or to any OTLP-compatible collector (Jaeger, Zipkin, Grafana Tempo, Honeycomb) by setting `pg_ripple.tracing_exporter = 'otlp'`. Tracing is zero-overhead when disabled.
