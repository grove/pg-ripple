# v0.50.0 — Developer Experience & GraphRAG Polish

**Effort estimate: 4–6 person-weeks**

**Goal:** Two targeted improvements that make pg_ripple significantly easier to use for developers and AI/LLM application builders. The `explain_sparql()` function, already improved in v0.40.0, gets richer output including per-operator actual row counts with decoded IRIs — making it a practical debugging tool. The new `rag_context()` function bundles the complete RAG retrieval pipeline into a single call.

---

## Enhanced `explain_sparql()` — Decoded Operator Details

**Every node in the explain output shows actual IRIs, not raw internal integers.**

The `explain_sparql()` function returns a JSONB document showing how a SPARQL query was executed. Previously, predicate names, class names, and filter constants appeared as raw 64-bit integers in the algebra section — unhelpful unless you happened to know the dictionary encoding. This release adds full IRI decoding to the algebra tree: every predicate, class, and IRI constant is shown by its full IRI string (or a prefix-abbreviated form using registered prefixes). The generated SQL section also substitutes decoded IRIs. Combined with actual row counts from `EXPLAIN (ANALYZE)`, this gives a complete human-readable picture of exactly what the query engine did and where the time went.

---

## `rag_context()` — Complete RAG Pipeline in One Call

**Vector recall, graph expansion, JSON-LD context assembly, and optional LLM call — one function.**

Building a Retrieval-Augmented Generation pipeline with pg_ripple previously required calling several functions in sequence: `similar_entities()` for vector recall, custom SPARQL for graph expansion, `export_jsonld_framed()` for context assembly, and then an external LLM call. The new `pg_ripple.rag_context(question TEXT, k INT)` function does all of this: it converts the question to an embedding, finds the top-k most similar entities via HNSW search, executes a configurable SPARQL expansion query to retrieve related triples, assembles a JSON-LD context document from the results, and optionally calls the configured LLM (if `pg_ripple.llm_endpoint` is set) with the context and question to produce a final answer.

---

## Per-Operator Actual Rows in Plan Cache Reports

**Cache hit/miss information is now included at the operator level, not just the query level.**

When `explain_sparql(query, analyze := true)` is called, the `cache_hit` flag in the returned JSON previously indicated only whether the whole plan was served from the plan cache. The enhanced version reports cache status at the plan-node level: each operator in the algebra tree shows whether its sub-plan was satisfied from a cached intermediate result (when incremental query caching is enabled). For repeated queries over slowly changing data, this cache sub-plan information guides operators to set `sparql_cache_ttl` appropriately for each class of query.
