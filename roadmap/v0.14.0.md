# v0.14.0 — Administrative & Operational Readiness

**Effort estimate: 4–6 person-weeks**

**Goal:** A knowledge graph that cannot be safely maintained, backed up, or secured is not ready for production. This release adds the operational toolkit: maintenance commands for reclaiming space and rebuilding indexes, row-level security so different users can see only their authorised graphs, extension upgrade scripts, and comprehensive documentation for operators.

---

## Maintenance Commands

**Reclaim space, rebuild indexes, and compact the dictionary.**

Over time, deleted triples leave tombstones in the VP tables, the dictionary accumulates entries for terms no longer referenced, and VP table statistics drift from reality. Four new functions address this: `pg_ripple.vacuum()` removes tombstones and triggers a PostgreSQL VACUUM; `pg_ripple.reindex()` rebuilds all VP table indexes; `pg_ripple.compact()` reorganises the storage to reclaim fragmentation; and `pg_ripple.vacuum_dictionary()` removes dictionary entries with no remaining references. These should be run periodically, or after large bulk delete operations.

---

## Graph-Level Row-Level Security

**Control which users and roles can see which named graphs.**

The named graph system makes it straightforward to implement access control at the graph level. This release integrates with PostgreSQL's Row-Level Security (RLS) mechanism so that the `pg_ripple.sparql()` function and all query paths automatically filter out named graphs that the current role does not have SELECT privilege on. An administrator can grant access to specific named graph IRIs to specific roles, and those roles will only ever see the authorised subset of the knowledge graph.

---

## Extension Upgrade Scripts

**Users on older versions can upgrade without dropping and recreating the extension.**

PostgreSQL's extension upgrade mechanism requires explicit migration scripts that transform the schema from one version to the next. This release establishes the convention of shipping a `sql/pg_ripple--X.Y.Z--A.B.C.sql` file with every release, ensuring that users on any prior version can run `ALTER EXTENSION pg_ripple UPDATE` and arrive at the current version cleanly, without losing any data.

---

## Operator Documentation

**Comprehensive guides for backup, recovery, security, and upgrades.**

Four reference documents cover the most common operational concerns: backup and restore (how `pg_dump` and `pg_restore` interact with pg_ripple's extension tables and VP tables), error reference (every error code with its cause and remedy), security guide (privilege model, RLS configuration, SPARQL injection prevention), and contributing guide (how to set up the development environment and submit changes). These documents are written for database administrators who may not be familiar with RDF or SPARQL.
