## GeoSPARQL 1.1

**First-class spatial queries over RDF data.**
The v0.54.0 batteries-included image already bundles PostGIS 3.4. v0.56.0 wires the GeoSPARQL 1.1 function vocabulary (`geof:distance`, `geof:within`, `geof:intersects`, `geof:buffer`, `geo:asWKT`, `geo:hasSpatialAccuracy`) as SPARQL `FILTER` functions that delegate to PostGIS `ST_*`. Users can mix spatial predicates and graph traversal in a single SPARQL query with no extension beyond the batteries image.

## SPARQL Entailment Regime Tests

**Automated conformance for RDFS and OWL 2 RL entailment under SPARQL.**
The Datalog layer has implemented OWL 2 RL at 100 % since v0.51.0, but there has never been a formal test that proves the combined SPARQL+Datalog stack satisfies the W3C SPARQL 1.1 Entailment Regimes recommendation. v0.56.0 adds a `tests/sparql_entailment/` suite covering the RDFS and OWL 2 RL Entailment subsets.

## Apache Arrow / Flight Bulk Export

**Stream large graph subsets at columnar speed.**
`pg_ripple.export_arrow_flight(graph_iri TEXT)` streams all triples in a named graph (or the default graph) as Apache Arrow record batches via an Arrow Flight endpoint embedded in `pg_ripple_http`. Clients using PyArrow, DuckDB, or Pandas can consume the stream without materializing an intermediate file.

> **Deferred to v0.59.0**: dbt Adapter (`dbt-pg-ripple` Python package).

## Operational Depth

- **Federation circuit breaker**: per-endpoint failure counter; circuit opens after N consecutive failures with exponential backoff.
- **DDL event trigger**: a PostgreSQL `EVENT TRIGGER` warns when a `_pg_ripple.*` relation is dropped outside `pg_ripple.vacuum()`.
- **SID runway monitor**: `pg_ripple.sid_runway()` returns the estimated years until `statement_id_seq` wraps, at the current insert rate.
- **BRIN re-summarize after merge**: the merge worker calls `brin_summarize_new_values()` after renaming the new main table so BRIN page-range summaries are valid immediately.
- **SPARQL audit log**: `pg_ripple.audit_log_enabled` GUC + `_pg_ripple.audit_log` table recording every SPARQL UPDATE / DELETE / DROP / CLEAR with timestamp, role, txid, query text, and affected-predicate IDs.
