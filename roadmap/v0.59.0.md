# v0.59.0 — Citus: SPARQL Shard-Pruning & Rebalance Coordination

> **Full technical details:** [v0.59.0-full.md](v0.59.0-full.md)

**Status: Planned** | **Scope: Medium**

> Complete the Citus query-execution story: push bound subject patterns to the
> correct shard instead of broadcasting to all workers, and make
> `citus_rebalance()` a fully observable, coordinator-safe operation.

---

## What is this?

v0.58.0 shipped Citus horizontal sharding of VP tables. But SPARQL queries
with a bound subject (the most common real-world query pattern) still fan
out to every worker — they don't yet benefit from Citus's shard-pruning
capabilities. v0.59.0 changes that.

---

## SPARQL federation shard-pruning

When a SPARQL query binds a subject term, pg_ripple can determine which
Citus shard holds those triples at compile time:

1. Encode the bound IRI to its `i64` subject ID via the dictionary.
2. Compute the shard assignment: `shard = subject_id % shard_count` (matching
   Citus hash distribution).
3. Generate a SQL plan that targets only the shard tables on the relevant
   worker instead of using the logical (coordinator-fan-out) table.

**Example**: a SPARQL query with `?x = <http://example.org/Alice>` that
previously scanned all 32 shards across 4 workers now routes to a single
worker and reads 1 shard. Expected speedup: 10–100× for large graphs.

Shard-pruning activates automatically when `pg_ripple.citus_sharding_enabled = on`
and the query planner detects a bound subject in the algebra tree. No
configuration required.

---

## Rebalance coordination improvements

`pg_ripple.citus_rebalance()` gains two improvements:

1. **NOTIFY around fence acquire**: emits `pg_ripple.merge_start` before
   blocking on the fence and `pg_ripple.merge_end` after releasing it. This
   gives pg-trickle and monitoring tools a signal that a rebalance is pending.

2. **Progress reporting**: returns a set of rows (`shard_id`, `from_node`,
   `to_node`, `status`) so operators can track the rebalance move plan before
   it executes.

---

## EXPLAIN support for Citus queries

`pg_ripple.explain_sparql()` gains a `citus := true` output section showing:

- Which shards were pruned and which were targeted.
- Whether the query used the coordinator or was pushed down to workers.
- Per-shard row estimates.

---

## Scope

v0.59.0 is a medium-sized release. Shard-pruning requires a new code path
in the SPARQL-to-SQL translator that intercepts bound subject patterns and
rewrites the VP table reference to a shard-qualified name. All changes are
gated behind Citus detection and do not affect single-node deployments.
