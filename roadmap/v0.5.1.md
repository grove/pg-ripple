# v0.5.1 — SPARQL Advanced (Storage, Serialisation & Write)

**Effort estimate: 3–5 person-weeks**

**Goal:** This patch release completes the SPARQL story by adding the write side (SPARQL Update), the construction side (CONSTRUCT and DESCRIBE query forms), inline value encoding for numbers and dates, and full-text search on string literals. Together with v0.5.0, it rounds out the SPARQL 1.1 feature set that most applications need before going to production.

---

## SPARQL Update: INSERT DATA and DELETE DATA

**Write new triples and remove existing ones using SPARQL syntax.**

Until now, the only way to add or remove triples was through PostgreSQL SQL function calls. SPARQL 1.1 Update defines a standard write syntax — `INSERT DATA { … }` and `DELETE DATA { … }` — that many SPARQL client libraries and tools speak natively. This release implements both operations, translating them to the same dictionary-encode-then-VP-table-insert/delete path used by the SQL API.

---

## CONSTRUCT and DESCRIBE Query Forms

**Build a new RDF graph as the output of a SPARQL query.**

SPARQL's SELECT form returns a table of variable bindings. CONSTRUCT and DESCRIBE return a new RDF graph instead. CONSTRUCT takes a template you define and materialises triples for every pattern match. DESCRIBE returns a concise bounded description — all facts about one or more resources. Both forms are useful for transforming or exporting subgraphs: "build a new graph containing only the facts about people in Berlin" can be expressed as a single CONSTRUCT query.

---

## Inline Value Encoding

**Numbers and dates are stored and compared as numbers, not strings.**

A naive implementation would store all literals — including `42`, `3.14`, and `2024-01-01` — as opaque strings in the dictionary, then look them up by string equality. This works but prevents the database engine from using numeric or date indexes and comparison operators. This release introduces inline encoding: small integers, decimals, and date/time values are packed directly into the dictionary ID using reserved bit patterns, so comparisons like `FILTER(?age > 18)` compile to numeric SQL comparisons rather than string lookups.

---

## Full-Text Search on Literals

**Find triples where the object contains a keyword, using a GIN index.**

For string literal objects, pg_ripple can now create a GIN (generalised inverted index) that enables fast keyword search with PostgreSQL's `@@` text-search operator. This makes it practical to answer questions like "find all entities whose description contains the word 'renewable'" across millions of triples without a sequential scan. The index is created per-predicate on the decoded literal values.

---

## Dual-Space ID Model

**The design is future-proofed for inline numeric encoding alongside dictionary encoding.**

The encoding system for dictionary IDs is formally split into two spaces: inline IDs (numeric literals, booleans, dates packed into the ID value itself) and dictionary IDs (strings looked up in the `_pg_ripple.dictionary` table). This distinction is maintained throughout the codebase from this release onwards, ensuring that future literal types can be added as new inline ranges without changing the storage schema.
