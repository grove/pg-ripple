# v0.61.0 — Ecosystem Depth & Polish

> **Full technical details:** [v0.61.0-full.md](v0.61.0-full.md)

**Status: Planned** | **Scope: Large**

> v0.61.0 widens pg_ripple's surface for enterprise teams: fine-grained
> access control so different users only see the graphs they are allowed to
> see, GDPR right-to-erasure at the push of a button, the ability to ask
> *why* a fact was inferred, and native connectors for the modern data-stack
> tools that data engineers already use every day.

---

## What is this?

With the foundation solid (v0.60.0 closed all remaining production blockers),
v0.61.0 is about making pg_ripple useful to a broader set of enterprise
adopters. That means compliance tooling that legal and privacy teams require,
explainability features that data engineers need for debugging, and connectors
to the ecosystem tools (dbt, Kafka) that already run in most data platforms.

---

## Per-graph access control

pg_ripple stores RDF data in named graphs. Until now, any database user who
could query the extension could read all graphs. v0.61.0 brings proper
row-level security: each named graph can be granted or revoked for individual
database roles.

New SQL commands:
```sql
SELECT pg_ripple.grant_graph('<https://hr.example.org/>', 'hr_team');
SELECT pg_ripple.revoke_graph('<https://hr.example.org/>', 'contractor_role');
```

Once a graph is restricted, PostgreSQL's own row-level security engine
ensures that a user in `contractor_role` receives zero rows from that graph
in any query — SPARQL, Datalog, or plain SQL. No application code changes
are required.

## GDPR right-to-erasure

Under GDPR (and similar regulations), a data subject can request that all
personal data about them be deleted. In a knowledge graph this is
non-trivial: the subject's IRI might appear as a subject, object, or graph
name in dozens of tables, plus in embeddings, provenance records, and audit
logs.

v0.61.0 provides a single function:
```sql
SELECT * FROM pg_ripple.erase_subject('<https://people.example.org/Alice>');
```

It removes every trace of the subject across all storage layers — VP tables,
the dictionary, knowledge-graph embeddings, the PROV-O provenance graph, and
the audit log — in a single atomic transaction. It returns a per-table row
count so you have a documented erasure record ready for the DPO.

## Inference explainability

pg_ripple's Datalog engine can automatically infer new facts from existing
data using ontology rules (for example: if Alice `is_a` Person, and every
Person `has_a` Birthday, then Alice `has_a` Birthday). But when something
unexpected appears in the inferred data, it can be hard to understand why.

`pg_ripple.explain_inference(subject, predicate, object, graph)` returns the
full derivation chain as a JSON tree: which rules fired, in which order,
and which base facts they consumed. Think of it as `EXPLAIN ANALYZE` for
reasoning.

## SHACL rules that fire automatically

SHACL (Shapes Constraint Language) describes what valid RDF data looks like.
SHACL-AF (Advanced Features) extends this with *rules*: if a shape matches,
also infer these additional triples. v0.61.0 implements the bridge that
compiles SHACL-AF `sh:rule` patterns into Datalog rules and runs them
automatically whenever new data is loaded. This allows data modellers to
express both validation *and* enrichment in a single SHACL shapes graph.

## dbt adapter

[dbt](https://www.getdbt.com) (data build tool) is the dominant SQL
transformation framework in the modern data stack. It lets data engineers
write modular, version-controlled transformations as SQL `SELECT` statements.

v0.61.0 publishes a `dbt-pg-ripple` adapter (in `clients/dbt-pg-ripple/`)
that adds SPARQL-aware macros: `sparql_model`, `sparql_source`, and
`sparql_ref`. Data engineers can now build dbt models that mix SQL and SPARQL
transformations in the same project, with full dbt lineage tracking.

## Kafka CDC sink

pg_ripple already emits change-data-capture events on a PostgreSQL NOTIFY
channel when triples are inserted or deleted (`pg_ripple_cdc_lifecycle`).
v0.61.0 adds an optional Kafka producer that bridges that channel directly
to a Kafka topic. Enabling it requires two configuration parameters:

```sql
SET pg_ripple.kafka_brokers = 'broker1:9092,broker2:9092';
SET pg_ripple.kafka_topic  = 'rdf.changes';
```

Downstream systems (Flink, Spark Streaming, ksqlDB) can then subscribe to
the topic and react to RDF graph changes in real time.

## Distributed tracing end-to-end

When a request arrives at the `pg_ripple_http` HTTP service, modern
observability platforms (Datadog, Honeycomb, Grafana Tempo) track it with a
`traceparent` header. Until now that trace ID was lost as soon as the request
reached PostgreSQL.

v0.61.0 extracts the `traceparent` header in the HTTP layer and forwards it
through a session variable into the pg_ripple extension. Every SPARQL and
Datalog query span is now tagged with the originating trace ID, giving you a
unbroken trace from the load balancer all the way through the query engine.

---

## Richer federation observability

`pg_ripple.federation_call_stats()` already shows call counts per remote
SPARQL endpoint. v0.61.0 enriches it with per-endpoint latency percentiles,
error counts, blocked-request counts, and the timestamp of the last error:

```sql
SELECT endpoint, calls, errors, p50_ms, p95_ms, last_error_at
FROM pg_ripple.federation_call_stats();
```

This makes it straightforward to identify which remote endpoint is the
bottleneck in a federated query.

---

## Scope

v0.61.0 is a large release spanning access control, compliance, explainability,
and ecosystem integrations. The storage format and query language are unchanged.
All new features are opt-in or additive. Upgrading from v0.60.0 requires no
changes to existing queries or schemas.
