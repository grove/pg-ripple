# v0.63.0 — SPARQL CONSTRUCT Writeback Rules

> **Full technical details:** [v0.63.0-full.md](v0.63.0-full.md)

**Status: Released ✅** | **Scope: Large**

> v0.63.0 closes a critical gap in the data-transformation story: SPARQL
> CONSTRUCT queries can now write their output *back into the triplestore*
> as first-class VP-table triples, and that derived data is maintained
> incrementally whenever the source graph changes — without a full recompute.

---

## What is this?

pg_ripple already lets you run a SPARQL CONSTRUCT query to produce new triples
on demand, and it lets you store those triples in an external stream table
(v0.18.0). What it cannot do today is route the derived triples back into the
VP storage layer as *persistent, queryable facts* — kept automatically
up-to-date as raw input data evolves.

v0.63.0 introduces **CONSTRUCT writeback rules**: named SPARQL CONSTRUCT
queries that are registered once, run incrementally on each write transaction,
and write their output directly into a target named graph inside the
triplestore (using `source = 1`, the same tag Datalog uses for inferred
triples). Any subsequent SPARQL query against that graph sees a fully
consistent, live view of the derived model.

---

## Raw-to-canonical pipelines

A canonical data model is a clean, stable representation of a domain
derived from messier raw data. In a typical knowledge-graph workflow:

1. Raw events or records arrive in `GRAPH <urn:raw>` — heterogeneous,
   with inconsistent vocabularies and ad-hoc structure.
2. A CONSTRUCT writeback rule transforms that raw data into a normalised
   canonical graph, e.g. `GRAPH <urn:canonical>`.
3. All downstream SPARQL queries and Datalog rules target `<urn:canonical>`.
4. When raw data changes, only the *delta* is reprocessed — new canonical
   triples are inserted, obsolete ones are retracted.

```sql
SELECT pg_ripple.create_construct_rule(
    name         := 'raw_to_canonical',
    sparql       := '
        CONSTRUCT {
            ?s <https://schema.org/name>  ?name .
            ?s <https://schema.org/email> ?email .
            ?s a <https://schema.org/Person> .
        }
        WHERE {
            GRAPH <urn:raw> {
                ?s <urn:raw/fullName>  ?name ;
                   <urn:raw/emailAddr> ?email .
            }
        }
    ',
    target_graph := 'urn:canonical',
    mode         := 'incremental'
);
```

---

## Incremental maintenance

The key advance over a plain CONSTRUCT query is *incremental evaluation*:

- **Insert path**: when new raw triples are inserted, only the newly derivable
  canonical triples are computed and added to the target graph.
- **Delete path**: when raw triples are deleted, derived triples that depended
  solely on the deleted triples are retracted (Delete-Rederive, the same
  algorithm used for Datalog since v0.34.0).
- **Full recompute**: an explicit `refresh_construct_rule()` reruns the
  CONSTRUCT from scratch, useful after bulk loads or schema changes.

---

## Safety constraints

To keep the system predictable:

- A rule may not read from and write to the **same named graph** (cycle
  detection at rule-registration time).
- Multi-rule pipelines are stratified: if rule B reads from rule A's target
  graph, B always runs after A in the same transaction.
- Rules that cannot be stratified (mutual recursion through writeback) are
  rejected at registration time with a clear error message.

## Citus scalability: coordinator hot-spots & operational gaps

Following CITUS-20–29 (v0.61.0–v0.62.0), eight narrower but production-critical
limits are addressed in this release.

**SERVICE result shard pruning.** When a federated `SERVICE` clause returns
subject IRIs, the subsequent local VP join today treats those subjects as
unbound and fans out to all workers. v0.63.0 feeds the SERVICE result set
through the shard-pruning encoder before the join is emitted, so only the
relevant workers are queried.

**Streaming fan-out results.** Large unbound-subject queries materialise
all worker results at the coordinator before streaming to the client, creating
a memory ceiling. v0.63.0 wires the v0.40.0 SPARQL cursor infrastructure to
pipe worker result pages directly to the client in sharded mode, keeping the
coordinator footprint proportional to the page size rather than the full
result set.

**Approximate `COUNT(DISTINCT)` via HyperLogLog.** Exact distinct counts
require all raw values at the coordinator. A new opt-in `SPARQL_APPROX_DISTINCT`
hint emits `hll_add_agg` (via `pg_hll`) on workers and merges compact HLL
sketches at the coordinator, eliminating the data-transfer bottleneck for
analytics queries.

**Batched dictionary encoding under concurrent writes.** Every new
IRI/literal hits the `_pg_ripple.dictionary` reference table on the
coordinator. v0.63.0 batches all new terms from a single bulk-load call
into one `INSERT … ON CONFLICT DO NOTHING … RETURNING` before writing any
VP rows, reducing coordinator dictionary round-trips from one-per-triple to
one-per-batch.

**`statement_id_timeline` per-worker local tables.** Under high insert
rates from many workers, every inserted triple writes to a single coordinator
table. v0.63.0 introduces per-worker `_pg_ripple.statement_id_timeline_local`
tables (written by the worker at insert time) and a coordinator view that
UNION ALLs them, eliminating the per-triple coordinator write.

**Non-blocking VP promotion.** `create_distributed_table()` during
predicate promotion holds an `AccessExclusiveLock` that stalls all queries
to that predicate. v0.63.0 implements a shadow-table promotion: new writes
go directly to the new distributed table while existing `vp_rare` rows are
migrated in the background; the logical view is swapped atomically only at
the end.

**Per-graph RLS propagation test.** `grant_graph()` / `revoke_graph()`
update the `_pg_ripple.tenant_graphs` reference table and rely on Citus DDL
propagation. A new CI integration test verifies that a `revoke_graph()` call
is enforced on every worker before the function returns, preventing a
time-of-check / time-of-use race.

**Per-worker BRIN summarise after merge.** The merge worker calls
`brin_summarize_new_values` on the coordinator-side logical table. In sharded
mode each worker shard needs its own call. v0.63.0 issues
`run_command_on_shards` to invoke `brin_summarize_new_values` on every worker
shard after a merge cycle, keeping shard BRIN indexes current and first-scan
performance consistent across the cluster.
