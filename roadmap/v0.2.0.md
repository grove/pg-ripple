# v0.2.0 — Bulk Loading & Named Graphs

**Effort estimate: 3–5 person-weeks**

**Goal:** Loading a knowledge graph one triple at a time is impractical. This release adds the ability to pour in large RDF files in a single call, introduces named graphs so you can keep logically separate collections of facts in the same database, and adds the first export capability so you can verify what you loaded.

---

## Loading Large Files in One Go

**A single function call loads thousands of triples from an RDF file.**

The workhorse feature of this release is bulk loading: `load_turtle`, `load_ntriples`, `load_trig`, `load_rdfxml`, and `load_nquads` each accept the content of an RDF file as text and import everything into pg_ripple's VP tables in one efficient batch. The implementation pipelines encoding through the dictionary and uses PostgreSQL's `INSERT … ON CONFLICT DO NOTHING … RETURNING` pattern so that duplicate triples are silently skipped and the dictionary is updated in a single round-trip rather than one call per triple. On typical hardware, this approach sustains over 50,000 triples per second for standard RDF formats.

---

## Named Graphs

**Facts can be organised into labelled collections.**

Real knowledge graphs often mix data from multiple sources — one graph for public facts, another for internal annotations, another for staging data not yet reviewed. Named graphs provide that separation: every triple belongs to a named graph identified by an IRI, and the default graph (used when no name is specified) has ID zero. Named graphs are stored in the same VP tables via a `g` column; no extra indirection is needed. TriG and N-Quads formats carry graph names natively and are supported from this release onwards.

---

## Rare-Predicate Consolidation

**Predicates used only rarely stay in a shared table instead of getting their own.**

Creating a dedicated VP table for a predicate that appears in only two or three triples would cause catalog bloat — hundreds of tiny tables that slow down schema introspection and eat up PostgreSQL's relation-cache memory. This release introduces a `vp_rare` consolidation table that stores triples for any predicate below a configurable threshold (default: 1,000 triples). When a predicate grows past that threshold it is automatically promoted to its own dedicated VP table. This keeps the catalog tidy without sacrificing the query performance benefits of vertical partitioning.

---

## N-Triples Export

**You can export any graph as N-Triples to verify what is stored.**

The first export function — `export_ntriples` — converts any named graph back to N-Triples format. This is primarily useful for verifying that a bulk load succeeded and that the data round-trips correctly through the dictionary encoder. It is not yet streaming; for very large graphs the full result is materialised in memory before being returned to the caller.

---

## File-Path Variants

**Superusers can load directly from server-side file paths.**

Companion `load_*_file` functions accept a file path rather than inline text content, reading the file directly from the server filesystem. These functions require superuser privileges to prevent unauthorised file access, and they validate that the path falls within PostgreSQL's allowed file-access directories — the same access model used by PostgreSQL's own `COPY FROM` command.
