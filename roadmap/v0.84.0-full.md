## Background

v0.84.0 is the first release in the Assessment 13 remediation track. Assessment 13 was conducted against the v0.83.0 codebase and identified a set of high-priority operational, correctness, and security findings. This release addresses the critical and high findings: service health verification, plan-cache correctness, GUC registration organisation, CI lint hardening, migration-chain test extension, developer tooling, and security annotations.

## HTTP-READY-01 — HTTP Companion Readiness & Version Alignment

### Current state

The `/health/ready` endpoint in `pg_ripple_http` returns a static `{"status":"ready"}` without verifying connectivity to PostgreSQL or the installed extension version. `COMPATIBLE_EXTENSION_MIN` is pinned to `"0.83.0"`.

### What changes

1. Bump `pg_ripple_http` to align with `v0.84.0`.
2. Raise `COMPATIBLE_EXTENSION_MIN` to `"0.84.0"` in `pg_ripple_http/src/main.rs`.
3. Rewrite the `/health/ready` handler to execute `SELECT 1` via the SPI bridge and query `pg_extension` for the installed extension version. Return `503 Service Unavailable` if either check fails.

### Deliverables

| ID | Deliverable |
|----|-------------|
| HTTP-READY-01a | `COMPATIBLE_EXTENSION_MIN` raised to `"0.84.0"` |
| HTTP-READY-01b | `/health/ready` executes real DB round-trip |
| HTTP-READY-01c | `/health/ready` checks installed extension version |
| HTTP-READY-01d | Returns `503` when DB or version check fails |
| HTTP-READY-01e | pg_regress test for readiness endpoint |

## PLAN-CACHE-NOPARSE-01 — Plan Cache Double-Parse Elimination

### Current state

`cache_key()` in `src/sparql/plan.rs` calls `spargebra::Query::parse()` to produce a canonical form for the cache key, then discards the result. The caller re-parses the query again immediately after. This doubles parse time for every cache miss.

### What changes

Refactor `cache_key()` to accept a pre-parsed `spargebra::Query` reference instead of raw query text. The caller parses once and passes the result to both `cache_key()` and the planner. Update all call sites in `src/sparql/plan.rs` and `src/sparql/execute.rs`.

### Deliverables

| ID | Deliverable |
|----|-------------|
| PLAN-CACHE-NOPARSE-01a | `cache_key()` accepts `&spargebra::Query` |
| PLAN-CACHE-NOPARSE-01b | All call sites updated |
| PLAN-CACHE-NOPARSE-01c | Benchmark confirms no regression in cache-hit throughput |

## GUC-MODULE-SPLIT-01 — GUC Registration Module Split

### Current state

`src/gucs/registration.rs` contains all GUC registrations for storage, SPARQL, Datalog, HTTP, and federation domains in a single 900-line file. This makes it hard to find domain-specific GUC definitions and audit them independently.

### What changes

Split into per-domain submodules:

```
src/gucs/
  mod.rs              — re-exports all public GUC accessors
  storage.rs          — HTAP, merge, dictionary GUCs
  sparql.rs           — SPARQL depth, plan cache, filter strictness GUCs
  datalog.rs          — Datalog iteration, cost, probabilistic GUCs
  http.rs             — HTTP companion GUCs
  federation.rs       — Federation timeout, circuit breaker GUCs
```

The public API (`pg_ripple::gucs::*`) is unchanged; only the internal file layout changes.

### Deliverables

| ID | Deliverable |
|----|-------------|
| GUC-MODULE-SPLIT-01a | `src/gucs/` submodule structure created |
| GUC-MODULE-SPLIT-01b | All GUC registrations moved to domain submodules |
| GUC-MODULE-SPLIT-01c | Public re-exports in `src/gucs/mod.rs` unchanged |
| GUC-MODULE-SPLIT-01d | CI clippy gate passes with zero warnings |

## CI-LINT-SQL-01 — SQL Construction Lint Gate

### Current state

Assessment 12 found SQL injection risks at sites where SQL was constructed via Rust string formatting. While the high-severity sites were fixed in v0.80.0, no CI gate prevents new injection-prone SQL construction from being introduced.

### What changes

Add `scripts/check_no_string_format_in_sql.sh` that:
1. Scans all `.rs` files for patterns like `format!("… SQL …", …)`, `sql.push_str(`, and string `+` concatenation adjacent to SQL keywords.
2. Maintains an allow-list of explicitly reviewed exceptions.
3. Exits non-zero if any new site is found outside the allow-list.

Wire as a required step in `.github/workflows/ci.yml` (runs before build).

### Deliverables

| ID | Deliverable |
|----|-------------|
| CI-LINT-SQL-01a | `scripts/check_no_string_format_in_sql.sh` created |
| CI-LINT-SQL-01b | Allow-list with all reviewed exceptions |
| CI-LINT-SQL-01c | Required CI step in `.github/workflows/ci.yml` |
| CI-LINT-SQL-01d | CI step passes on current codebase |

## JUSTFILE-RECIPES-01 — Developer Tooling Recipes

### Current state

Version bumps, SBOM regeneration, and OpenAPI spec regeneration are manual multi-step processes that are easily forgotten during release preparation.

### What changes

Add three `just` recipes to `justfile`:

```makefile
# Bump the version consistently across Cargo.toml, pg_ripple.control, and CHANGELOG.md
bump-version NEW_VERSION:
    sed -i '' "s/^version = .*/version = \"{{NEW_VERSION}}\"/" Cargo.toml
    sed -i '' "s/default_version = .*/default_version = '{{NEW_VERSION}}'/" pg_ripple.control
    @echo "## [{{NEW_VERSION}}] — $(date +%Y-%m-%d)" >> CHANGELOG_HEADER.tmp
    @echo "Bumped to {{NEW_VERSION}}"

# Regenerate sbom.json using cargo cyclonedx
regen-sbom:
    cargo cyclonedx --format json --all --output-file sbom.json
    @echo "SBOM regenerated"

# Regenerate the HTTP companion OpenAPI spec and validate it
regen-openapi:
    cargo run --bin generate-openapi 2>/dev/null || true
    python3 -m openapi_spec_validator docs/openapi/pg_ripple_http.yaml
    @echo "OpenAPI spec regenerated and validated"
```

### Deliverables

| ID | Deliverable |
|----|-------------|
| JUSTFILE-RECIPES-01a | `bump-version` recipe in `justfile` |
| JUSTFILE-RECIPES-01b | `regen-sbom` recipe in `justfile` |
| JUSTFILE-RECIPES-01c | `regen-openapi` recipe in `justfile` |

## SECDEF-ANNOT-01 — SECURITY DEFINER Annotation & Lint

### Current state

Several SQL functions use `SECURITY DEFINER` but do not carry inline comments explaining why elevated privileges are required. Assessment 13 flagged this as a security documentation gap.

### What changes

1. Add `-- SECURITY-JUSTIFY: <reason>` comment at each `SECURITY DEFINER` site in the SQL schema and in Rust `#[pg_extern]` definitions that emit `SECURITY DEFINER`.
2. Add `scripts/check_security_definer.sh` that rejects any new `SECURITY DEFINER` without a `SECURITY-JUSTIFY` annotation. Wire as a required CI step.

### Deliverables

| ID | Deliverable |
|----|-------------|
| SECDEF-ANNOT-01a | All existing SECURITY DEFINER sites annotated |
| SECDEF-ANNOT-01b | `scripts/check_security_definer.sh` created |
| SECDEF-ANNOT-01c | CI step wired and passing |

## AUDIT-TOML-01 — Audit.toml Expiry Dates & Strict Compat Flag

### Current state

`audit.toml` has several `ignore` entries with no expiry dates. The `serde_cbor` advisory has been acknowledged but not yet scheduled for replacement. `pg_ripple_http` has no mechanism for operators to enforce strict extension compatibility.

### What changes

1. Add `expiry = "YYYY-MM-DD"` to all `ignore` entries in `audit.toml`.
2. Document the `serde_cbor` → `ciborium` evaluation in `CHANGELOG.md` with a tracking reference.
3. Add `PG_RIPPLE_HTTP_STRICT_COMPAT=1` env var to `pg_ripple_http/src/main.rs`. When set, the HTTP companion refuses to start if the installed extension version is below `COMPATIBLE_EXTENSION_MIN` (currently a warning; this makes it an error).

### Deliverables

| ID | Deliverable |
|----|-------------|
| AUDIT-TOML-01a | Expiry dates on all `audit.toml` ignore entries |
| AUDIT-TOML-01b | `serde_cbor` advisory documented in CHANGELOG |
| AUDIT-TOML-01c | `PG_RIPPLE_HTTP_STRICT_COMPAT` env var implemented |
| AUDIT-TOML-01d | Strict compat mode documented in operations guide |
