# v0.22.0 — Storage Correctness & Security Hardening

**Effort estimate: 6–8 person-weeks**

**Goal:** This release addresses the most serious data-integrity and security issues found in the v0.20.0 audit. The headline fix prevents a subtle bug where rolling back a database transaction could plant ghost term IDs in the dictionary that corrupt future queries. Two race conditions in the background merge process are also closed. The HTTP service's security controls are made actually effective rather than nominally present.

---

## Dictionary Cache Rollback Fix

**Rolling back a transaction no longer plants phantom term IDs.**

pg_ripple's dictionary cache is designed to speed up term encoding by remembering recently seen terms. But if you insert a triple, the dictionary encodes and caches the new terms, and then you roll back the transaction, the cached IDs remain in the per-connection cache. The next time the same term is encountered, the cache returns the ID that was created by the rolled-back transaction — an ID that no longer exists in the dictionary table. This creates phantom references that make facts appear to exist when they do not, or vanish when they should exist. The fix registers a PostgreSQL transaction-boundary callback that purges the in-process cache on every rollback.

---

## HTAP Merge Race Condition Fixes

**Two scenarios where concurrent writes and merges could lose or duplicate data are closed.**

The background merge worker and concurrent write transactions can interact in two problematic ways. First, if a delete arrives during the precise window when the merge worker is swapping old and new main partitions, the delete could apply to a partition that no longer exists, silently losing the deletion. Second, tombstone entries (which mark deleted triples) were previously cleared with a simple TRUNCATE, which could discard tombstones for deletes that arrived after the merge snapshot — causing deleted triples to reappear after the merge. Both races are fixed with proper advisory locking and epoch-fenced tombstone cleanup.

---

## Shared-Memory Cache Redesign

**The cache now handles large vocabularies without thrashing.**

The shared-memory dictionary cache used a direct-mapped design where any two terms that hash to the same slot would evict each other. For a vocabulary with many frequently used terms, this caused excessive eviction and low hit rates. The redesigned cache is 4-way set-associative: each slot has four ways, so four different terms can coexist at the same hash position. This drops the collision eviction rate from about 15% to under 1% for typical 5,000-term working sets.

---

## HTTP Service Security

**Rate limiting, error redaction, and constant-time authentication now work correctly.**

Three security controls in the `pg_ripple_http` companion service were present in code but not effective in practice. Rate limiting (preventing one client from flooding the server with requests) is now enforced using per-IP token buckets that actually reject excess requests with HTTP 429 responses. Error messages to API clients no longer include internal PostgreSQL error text, schema names, or file paths — only a category code and a trace ID. Authentication token comparison is now done with a constant-time function to prevent timing-based attacks that could reveal the token character by character.

---

## Internal Schema Privilege Lockdown

**The `_pg_ripple` internal schema is no longer accessible to unprivileged users.**

The internal schema where VP tables, the dictionary, and extension metadata live was previously accessible to any authenticated database user. This release explicitly revokes all privileges from PUBLIC on the `_pg_ripple` schema, its tables, and its sequences. The public `pg_ripple.*` functions remain accessible and continue to work — they run with the appropriate elevated privileges internally. Unprivileged users now get a clear permission error if they try to query internal tables directly.
