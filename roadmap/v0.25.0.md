# v0.25.0 — GeoSPARQL & Architectural Polish

**Effort estimate: 6–8 person-weeks**

**Goal:** Geographic data is everywhere in modern knowledge graphs — locations, boundaries, distances. This release connects pg_ripple to PostGIS through the GeoSPARQL 1.1 standard, enabling spatial queries in SPARQL. It also closes a set of medium and low priority issues from the v0.20.0 audit: catalog stability, strict bulk loading, federation URL safety, and a health-check function for operators.

---

## GeoSPARQL via PostGIS

**SPARQL queries can now filter and compute over geographic data.**

GeoSPARQL is the OGC standard for spatial queries on RDF data. When PostGIS is installed alongside pg_ripple, SPARQL FILTER expressions can now call geographic functions: `geo:sfIntersects` (do two geometries overlap?), `geo:sfContains` (is one entirely inside another?), `geo:sfWithin`, and `geof:distance` (how far apart are two locations?). These translate directly to PostGIS functions like `ST_Intersects` and `ST_Distance`. Geographic data is stored as `geo:wktLiteral` typed literals (Well-Known Text format), which pg_ripple recognises and passes to PostGIS for evaluation. If PostGIS is not installed, these functions return NULL with a WARNING rather than causing errors.

## Catalog OID Stability

**The predicate catalog now stores table names, not fragile internal object IDs.**

PostgreSQL assigns each table an internal object ID (OID) when it is created. These OIDs are not stable across `pg_dump` / `pg_restore` cycles — after a restore, the same table may have a completely different OID. The predicate catalog (`_pg_ripple.predicates`) previously stored OIDs, which meant that after a restore, every dynamic SQL query that looked up a VP table by OID could silently redirect to the wrong table or fail entirely. This release adds `schema_name` and `table_name` columns to the catalog and uses those for all dynamic SQL lookups, making OID drift a non-issue.

## Bulk Load Strict Mode

**A `strict` parameter makes bulk loads fail fast on any malformed triple.**

The default bulk load behaviour is lenient: malformed triples emit a WARNING and are skipped, but the rest of the load continues. For use cases where partial loads are unacceptable — data migrations, ETL pipelines with strict data contracts, compliance scenarios — this is dangerous. The new `strict = true` parameter causes the entire load to be rolled back if any triple is malformed, with an error identifying the line number and the offending content.

## Federation SSRF Scheme Validation

**Endpoint URLs must use `http://` or `https://` — other schemes are rejected at registration time.**

A belt-and-braces defence against server-side request forgery: when registering a federation endpoint with `pg_ripple.register_endpoint()`, the URL is checked to ensure its scheme is `http` or `https`. `file://`, `gopher://`, and any other non-HTTP scheme are rejected with a clear error. This validation happens at registration time, before any query is attempted against the endpoint.

## `canary()` Health Function

**A single function call reports the health of the running pg_ripple instance.**

The `pg_ripple.canary()` function returns a JSON object summarising the state of all key subsystems: whether the merge worker is running or stalled, the dictionary cache hit rate, whether the predicate catalog is consistent with the actual VP tables, and whether any orphaned rows exist in `vp_rare` for predicates that have already been promoted. This is designed for integration with monitoring dashboards, alerting pipelines, and CI smoke tests.
