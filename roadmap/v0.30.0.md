# v0.30.0 — Datalog Aggregation & Compiled Rule Plans

**Effort estimate: 5–7 person-weeks**

**Goal:** Two capabilities that unlock analytics and operational efficiency in the Datalog engine. First, rules can now aggregate facts — count friends, compute maximum scores, sum weights — enabling graph analytics as rules. Second, the generated SQL for each rule set is cached in shared memory, so repeated inference calls (the common case after data loads) skip the translation step entirely.

---

## Aggregation in Rule Bodies

**Rules can count, sum, average, and find extremes — not just derive new facts.**

Standard Datalog can only derive atomic facts: "Alice is an ancestor of Bob". Datalog with aggregation (sometimes called Datalog^agg) allows rules that compute aggregate values: "Alice has 47 friends", "the maximum salary in the engineering department is $150,000", "the average response time for requests tagged 'critical' is 230ms". The aggregation is stratified — aggregated predicates must be fully computed before any rule uses their result — ensuring the fixpoint is well-defined. The `pg_ripple.infer_agg()` function runs the rule set in aggregation-aware mode.

---

## Aggregation-Stratification Safety Check

**The engine rejects rule programs where aggregation would be unsafe.**

The guarantee that Datalog^agg produces a unique answer depends on aggregated predicates not being recursively defined through their own aggregation. If a rule says "count the number of entities of type T" but "type T" is itself derived by another rule that depends on that count, the result would be undefined. The stratification checker detects this at rule-load time and raises a clear error naming the cycle, rather than producing inconsistent results silently.

---

## Compiled Rule Plan Cache

**Generated SQL for each rule set is stored in shared memory and reused.**

Every call to `pg_ripple.infer()` previously repeated the full rule-to-SQL compilation: parsing the rule text, resolving predicate names to dictionary IDs, building the SQL template, formatting it with current schema information. For a rule set with fifty rules, this overhead was measurable — especially for applications that call `infer()` after every batch load. The plan cache stores the compiled SQL for each rule set in a fixed-size shared-memory LRU keyed on the rule set name and schema version. Repeated calls find the cached SQL and skip straight to execution.

---

## SPARQL On-Demand Benefits from Plan Cache

**SPARQL queries that trigger Datalog inference get faster on repeat executions.**

When a SPARQL query references a derived predicate (one that the Datalog engine produces), the engine can inline the derivation as a SQL CTE rather than requiring the full fixpoint to be pre-materialised. The CTE SQL is now served from the rule plan cache rather than regenerated on every query, making the second and subsequent queries that trigger the same inference significantly faster. This is the most common production mode: query-time inference without pre-materialisation.
