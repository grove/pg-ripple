## Release theme

v0.73.0 prepares the project for v1.0.0 by tracking protocol evolution (SPARQL 1.2) and adding the live subscription capability that has been listed as a gap in multiple assessments. The ecosystem hardening items (CONTRIBUTING.md, Helm chart SHA, feature taxonomy) improve onboarding and operational confidence.

## Deliverables

### SPARQL12-01 — SPARQL 1.2 design document

**Root cause**: Assessment 10 Dimension 12 noted SPARQL 1.2 (currently in W3C Working Group review) is not tracked.

**Work items**:

1. Create `plans/sparql12_tracking.md` covering:
   - Key SPARQL 1.2 changes vs 1.1: RDF-star first-class support (RDF 1.2 alignment), `LATERAL` joins, updated `VALUES` syntax, additional built-in functions.
   - Compatibility assessment: which changes are backward-compatible; which require new grammar tokens; which require changes to `spargebra`.
   - Migration path: update `spargebra` crate when SPARQL 1.2 support lands upstream; run W3C SPARQL 1.2 test suite.
   - Tentative timeline: track as a post-v1.0.0 feature unless `spargebra` ships 1.2 support before v1.0.0.
2. Add a tracking entry to `feature_status.rs` for `sparql_12` with status `planned`.

### SUB-01 — Live SPARQL subscription API

**Root cause**: Assessment 10 Dimension 12 / Recommended item 18. No live query subscription API exists.

**Work items**:

1. Add SQL function `pg_ripple.subscribe_sparql(subscription_id TEXT, query TEXT, graph_iri TEXT) → VOID` that registers a SPARQL SELECT subscription in `_pg_ripple.subscriptions` catalog table.
2. Add `pg_ripple.unsubscribe_sparql(subscription_id TEXT) → VOID`.
3. In the mutation journal `on_graph_write` / `on_graph_delete` path, after CWB hooks run, check if any subscriptions depend on the affected graph. If so, call `pg_notify('pg_ripple_subscription_' || subscription_id, ...)` with the new query result as JSONB payload.
4. In `pg_ripple_http`, add `GET /subscribe/:subscription_id` SSE endpoint:
   - Listens on PostgreSQL `LISTEN pg_ripple_subscription_<id>`.
   - Forwards each notification as a Server-Sent Event.
5. Document in `docs/src/features/live-subscriptions.md`.
6. Add `tests/pg_regress/sql/subscriptions.sql` smoke test (register, fire, unregister).

**Note**: This is a prototype / experimental implementation. The `pg_notify` payload size limit (8 KB) means very large result sets send a "changed" signal only; clients must re-query.

### JSONLD-INGEST-02 — Multi-subject JSON-LD document ingest

**Root cause**: Carried forward from v0.72.0 deferred corner case **RT-03**.
`json_to_ntriples_and_load(payload, subject_iri, ...)` requires a single
subject IRI parameter, so callers cannot ingest a JSON-LD document that
contains multiple top-level nodes under `@graph`. The outbound side already
emits multi-graph documents, so the asymmetry is purely on the ingest path.

**Fix**: Add a new SQL function `pg_ripple.json_ld_load(document JSONB, default_graph TEXT DEFAULT NULL) RETURNS BIGINT` that:

1. Parses `document` as a JSON-LD value using `oxjsonld` (already in the dependency tree via `oxrdf`).
2. Walks `@graph` (or treats the document as a single node when no `@graph` is present).
3. For each top-level node, requires an `@id` (raise a descriptive error if missing).
4. Routes triples to the named graph identified by `@id` at the document's outer level (if present), otherwise to `default_graph`, otherwise to the default graph.
5. Returns the total number of triples loaded.

**Test**: Add `tests/pg_regress/sql/jsonld_ingest_multi_graph.sql`:
- Document with `@graph` of 3 nodes; verify all triples loaded under the right subjects.
- Document with named outer `@id` (a graph IRI); verify triples land in that named graph.
- Document missing top-level `@id` on a node; verify descriptive error.
- Backward compat: single-node document with `@id` works without `default_graph`.

**Doc update**: Note in `docs/src/operations/pg-trickle-relay.md` that for multi-subject inbound payloads the trigger should call `json_ld_load()` instead of looping over `json_to_ntriples_and_load()`.
