# v0.62.0 — Query Frontier

> **Full technical details:** [v0.62.0-full.md](v0.62.0-full.md)

**Status: Planned** | **Scope: Very Large**

> v0.62.0 delivers a browser-based visual graph explorer, activates a
> dormant query optimizer that can speed up certain complex queries by
> 5–100×, adds a high-speed bulk-export path for data-pipeline consumers,
> and closes the remaining Citus scalability limits identified after v0.61.0.

---

## What is this?

v0.62.0 rounds out the query-engine feature set with tools that make
pg_ripple easier to explore, easier to integrate into data pipelines, and
faster on the most complex graph patterns. Cypher/GQL support (both
read-only and write operations) is delivered in v1.1.0.

---

## Apache Arrow Flight bulk export

[Apache Arrow Flight](https://arrow.apache.org/docs/format/Flight.html) is
a high-performance binary streaming protocol designed for moving large
datasets between systems at wire speed. It is natively supported by Pandas,
Polars, DuckDB, Spark, and most modern data warehouse connectors.

v0.62.0 adds an Arrow Flight endpoint to `pg_ripple_http`. A data engineer
can export an entire named graph — potentially hundreds of millions of
triples — with a single call:

```sql
SELECT pg_ripple.export_arrow_flight('<https://mygraph.example.org/>');
```

and stream the result directly into a Pandas DataFrame or a Spark job. The
benchmark target is ≥ 500,000 triples per second on localhost.

## WCOJ planner activated

Since v0.36.0, pg_ripple has contained an implementation of the
Leapfrog-Triejoin (WCOJ) join algorithm — a technique that handles cyclic
graph patterns (triangle queries, 4-cycles, etc.) in time proportional to
the *output* size rather than the Cartesian product of the inputs. On dense
graphs these patterns can be 5–100× faster than standard hash-join.

The algorithm was implemented but never connected to the live query planner.
v0.62.0 wires it in: the planner now analyses each multi-triple pattern,
detects whether it contains a cycle, and automatically switches to the WCOJ
strategy when it does. There are no new SQL functions or configuration
parameters — the speedup is automatic.

A WatDiv benchmark regression test is added to CI that must show ≥ 5×
improvement on cyclic-pattern templates to pass.

## Visual graph explorer

`pg_ripple_http` gains a new `/explorer` route that serves a browser-based
visual graph explorer — no installation required, no external service.

You open `http://your-server/explorer` in a browser, type a SPARQL
`CONSTRUCT` query into the text box, and the results are rendered as an
interactive force-directed graph. Clicking any node expands its
neighbourhood with another query. The explorer uses the same authentication
as the API — it can see exactly the graphs the current token is allowed
to access.

This makes pg_ripple accessible to analysts and domain experts who want to
explore data visually without writing code.

## Citus scalability: remaining limits

**Property-path coordinator push-down mitigation.** `WITH RECURSIVE` CTEs
cannot be executed on Citus workers — every `rdfs:subClassOf+` or `foaf:knows*`
traversal today ships all intermediate hop results to the coordinator at each
iteration. v0.62.0 adds a rewrite: when the start node is bound and shard
pruning resolves it to a single worker, the entire recursive CTE is sent to
that worker via `run_command_on_workers` and only the final result set travels
back to the coordinator. For multi-hop traversals anchored on a known entity
this eliminates the per-hop network round-trip entirely.

**`vp_rare` cold-entry archival.** `vp_rare` is a Citus reference table —
fully replicated to every worker. Rare predicates that have been superseded
or deleted are never removed from it today, so the table grows without bound.
v0.62.0 adds a `pg_ripple.vacuum_vp_rare()` function that tombstones entries
where the predicate has zero live triples and removes them from the reference
table, shrinking the per-worker replica.

**Tiered dictionary cache.** The dictionary is also a reference table,
replicated to every worker. At billions of unique IRIs and literals it becomes
a multi-GB table on every node. v0.62.0 introduces a tiered strategy: the
top-N most-recently-used entries remain in the shared-memory LRU (existing
behaviour); the next tier lives on the coordinator and is fetched on miss;
the cold tail is resolved by a worker-local lookup that avoids the
cross-node round-trip for terms that are only ever queried on one worker.
Controlled by a new GUC `pg_ripple.dictionary_tier_threshold` (default:
disabled).

**Datalog inference worker dispatch.** Semi-naive fixpoint today runs entirely
at the coordinator: each `INSERT … SELECT` in an inference iteration is
orchestrated by the coordinator even though the heavy `SELECT` can be
parallelised across workers. v0.62.0 emits inference iteration SQL with
`/*+ Parallel(vp_N 4) */` hints and, when sharding is enabled, wraps each
stratum iteration in a Citus `run_command_on_all_nodes` so workers execute
their shard slice of the inference join in parallel. The coordinator only
collects the count of newly derived rows to determine fixpoint termination.

**Live shard rebalance (non-blocking bulk load).** Today `citus_rebalance()`
acquires the merge-fence advisory lock, causing bulk loads to pause for the
duration of the rebalance. v0.62.0 adds a
`pg_ripple.citus_live_rebalance()` variant that uses Citus's copy-based
shard migration: the source shard remains writable during the copy phase;
only a brief AccessShareLock is held during the final cutover. Bulk loads
continue at full speed throughout.

**Multi-hop shard pruning carry-forward.** CITUS-20 (v0.61.0) prunes on a
bound start subject or object. For property paths with a bound start node,
subsequent hops expand to a new set of intermediate subjects that are still
unknown at compile time, so each hop fans out. v0.62.0 adds runtime
carry-forward: after the first hop resolves a set of intermediate subject IDs,
those IDs are encoded and passed as a shard-pruning hint to the next hop's
VP table scan, progressively restricting the fan-out as the path narrows.

## Scope

v0.62.0 is a very large release: Arrow Flight, WCOJ planner integration,
visual explorer, and six targeted Citus scalability improvements. All changes
are additive or opt-in. Upgrading from v0.61.0 requires no changes to existing
queries, schemas, or application code.
