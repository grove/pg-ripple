# v0.34.0 — Bounded-Depth Termination & Incremental Retraction (DRed)

**Effort estimate: 5–7 person-weeks**

**Goal:** Two production-critical Datalog capabilities. Bounded-depth termination lets the engine stop early when a known maximum hierarchy depth makes further iterations impossible — shaving 20–50% off long fixpoint loops. Delete-Rederive (DRed) means deleting a base triple no longer requires re-materialising the entire derived closure: only the affected derived facts are surgically removed and re-derived, keeping materialized views correct in milliseconds after deletes.

---

## Bounded-Depth Early Termination

**When hierarchies have a known depth limit, inference stops early.**

Many ontologies have bounded hierarchies: a SHACL shape might declare that class hierarchies are at most five levels deep, or a user might know their data has no chain longer than ten hops. If the inference engine knows the maximum possible derivation depth, it can stop the fixpoint loop when that depth is reached — even if the standard "did anything change?" termination check would continue for another iteration or two. For deep but bounded hierarchies, this saves multiple full-graph scans and can reduce inference time by 20–50%.

---

## DRed: Delete-Rederive Algorithm

**Deleting a base triple only updates the derived facts that actually depended on it.**

Before DRed, deleting a triple from the base graph required a full re-materialization: discard all derived facts and re-run inference from scratch. For large rule sets on million-triple datasets this could take minutes. DRed (published by Gupta, Katiyar and Sagiv in 1993, and used in production by systems like RDFox) takes a surgical approach: over-delete all derived facts that might have depended on the deleted triple, then re-derive any that survive through alternative derivation paths. Only the net-removed derived facts are permanently gone. For most real-world deletes, DRed completes in milliseconds rather than minutes.

---

## Incremental Rule Updates

**Adding or removing a single rule does not require re-running full inference.**

Beyond incremental data updates, this release also makes rule updates incremental. The `pg_ripple.add_rule()` function adds a single rule to a live rule set and runs only the one additional iteration needed for the new rule — not a full recompute of the whole rule set. The `pg_ripple.remove_rule()` function removes a rule and uses DRed to retract the derived facts that were solely supported by that rule. For production systems that evolve their rule sets gradually, this makes live rule management practical.

---

## DRed Safety: Graceful Fallback for Complex Cases

**When DRed cannot safely determine the minimal retraction, it falls back to full recomputation.**

DRed's guarantee of minimally retracting only the right derived facts holds for most rule structures. For unusual patterns involving self-referential derivations where DRed cannot resolve the dependency safely, it detects the situation (error code `PT530`) and falls back to a full re-materialization automatically. From the application's perspective, the result is always correct — DRed just provides the fast path when it is safe to use.
