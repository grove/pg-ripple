# v0.77.0 - Bidirectional Integration Primitives

> **Full technical details:** [v0.77.0-full.md](v0.77.0-full.md)

**Status: Planned** | **Scope: Medium-Large**

> v0.77.0 adds the generic primitives needed for bidirectional integration between pg_ripple and external systems: source attribution via named graphs, diff-mode ingest with RDF-star change timestamps, declarative conflict resolution (including time-aware `latest_wins`), SHACL-driven upsert, symmetric delete propagation, loop-safe subscriptions, object-level CAS events, target-assigned ID rendezvous, a stable transport contract, and per-graph observability. A CRM ⇄ ERP relay is one worked example; mesh, peer-to-peer, and federated topologies use the same primitives.

---

## What is this?

The relay pattern in [pg-trickle-relay.md](../docs/src/operations/pg-trickle-relay.md) lets pg_ripple sit in the middle of a bidirectional integration: many sources of JSON data feed in, are reasoned over, and many destinations consume the enriched results. Inbound and outbound mechanics are solid; `register_json_mapping` (v0.73.0) closes the symmetric-mapping gap. What is still missing is the machinery that makes the *bidirectional* loop safe.

Independent of topology, eight problems need solving:

1. **Source attribution** — knowing which external system contributed each triple.
2. **Per-triple change timestamps** — deriving when each individual fact last changed, given that source payloads carry only one root-level `lastModified` covering the whole row.
3. **Conflict resolution** — choosing which source wins when two disagree, ideally based on real source-event time rather than pg_ripple-ingest order.
4. **Echo-aware comparison** — when an external system stores at lower precision, the round-trip echo would otherwise be treated as a fresh change every cycle.
5. **Upsert semantics** — treating an updated payload as an update rather than two separate inserts.
6. **Cross-source reference resolution** — when CRM emits a contact referencing `crm:company-7` and ERP emits the same logical company as `erp:account-3`, an outbound push to CRM must express the reference as `crm:company-7`.
7. **Delete propagation + loop prevention** — propagating deletes outbound and stopping change events from cycling back to their origin.
8. **Compare-and-swap on outbound apply** — receivers must be able to detect concurrent external changes between pg_ripple's emit time and the receiver's apply time, without re-implementing CAS in every relay client.

An additional concern — target-assigned ID feedback — is handled by a typed rendezvous primitive: pg_ripple owns a `pending_linkbacks` table and exposes `pg_ripple.record_linkback(event_id, target_id => '<bare-id>')`. The relay creates the entity in the target system and forwards the bare ID it gets back; pg_ripple expands it through the target graph's `iri_pattern` (the same one used inbound by `ingest_json`), atomically writes the `owl:sameAs` triple, and flushes any events buffered for the same subject. A literal-IRI escape hatch (`target_iri`) is available for target systems whose responses already carry canonical URLs. Crash recovery and multi-relay coordination fall out for free; no relay-local pending-ID map and no target-system IRI template duplicated in the relay.

Finally, the transport for outbound events reuses the **pg-trickle outbox/inbox** pattern that v0.52.0 already shipped: the bridge writer emits the new event shape into operator-named outbox tables; pg-trickle's relay handles delivery, retry, and dead-letter. Receiver feedback (linkback, divergence, abandon) flows back via pg-trickle inbox tables whose `AFTER INSERT` triggers call `record_linkback` / `reconciliation_enqueue` / `abandon_linkback`. The wire payload (flat JSON, top-level `version` discriminator) and the side-band SQL helpers are the frozen contract; relays in any language target a stable surface.

RDF already has a built-in identifier for "where did this come from": the **named graph IRI**. v0.77.0 leans on that for attribution and routing. For per-triple change timestamps it uses **RDF-star annotations** (storage and parser support has been in place since v0.4.0) written by an opt-in diff-mode ingest that compares the incoming payload against current state and stamps only the predicates whose values actually changed. For cross-source reference resolution it uses the **`owl:sameAs` equivalence closure** (already maintained by the inference engine since v0.7.0) combined with an `iri_pattern` declared per mapping. No endpoint catalog, no repurposed column, no new identifier scheme.

## Headline outcomes

- **Source attribution via named graphs** — `ingest_json` and `ingest_jsonld` accept a `graph_iri` parameter; that IRI is the source identifier. Use `<urn:source:crm>`, `<urn:tenant:acme:snapshot:2026-04>`, `<https://partner.example.com/feed>` — whatever the deployment needs.
- **Diff-mode ingest with RDF-star change timestamps** — `ingest_json(..., mode => 'diff')` reads the source's root `lastModified` (via a `timestamp_path` JSONPath in the mapping), diffs against current state, and writes per-triple annotations only for predicates whose values actually changed. Idempotent: replaying the same CDC event is a no-op.
- **Declarative conflict resolution** — `register_conflict_policy(predicate, strategy)` with `source_priority` (ordered list of graph IRIs, null fall-through), `latest_wins` (annotation-aware), `reject_on_conflict`, and explicit `union`. Conflict decisions are non-destructive: losing assertions remain in their source graphs while `_pg_ripple.conflict_winners` caches the resolved projection.
- **Echo-aware `normalize`** — optional restricted expression on conflict policies; the comparator runs on the normalised value and a normalised-equal round-trip echo becomes a no-op in the resolved projection even when newer.
- **Upsert ingest semantics** — `mode => 'upsert'` uses SHACL `sh:maxCount 1` on the registered shape to detect functional predicates and emit DELETE+INSERT atomically.
- **Symmetric delete + tombstone CDC** — `pg_ripple.delete_by_subject(mapping, subject_iri, graph_iri)`; CDC subscription bridge fires on tombstone inserts.
- **Cross-source reference resolution with late binding** — the hub exposes the `owl:sameAs` equivalence class; graph-backed subscriptions check if target graph `T` already holds an equivalent IRI and suppress the write when it does. External relay state remains the relay client's responsibility. No flip-flop, no canonical-IRI registry.
- **Loop-safe subscriptions** — `create_subscription(..., exclude_graphs => ARRAY['<iri>', ...])` is sugar for `FILTER(?g NOT IN (...))`; `propagation_depth` breaks cycles through `owl:sameAs`.
- **Object-level outbound events with optimistic-lock `base`** — outbound CDC emits one event per (subject, transaction); `after` carries the full JSON-LD frame and `base` carries a sparse diff of previous values for changed predicates. Receivers compare-and-swap each `base` key against their current state, turning at-least-once delivery into safely-idempotent application. No-op transactions emit nothing. Every event carries a UUID `event_id` and a `subject_resolved` flag.
- **Target-assigned ID rendezvous** — `pg_ripple.record_linkback(event_id, target_id => '<bare-id>')` is an atomic, idempotent API that expands the bare ID through the target graph's `iri_pattern`, writes `owl:sameAs` into the target graph, deletes the pending row, and flushes any subscription-buffered events for the same hub subject into the outbox. A literal `target_iri` argument is the escape hatch for target systems whose responses already carry canonical URLs. The bridge writer holds buffered events out of the outbox until linkback lands, so the relay never sees an unresolved INSERT followed by orphan UPDATEs. A configurable `linkback_timeout` GUC auto-abandons stuck pending entities with full operator visibility (`_pg_ripple.pending_linkbacks`, `iri_rewrite_misses`, WARNING log).
- **Outbound transport via the pg-trickle outbox (BIDI-OUTBOX-01 + BIDI-WIRE-01)** — v0.77.0 extends the v0.52.0 CDC bridge writer to emit the new event shape into operator-named outbox tables; pg-trickle's relay handles delivery, retries, and dead-letter exactly as it does for v0.52.0 events. No parallel queue, no `next_event`/`ack_event` machinery in pg_ripple. The wire payload is a flat JSON event shape with one top-level `version` discriminator (no envelope wrapper); the schema URI is advertised via HTTP `Link: rel="describedby"` on the side-band façade and referenced from `docs/src/operations/event-schema-v1.json`. A `pg_regress` round-trip validates every emitted payload against the schema.
- **Receiver feedback via the pg-trickle inbox (BIDI-INBOX-01)** — `pg_ripple.install_bidi_inbox()` provisions a pg-trickle-fed inbox table with an `AFTER INSERT` trigger that dispatches to `record_linkback`, `reconciliation_enqueue`, or `abandon_linkback` based on the payload's `action` field. Operators not running pg-trickle use the same SQL helpers via thin HTTP façades in `pg_ripple_http` (`/linkback`, `/divergence`, `/abandon`).
- **Per-graph observability** — `pg_ripple.graph_stats()` returns per-graph triple count, last-write timestamp, conflict-rejection count, and subscription count. Useful for tenancy and snapshot tracking too, not only bidirectional sources.
- **Migration script and performance budget** — `sql/pg_ripple--0.73.0--0.74.0.sql` creates the new catalog tables and registers the new function signatures so existing v0.73.0 deployments upgrade in place; a stated performance budget (≤ 2× vanilla ingest with policies, ≤ 5 ms median outbound rewrite) backed by a CI regression gate keeps the new hot-path work honest.
- **Documented patterns** — the relay doc gains a CRM ⇄ ERP walkthrough as one example; the spec calls out that the same primitives compose into mesh and federated topologies.
