# v1.1.0 — Ecosystem Integration: Cypher/GQL Transpiler, Jupyter, LangChain, Kafka, Materialized Views & dbt Adapter

**Effort estimate: 7–10 person-weeks**

**Goal:** Deliver full Cypher/GQL query support (read-only transpiler plus write operations) and the five ecosystem integration items deferred from v0.55.0 and v0.56.0 — a Jupyter SPARQL kernel, LangChain / LlamaIndex tool packages, a Kafka CDC sink, materialized SPARQL views, and a dbt adapter. Together these items open pg_ripple to the large community of property-graph (Neo4j/Cypher) users and make pg_ripple a first-class citizen in data-science notebooks, LLM application frameworks, event-streaming pipelines, analytics dashboards, and data-transformation pipelines.

## Cypher / GQL read-only queries

**Query pg_ripple with Neo4j-style Cypher patterns.**
A new `pg_ripple.cypher(query TEXT)` set-returning function accepts read-only openCypher / ISO GQL queries (the `MATCH … WHERE … RETURN` subset) and translates them to SPARQL algebra before handing them to the existing query engine.

A query like:

```cypher
MATCH (person:Person)-[:KNOWS]->(friend:Person)
WHERE person.name = 'Alice'
RETURN friend.name
```

is equivalent to a multi-triple SPARQL `SELECT` and is executed by the same optimized pipeline (Citus shard-pruning, WCOJ, federation). Existing Neo4j users can query their migrated data without rewriting queries. Data scientists who know Cypher from graph database courses can use pg_ripple immediately.

Write operations (`CREATE`, `SET`, `MERGE`, `DELETE`) are delivered in the same release — see below.

## Cypher / GQL write operations

**Create, update, and delete nodes and relationships using Cypher.**
Building on the read-only transpiler, v1.1.0 extends the `pg_ripple.cypher()` function to support Cypher mutation clauses:

- `CREATE (n:Person {name: 'Bob'})` — inserts the corresponding RDF triples.
- `MATCH (n:Person {name: 'Alice'}) SET n.age = 30` — updates property literals.
- `MATCH (n:Person {name: 'Alice'}) DELETE n` — retracts triples via the standard deletion path.
- `MERGE` patterns follow the RDF upsert semantics already provided by `pg_ripple.insert_triple`.

All writes go through the existing HTAP delta layer and are visible to SPARQL queries immediately.

**Why it matters**: Applications that were built against Neo4j can be pointed at pg_ripple without rewriting their Cypher mutation code.

## Jupyter SPARQL Kernel

**Write SPARQL directly in Jupyter notebooks.**
A Python package `pg-ripple-kernel` (published to PyPI) implements a Jupyter kernel spec. Users install it with `pip install pg-ripple-kernel`, then write `%%sparql` cells in any Jupyter environment. `SELECT` results render as pandas DataFrames; `CONSTRUCT` results render as formatted Turtle. The kernel connects to any configured `pg_ripple_http` endpoint, so it works equally well against a local Docker deployment or a remote cloud database.

**Why it matters**: Data scientists can mix Python analysis and SPARQL graph queries in a single notebook without switching tools, copying data, or writing HTTP boilerplate.

## LangChain / LlamaIndex Tool Integration

**Plug pg_ripple into any LLM application framework.**
A Python package `pg-ripple-tools` (published to PyPI) provides two ready-made connectors:
- `PgRippleSPARQLTool` — a LangChain `BaseTool` subclass that accepts a natural-language or SPARQL query and routes it to the `/sparql` endpoint, returning the result as a string the agent can read.
- `PgRippleQueryEngine` — a LlamaIndex `BaseQueryEngine` implementation for the same endpoint.

**Why it matters**: ChatGPT plugins, RAG systems, and autonomous agents can call pg_ripple as a knowledge-graph lookup step without any custom glue code. Developers drop in two lines of Python to give an LLM access to the entire graph.

## Kafka CDC Sink

**Stream every triple change to Kafka in real time.**
When `pg_ripple.kafka_bootstrap_servers` is set, the merge worker publishes every lifecycle event from `pg_ripple_cdc_lifecycle` to a Kafka topic (`{prefix}.rdf.changes`) as a JSON-encoded N-Triples batch. A companion `pg_ripple.kafka_topic_prefix` GUC controls the topic namespace. The integration uses the `rdkafka` Rust crate for reliability and back-pressure handling.

**Why it matters**: Downstream systems — Elasticsearch indices, data warehouses, event logs, CDC pipelines — can consume triple changes as they happen, enabling live analytics and audit trails without polling the database.

## Materialized SPARQL Views

**Pre-compute expensive query results and cache them.**
Three new SQL functions provide a complete materialized-view lifecycle:
- `pg_ripple.create_sparql_view(name, query, refresh_on_merge := true)` — runs the query once and stores results in `_pg_ripple.sparql_mv_{name}`; registers in `_pg_ripple.sparql_mv_catalog`.
- `pg_ripple.refresh_sparql_view(name)` — manual refresh at any time.
- `pg_ripple.drop_sparql_view(name)` — clean removal including the catalog entry.

When `refresh_on_merge = true`, the merge worker automatically refreshes the view after each successful cycle, so the cached results stay in sync with the latest data.

**Why it matters**: Aggregations that would take seconds to recompute are served from a pre-built table in milliseconds. Dashboards and API endpoints that hit the same query repeatedly benefit immediately.

## dbt Adapter

**Make RDF data accessible to the dbt transformation ecosystem.**
A `dbt-pg-ripple` Python package provides SPARQL-aware `{{ sparql('…') }}` macros that execute queries via `pg_ripple_http` and return result sets as dbt models. Users can combine traditional SQL transformations with SPARQL graph queries in a single dbt project. A `{{ sparql_construct('…') }}` macro returns Turtle for graph-aware pipelines.

**Why it matters**: dbt is the dominant data-transformation framework in the modern data stack. This adapter lets dbt users query pg_ripple graphs as sources in their existing projects without any custom adapter code.
