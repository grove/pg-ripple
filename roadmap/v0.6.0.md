# v0.6.0 — HTAP Architecture

**Effort estimate: 5–7 person-weeks**

**Goal:** As pg_ripple moves toward production workloads, the single-table VP design creates a tension: writes want to hold locks, but reads want consistent access without waiting. This release introduces the HTAP (Hybrid Transactional/Analytical Processing) storage architecture — a write-optimised delta partition alongside a read-optimised main partition — so that concurrent reads and writes no longer block each other. It also adds change notifications, shared-memory caching, and optional integration with the pg-trickle statistics extension.

---

## Split Write Path from Read Path

**Writes go to a fast "delta" table; reads query the "main" archive plus the delta.**

Each VP table is now split into two physical parts: a `_delta` heap table that receives all new inserts (small, always up to date), and a `_main` table that holds the bulk of historical data (large, read-optimised with a BRIN index). A VP view merges both together for queries. Writes only ever touch the small delta table — no locks on the large main table. The background merge worker periodically consolidates delta into main, keeping the delta small.

---

## Background Merge Worker

**A separate process automatically keeps the storage layers in sync.**

A PostgreSQL background worker process monitors the delta tables and merges them into main on a configurable schedule (controlled by the `pg_ripple.merge_interval_secs` and `pg_ripple.merge_threshold` configuration parameters). The merge is done by creating a fresh copy of the main table with the new rows included, then atomically swapping it in. This means reads see a consistent snapshot throughout, with no partial merges visible. After this release, pg_ripple must be listed in the PostgreSQL `shared_preload_libraries` setting so the worker starts automatically.

---

## Change Data Capture (CDC)

**Applications can subscribe to a live stream of triple changes.**

When a triple is inserted or deleted, pg_ripple now publishes a notification on a PostgreSQL `NOTIFY` channel. Any connected client that runs `LISTEN pg_ripple_cdc` receives a JSON payload describing the change: the operation (add or remove) and the encoded triple IDs. This is the first step toward reactive applications that update dashboards, trigger downstream processing, or feed event-streaming platforms in real time.

---

## Shared-Memory Dictionary Cache

**All database connections share a single in-memory term cache.**

Previously, each PostgreSQL connection had its own private dictionary cache, which meant that a frequently used IRI like `rdf:type` was encoded freshly for every new connection. This release allocates a fixed-size shared-memory segment (controlled by the `pg_ripple.dictionary_cache_size` GUC) that all backends read and write together. A connection that just encoded `rdf:type` populates the shared cache, and subsequent connections find it there without a database round-trip. The cache uses a set-associative design to minimise collision evictions.

---

## Bloom Filter for Fast Delta Skip

**Queries skip the delta table entirely when they know it is empty for a predicate.**

For predicates where no recent writes have occurred, the query path can skip the delta table scan entirely. A per-predicate bloom filter in shared memory tracks which predicates have entries in delta. A false positive (the filter says "check delta" when the delta is actually empty) costs a cheap no-result scan. A false negative (skipping delta when it has data) is prevented by the design: the filter is only cleared after a confirmed merge. This keeps the common read-heavy case fast.

---

## Optional pg-trickle Integration

**Live statistics can flow to the pg-trickle event-streaming extension.**

If the optional pg-trickle extension is installed alongside pg_ripple, the merge worker can publish merge cycle statistics (how many triples were merged, how long it took, predicate-level counts) as trickle relay events. This is purely optional: pg_ripple works fully without pg-trickle. The integration is detected at runtime and enabled automatically.
