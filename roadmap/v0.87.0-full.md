# v0.87.0 Full Details — Uncertain Knowledge Engine

**Parent:** [v0.87.0.md](v0.87.0.md)
**Status: Planned** | **Scope: Very Large**

---

## Background

Classical knowledge graph systems reason in binary: a fact is either true or false, a constraint either passes or fails, a class member either belongs or does not. Real-world knowledge is graded. Facts extracted from text carry extraction confidence. Sensor readings carry measurement uncertainty. Regulatory rules have degrees of compliance. Ontologies often encode probabilistic rather than definitive relationships.

v0.87.0 introduces a graded confidence layer that composes with the existing VP storage, SPARQL engine, Datalog evaluator, and SHACL validator without replacing any of them. The key architectural decision is a side table — `_pg_ripple.confidence` — keyed on the globally-unique statement ID (SID) that VP tables already carry. The hot path is completely unaffected when probabilistic features are disabled.

All five features share this infrastructure:

| Feature | Complexity |
|---|---|
| Probabilistic Datalog (`@weight` rules) | Large |
| Fuzzy SPARQL filtering (`pg:fuzzy_match`, `pg:token_set_ratio`) | Medium |
| Confidence-threshold property paths (`pg:confPath`) | Medium |
| Soft SHACL scoring (`sh:severityWeight`, `shacl_score()`) | Medium |
| Provenance-weighted confidence (PROV-O source trust) | Small (given Feature 1) |

Research basis: [plans/probabilistic-features.md](../plans/probabilistic-features.md).

## CONF-TABLE-01 — Confidence Side Table

### Current state

VP tables store `(s, o, g, i, source)` with `i` as a globally-unique SID. There is no mechanism to attach a numeric confidence to a triple. The `probabilistic_datalog` GUC stub exists (added v0.57.0) but is a no-op.

### What changes

Add to the `sql/pg_ripple--0.86.0--0.87.0.sql` migration script:

```sql
CREATE TABLE IF NOT EXISTS _pg_ripple.confidence (
    statement_id  BIGINT      NOT NULL,
    confidence    FLOAT8      NOT NULL CHECK (confidence >= 0.0 AND confidence <= 1.0),
    model         TEXT        NOT NULL DEFAULT 'datalog',
    asserted_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    PRIMARY KEY   (statement_id, model)
);
CREATE INDEX IF NOT EXISTS confidence_stmt_idx
    ON _pg_ripple.confidence (statement_id);
```

The `model` column identifies the source of the confidence value (`'datalog'`, `'prov'`, `'explicit'`, custom). The `FLOAT8 CHECK` guards invariants at the schema level. The separate index on `statement_id` supports per-triple lookups used by `pg:confidence()`.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-TABLE-01a | `_pg_ripple.confidence` table in migration script `sql/pg_ripple--0.86.0--0.87.0.sql` |
| CONF-TABLE-01b | `confidence_stmt_idx` GIN index |
| CONF-TABLE-01c | Table documented in storage reference |

## PROB-DATALOG-01 — Probabilistic Datalog

### Current state

The `pg_ripple.probabilistic_datalog` GUC exists but is a no-op. The Datalog rule parser (`src/datalog/parse.rs`) does not accept rule annotations. The semi-naive evaluator (`src/datalog/seminaive.rs`) generates plain SQL CTEs with no confidence arithmetic.

### Mathematical model

**Conjunction:** confidences multiply. A rule at weight `w` applied to body atoms with confidences `c₁, c₂, …, cₙ` produces:

```
conf(head) = w × c₁ × c₂ × … × cₙ
```

A rule at weight 0.9 with body atoms at 0.95 and 0.80 yields `0.9 × 0.95 × 0.80 = 0.684`.

**Noisy-OR (multiple derivation paths):** when a fact is derived via `k` independent rule firings at confidences `c₁ … cₖ`, the combined confidence is:

```
conf(head) = 1 − ∏ᵢ (1 − cᵢ)
```

Two independent paths at 0.7 yield `1 − 0.3 × 0.3 = 0.91`.

**Explicit facts:** triples inserted without a confidence have `confidence = 1.0` by default.

This model is provably sound for acyclic rule sets. For cyclic sets, a guard GUC (`pg_ripple.prob_datalog_cyclic`, default off) is required to opt in to approximate evaluation.

### What changes

1. **Grammar** — add `@weight(FLOAT)` to the Datalog rule annotation parser in `src/datalog/parse.rs`. Float must be in `[0.0, 1.0]`; values outside range are rejected at parse time.
2. **SQL generation** — when `probabilistic_datalog = on`, the semi-naive CTE generator in `src/datalog/seminaive.rs` emits:
   ```sql
   INSERT INTO _pg_ripple.confidence (statement_id, confidence, model)
   SELECT i, $rule_weight * COALESCE(c1.confidence, 1.0) * COALESCE(c2.confidence, 1.0), 'datalog'
   FROM vp_{p} JOIN _pg_ripple.confidence c1 ON c1.statement_id = vp1.i ...
   ON CONFLICT (statement_id, model) DO UPDATE
     SET confidence = 1 - (1 - EXCLUDED.confidence) * (1 - _pg_ripple.confidence.confidence)
   ```
   The `ON CONFLICT` clause implements noisy-OR in-database.
3. **`explain_inference()`** — include per-step confidence in derivation tree JSON output.
4. **GUC** — activate the `pg_ripple.probabilistic_datalog` GUC (currently a stub). Add `pg_ripple.prob_datalog_cyclic` GUC (bool, default off) for cyclic rule sets.

### SQL surface

```sql
SET pg_ripple.probabilistic_datalog = on;

SELECT pg_ripple.add_rule(
  'budget_approver(?x) :- decision_maker(?x), dept(?x, finance). @weight(0.85)'
);

SELECT pg_ripple.load_triples_with_confidence(
  '<ex:alice> <ex:isDecisionMaker> "true" .',
  confidence => 0.90
);

-- Query derived confidence
SELECT * FROM pg_ripple.sparql('
  SELECT ?approver ?conf WHERE {
    ?approver ex:isBudgetApprover true .
    BIND(pg:confidence(?approver, ex:isBudgetApprover, true) AS ?conf)
    FILTER(?conf > 0.5)
  } ORDER BY DESC(?conf)
');
```

## SPARQL-CONF-FN-01 — pg:confidence() SPARQL Function

### Current state

There is no SPARQL mechanism to retrieve the confidence of a derived triple. Users cannot filter or sort by confidence scores in SPARQL queries.

### What changes

Add `pg:confidence(?s, ?p, ?o)` to the SPARQL function dispatch table in `src/sparql/plan.rs`. The function compiles to:

```sql
COALESCE(
  (SELECT MAX(confidence) FROM _pg_ripple.confidence
   WHERE statement_id = (
     SELECT i FROM _pg_ripple.vp_{p_id}
     WHERE s = {s_enc} AND o = {o_enc}
     LIMIT 1
   )
  ),
  1.0  -- explicit facts without a confidence entry have confidence 1.0
)
```

The three arguments can be variables or bound terms. When all three are bound, the function performs a single SID lookup; when variables are unbound, confidence is joined lazily in the SELECT clause.

### Deliverables

| ID | Deliverable |
|----|-------------|
| SPARQL-CONF-FN-01a | `pg:confidence(?s, ?p, ?o)` in SPARQL function dispatch |
| SPARQL-CONF-FN-01b | Compiles to correct SQL for bound and unbound variable patterns |
| SPARQL-CONF-FN-01c | pg_regress test for `FILTER`, `BIND`, and `ORDER BY` usage |
| SPARQL-CONF-FN-01d | `explain_sparql()` output includes confidence join node when `pg:confidence()` appears in query |

```sql
pg_ripple.load_triples_with_confidence(
```

```sql
INSERT INTO _pg_ripple.confidence (statement_id, confidence, model)

FROM _pg_ripple.mutation_journal
WHERE txid = txid_current()
ON CONFLICT DO NOTHING;
```

## FUZZY-SPARQL-01 — Fuzzy SPARQL Filtering

### Current state

SPARQL `FILTER` supports only exact string equality and regex matching. There is no trigram similarity function and no trigram index on the dictionary table.

### What changes

1. **New SPARQL functions** added to the dispatch table:
   - `pg:fuzzy_match(a, b)` → `pg_trgm.similarity(a, b)` (trigram overlap; 0.0–1.0)
   - `pg:token_set_ratio(a, b)` → `pg_trgm.word_similarity(a, b)` (word-set similarity; useful for proper nouns, company names)
2. **GUC** — `pg_ripple.default_fuzzy_threshold` (FLOAT8, default 0.7). Used when no explicit threshold is provided in path queries.
3. **Migration script** — creates a GIN trigram index on `_pg_ripple.dictionary (value gin_trgm_ops)` if `pg_trgm` is available:
   ```sql
   DO $$ BEGIN
     IF EXISTS (SELECT 1 FROM pg_extension WHERE extname = 'pg_trgm') THEN
       CREATE INDEX IF NOT EXISTS dict_trgm_idx
         ON _pg_ripple.dictionary USING GIN (value gin_trgm_ops);
     END IF;
   END; $$;
   ```
   When this index exists, PostgreSQL's planner can push similarity thresholds to an index scan.

### Example usage

```sparql
SELECT ?person ?name WHERE {
  ?person foaf:name ?name .
  FILTER(pg:fuzzy_match(?name, "John Smith") > 0.8)
}

SELECT ?org ?name WHERE {
  ?org rdf:type org:Organization .
  ?org rdfs:label ?name .
  FILTER(pg:token_set_ratio(?name, "Microsoft Corporation") > 0.9)
}
```

### Deliverables

| ID | Deliverable |
|----|-------------|
| FUZZY-SPARQL-01a | `pg:fuzzy_match()` and `pg:token_set_ratio()` in SPARQL dispatch |
| FUZZY-SPARQL-01b | Require `pg_trgm` at function call time (error if unavailable) |
| FUZZY-SPARQL-01c | Conditional GIN trigram index in migration script |
| FUZZY-SPARQL-01d | `pg_ripple.default_fuzzy_threshold` GUC |
| FUZZY-SPARQL-01e | pg_regress tests |

## FUZZY-PATH-01 — Confidence-Threshold Property Paths

### Current state

SPARQL property paths compile to `WITH RECURSIVE … CYCLE` CTEs. All edges in the traversal are treated as equally valid; there is no mechanism to restrict traversal to edges above a confidence threshold.

### What changes

Add `pg:confPath(predicate, min_confidence)` as a new path operator in the SPARQL path grammar and the SPARQL → SQL translator in `src/sparql/plan.rs`. The generated SQL is a `WITH RECURSIVE` CTE that joins against `_pg_ripple.confidence` in the recursive edge condition:

```sql
WITH RECURSIVE conf_path(start, node, depth) AS (
  SELECT s, o, 1
  FROM _pg_ripple.vp_{p_id} vp
  JOIN _pg_ripple.confidence c ON c.statement_id = vp.i
  WHERE vp.s = {subject_enc}
    AND c.confidence >= {min_confidence}
  UNION ALL
  SELECT cp.start, vp.o, cp.depth + 1
  FROM conf_path cp
  JOIN _pg_ripple.vp_{p_id} vp ON vp.s = cp.node
  JOIN _pg_ripple.confidence c ON c.statement_id = vp.i
  WHERE c.confidence >= {min_confidence}
    AND cp.depth < {max_depth}
) CYCLE node SET is_cycle USING path
SELECT * FROM conf_path WHERE NOT is_cycle
```

Edges without a confidence row in `_pg_ripple.confidence` are treated as `confidence = 1.0` (explicit facts are always confident).

### Example usage

```sparql
# Follow only high-confidence connections
SELECT ?start ?end WHERE {
  ?start pg:confPath(ex:relatedTo, 0.7)+ ?end
}

# With GUC default threshold
SELECT ?start ?end WHERE {
  ?start pg:confPath(ex:relatedTo)+ ?end
}
```

### Deliverables

| ID | Deliverable |
|----|-------------|
| FUZZY-PATH-01a | `pg:confPath(pred, threshold)` in path grammar and SQL translator |
| FUZZY-PATH-01b | Default threshold from `pg_ripple.default_fuzzy_threshold` GUC when omitted |
| FUZZY-PATH-01c | Edges without confidence entry treated as 1.0 |
| FUZZY-PATH-01d | pg_regress tests for single-hop and multi-hop traversal |

## SOFT-SHACL-01 — Soft SHACL Scoring

### Current state

SHACL validation is binary: a focus node either satisfies or violates a constraint. The `shacl_validate()` function returns `sh:ValidationResult` rows with a categorical `sh:resultSeverity`. There is no aggregate quality score.

### Mathematical model

Each shape declares an optional `sh:severityWeight xsd:decimal` (default 1.0). The composite quality score is:

```
score = 1 − (Σᵢ wᵢ × violationsᵢ) / (Σᵢ wᵢ × applicableᵢ)
```

This produces 1.0 for perfect data, 0.0 for total failure, and meaningful intermediate values.

### What changes

1. **Shape store** — recognise `sh:severityWeight` as a catalogued predicate in `src/shacl/`.
2. **`pg_ripple.shacl_score(graph_iri TEXT) RETURNS FLOAT8`** — runs the existing validation pipeline and aggregates with the weighted formula.
3. **`pg_ripple.shacl_report_scored(graph_iri TEXT) RETURNS TABLE (focus_node TEXT, shape_iri TEXT, result_severity TEXT, result_severity_score FLOAT8, message TEXT)`** — per-result scores alongside existing categorical severity.
4. **Optional monitoring table** — `_pg_ripple.shacl_score_log (graph_iri TEXT, score FLOAT8, measured_at TIMESTAMPTZ)` populated by a new `log_shacl_score(graph_iri TEXT)` helper for continuous quality monitoring.

### Deliverables

| ID | Deliverable |
|----|-------------|
| SOFT-SHACL-01a | `sh:severityWeight` recognised in shape store |
| SOFT-SHACL-01b | `pg_ripple.shacl_score()` SQL function |
| SOFT-SHACL-01c | `pg_ripple.shacl_report_scored()` SQL function |
| SOFT-SHACL-01d | `_pg_ripple.shacl_score_log` table and `log_shacl_score()` helper |

## PROV-CONF-01 — Provenance-Weighted Confidence

### Current state

pg_ripple supports PROV-O provenance (since v0.57.0): triples record `prov:wasDerivedFrom` and `prov:wasAttributedTo` back to their source. There is no mechanism to convert these provenance relationships into triple confidence scores automatically.

### What changes

1. **`pg:sourceTrust FLOAT8`** registered as a special predicate in `_pg_ripple.predicates`.
2. **Built-in Datalog rule template** (auto-registered when `pg_ripple.prov_confidence = on`):
   ```
   triple_confidence(?s, ?p, ?o, ?trust) :-
     quad(?s, ?p, ?o, ?g),
     named_graph_source(?g, ?src),
     source_trust(?src, ?trust). @weight(1.0)
   ```
   During `run_inference()`, this rule propagates trust scores from source declarations to `_pg_ripple.confidence` with `model = 'prov'`.
3. **Noisy-OR** — when a triple appears in multiple named graphs from different sources, the existing noisy-OR merge combines the independent attestations automatically.
4. **`pg_ripple.prov_confidence` GUC** (bool, default off) — enables the feature.

### Example usage

```sparql
-- Declare source trust scores
INSERT DATA {
  GRAPH ex:provenance {
    ex:WikidataExtractor  pg:sourceTrust 0.95 .
    ex:NLPPipeline        pg:sourceTrust 0.70 .
    ex:ManualEntry        pg:sourceTrust 0.99 .
  }
}
```

```sql
SET pg_ripple.prov_confidence = on;
SELECT pg_ripple.run_inference();

-- Query with confidence derived from provenance
SELECT * FROM pg_ripple.sparql('
  SELECT ?s ?conf WHERE {
    ?s ex:hasProperty ?o .
    BIND(pg:confidence(?s, ex:hasProperty, ?o) AS ?conf)
  } ORDER BY DESC(?conf)
');
```

### Deliverables

| ID | Deliverable |
|----|-------------|
| PROV-CONF-01a | `pg:sourceTrust` registered in predicate catalog |
| PROV-CONF-01b | Built-in Datalog rule template |
| PROV-CONF-01c | `pg_ripple.prov_confidence` GUC |
| PROV-CONF-01d | Noisy-OR combines multi-source attestations |
| PROV-CONF-01e | pg_regress test |

## CONF-GC-01 — Confidence Row Lifecycle and Garbage Collection

### Current state

There is no mechanism to clean up `_pg_ripple.confidence` rows when the corresponding triples are deleted via `DELETE DATA`, `SPARQL Update`, or Datalog `retract_inferred()`. Orphaned rows accumulate silently, causing `pg:confidence()` to return stale scores for SIDs that are later reused by a different statement.

### What changes

1. **Retraction cascade** — the SPARQL Update DELETE path (`src/sparql/execute.rs`) and `retract_inferred()` (`src/construct_rules/retract.rs`) must `DELETE FROM _pg_ripple.confidence WHERE statement_id = $sid` for each retired SID. This is a correctness requirement: SIDs are reused by `statement_id_seq` rollover (rare but possible), and stale rows would contaminate new statements.

2. **HTAP tombstone path** — when the merge worker copies tombstones into the main partition, it must also delete the corresponding confidence rows. Add a `DELETE FROM _pg_ripple.confidence WHERE statement_id = ANY($tombstone_sids)` step to the merge routine in `src/storage/promote.rs`.

3. **`vacuum_confidence()` helper** — periodic orphan cleanup for rows that missed the cascade (e.g., direct VP table manipulation in tests):
   ```sql
   CREATE OR REPLACE FUNCTION pg_ripple.vacuum_confidence() RETURNS BIGINT
   LANGUAGE plpgsql AS $$
   DECLARE deleted BIGINT;
   BEGIN
     DELETE FROM _pg_ripple.confidence c
     WHERE NOT EXISTS (
       SELECT 1 FROM _pg_ripple.predicates p
       JOIN LATERAL (
         SELECT 1 FROM _pg_ripple.vp_rare WHERE i = c.statement_id
         UNION ALL
         SELECT 1 FROM pg_class cls
           JOIN pg_namespace ns ON cls.relnamespace = ns.oid
           -- delegate to per-VP table EXISTS check via dynamic SQL
       ) sub ON true
     );
     GET DIAGNOSTICS deleted = ROW_COUNT;
     RETURN deleted;
   END; $$;
   ```
   The implementation uses a single-pass JOIN against `_pg_ripple.predicates` and all VP tables; the exact SQL is generated dynamically against the current VP catalog. Called manually or by the vacuum extension hook.

4. **`pg_dump` round-trip** — `_pg_ripple.confidence` must be included in `pg_dump` output alongside the VP tables so confidence scores survive backup/restore.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-GC-01a | Retraction cascade in `sparql/execute.rs` DELETE path |
| CONF-GC-01b | Retraction cascade in `construct_rules/retract.rs` |
| CONF-GC-01c | Merge worker tombstone cleanup in `storage/promote.rs` |
| CONF-GC-01d | `pg_ripple.vacuum_confidence() RETURNS BIGINT` helper function |
| CONF-GC-01e | pg_regress test: delete triple, assert confidence row gone |
| CONF-GC-01f | pg_dump round-trip CI assertion includes confidence rows |

## CONF-HTAP-01 — HTAP Merge and Confidence Row Durability

### Current state

The HTAP merge worker (`src/storage/promote.rs`) combines `vp_{id}_delta` + `vp_{id}_main` (minus tombstones) into a fresh `vp_{id}_main`. The merge is transactional at the VP table level. The `_pg_ripple.confidence` side table is not part of the merge transaction, creating a window where a read could see the new main partition but not the corresponding confidence rows.

### What changes

Wrap confidence row migration inside the same transaction as the VP table `RENAME`:

```sql
BEGIN;
  -- existing: INSERT INTO vp_{id}_main SELECT * FROM new_main;
  -- existing: DROP TABLE vp_{id}_delta;
  -- new: ensure confidence rows referencing delta SIDs survive in main
  --      (they already reference SIDs by value, so no action needed for
  --       surviving rows; only tombstoned SIDs need cleanup — see CONF-GC-01c)
  DELETE FROM _pg_ripple.confidence
  WHERE statement_id = ANY(SELECT i FROM vp_{id}_tombstones);
COMMIT;
```

Because SIDs are global and immutable (never rewritten during merge), confidence rows for surviving triples require no migration — they continue to reference the same SID. Only tombstoned SIDs need cleanup, which is now done inside the merge transaction.

Add an explicit note to the HTAP architecture documentation that `_pg_ripple.confidence` is SID-keyed and therefore merge-stable for non-tombstoned rows.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-HTAP-01a | Merge worker deletes tombstoned SIDs from confidence table in-transaction |
| CONF-HTAP-01b | HTAP architecture doc section notes confidence merge-stability |
| CONF-HTAP-01c | pg_regress test: merge under load with probabilistic_datalog on; assert no stale confidence rows |

## CONF-EXPORT-01 — Confidence Serialization in RDF Export

### Current state

The export functions (`src/export/`) serialize triples as Turtle, N-Triples, N-Quads, and JSON-LD. Confidence scores are stored in `_pg_ripple.confidence` but are not exposed in any serialization format.

### What changes

Define the representation convention and implement optional export:

1. **Representation** — use RDF* reification (already supported since v0.4.0). A triple `<s> <p> <o>` with confidence `c` serializes as:
   ```turtle
   << <s> <p> <o> >> pg:confidence "0.85"^^xsd:float .
   ```
   For N-Quads and N-Triples (no RDF* support), fall back to a separate named graph:
   ```
   <_:conf1> <http://pg-ripple.io/ns#aboutStatement> << <s> <p> <o> >> .
   <_:conf1> <http://pg-ripple.io/ns#confidence> "0.85"^^xsd:float .
   ```

2. **`export_turtle_with_confidence(graph_uri TEXT DEFAULT NULL) RETURNS TEXT`** — variant of `export_turtle()` that JOINs `_pg_ripple.confidence` and emits RDF* confidence annotations for triples that have confidence rows. Controlled by `pg_ripple.export_confidence` GUC (bool, default off) so existing export calls are unaffected.

3. **JSON-LD** — confidence appears as a `@annotation` block on the affected node (JSON-LD 1.1 annotations). Emitted only when `pg_ripple.export_confidence = on`.

## CONF-PERF-01 — Confidence Layer Performance Regression Tests

### Current state

## CONF-RLS-01 — Multi-Tenant Row-Level Security on Confidence Table

### Current state

Named-graph RLS policies on VP tables (v0.55.0, v0.61.0) restrict data access by named-graph ownership. `_pg_ripple.confidence` has no RLS policies. In a multi-tenant deployment Tenant A can read confidence scores for triples in Tenant B's named graphs and insert confidence rows for SIDs in Tenant B's graphs.

### What changes

1. **Enable RLS** — `ALTER TABLE _pg_ripple.confidence ENABLE ROW LEVEL SECURITY` in the v0.87.0 migration script.
2. **Read policy** — `confidence_read` policy allows SELECT only when the current role has SELECT access to the named graph that contains the referenced SID, using the same graph-ownership predicate applied to VP table RLS. Superuser bypass via `pg_ripple.skip_rls = on` (existing GUC) applies.
3. **Write policy** — `confidence_write` policy allows INSERT/UPDATE/DELETE only for the graph owner role and superusers, matching the VP-table write policy.
4. **Owner bypass** — the extension's own SPI calls (inference, retraction, bulk load) run as the extension owner with `pg_ripple.skip_rls = on` so internal operations are unaffected.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-RLS-01a | `ALTER TABLE _pg_ripple.confidence ENABLE ROW LEVEL SECURITY` in migration script |
| CONF-RLS-01b | `confidence_read` RLS policy |
| CONF-RLS-01c | `confidence_write` RLS policy |
| CONF-RLS-01d | pg_regress test: cross-tenant confidence isolation (Tenant A cannot read Tenant B's confidence rows) |
| CONF-RLS-01e | RLS policies documented in security reference |

## CONF-FED-01 — Federation Planner Handling for Confidence Extension Functions

### Current state

The FedX-style federation planner (v0.42.0) rewrites SERVICE clauses by partitioning the SPARQL algebra into local and remote sub-plans. The four new extension functions — `pg:confidence()`, `pg:fuzzy_match()`, `pg:token_set_ratio()`, and `pg:confPath()` — are specific to this PostgreSQL extension and have no row in the federation planner's function-classification table. Without a classification, the planner may attempt to push them into a remote SERVICE clause, which would fail at the remote endpoint or produce silently incorrect results.

### What changes

1. **Local-only classification** — add `pg:confidence`, `pg:fuzzy_match`, `pg:token_set_ratio`, and `pg:confPath` to the local-only function registry in `src/sparql/plan.rs`. Any algebra sub-tree containing these functions is classified `Local` and never pushed inside a `SERVICE` clause.

2. **Correct query splitting** — when a query mixes confidence functions and SERVICE patterns, the planner fetches remote bindings first, then evaluates the confidence function as a local post-processing step using the bound values:
   ```sparql
   SELECT ?s ?conf WHERE {
     SERVICE <http://remote.example.org/sparql> { ?s ex:hasFact ?o . }
     BIND(pg:confidence(?s, ex:hasFact, ?o) AS ?conf)
   }
   ```
   The `BIND` triple is evaluated locally after the SERVICE result set is available.

3. **Error PT0305** — if `pg:confidence()` or any other `pg:` confidence function appears *inside* a SERVICE clause body (rather than as a post-processing step), the planner emits error `PT0305`.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-FED-01a | `pg:confidence`, `pg:fuzzy_match`, `pg:token_set_ratio`, `pg:confPath` in local-only function registry |
| CONF-FED-01b | Planner splits mixed queries: remote fetch first, local confidence join second |
| CONF-FED-01c | `PT0305` error for confidence function inside SERVICE clause body |
| CONF-FED-01d | pg_regress test: mixed local-confidence + remote-SERVICE query produces correct results |

## CONF-CWB-01 — CONSTRUCT Writeback Rule Confidence Propagation

### Current state

SPARQL CONSTRUCT writeback rules (v0.63.0) derive new triples and materialize them in a target named graph; delta maintenance and Delete-Rederive retraction are fully implemented (v0.65.0). When source triples matched by the CONSTRUCT body pattern have confidence scores, the confidence of the derived triples is currently undefined — they are inserted with no confidence row, making them implicitly `confidence = 1.0` regardless of source certainty.

### What changes

Define explicit semantics controlled by a new GUC:

**`pg_ripple.cwb_confidence_propagation` TEXT** (default `'explicit'`):
- `'explicit'` — derived CWB triples get no confidence row (implicit 1.0). No change from current behavior.
- `'inherit'` — derived CWB triples inherit confidence from their sources:
  ```
  conf(derived) = rule_weight × min(conf(source triples matched by body))
  ```
  If no source triples have confidence rows, the derived triple also gets no confidence row (implicit 1.0). If the CONSTRUCT rule has no `@weight` annotation, `rule_weight = 1.0`.

When `'inherit'` is active, the CONSTRUCT rule delta compiler (`src/construct_rules/delta.rs`) emits an additional INSERT into `_pg_ripple.confidence` after each derivation batch:

```sql
INSERT INTO _pg_ripple.confidence (statement_id, confidence, model)
SELECT new_triple.i,
  $rule_weight * MIN(COALESCE(src.confidence, 1.0)),
  'cwb'
FROM {construct_delta_result} new_triple
LEFT JOIN _pg_ripple.confidence src ON src.statement_id = ANY($source_sids)
GROUP BY new_triple.i
ON CONFLICT (statement_id, model) DO UPDATE
  SET confidence = LEAST(EXCLUDED.confidence, _pg_ripple.confidence.confidence);
```

The retraction path in `src/construct_rules/retract.rs` already deletes confidence rows when triples are retracted (CONF-GC-01b).

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-CWB-01a | `pg_ripple.cwb_confidence_propagation` GUC (`'explicit'` \| `'inherit'`, default `'explicit'`) |
| CONF-CWB-01b | Delta compiler emits confidence INSERT when mode is `'inherit'` |
| CONF-CWB-01c | Semantics documented in the uncertain-knowledge docs chapter |
| CONF-CWB-01d | pg_regress test: `'inherit'` mode propagates correct confidence from body-pattern sources |

## CONF-CYCLIC-01 — Cyclic Probabilistic Datalog: Convergence and Iteration Bounds

### Current state

`PROB-DATALOG-01` enables cyclic rule evaluation via `pg_ripple.prob_datalog_cyclic = on`, but provides no specification for:
- How many semi-naive iterations are allowed before the evaluator gives up
- What convergence criterion terminates iteration early
- What the engine does when the maximum iteration count is reached (error, warning, or partial result)

Without convergence bounds, a cyclic probabilistic rule set can run indefinitely in production, consuming all available CPU and memory.

### Mathematical model

For cyclic rule sets, the noisy-OR aggregation may not converge to a fixed point. The standard approach is **value-based convergence**: iteration stops when the maximum absolute confidence delta across all `_pg_ripple.confidence` rows changes by less than ε in a single iteration:

```
max(|conf_new(s, p, o) − conf_old(s, p, o)|) < ε
```

For rule sets that provably do not converge (e.g., rules that amplify confidence), iteration stops at `max_iterations` and the partial result is used with a warning.

### What changes

1. **`pg_ripple.prob_datalog_max_iterations` GUC** (INTEGER, default 100, min 1, max 10,000) — maximum number of semi-naive inference rounds when `prob_datalog_cyclic = on`. After this limit, the evaluator emits a `WARNING` and returns the current partial result.

2. **`pg_ripple.prob_datalog_convergence_delta` GUC** (FLOAT8, default 0.001, range [0.0, 1.0]) — early-exit threshold. If the maximum confidence change in a round is below this value, iteration terminates as converged.

3. **Convergence tracking** — the semi-naive evaluator (`src/datalog/seminaive.rs`) tracks the maximum delta in each round:
   ```sql
   -- After each INSERT ... ON CONFLICT DO UPDATE:
   SELECT MAX(ABS(new_conf - old_conf)) INTO v_delta
   FROM _pg_ripple.confidence ...;
   -- Exit loop when v_delta < convergence_delta OR round >= max_iterations
   ```

4. **`explain_inference()` convergence report** — when `prob_datalog_cyclic = on`, the JSON output includes `"iterations": N`, `"converged": true/false`, and `"final_delta": F`.

5. **Error `PT0307`** — if `max_iterations` is reached without convergence, the evaluator emits `WARNING PT0307` (not an ERROR) so partial results are still usable. Operators can set `pg_ripple.prob_datalog_cyclic_strict = on` to promote this to an ERROR.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-CYCLIC-01a | `pg_ripple.prob_datalog_max_iterations` GUC (INTEGER, default 100) |
| CONF-CYCLIC-01b | `pg_ripple.prob_datalog_convergence_delta` GUC (FLOAT8, default 0.001) |
| CONF-CYCLIC-01c | `pg_ripple.prob_datalog_cyclic_strict` GUC (bool, default off) |
| CONF-CYCLIC-01d | Semi-naive evaluator tracks per-round max delta and exits on convergence |
| CONF-CYCLIC-01e | `explain_inference()` includes `iterations`, `converged`, `final_delta` fields when cyclic mode on |
| CONF-CYCLIC-01f | `PT0307` warning (or error when strict) on max-iterations reached |
| CONF-CYCLIC-01g | pg_regress test: converging rule set exits early; non-converging rule set hits limit and warns |

## CONF-HTTP-01 — pg_ripple_http REST API for Uncertain-Knowledge Functions

### Current state

v0.39.0 established the pattern that all SQL functions exposed by the extension are also reachable via the `pg_ripple_http` REST companion. The new uncertain-knowledge functions — `load_triples_with_confidence()`, `shacl_score()`, `shacl_report_scored()`, `vacuum_confidence()`, and `pg:confidence()` SPARQL queries — are not exposed via any HTTP endpoint. This makes the uncertain-knowledge engine inaccessible from non-SQL clients.

### What changes

Add to `pg_ripple_http/src/routing.rs`:

| Method | Endpoint | Delegates to |
|--------|----------|-------------|
| `POST` | `/confidence/load` | `pg_ripple.load_triples_with_confidence($data, $confidence, $format, $graph_uri)` |
| `GET` | `/confidence/shacl-score` | `pg_ripple.shacl_score($graph_iri)` |
| `GET` | `/confidence/shacl-report` | `pg_ripple.shacl_report_scored($graph_iri)` — returns JSON array |
| `POST` | `/confidence/vacuum` | `pg_ripple.vacuum_confidence()` — returns `{"deleted": N}` |

Request/response format:

```http
POST /confidence/load
Content-Type: application/n-triples
X-Confidence: 0.85
X-Graph-Uri: http://example.org/trusted-data

<http://ex.org/alice> <http://ex.org/knows> <http://ex.org/bob> .
```

```http
GET /confidence/shacl-score?graph=http%3A%2F%2Fex.org%2Fdata
→ 200 OK
{"score": 0.934}
```

Authentication follows the existing `check_auth()` middleware (Bearer token). All four endpoints are documented in the OpenAPI spec (`docs/src/operations/openapi.yaml`).

## CONF-EXPLAIN-01 — explain_datalog() Per-Step Confidence Output

### Current state

`explain_datalog()` (added v0.40.0) produces a plan tree showing rule bodies, estimated cardinalities, and index usage for the Datalog SQL translation. `explain_inference()` (added v0.61.0) produces a derivation tree showing which rules fired and which facts they derived.

`PROB-DATALOG-01f` specifies that `explain_inference()` includes per-step confidence values when `probabilistic_datalog = on`. However `explain_datalog()` is a distinct function that shows the *query plan* level (SQL execution plan), not the derivation level. It should also include confidence-related plan nodes when `probabilistic_datalog = on`.

### What changes

When `probabilistic_datalog = on`, `explain_datalog(rule_id TEXT)` includes:

1. **Confidence join nodes** — each body-atom scan in the plan tree shows whether a `_pg_ripple.confidence` JOIN is present and its estimated cost:
   ```json
   {
     "node": "BodyAtomScan",
     "predicate": "ex:knows",
     "confidence_join": true,
     "confidence_join_cost": 0.03,
     "index": "confidence_stmt_idx"
   }
   ```

2. **Noisy-OR aggregation node** — the final `ON CONFLICT DO UPDATE` step is annotated as a noisy-OR merge:
   ```json
   {
     "node": "ConfidenceWrite",
     "mode": "noisy_or",
     "conflict_action": "UPDATE SET confidence = 1 - (1 - EXCLUDED.confidence) * (1 - confidence.confidence)"
   }
   ```

3. **Rule weight annotation** — the top-level rule node shows the `@weight` value if present:
   ```json
   {
     "node": "Rule",
     "rule_id": "...",
     "weight": 0.85,
     "probabilistic": true
   }
   ```

### Deliverables

| ID | Deliverable |
|----|-------------|
| CONF-EXPLAIN-01a | `explain_datalog()` includes confidence join nodes when `probabilistic_datalog = on` |
| CONF-EXPLAIN-01b | Noisy-OR aggregation node annotated in plan output |
| CONF-EXPLAIN-01c | Rule `@weight` appears in top-level rule node |
| CONF-EXPLAIN-01d | pg_regress test: `explain_datalog()` output for a weighted rule includes expected JSON fields |
