# v0.65.0 - CONSTRUCT Writeback Correctness Closure

> **Full technical details:** [v0.65.0-full.md](v0.65.0-full.md)

**Status: Released** | **Scope: Very Large**

> v0.65.0 closes the correctness gap in v0.63.0 CONSTRUCT writeback. It turns manual full recompute into real delta-maintained derived graphs, fixes promoted-predicate retraction, captures exact provenance, and builds the behavior test matrix promised by the original v0.63.0 roadmap.

---

## What is this?

CONSTRUCT writeback is the core raw-to-canonical pipeline story. The assessment found that the API and catalogs exist, but source writes do not automatically maintain target graphs. This release implements the missing maintenance kernel and validates every lifecycle path.

## Headline outcomes

- Source graph inserts and deletes update dependent CONSTRUCT targets in the same transaction.
- Promoted HTAP predicates retract through delta/tombstone-aware storage paths, not deletes against read views.
- Provenance records exactly the triples produced by each rule execution.
- `mode` values are validated and documented.
- The full CWB test matrix from v0.63.0 is implemented.

## Assessment findings covered

- C1: CONSTRUCT writeback is not incremental despite release claims.
- C2: promoted HTAP predicate retraction can silently fail.
- P1: full recompute provenance captures all target `source = 1` triples, not exact inserted rows.
- S3/A2: user-provided construct-rule values are inserted with manual SQL escaping.
- SC2: actual tests do not cover the promised v0.63 CWB matrix.
- Operational gaps: construct-rule freshness, provenance health, pending deltas, failed rules, and retry controls.
- Recommended new features: Delta Maintenance Kernel and Canonical Graph Pipeline UI/API foundation.
