# v0.39.0 — Datalog HTTP API

**Effort estimate: 3–5 person-weeks**

**Goal:** The `pg_ripple_http` companion service speaks SPARQL over HTTP. This release adds a `/datalog` namespace that exposes all Datalog SQL functions — rule management, inference, goal queries, constraint checking, and monitoring — as REST endpoints. Any HTTP client, without a PostgreSQL driver, can now manage and query the reasoning engine.

---

## Rule Management Endpoints

**Load, list, add, remove, enable, and disable rule sets over HTTP.**

The `/datalog/rules/{rule_set}` family of endpoints mirrors the SQL functions for managing rule sets. `POST /datalog/rules/custom` with a Datalog rule body loads a new rule set. `GET /datalog/rules` lists all loaded rule sets. `DELETE /datalog/rules/custom` drops a rule set. Individual rules can be added with `POST /datalog/rules/custom/add` and removed with `DELETE /datalog/rules/custom/{rule_id}`. All operations are authenticated with the same bearer token as the SPARQL endpoints.

---

## Inference Endpoints

**Trigger full inference, goal-directed inference, or aggregation inference over HTTP.**

`POST /datalog/infer/{rule_set}` triggers `pg_ripple.infer()` and returns the count of derived triples. `POST /datalog/infer/{rule_set}/stats` triggers `infer_with_stats()` and returns the full statistics JSON including iteration counts and parallelism information. Aggregation inference (`/agg`), well-founded semantics (`/wfs`), demand transformation (`/demand`), and lattice inference (`/lattice`) are also exposed as separate endpoints, matching the full SQL API surface.

---

## Query and Constraint Endpoints

**Execute goal queries and check integrity constraints without a database driver.**

`POST /datalog/query/{rule_set}` accepts a Datalog goal pattern as the request body and returns matching derived facts. `GET /datalog/constraints` runs `pg_ripple.check_constraints()` and returns any violations as a JSON array. `GET /datalog/constraints/{rule_set}` checks only the constraints in a specific rule set. These endpoints are useful for compliance checks, monitoring dashboards, and automated data quality gates in CI pipelines.

---

## Admin and Statistics Endpoints

**View cache statistics, manage lattice types, and list live Datalog views.**

`GET /datalog/stats/cache` returns the rule plan cache hit rate and entry count. `GET /datalog/lattices` lists all registered custom lattice types. `POST /datalog/lattices` creates a new lattice type. `GET /datalog/views` lists all registered incremental Datalog views. `POST /datalog/views` and `DELETE /datalog/views/{name}` create and drop views. The full Datalog management surface is now accessible via REST, making pg_ripple scriptable from shell scripts, CI pipelines, and any language with an HTTP library.

---

## Security

**All endpoints use the same auth and parameterized-query guarantees as the SPARQL endpoints.**

Every Datalog endpoint calls `check_auth()` to verify the bearer token. Write endpoints (rule loading, inference triggers) can additionally require a separate write-specific token configured via the `PG_RIPPLE_HTTP_DATALOG_WRITE_TOKEN` environment variable. All SQL calls use parameterized queries — no user input is ever interpolated directly into SQL strings — providing the same SQL injection prevention as the SPARQL HTTP endpoints.
