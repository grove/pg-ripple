# v0.43.0 — WatDiv & Apache Jena Conformance Suites

**Effort estimate: 5–7 person-weeks**

**Goal:** The W3C test suite from v0.41.0 covers the SPARQL standard's own test cases. This release adds two complementary suites: the Apache Jena suite (~1,000 tests) covers real-world edge cases involving data types, blank nodes, and string handling not covered by W3C tests; WatDiv (100 query templates over 10M–100M triples) measures real-world query performance across the five structural query categories that routinely trip up triple-store implementations.

---

## Apache Jena Conformance Adapter

**~1,000 tests covering date/time, type coercion, blank-node scoping, and string functions.**

The Apache Jena project maintains a SPARQL conformance suite that goes significantly beyond the W3C's own test collection. It covers numeric type promotion (integer ÷ integer → decimal), date and time arithmetic (xsd:date minus xsd:duration), blank-node scoping across named graphs, case-insensitive string matching, URI normalization, and dozens of other practical behaviours. pg_ripple's Jena adapter loads each test's data, executes the query, and compares results. Results are reported separately from the W3C suite, with their own pass/fail/skip counts. The target at release is ≥90% pass rate; ≥95% is required before any future v1.0.0 release.

---

## WatDiv Performance Benchmark Harness

**100 query templates over synthetic linked data, from 10 million to 100 million triples.**

WatDiv (Waterloo SPARQL Diversity Test Suite) is the academic standard for measuring SPARQL engine performance across diverse query patterns. It generates synthetic linked data and 100 query templates covering five categories: star queries (many triples sharing one subject), chain queries (sequential property traversals), snowflake queries (star plus chains), complex linear queries, and complex queries with cycles. For each category, WatDiv reports median latency, 99th percentile latency, and total query time. These baselines are stored in CI as `watdiv_baseline.json` and any regression exceeding 20% blocks merge.

## Unified Conformance Runner

**A single command runs all four suites: W3C, Jena, WatDiv, and LUBM (the next release).**

The four conformance suites all use the same underlying fixture loader, named-graph isolation mechanism, and result comparison engine. This release extracts these shared utilities into a `conformance/` library so that adding a new suite (as LUBM is added in v0.44.0) requires writing only the test adapter, not reimplementing graph loading and result comparison. The `scripts/fetch_conformance_tests.sh` helper downloads all suite dependencies from authoritative sources and verifies checksums.

---

## Regressions Are Blocking

**A conformance regression — a test that was passing and now fails — blocks merge.**

New failures are never automatically added to the known-failures manifest. If a previously passing test starts failing, CI reports a regression failure and blocks the pull request. This ensures that new optimisations and refactors cannot silently break existing correct behaviour. Fixing a known failure removes it from the manifest; that removal is part of the same PR that fixes the underlying bug.
