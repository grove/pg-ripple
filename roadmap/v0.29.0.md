# v0.29.0 — Datalog Optimization: Magic Sets & Cost-Based Compilation

**Effort estimate: 5–7 person-weeks**

**Goal:** The Datalog inference engine from v0.10.0 computes all possible consequences of all rules. This is powerful but wasteful when you only need the facts relevant to a specific question. This release adds magic sets — the classical Datalog optimization that derives only the facts needed to answer a targeted query — along with cost-based join reordering, redundant-rule elimination, and smarter negation compilation.

---

## Magic Sets Transformation

**The engine derives only the facts needed to answer your specific question.**

Magic sets (named after a 1986 paper by Bancilhon and colleagues) rewrites a rule program given a specific goal query so that it only derives facts that could possibly contribute to the answer. If you ask "what are all the ancestors of Alice?", magic sets rewrites the transitivity rules to propagate only along the ancestor chain of Alice — not the entire population. For large ontologies where full materialization would compute millions of unnecessary facts, goal-directed inference with magic sets can be 10× to 1000× faster. The `pg_ripple.infer_goal(rule_set, goal)` function applies this automatically.

---

## Cost-Based Body Atom Reordering

**Rules are compiled with their joins ordered by estimated selectivity.**

When a Datalog rule has multiple body atoms (conditions), the order in which they are joined in the generated SQL matters enormously for performance. This release queries PostgreSQL's statistics (`pg_class.reltuples`) at rule-compilation time to estimate the selectivity of each atom, and orders the join sequence from most to least selective. This mirrors what PostgreSQL's own query planner does, applied one level earlier at the Datalog-to-SQL compilation stage.

---

## Subsumption Checking

**Redundant rules that are covered by a more general rule are automatically eliminated.**

If rule A says "every mammal is an animal" and rule B says "every dog is an animal" (while "every dog is a mammal" is also a rule), then rule B is subsumed by the combination of the dog-is-mammal and mammal-is-animal rules — it never derives anything that the other rules would not already derive. Detecting and eliminating these redundant rules reduces the number of SQL statements executed per fixpoint iteration, speeding up inference without changing results.

---

## Anti-Join Negation

**Negated conditions compile to SQL LEFT JOIN patterns for better plan quality.**

Datalog rules with negation ("`?x` is trusted unless `?x` is on the blocklist") compile to SQL NOT EXISTS subqueries by default. For large VP tables, PostgreSQL often generates better plans from a `LEFT JOIN … WHERE … IS NULL` anti-join pattern instead. This release chooses between the two forms at compilation time based on estimated table size, defaulting to anti-join for predicates with more than 1,000 rows.

---

## Predicate-Filter Pushdown

**Arithmetic guards are moved as early as possible in the generated SQL.**

When a Datalog rule contains a condition like `?age > 18`, it is most efficient to apply that filter as soon as the age variable is bound — immediately after the join that produces it. Previously, filter conditions were sometimes placed at the outermost WHERE clause of the SQL statement, meaning many rows were produced and then discarded. Pushdown moves each filter to its earliest possible position, often converting it into a JOIN ON condition that the index can exploit.
