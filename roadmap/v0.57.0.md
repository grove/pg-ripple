# v0.57.0 — Reasoning Platform & AI Integration

**Effort estimate: 8–12 person-weeks**

**Goal:** Elevate pg_ripple from OWL 2 RL to a multi-profile reasoning platform supporting OWL 2 EL and OWL 2 QL. Deliver the first production-grade knowledge-graph embedding (KGE) capability, automated entity alignment, LLM-augmented SPARQL repair, and automated ontology mapping. Also add multi-tenant named-graph isolation via PostgreSQL row-level security.

## OWL 2 EL Profile

**For biomedical and large-scale ontologies.**
The OWL 2 EL profile is optimised for ontologies with very large numbers of classes and properties, such as SNOMED CT, GO, and ChEBI. Unlike OWL 2 RL (which pg_ripple has implemented since v0.48.0), EL supports existential restrictions and can handle millions of axioms without exponential blowup. v0.57.0 implements the core EL reasoning algorithm (classification + instance checking) via Datalog rule compilation.

## OWL 2 QL Profile (Query Rewriting)

**For ontology-mediated query answering over relational data.**
OWL 2 QL (DL-Lite) enables query rewriting: instead of materialising all inferred triples, the reasoner rewrites a SPARQL query into an equivalent (potentially larger) SPARQL query that holds in the base graph alone. This is the approach used by Ontop, a world-class SPARQL-to-SQL rewriter. v0.57.0 implements first-order rewriting for the QL profile's supported axiom types.

## Knowledge-Graph Embeddings (KGE)

**Entity representations that capture graph structure.**
A new background worker (`kge_worker`) iterates over VP tables and computes TransE embeddings (with a RotatE/ComplEx extension point) for all entities. Results are stored in `_pg_ripple.kge_embeddings (entity_id BIGINT, embedding vector)` and indexed with HNSW. These embeddings enable entity similarity queries that are purely structural (not dependent on label text) and serve as a foundation for link-prediction and entity-alignment workflows.

## Entity Alignment via Embedding Similarity

**Automatically discover `owl:sameAs` candidates.**
`pg_ripple.find_alignments(source_graph TEXT, target_graph TEXT, threshold FLOAT DEFAULT 0.85)` uses KGE embeddings to propose `owl:sameAs` pairs between two named graphs. Results are returned as a table of candidate pairs with confidence scores and can feed the existing `apply_sameas_candidates()` workflow.

## LLM-Augmented SPARQL Repair

**Help users fix broken queries.**
When a SPARQL query fails to parse or returns zero results for a pattern that looks like it should match, pg_ripple can send the query + error + schema digest to the configured LLM endpoint and return a corrected query suggestion. This is an optional, explicit call via `pg_ripple.repair_sparql(query TEXT, error TEXT) RETURNS TEXT` — pg_ripple never auto-executes LLM-suggested queries.

## Multi-Tenant Named-Graph Isolation

**Multiple tenants on one database.**
`pg_ripple.grant_graph_access()` (delivered in v0.55.0) creates RLS policies per named graph. v0.57.0 extends this with a tenant-management layer: `pg_ripple.create_tenant(name TEXT, graph_iri TEXT)` creates a PostgreSQL role, maps it to a named graph, sets RLS policies on all VP tables, and creates a quota-enforcing `AFTER INSERT` trigger.
