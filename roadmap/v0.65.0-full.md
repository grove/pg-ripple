## Release theme

v0.65.0 restores semantic correctness to the raw-to-canonical pipeline story. The release makes CONSTRUCT writeback transactional, delta-aware, provenance-exact, and tested across rare predicates, promoted predicates, shared target graphs, post-merge storage, and refresh/drop lifecycle operations. It also creates the derivation kernel that future SHACL-AF rules and materialized SPARQL views can reuse.

## Deliverables

### CWB-FIX-01 - Delta maintenance kernel

Create an internal derived-triple API used by CONSTRUCT rules first and later by SHACL-AF rules and materialized SPARQL views:

```rust
begin_derivation_run(rule_id, source_delta)
insert_derived(rule_id, pred_id, s, o, g, provenance)
retract_derived(rule_id, pred_id, s, o, g)
commit_derivation_run(rule_id)
```

Requirements:

- All writes use existing VP storage APIs and integer-encoded terms.
- Insert path records exact provenance for rows inserted by the current run.
- Delete path removes only triples no remaining rule supports.
- Errors propagate to callers; no warning-only failure for correctness-critical retraction.

Verification:

- Unit tests for reference-count semantics.
- pg_regress tests for shared-target rules.

### CWB-FIX-02 - Source graph write hooks

Wire CONSTRUCT maintenance to source graph changes. The implementation may use statement-level triggers, CDC bridge hooks, or a storage-layer dispatch point, but it must run inside the same transaction as the source write.

Requirements:

- On source insert, evaluate only affected rules in `rule_order`.
- On source delete, run Delete-Rederive or equivalent candidate/rederive/retract logic.
- Graph dependencies come from `_pg_ripple.construct_rules.source_graphs`.
- Bulk loads batch rule maintenance instead of firing one derivation per triple.

Verification:

- Insert source triples after rule creation and verify derived triples appear without `refresh_construct_rule()`.
- Delete source triples and verify orphaned derived triples are retracted.
- Multi-rule pipeline runs in topological order.

### CWB-FIX-03 - HTAP-aware promoted-predicate retraction

Replace direct `DELETE FROM _pg_ripple.vp_{pred_id}` against read views with storage operations that know HTAP layout:

- Delete delta-resident triples from `vp_{id}_delta`.
- Insert tombstones for main-resident triples in `vp_{id}_tombstones`.
- Preserve rare-predicate behavior through `vp_rare` deletes.
- Handle post-merge triples correctly.

Verification:

- Create a rule whose target predicate is promoted.
- Merge the predicate.
- Drop/refresh the rule and verify target triples disappear.
- Assert no stale `source = 1` triples remain in delta, main, or `vp_rare`.

### CWB-FIX-04 - Exact provenance capture

Stop recording provenance by scanning all `source = 1` triples in the target graph. Use one of:

- `INSERT ... RETURNING` into a staging table.
- A run-id scoped temporary table.
- A derivation-kernel return structure containing inserted quads.

Requirements:

- Pre-existing inferred triples are not attributed to a new rule.
- Refresh/drop cannot remove triples not derived by that rule.
- Provenance writes are idempotent and conflict-safe.

Verification:

- Seed target graph with unrelated `source = 1` triples, create a rule, then drop it. Unrelated triples remain.

### CWB-FIX-05 - Parameterized SQL and mode validation

Replace manual escaping in construct-rule catalog writes with `Spi::run_with_args` or equivalent parameterized SPI calls. Validate `mode IN ('incremental', 'full')`.

Verification:

- Inputs with quotes, backslashes, and long SPARQL text round-trip correctly.
- Invalid mode raises a PostgreSQL-style lowercase error with no trailing period.

### CWB-FIX-06 - Shared-target reference-count semantics

Ensure two or more rules can write the same derived triple to the same graph. Dropping or refreshing one rule must preserve triples still supported by another rule.

Verification:

- Two rules produce the same `(pred_id, s, o, g)`.
- Drop rule A with `retract := true`.
- Rule B's triple remains queryable and has provenance.

### CWB-FIX-07 - Construct-rule observability

Add rule health reporting through `list_construct_rules()` and a new detailed view/function:

- last successful incremental run
- last full refresh
- pending source delta count
- failed run count
- last error code/message
- derived triple count by rule
- stale target graph flag
- provenance over-attribution detector
- retraction failure count

Verification:

- pg_regress verifies counters update on success and controlled failure.
- Metrics are exposed to `pg_ripple_http /metrics` when HTTP service is enabled.

### CWB-FIX-08 - Full CWB test matrix

Implement the test matrix originally listed in `roadmap/v0.63.0-full.md`:

- create rule, verify target graph populated
- insert into source graph, verify derived triples appear
- delete from source graph, verify orphaned derived triples retract
- `refresh_construct_rule()` from scratch
- self-cycle rejected
- two-rule pipeline stratification
- mutual cycle rejected
- drop with `retract := true`
- drop with `retract := false`
- shared target preservation
- promoted predicate retraction
- post-merge retraction
- `list_construct_rules()` metadata
- `explain_construct_rule()` delta SQL and source graphs
- SELECT query rejected
- unbound variable rejected
- blank node in CONSTRUCT template rejected

Verification:

- `construct_rules.sql` becomes behavior-first, not only catalog/API existence checks.

### CWB-FIX-09 - SHACL rule bridge foundation

Once the derivation kernel is stable, route supported `sh:TripleRule` and future `sh:SPARQLRule` execution through the same provenance and retraction path.

Requirements:

- `sh:SPARQLRule` remains `planned` in feature status unless executed end to end.
- Unsupported rule forms emit documented PT codes and degrade visibly.

### CWB-FIX-10 - Canonical pipeline introspection API

Add `pg_ripple.construct_pipeline_status()` returning dependency graph, rule order, last run state, pending deltas, derived triple counts, and failed rules.

This is the API foundation for a future UI; no web UI is required in v0.65.0.

## Conflicts and risks

- Incremental maintenance touches storage, SPARQL translation, DRed-style retraction, and provenance. It is high regression risk and must land behind behavior tests.
- Statement-level triggers can add write latency. Bulk-load batching is required to keep ingestion acceptable.
- A pure trigger design may not cover all write paths if some paths bypass normal VP inserts. Prefer a storage-layer dispatch point where possible.
- Schema changes may be needed for run ids, rule status, or provenance metadata. Add migration scripts even if some changes are internal.

## Exit criteria

- Source insert/delete updates target graphs without manual refresh.
- Promoted and post-merge target predicates retract correctly.
- Provenance records only exact rule-produced triples.
- Invalid construct modes fail fast.
- Full CWB test matrix passes.
- `feature_status()` can move `construct_writeback` from `manual_refresh` to `implemented` only after all required tests pass.
