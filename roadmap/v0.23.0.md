# v0.23.0 — SHACL Core Completion & SPARQL Diagnostics

**Effort estimate: 6–8 person-weeks**

**Goal:** This release closes the remaining gaps in SHACL Core constraint coverage — adding eight constraint types that were silently ignored before — and introduces a diagnostic function that lets developers see exactly what SQL their SPARQL queries generate. The Datalog engine also receives three correctness fixes for edge cases that previously produced silent wrong results.

---

## Eight New SHACL Constraints

**Common data quality rules that were previously ignored now work correctly.**

Until this release, writing a SHACL shape with `sh:hasValue`, `sh:closed`, `sh:nodeKind`, `sh:languageIn`, `sh:uniqueLang`, `sh:lessThan`, or `sh:greaterThan` would not actually enforce those constraints — they were parsed and stored but never evaluated. For anyone relying on these constraints to protect data quality, the implication was serious: invalid data passed through without any warning. This release implements all eight, along with proper support for property path expressions in shapes (`sh:inversePath`, `sh:alternativePath`, sequential paths, and recursive paths).

---

## `explain_sparql()` Function

**See exactly what SQL your SPARQL query generates, with a full execution plan.**

The `pg_ripple.explain_sparql(query TEXT, format TEXT)` function is the most-requested developer tool since `sparql_explain()` was added in v0.3.0. You give it a SPARQL query string and it returns either: the raw SQL that the translator generated (format `'sql'`); a full `EXPLAIN ANALYZE` plan in text or JSON format (format `'text'` or `'json'`); or the internal algebra tree showing how the query was parsed and optimised (format `'sparql_algebra'`). This makes diagnosing slow queries far more tractable — you can see exactly which join order was chosen, which indexes were used, and where the time was spent.

## SHACL Query-Optimisation Hints Verified

**Shape constraints now influence the SPARQL query planner.**

When a SHACL shape declares `sh:maxCount 1` for a predicate (meaning each subject has at most one value), the SPARQL query engine can skip the `DISTINCT` operation that would otherwise be needed on that predicate's join — because duplicates are structurally impossible. When `sh:minCount 1` guarantees a value always exists, OPTIONAL joins can be upgraded to INNER JOINs. These hints were described in earlier documentation but not actually applied; this release verifies they are wired in and adds a test that confirms the query plan changes.

---

## Datalog Engine Correctness Fixes

**Three edge cases in the rule engine now produce errors instead of silent wrong results.**

Division by zero in a Datalog arithmetic expression now emits a `NOTICE` message naming the rule that caused it rather than crashing or silently propagating NULL. Rules with unbound variables (variables that appear in the rule head but not in any positive body literal) now raise a compile-time `SYNTAX_ERROR` rather than generating SQL that silently matches nothing. And cyclic negation — a rule program where rule A negates B, B negates C, and C negates A — is now correctly detected using Tarjan's algorithm and rejected with a clear "unstratifiable negation cycle" error naming all three rules.
