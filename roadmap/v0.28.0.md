# v0.28.0 — Advanced Hybrid Search & RAG Pipeline

**Effort estimate: 5–8 person-weeks**

**Goal:** The vector embedding foundation from v0.27.0 enables a complete hybrid search and RAG (Retrieval-Augmented Generation) pipeline. This release adds the fusion algorithm that combines SPARQL and vector results, a background worker that keeps embeddings up to date as new data arrives, and an end-to-end `rag_retrieve()` function that takes a natural language question and returns structured context ready for an LLM.

---

## Reciprocal Rank Fusion

**Combine graph query results and similarity results into a single ranked list.**

SPARQL queries and vector similarity searches each produce a ranked list of relevant entities, but they rank by different criteria — SPARQL ranks by structural graph constraints, vector search ranks by semantic similarity. Neither list alone is optimal. Reciprocal Rank Fusion (RRF) is the standard algorithm for merging ranked lists: each entity's final score is the sum of its reciprocal ranks across both lists (formula: $\sum_{r} \frac{1}{60 + r(d)}$). The `pg_ripple.hybrid_search()` function executes both retrievals in parallel and fuses the results, with a configurable blend parameter that controls how much weight to give to each list.

---

## Incremental Embedding Background Worker

**New entities are automatically embedded as they are added to the graph.**

When `pg_ripple.auto_embed = true`, a trigger on the dictionary table adds new entity IDs to an embedding queue. The background worker drains this queue in configurable batches, calling the embedding API and storing the results. Applications that add entities continuously — IoT device registrations, document ingestion, event processing — get embeddings without any manual batch-embed calls.

---

## Graph-Contextualised Embeddings

**Embeddings can encode an entity's neighbourhood, not just its label.**

The `pg_ripple.contextualize_entity()` function generates a text description of an entity that includes its types, its label, and the labels of its immediate neighbours in the knowledge graph. Embedding this richer text — rather than just the entity's label — produces a vector that captures relational context: two entities that have similar roles in the graph end up close in vector space, even if their names differ. This is controlled by the `pg_ripple.use_graph_context` GUC.

---

## End-to-End RAG Retrieval

**A single function call takes a question and returns structured LLM context.**

The `pg_ripple.rag_retrieve(question, sparql_filter, k)` function orchestrates a complete RAG retrieval pipeline: embed the question, find the k nearest entities by vector similarity, apply the optional SPARQL filter to refine the candidates, gather graph-contextualised descriptions for each surviving entity, and return them as JSONB rows ready for assembly into an LLM system prompt. An optional `output_format := 'jsonld'` parameter formats the context as JSON-LD, which some LLMs handle especially well.

---

## REST Endpoint for RAG

**The HTTP companion service exposes RAG retrieval to non-SQL clients.**

A new `POST /rag` endpoint on `pg_ripple_http` accepts a JSON body with the question and optional parameters, calls `pg_ripple.rag_retrieve()`, and returns the assembled context as a plain-text string ready to paste into an LLM system prompt. This bridges pg_ripple to LangChain, LlamaIndex, and any other framework that calls RAG backends via HTTP.
