# v0.31.0 — Entity Resolution & Demand Transformation

**Effort estimate: 5–7 person-weeks**

**Goal:** Real knowledge graphs contain duplicate references to the same real-world entity — "Alice Smith", "A. Smith", "Prof. Smith" all referring to the same person. This release adds `owl:sameAs` canonicalization so that all facts about any alias are automatically visible through any other alias. It also generalises magic sets into a more powerful demand transformation that works across entire multi-predicate rule programs.

---

## `owl:sameAs` Entity Canonicalization

**When two names are declared the same, facts about one automatically apply to the other.**

The RDF way of saying "these two identifiers refer to the same entity" is the `owl:sameAs` predicate. Before this release, asserting `ex:Alice owl:sameAs ex:A.Smith` had no automatic effect: queries asking about `ex:A.Smith` would not find facts stored under `ex:Alice`. This release adds a pre-pass that computes equivalence classes over all `owl:sameAs` assertions and rewrites triple patterns to use the canonical representative (the entity with the lowest dictionary ID). All facts about any alias become visible through any other alias, transparently.

---

## SPARQL Rewriting for Non-Canonical Entities

**SPARQL queries that reference an alias automatically see the canonical entity's facts.**

The canonicalization extends to the SPARQL query path: when a query references an entity that is known to be an alias (via `owl:sameAs`), the query is automatically rewritten to use the canonical ID. This means application code does not need to know which identifier is canonical — it can use any IRI that refers to the entity and get the complete picture.

---

## Demand Transformation

**A generalisation of magic sets that works across entire programs, not just single predicates.**

Magic sets (v0.29.0) work predicate by predicate. Demand transformation generalises the idea to entire rule programs: given a set of goal patterns for multiple predicates simultaneously, it computes the minimal subset of derivations needed to satisfy all goals at once and applies that constraint to every rule in the program. For SPARQL queries that reference several derived predicates, demand transformation can reduce the combined derivation work by computing a joint "demand" set rather than applying magic sets independently per predicate. The `pg_ripple.infer_demand(rule_set, demands)` function exposes this capability directly.

---

## SPARQL On-Demand with Demand Transformation

**SPARQL queries that trigger multiple derived predicates run with smaller, faster CTEs.**

When a SPARQL query uses on-demand Datalog mode and references more than one derived predicate, the demand transformation computes the minimal joint demand and applies it to all the inline CTEs generated for that query. The result is smaller CTEs, fewer rows processed in joins, and lower total query cost. For analytical queries that pull together several inferred relationship types, this can reduce query time by 50% or more compared to per-predicate magic sets applied independently.
