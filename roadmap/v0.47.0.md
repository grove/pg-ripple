# v0.47.0 — SHACL Truthfulness, Dead-Code Activation & Architecture Refactor

**Effort estimate: 8–11 person-weeks**

**Goal:** Four SHACL constraints that were being parsed but silently ignored are now correctly enforced. Two Rust functions that were implemented but never connected to any caller are wired in. The SQL generation module — which had grown beyond maintainable size — is split into focused units. Five new fuzz targets cover parsers not previously fuzzed. CI hygiene (dependency audits, privilege checks) is tightened.

---

## Four Previously-Silent SHACL Constraints Now Enforced

**`sh:closed`, `sh:uniqueLang`, `sh:pattern`, and `sh:lessThanOrEquals` now actually run.**

These four constraints were syntactically parsed from SHACL shape files and silently discarded during validation — a shape declaring `sh:closed true` would always appear to pass even if the data contained disallowed properties. With this release all four are enforced: `sh:closed` rejects any property not listed in `sh:property` or `sh:ignoredProperties`; `sh:uniqueLang` rejects a focus node that has two string literals with the same language tag for the same property; `sh:pattern` validates string values against a regex; `sh:lessThanOrEquals` checks that one property's values are numerically or lexicographically ≤ another's.

---

## SID Range Pre-allocation Wired In

**The SID pre-allocation function was implemented but never called — now it is.**

The `preallocate_sid_ranges()` function, introduced to allow parallel Datalog workers to claim blocks of statement identifiers without sequence contention, was written but never connected to the parallel inference code path. This release wires it into the parallel stratum evaluator (v0.35.0): each worker pre-allocates its SID block at dispatch time and uses the pre-allocated range without further sequence calls. Under high-parallelism inference, sequence contention was the second biggest bottleneck after the data scan itself.

---

## SQL Generation Module Split

**The 3,600-line SQL generation file is split into focused per-algebra-node units.**

The `sqlgen.rs` file that translates SPARQL algebra into SQL had grown to 3,600 lines across dozens of mixed responsibilities. It is now split into separate modules: `bgp.rs` handles basic graph pattern compilation, `join.rs` handles inner joins and hash joins, `left_join.rs` handles OPTIONAL, `path.rs` handles property paths, `aggregate.rs` handles GROUP BY and HAVING, and `update.rs` handles SPARQL Update operations. Each module's file stays under 800 lines. Cross-module function calls use a shared `Context` struct to pass the predicate catalog, GUC settings, and query-level statistics.

---

## Five New Fuzz Targets

**The SPARQL, Turtle, Datalog, SHACL, and dictionary hash parsers are now fuzz-tested.**

The federation result decoder was fuzz-tested in v0.46.0. This release adds five more targets: the SPARQL parser (which turns SPARQL query text into an algebra tree), the Turtle parser (which loads RDF data), the Datalog rule parser, the SHACL shape parser, and the XXH3 dictionary hash function (to catch any collision or hash-stability bugs). Together these cover every external-input parsing surface in pg_ripple.

---

## Promotion-Race Stress Test

**50 concurrent transactions simultaneously cross the rare-predicate promotion threshold.**

The rare-predicate promotion (moving a predicate from the `vp_rare` consolidated table to its own dedicated VP table when its triple count crosses the threshold) was made idempotent and serialised in v0.37.0. This release adds a stress test that verifies the fix: 50 goroutines all insert the 1,001st triple for the same predicate simultaneously, then verify that exactly one new VP table exists and all triples are present.
