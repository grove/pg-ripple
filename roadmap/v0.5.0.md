# v0.5.0 — SPARQL Advanced (Query Completeness)

**Effort estimate: 4–6 person-weeks**

**Goal:** The core SPARQL engine from v0.3.0 handles everyday queries, but the full SPARQL 1.1 standard includes significantly more expressive power. This release adds the features that unlock analytical queries: graph traversal along paths of arbitrary length, combining multiple query branches, aggregation (counting, summing, averaging), subqueries, and more. It is a pure query-engine improvement with no storage changes.

---

## Property Paths

**Traverse a chain of relationships of any length in a single query.**

Property paths let you write "find all ancestors of Alice" or "find everything reachable from this concept within three steps" without knowing in advance how many hops are involved. The `*` operator (zero or more), `+` (one or more), and `/` (sequential chain) operators are all implemented. Under the hood, each path compiles to a `WITH RECURSIVE … CYCLE` SQL expression using PostgreSQL 18's native `CYCLE` clause, which provides hash-based cycle detection so that circular graphs do not cause infinite loops.

---

## UNION and MINUS

**Combine results from multiple patterns or exclude a pattern's matches.**

UNION lets you merge results from two or more independent query patterns into one result set — useful when the data you are looking for might be stored under different predicates in different parts of the graph. MINUS does the opposite: it removes from one result set any row that also appears in another pattern. Both translate directly to SQL UNION ALL and EXCEPT operations on the generated subqueries.

---

## Aggregates

**Count, sum, average, find the minimum or maximum across a result set.**

The GROUP BY, COUNT, SUM, AVG, MIN, and MAX aggregations from SPARQL 1.1 are all implemented. GROUP_CONCAT (joining multiple values into a delimited string) is also supported. These translate to their direct SQL equivalents, letting PostgreSQL's own aggregate engine handle the computation. HAVING filters on aggregate results are supported too.

---

## Subqueries, BIND, and VALUES

**Build complex queries from simpler parts.**

Subqueries let you use the result of one SELECT as input to an outer SELECT, enabling multi-step filtering and transformation. BIND assigns the result of an expression to a new variable for use later in the query. VALUES provides a literal inline table of values, useful for "find facts about any of these specific entities" without constructing a complex FILTER. All three translate cleanly to SQL subqueries and CTEs.
