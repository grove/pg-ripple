# v0.55.0 — Security Hardening & v1.0.0 Unblocking

**Effort estimate: 6–8 person-weeks**

**Goal:** Close every finding identified in the v6 assessment that blocks a credible v1.0.0 tag — federation SSRF, the HTAP merge race window, the error-catalog drift, and the absence of mid-merge crash-recovery and concurrent-write stress tests. Also delivers Quick-Win ecosystem features (VoID, Service Description, read-replica routing, OpenAPI spec) and security improvements (named-graph RLS, GDPR erasure, `copy_rdf_from` path policy). The Jupyter kernel, LangChain integration, Kafka CDC sink, and materialized SPARQL views are deferred to [v0.59.0](v0.59.0.md).

## Critical: Federation SSRF Allowlist

**Prevent `SERVICE` queries from reaching cloud metadata and internal services.**
Any SPARQL query containing a `SERVICE <http://169.254.169.254/...>` clause currently makes the server issue a network request to that address — a well-known cloud-instance metadata endpoint. v0.55.0 adds a `pg_ripple.federation_endpoint_policy` GUC (`default-deny` / `allowlist` / `open`) that by default blocks requests to RFC-1918 ranges, link-local addresses, loopback, and `file://` URIs unless the operator explicitly allowlists them.

## High: HTAP Merge Race Elimination

**Reads should never fail mid-merge.**
The merge worker currently drops the VP view then creates a new one, leaving a window where any SPARQL query touching that predicate will fail with a "relation does not exist" error. v0.55.0 replaces the `CREATE OR REPLACE VIEW` approach with a lock-protected table rename that eliminates the existence gap entirely. A new `tests/crash_recovery/merge_kill.sh` stress test kills the merge worker mid-cycle and verifies the database is consistent on restart.

## High: Error-Catalog Completion

**Every PT-error code must be documented.**
Seventeen PT-error codes used in source — including `PT440` (the v0.51.0 SPARQL DoS code), `PT480`, `PT481`, and `PT800` — are absent from `docs/src/reference/error-catalog.md`. A new CI gate runs `scripts/check_pt_codes.sh` on every PR to prevent future drift.

## Code Quality

**Close the dead-code and safety-comment backlog.**
The `src/shacl/mod.rs` god-module (2,761 lines) is split along parser / validator / hints / AF-rules / SPI boundaries. The `src/datalog/mod.rs` file is genuinely trimmed after the v0.51.0 delegation modules proved hollow. All `unsafe` blocks gain `// SAFETY:` comments. The two replication-worker `unwrap()` calls are converted to explicit error handling. The parallel-strata `execute_with_savepoint()` helper is wired into the coordinator.

## Storage & Performance

**Make tombstone GC, concurrent promotion, and baseline CI gates real.**
Tombstone rows are now pruned atomically at the end of every successful merge cycle. A `pg_advisory_xact_lock` call serializes concurrent rare-predicate promotions. Both the merge-throughput and vector-index baselines gain CI comparison jobs that warn on >15 % regression.

## Standards & Compliance Quick Wins

**VoID, SPARQL Service Description, named-graph RLS, and GDPR erasure.**
- `pg_ripple_http /void` auto-publishes a VoID dataset description.
- `pg_ripple_http /service` returns a Turtle SPARQL Service Description document.
- Row-level security policies on named graphs let multiple tenants share a single pg_ripple instance.
- `pg_ripple.erase_subject(iri)` removes all triples about a subject across VP tables, the dictionary, and the embeddings table.

## Ecosystem Quick Wins

- **Read-replica routing**: route `SELECT`/`CONSTRUCT`/`ASK` to a replica via `pg_ripple.read_replica_dsn`.
- **OpenAPI spec**: `utoipa`-annotated handlers publish `docs/src/reference/openapi.yaml`.

> **Deferred to v0.59.0**: Jupyter SPARQL kernel, LangChain / LlamaIndex tool, Kafka CDC sink, materialized SPARQL views.
