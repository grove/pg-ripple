## v1.1.0 — Ecosystem Integration: Jupyter, LangChain, Kafka, Materialized Views & dbt Adapter

**Theme**: Deliver the five ecosystem integration items deferred from v0.55.0 and v0.56.0.

> **In plain language:** v0.55.0 focused on security hardening and v0.56.0 on standards completeness — both were already large. These five items are cohesive ecosystem features that form a natural release of their own. Together they make pg_ripple usable from notebooks, LLM frameworks, event-streaming pipelines, analytics dashboards, and data-transformation pipelines without any custom integration code.
>
> **Effort estimate: 4–6 person-weeks**

### Deliverables

#### Feature L-2.3 — Materialized SPARQL Views (Medium)

- [ ] Implement `pg_ripple.create_sparql_view(name TEXT, query TEXT, refresh_on_merge BOOL DEFAULT TRUE)` that executes the SELECT query and materializes results into `_pg_ripple.sparql_mv_{name}`; registers the view in `_pg_ripple.sparql_mv_catalog`
- [ ] `pg_ripple.refresh_sparql_view(name TEXT)` for manual refresh — re-runs the stored query and replaces the materialized table atomically
- [ ] `pg_ripple.drop_sparql_view(name TEXT)` — drops the materialized table and removes the catalog entry
- [ ] Merge worker integration: after a successful merge cycle, call `refresh_sparql_view()` for each catalog entry where `refresh_on_merge = TRUE`
- [ ] pg_regress test: `tests/pg_regress/sql/sparql_mv.sql` — create a view; insert triples; trigger merge; verify refreshed results; drop view

#### Feature L-6.2 — Kafka CDC Sink (Medium)

- [ ] Add `rdkafka` Rust crate dependency (feature-gated behind `kafka` Cargo feature)
- [ ] New GUC `pg_ripple.kafka_bootstrap_servers` (text, default empty)
- [ ] New GUC `pg_ripple.kafka_topic_prefix` (text, default `pg_ripple`)
- [ ] When `kafka_bootstrap_servers` is non-empty, the merge worker publishes each event from `pg_ripple_cdc_lifecycle` to `{prefix}.rdf.changes` as a JSON payload containing an N-Triples batch and event metadata (`event_type`, `graph_iri`, `timestamp`, `statement_count`)
- [ ] Backpressure: if the Kafka producer queue is full, emit PT530 warning and continue without blocking the merge cycle
- [ ] `examples/kafka_cdc_sink.sql`: configuration example, consumer schema, and Avro schema stub
- [ ] `docs/src/operations/kafka-cdc.md`: setup guide for Confluent Cloud and self-hosted Kafka

#### Feature L-6.4 — Jupyter SPARQL Kernel (Quick Win)

- [ ] Python package `pg-ripple-kernel` in a new top-level `python/pg_ripple_kernel/` directory
- [ ] Implements Jupyter `KernelApp` spec: `%%sparql` magic routes cells to a configured `pg_ripple_http` endpoint via the `/sparql` HTTP endpoint
- [ ] `SELECT` results rendered as `pandas.DataFrame`; `CONSTRUCT`/`DESCRIBE` rendered as Turtle; `ASK` rendered as a boolean badge
- [ ] Connection configured via `PGRIPPLE_HTTP_URL` environment variable or `%config PgRippleKernel.url = "http://localhost:7474"` magic
- [ ] Published to PyPI as `pg-ripple-kernel`; minimum Python 3.10, Jupyter 4+
- [ ] `docs/src/user-guide/jupyter.md`: installation and usage guide with screenshot stubs

#### Feature L-6.5 — LangChain / LlamaIndex Tool Integration (Quick Win)

- [ ] Python package `pg-ripple-tools` in `python/pg_ripple_tools/`
- [ ] `PgRippleSPARQLTool(BaseTool)`: LangChain tool that accepts a SPARQL query string (or free-text passed through `sparql_from_nl()` if `nl_mode=True`), calls `/sparql`, returns JSON result string
- [ ] `PgRippleQueryEngine(BaseQueryEngine)`: LlamaIndex engine wrapping the same HTTP endpoint
- [ ] Both classes accept `endpoint_url`, `timeout_seconds`, and `max_results` constructor arguments
- [ ] Published to PyPI as `pg-ripple-tools`; requires `langchain-core >= 0.2` or `llama-index-core >= 0.10`
- [ ] `examples/llm_workflow.sql` updated with cross-reference to the Python package
- [ ] `docs/src/user-guide/llm-tools.md`: integration guide with LangChain agent and LlamaIndex RAG examples

#### Feature L-6.3 — dbt Adapter (Medium)

- [ ] Python package `dbt-pg-ripple` (published to PyPI under `grove/` org) with:
  - `{{ sparql(query) }}` macro executing via `pg_ripple_http /sparql` and returning as a dbt relation
  - `{{ sparql_construct(query) }}` macro returning Turtle
  - Adapter config: `pg_ripple_url`, `timeout_seconds`
- [ ] `examples/dbt_adapter/` directory with an annotated `dbt_project.yml` and sample model
- [ ] `docs/src/user-guide/dbt-adapter.md`: setup guide covering profiles, macros, and incremental models

### Migration Script

`sql/pg_ripple--0.58.0--1.1.0.sql`:
- Creates `_pg_ripple.sparql_mv_catalog (name TEXT PRIMARY KEY, query TEXT, refresh_on_merge BOOL NOT NULL DEFAULT TRUE, last_refreshed TIMESTAMPTZ)` for materialized SPARQL views
- No other schema changes (Kafka sink and Python packages are purely additive external artifacts)

### Exit Criteria

- `pg_ripple.create_sparql_view('test', 'SELECT * WHERE { ?s ?p ?o }')` succeeds and populates `_pg_ripple.sparql_mv_test`.
- Merge cycle with `refresh_on_merge = true` updates the materialized table without errors.
- `pg-ripple-kernel` installs cleanly in a fresh Jupyter environment and renders a `%%sparql SELECT` cell as a DataFrame.
- `pg-ripple-tools` `PgRippleSPARQLTool` passes a LangChain `tool.run("SELECT * WHERE {?s ?p ?o} LIMIT 1")` end-to-end test.
- Kafka integration compiles cleanly with the `kafka` feature flag enabled.
- `dbt-pg-ripple` `{{ sparql('SELECT * WHERE {?s ?p ?o} LIMIT 1') }}` runs successfully in a dbt project against a live `pg_ripple_http` instance.
- All pg_regress tests continue to pass.
