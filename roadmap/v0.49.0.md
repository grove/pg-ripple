# v0.49.0 — AI & LLM Integration

**Effort estimate: 5–7 person-weeks**

**Goal:** Two AI-powered capabilities that make pg_ripple accessible to non-technical users and automate common knowledge graph management tasks. Natural language to SPARQL converts a plain English question into an executable SPARQL query using any OpenAI-compatible LLM. Embedding-based entity matching surfaces candidate `owl:sameAs` relationships — pairs of entities that likely refer to the same real-world thing — by finding semantically similar entity descriptions in the vector index.

---

## Natural Language to SPARQL

**Ask a question in plain English and get back a runnable SPARQL query.**

`pg_ripple.sparql_from_nl('Which medications are contraindicated for patients with heart disease?')` calls a configured LLM with a structured prompt that includes the VoID description of the knowledge graph (the list of predicates, their counts, and sample values) and, optionally, the relevant SHACL shapes. The LLM returns a SPARQL query using function calling, which pg_ripple validates syntactically before returning it. If the LLM returns an invalid query, pg_ripple retries with an error description up to three times. The result is a text string that can be passed directly to `pg_ripple.sparql()`.

---

## Configuration for LLM Connection

**The LLM endpoint, model name, and API key are configured as PostgreSQL GUC parameters.**

`pg_ripple.llm_endpoint` accepts any OpenAI-compatible API base URL — the official OpenAI API, Azure OpenAI, Ollama, and others. `pg_ripple.llm_model` selects the model (for example, `gpt-4o` or `llama3`). The API key is never stored in a GUC (which would be visible in `pg_settings`); instead, `pg_ripple.llm_api_key_env` names an environment variable from which the key is read at connection time. `pg_ripple.llm_include_shapes` controls whether SHACL shapes are sent alongside VoID statistics in the prompt.

---

## Few-Shot Examples for Accuracy

**Operator-provided example question/query pairs are included in LLM prompts.**

The `_pg_ripple.llm_examples` table stores (question, sparql) pairs that the operator curates for their specific knowledge graph. When `sparql_from_nl()` is called, all rows from this table are included in the prompt as few-shot examples. This dramatically improves accuracy on domain-specific queries: an operator who adds 10–20 representative question/answer pairs can typically achieve >90% query correctness on similar questions. Examples are stored as ordinary table rows and can be inserted, updated, or deleted with standard SQL.

---

## Embedding-Based `owl:sameAs` Candidate Generation

**"These two entities probably refer to the same person" — discovered automatically.**

`pg_ripple.suggest_sameas(threshold)` uses the vector embeddings stored by `embed_entities()` (v0.27.0) to find pairs of entities whose embeddings are very similar — close in the high-dimensional vector space that represents their descriptions. It returns a table of candidate pairs with a similarity score. The operator reviews the candidates and calls `apply_sameas_candidates(min_score)` to assert `owl:sameAs` triples for all pairs above the threshold. Datalog inference (v0.31.0) then propagates the implications of these assertions through the knowledge graph.
