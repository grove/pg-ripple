# v0.46.0 — Property-Based Testing, Fuzz Hardening & OWL 2 RL Conformance

**Effort estimate: 5–7 person-weeks**

**Goal:** Automated randomised testing catches bugs that human-written tests miss. This release integrates two complementary forms of randomised testing — property-based tests that verify algebraic invariants, and fuzz tests that try to crash the input parsers — plus the W3C OWL 2 RL conformance suite, a TopN query optimisation, and CI gates that prevent known-bad Rust crate versions from entering the codebase.

---

## Property-Based Testing with `proptest`

**Randomly generated inputs verify that round-trips and invariants always hold.**

Property-based testing (a technique popularised by QuickCheck and used in production by Cloudflare, AWS, and others) generates hundreds of random inputs for each tested property and verifies that an invariant holds for all of them. Three critical invariants are tested: (1) SPARQL algebra round-trip — any parsed SPARQL query re-serialised and re-parsed should produce the same algebra; (2) dictionary encode/decode — any IRI, blank node, or literal encoded to an integer and decoded back should equal the original; (3) JSON-LD framing round-trip — a framed JSON-LD document re-loaded and re-framed should be structurally identical. Property tests run in under 30 seconds and are included in the standard `cargo pgrx test` suite.

## Fuzz Testing of the Federation Result Decoder

**Malformed responses from remote SPARQL endpoints can no longer crash the local server.**

Federation endpoints are under no obligation to return well-formed SPARQL results XML or JSON. A remote endpoint could time out mid-response, return a truncated JSON object, or — in an adversarial scenario — return crafted input intended to trigger a parser bug. A `cargo-fuzz` target runs the federation result decoder against millions of randomly mutated byte sequences, finding any input that causes a panic or crash. The fuzz corpus includes all previous real-world test suite fixtures plus the mutations discovered during fuzzing.

## W3C OWL 2 RL Conformance Suite

**The official OWL 2 RL test suite measures reasoning correctness against W3C's own test cases.**

The W3C publishes an OWL 2 RL conformance test suite with test cases for all OWL 2 RL entailment patterns. This release adds the suite to CI as an informational check — failures are reported but do not block merge — until the pass rate reaches ≥95%, at which point it becomes required. The target at release is ≥80% pass with all LUBM-relevant rules covered.

## TopN Push-Down

**`ORDER BY … LIMIT N` queries no longer materialise all results before sorting.**

Previously, a SPARQL query with `ORDER BY ?score LIMIT 10` was executed by materialising all results, sorting them, and then discarding all but the top 10. For queries with millions of results, this was extremely wasteful. TopN push-down rewrites the generated SQL to use `ORDER BY … LIMIT 10` at each stage of the plan, allowing PostgreSQL's planner to use a TopN sort algorithm that maintains only the 10 best rows at any time. For LIMIT values ≤ 1,000 over un-aggregated queries, the result is typically a 10×–100× improvement in both time and memory.

## Dependency Security Gates

**CI now blocks on known-vulnerable and duplicate Rust crate versions.**

`cargo audit` checks all dependencies against the RustSec advisory database; any advisory rated HIGH or CRITICAL blocks CI. `cargo deny` enforces a license allowlist (MIT, Apache-2.0, ISC, and BSD variants only) and reports duplicate crate versions that would bloat the binary. A new `rustdoc` lint gate ensures all public SQL-facing functions have documentation comments. These three checks run on every push to main and on all pull requests.
