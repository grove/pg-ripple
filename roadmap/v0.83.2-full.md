## Background

v0.83.2 delivers the 22 medium-priority findings from Assessment 13 that address
correctness gaps, performance micro-optimisations, and code-quality improvements.
v0.83.1 must land first because P13-01 (plan cache double-parse) changes the
`cache_key` signature that several correctness fixes depend on.

## Correctness Findings

### C13-02 — `strict_dictionary` Not Respected in `batch_decode`

`batch_decode` in `src/sparql/decode.rs:97–107` silently emits an empty string when
a dictionary ID has no corresponding entry.  When `pg_ripple.strict_dictionary = on`
the correct behaviour is to raise an error, matching the scalar path.

**Fix**: Check `pg_ripple.strict_dictionary` at the top of `batch_decode`; if enabled,
iterate the result rows first and raise `pgrx::error!` for any NULL value returned by
the `WHERE id = ANY($1)` lookup before assembling the output slice.

---

### C13-03 — RDF-star Blank Node in Quoted Triple

Blank nodes inside quoted triples (`<< _:b0 :p :o >>`) are serialised differently
by different codepaths: the Turtle exporter uses `_:b0`, the N-Triples exporter uses
`_:genid0`.  The round-trip test does not cover this case.

**Fix**: Add a round-trip test; normalise serialisation to `_:genid<hash>` in both
paths.

---

### C13-04 — `execute_drop` / `execute_clear` Mutation Journal Precondition

`execute_drop()` (line 967) and `execute_clear()` (line 995) in
`src/sparql/execute.rs` rely on the outer transaction to have opened the mutation
journal before they write.  If called outside a transaction (e.g. from a background
worker) the journal write is silently lost.

**Fix**: Add `debug_assert!(mutation_journal_is_open())` at the top of each function
and document the calling convention in a `// INVARIANT:` comment.

---

### C13-05 — Plan Cache Key INFERENCE_MODE Normalisation

The plan cache key includes the raw `pg_ripple.inference_mode` GUC string.
If two sessions set it as `"RDFS"` vs `" rdfs "` they get different cache entries
for semantically identical plans.

**Fix**: Trim and lowercase the GUC value when constructing the cache key.

---

### C13-06 — `GRAPH ?g` Default-Graph Semantics

A `GRAPH ?g { … }` pattern should bind `?g` only to named graphs (graph IDs > 0),
never to the default graph (ID 0).  The current implementation does not filter out
ID 0 from VP table scans.

**Fix**: Add `AND g != 0` to all VP table scans generated for `GRAPH ?g` patterns.
Add a regression test with explicit default-graph triples.

---

### C13-07 — `batch_decode` Warning Guard

The existing `batch_decode` warning at `decode.rs:107` fires when `id <= 0`, but
the only invalid sentinel value is exactly `0` (negative IDs are legitimate for
reverse-encoded hash collisions in some code paths).

**Fix**: Change the guard to `id == 0`.

---

### C13-08 — `encode_token` Typed Literals in `magic.rs`

`encode_token()` in `src/datalog/magic.rs` encodes typed literals by concatenating
the lexical form and datatype IRI without separating them by a canonical delimiter.
This can produce collisions between `"42"^^xsd:integer` and `"42^^xsd:integer"` (a
plain literal containing the caret sequence).

**Fix**: Use the standard N-Triples encoding: `"<lexical>"^^<datatype>` as the
canonical string before hashing.

---

### C13-09 — `parse_nt_triple` IRI Bounds

`parse_nt_triple()` does not validate that an IRI terminated with `>` is preceded
by `<` at a reasonable distance.  Malformed input can produce oversized or
zero-length IRI strings without error.

**Fix**: Add `MAX_IRI_LEN = 8192` guard and return `Err` when the terminating `>`
would require an IRI longer than the limit.

---

### C13-10 — `xsd:dateTime` Sub-Millisecond Precision

`format_inline()` for `xsd:dateTime` truncates to milliseconds.  The specification
allows arbitrary decimal fractions.  This is intentional but undocumented.

**Fix**: Add a note to `docs/src/reference/datatypes.md` stating the current
precision limit and how to work around it.

---

### C13-11 — `describe_cbd` Recursion Depth

`describe_cbd()` follows `owl:sameAs` and `rdf:type` links recursively with no depth
limit.  A graph with long equivalence chains can produce a very large result or
trigger a stack overflow.

**Fix**: Add `pg_ripple.describe_depth_limit` GUC (integer, default 64, minimum 1,
maximum 1024).  `describe_cbd` increments a counter on each recursive call and
raises `pgrx::error!` when the limit is reached.

## Performance Findings

### P13-02 — Encode Batch API

Dictionary encoding currently calls `SPI_execute_with_args` once per triple in the
hot insert path.  A batch variant sends all new terms in a single
`INSERT … VALUES ($1, $2), … ON CONFLICT DO NOTHING RETURNING` round-trip.

**Fix**: Introduce `encode_batch(terms: &[RdfTerm]) -> Vec<i64>` and use it in
`insert_triple`, `bulk_load`, and `construct_delta`.

---

### P13-03 — Merge-Worker Heartbeat Log Throttle

The HTAP merge background worker emits a log line on every iteration of its main
loop, including no-op iterations when no delta triples exist.  Under low write
workloads this floods the PostgreSQL log.

**Fix**: Track `last_log_time`; emit a heartbeat log at most once per 60 seconds.

---

### P13-04 — `SET LOCAL` Batching

Several code paths issue multiple `SET LOCAL` statements sequentially (e.g.
`SET LOCAL search_path`, `SET LOCAL pg_ripple.inference_mode`).  Each is a separate
SPI call with its own round-trip overhead.

**Fix**: Collect `SET LOCAL` pairs and emit them as a single `DO $$ BEGIN … END $$`
block or as a comma-separated `SET LOCAL a = x, b = y` (PG18 syntax).

---

### P13-06 — Intra-Stratum Cycle Pre-Check

The SCC evaluator in `src/datalog/stratify.rs` runs fixpoint iteration even for
strata known to be acyclic.  A pre-check on the dependency graph can skip the
fixpoint loop entirely for the common case of linear rule chains.

**Fix**: Before entering the fixpoint loop, verify that the current stratum's
dependency graph is a DAG.  If so, evaluate each rule exactly once in topological
order.

---

### P13-08 — HOT-Path Prometheus Metrics

Plan-cache hit/miss ratios and dictionary-encode batch vs single-call counters are
not tracked in Prometheus.  Operators cannot observe cache effectiveness without
reading PostgreSQL logs.

**Fix**: Register four Prometheus counters via the existing `src/stats/metrics.rs`
infrastructure:
- `pg_ripple_plan_cache_hits_total`
- `pg_ripple_plan_cache_misses_total`
- `pg_ripple_dict_encode_batch_calls_total`
- `pg_ripple_dict_encode_single_calls_total`

## Code Quality Findings

### Q13-02 — `src/schema.rs` Module Split (1 939 lines)

Split into:
- `schema/tables.rs` — `CREATE TABLE` DDL helpers
- `schema/indices.rs` — `CREATE INDEX` DDL helpers
- `schema/functions.rs` — `CREATE FUNCTION` / `CREATE OR REPLACE FUNCTION` helpers
- `schema/triggers.rs` — `CREATE TRIGGER` helpers

`schema/mod.rs` re-exports `ensure_schema()` and other public entry points.

---

### Q13-03 — `src/sparql/federation.rs` Module Split (1 693 lines)

Split into:
- `federation/planner.rs` — federated SERVICE query planning
- `federation/executor.rs` — HTTP call, response decoding, error handling
- `federation/cache.rs` — endpoint capability cache
- `federation/cost.rs` — cost estimation for federated joins

---

### Q13-04 — CI Module-Size Lint Gate

A shell script (`scripts/check_module_size.sh`) fails CI if any `.rs` file in `src/`
exceeds 1 800 lines.  This enforces the single-responsibility boundary established
by the Q13-01–Q13-03 splits and prevents future drift.

---

### Q13-05 — `#[allow(dead_code)]` Audit

Assessment 13 found 14 `#[allow(dead_code)]` annotations.  Several are on structs
or fields that are no longer referenced after previous module splits.

**Fix**: Remove dead_code annotations that are genuinely unused.  For those required
by upstream crates, add a `// UPSTREAM: <crate>` comment.

---

## Concurrency Findings

### CC13-01 — VP-Promotion Crash Recovery Regression Test

The VP-promotion path (`src/storage/promote.rs`) promotes rare-predicate triples to
a dedicated VP table atomically.  There is no test that simulates a crash mid-way
through promotion and verifies that catalogue consistency is restored on restart.

**Fix**: Add a `pg_test` that runs `promote_predicate()` in a transaction, panics
before commit, and then asserts that `_pg_ripple.predicates` and the VP table are
either both present or both absent after the next extension load.

---

### CC13-02 — Merge Fence Per-Predicate Lock

The HTAP merge background worker (`src/storage/merge.rs`) acquires a global advisory
lock (`pg_try_advisory_xact_lock(MERGE_LOCK_KEY)`) for the entire merge cycle.
This serialises merges across all predicates even when they are independent.

**Fix**: Replace the global lock with per-predicate advisory locks keyed by the VP
table OID.  Multiple predicates can then merge concurrently without contention.
