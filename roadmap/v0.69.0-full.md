## Release theme

v0.69.0 is a structural refactoring release. It does not add features or change external behavior. Its goal is to make the codebase maintainable for a v1.0.0 production freeze by ensuring each file has a single, nameable responsibility. The mutation journal from v0.67 and the shadow-table promotion from v0.68 have already established the contract boundaries; this release makes those boundaries explicit in the module hierarchy.

The risk of a refactoring release is behavioral regression. All work must be gated on a zero-diff pg_regress run.

## Deliverables

### ARCH-01 — Split `src/sparql/mod.rs`

Current state: approximately 1,500 lines mixing SPARQL text parsing, algebra optimization via `sparopt`, SQL generation, SPI execution, and result decoding.

Target structure:

```
src/sparql/
  mod.rs           — public re-exports and the pg_extern entry points only
  parse.rs         — text → spargebra algebra (calls spargebra::Query::parse)
  plan.rs          — spargebra algebra → sparopt algebra (calls sparopt optimizer)
  translate/       — (already exists; keep as is)
  execute.rs       — SPI execution, result fetching, error mapping
  cursor.rs        — (already exists; keep streaming cursor logic here)
  decode.rs        — SPI row → RdfTerm / QuerySolution decoding
```

Steps:
1. Move parsing helpers into `parse.rs`.
2. Move optimizer calls into `plan.rs`.
3. Move execution helpers into `execute.rs`.
4. Move row-decoding helpers into `decode.rs`.
5. Update `mod.rs` to re-export; ensure `pg_extern` function bodies stay in `mod.rs` or `execute.rs`.
6. Run `cargo test pg18` and `cargo pgrx regress pg18`; confirm zero diff.

### ARCH-02 — Split `pg_ripple_http/src/main.rs`

Current state: 2,206 lines mixing Axum routing, JWT/HMAC auth, SPI bridging, Arrow IPC encoding, and SSE streaming.

Target structure:

```
pg_ripple_http/src/
  main.rs          — Axum app construction, router wiring, signal handling
  routing.rs       — handler function signatures and route registration
  auth.rs          — JWT decode, HMAC ticket validation, session extraction
  spi_bridge.rs    — PostgreSQL SPI calls, query construction, row fetching
  arrow_encode.rs  — Arrow IPC record batch construction and streaming
  stream.rs        — SSE/chunked response helpers, back-pressure logic
```

Steps:
1. Extract auth helpers into `auth.rs`; update imports.
2. Extract SPI query helpers into `spi_bridge.rs`.
3. Extract Arrow batch builder into `arrow_encode.rs`.
4. Extract SSE helpers into `stream.rs`.
5. Move route handler functions into `routing.rs`; keep `main.rs` for app construction.
6. Confirm the HTTP service compiles and existing HTTP integration tests pass.

### ARCH-03 — Split `src/construct_rules.rs`

Current state: approximately 1,200+ lines combining rule catalog CRUD, incremental delta derivation, topological sort scheduler, and retraction logic.

Target structure:

```
src/construct_rules/
  mod.rs           — public API: on_graph_write, on_graph_delete, register_rule, list_rules
  catalog.rs       — rule CRUD against _pg_ripple.construct_rules catalog table
  delta.rs         — incremental delta derivation (insert path)
  retract.rs       — Delete-Rederive retraction (delete path)
  scheduler.rs     — topological sort, stratum ordering, cycle detection (replaces the panic)
```

Note: `scheduler.rs` is where the `panic!("in_degree missing for {rule}")` replaced by PANIC-01 in v0.67 lived. Ensure the new recoverable-error version is preserved in `scheduler.rs`.

Steps:
1. Create the `construct_rules/` directory and `mod.rs`.
2. Move catalog SQL helpers into `catalog.rs`.
3. Move delta derivation into `delta.rs`.
4. Move retraction into `retract.rs`.
5. Move topological sort into `scheduler.rs`.
6. Update all callsites in `dict_api.rs`, `storage/mod.rs`, and the mutation journal.
7. Run full test suite; confirm zero diff.

### ARCH-04 — Narrow `src/storage/mod.rs` public surface

After the mutation journal lands in v0.67, `storage/mod.rs` exposes both the journal-based API and the lower-level `insert_triple_by_ids` / `delete_triple_by_ids` / `batch_insert_encoded` functions that every write path must stop calling directly.

Steps:
1. Make `insert_triple_by_ids`, `delete_triple_by_ids`, and `batch_insert_encoded` `pub(crate)` and annotate each with `// Callers must go through storage::journal::record_mutation; direct calls are allowed only from the journal flush function`.
2. Rename or re-route any remaining direct callers (verify there are none after v0.67).
3. Move HTAP merge helpers into `src/storage/merge.rs` (already exists) and VP promotion into `src/storage/promote.rs` (new file).
4. Document the new public API in a `//!` module-level doc comment.

## Conflicts and risks

- A pure refactoring release is high-risk for subtle behavioral changes if module boundary assumptions are wrong. Use feature branches with intermediate pg_regress gates rather than one large PR.
- The `construct_rules` split interacts with the mutation journal (v0.67) and the MJOURNAL-02 rewiring. If v0.67 is not yet merged when v0.69 work begins, base the split on a branch that includes MJOURNAL-01 and MJOURNAL-02.
- The HTTP service split (ARCH-02) must preserve the existing Arrow streaming changes from FLIGHT-SEC-02 (v0.67) and any streaming additions from STREAM-01 (v0.68). Plan the split after those items land.

## Exit criteria

- `src/sparql/mod.rs` is < 300 lines; new sub-files each have a single named responsibility.
- `pg_ripple_http/src/main.rs` is < 300 lines; handler logic lives in `routing.rs`.
- `src/construct_rules.rs` is replaced by `src/construct_rules/` directory with four sub-files.
- `src/storage/mod.rs` direct-call functions are `pub(crate)` with mutation-journal-only caller comment.
- Full pg_regress suite passes with zero diff.
- AGENTS.md `src/` layout table is updated to reflect the new structure.
