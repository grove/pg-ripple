## Background

After v0.80.0 and v0.81.0 lock down correctness, security, and concurrency, v0.82.0 addresses the performance tier (P-01 through P-13), observability tier (O-01 through O-07), and the remaining security-hardening items (S-04 through S-15) from Assessment 12.

## CACHE-CAP-01 — Plan Cache Capacity GUC

### Current state

`src/sparql/plan_cache.rs` constructs the LRU with a hardcoded `LruCache::new(NonZeroUsize::new(1024).unwrap())`. Installations with few repeated queries waste memory; workloads with many distinct queries suffer evictions.

### What changes

Add `pg_ripple.plan_cache_capacity` GUC (integer, default 1024, min 64, max 65536). Initialise the LRU with this value. Document in configuration reference.

### Deliverables

| ID | Deliverable |

| CACHE-CAP-01a | `pg_ripple.plan_cache_capacity` GUC |
| CACHE-CAP-01b | LRU initialised from GUC value |
| CACHE-CAP-01c | GUC documented |

## DECODE-BIND-01 — Batch Decode Bind Parameter

### Current state

`src/dictionary/mod.rs` `batch_decode()` emits:
```sql
SELECT id, term FROM _pg_ripple.dictionary WHERE id IN (1234, 5678, …)
```
PostgreSQL generates a new query plan for each unique combination of IDs. Under high-concurrency workloads, this causes plan cache churn.

### What changes

Replace with:
```sql
SELECT id, term FROM _pg_ripple.dictionary WHERE id = ANY($1::bigint[])
```
using `Spi::run_with_args()` with a single `DatumArray<PgBuiltInOids::INT8OID>` parameter.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DECODE-BIND-01a | `ANY($1)` binding in `batch_decode()` |
| DECODE-BIND-01b | Benchmark: plan cache hit rate before/after |

## MERGE-PRED-01 — Merge Worker Predicate ID Cache

### Current state

The merge background worker (`src/storage/merge.rs`) queries `_pg_ripple.predicates` on every merge cycle to discover which predicates have pending delta rows. Under high-ingest workloads, this adds significant overhead to every merge wake-up.

### What changes

Maintain a `HashSet<PredicateId>` in the worker's local state. Refresh it from `_pg_ripple.predicates` only when a `NOTIFY pg_ripple_predicates_changed` message is received (already emitted by the promote path) or on a 60-second fallback timer.

### Deliverables

| ID | Deliverable |
|----|-------------|
| MERGE-PRED-01a | Predicate ID cache in merge worker state |
| MERGE-PRED-01b | Cache invalidated on NOTIFY |
| MERGE-PRED-01c | Benchmark: merge worker CPU before/after |

## FED-COST-01 — Federation Cost Model Statistics

### Current state

`src/sparql/federation.rs` federation cost estimates use static constants. A query planner cannot distinguish a fast local endpoint from a slow remote endpoint.

### What changes

Add table `_pg_ripple.federation_stats(endpoint_url TEXT PRIMARY KEY, p50_ms FLOAT, p95_ms FLOAT, row_estimate BIGINT, updated_at TIMESTAMPTZ)`. Update it in the federation result decoder after each successful call. The query planner reads `p50_ms` and `row_estimate` to cost federation scan nodes.

### Deliverables

| ID | Deliverable |

| FED-COST-01b | Stats updated after each federation call |
| FED-COST-01c | Planner reads stats for cost estimation |

## STATS-CACHE-01 — Predicate Stats Cache

### Current state

`pg_ripple.graph_stats()` scans every VP table on each call to compute triple counts. Under workloads with hundreds of predicates, this creates significant lock contention.

### What changes

Add materialised table `_pg_ripple.predicate_stats_cache(predicate_id BIGINT PRIMARY KEY, triple_count BIGINT, refreshed_at TIMESTAMPTZ)`. A background refresh task updates it every `pg_ripple.stats_refresh_interval_seconds` seconds (default: 300). `graph_stats()` reads from the cache; a new `pg_ripple.refresh_stats_cache()` SQL function forces an immediate refresh.

### Deliverables

| ID | Deliverable |
|----|-------------|
| STATS-CACHE-01a | `_pg_ripple.predicate_stats_cache` table (+ migration script) |
| STATS-CACHE-01b | Background refresh task |
| STATS-CACHE-01c | `pg_ripple.refresh_stats_cache()` function |
| STATS-CACHE-01d | `graph_stats()` reads from cache |

## ARROW-LIMIT-01 — Arrow Export Row Limit GUC

### Current state

The Arrow Flight export endpoint in `pg_ripple_http/src/arrow_encode.rs` has no row-count limit. A query returning hundreds of millions of rows can exhaust server memory.

### What changes

Add `pg_ripple.arrow_max_export_rows` GUC (default: 10,000,000). If the result set exceeds this limit:
1. The HTTP endpoint returns HTTP 400 with `{"error":"PT413","message":"result set exceeds arrow_max_export_rows limit of N"}`.
2. The GUC is documented in the configuration reference.

### Deliverables

| ARROW-LIMIT-01a | `pg_ripple.arrow_max_export_rows` GUC |

## PGSS-NORM-01 — pg_stat_statements Normalisation

### Current state

SPARQL queries passed to `pg_stat_statements` include literal values and IRIs verbatim, causing each distinct query to create a separate `pg_stat_statements` entry. A workload of 10,000 distinct SPARQL queries creates 10,000 entries.

### What changes

Before recording a query in `pg_stat_statements` (via `pgss_ProcessUtility`), normalise the SPARQL query text by replacing:
- String literals with `$S`
- IRI literals with `$I`
- Numeric literals with `$N`

Use the existing `sparql_normalise()` helper in `src/sparql/parse.rs`.

### Deliverables

| ID | Deliverable |
|----|-------------|
| PGSS-NORM-01a | Normalisation applied before pg_stat_statements recording |
| PGSS-NORM-01b | Test: 100 queries differing only in literal values produce 1 pg_stat_statements row |

## GUC-BOUNDS-01 — GUC Validator Bounds

### Current state

`vp_promotion_threshold`, `dictionary_cache_size`, and `merge_batch_size` GUCs have no min/max constraints. A user who sets `vp_promotion_threshold = -1` or `dictionary_cache_size = 0` triggers a panic in the promotion or dictionary subsystem.

### What changes

Add `min_val` and `max_val` to each GUC definition:
- `vp_promotion_threshold`: min 100, max 10,000,000
- `dictionary_cache_size`: min 1024, max 1,073,741,824 (1 GiB)
- `merge_batch_size`: min 100, max 100,000,000

### Deliverables

| ID | Deliverable |
|----|-------------|
| GUC-BOUNDS-01a | Bounds added to all three GUCs |
| GUC-BOUNDS-01b | Test: values below min and above max are rejected with clear error |

## EXPLAIN-ALG-01 — SPARQL Algebra Tree in EXPLAIN Output

### Current state

`pg_ripple.sparql_explain(query)` returns the generated SQL query plan but not the SPARQL algebra tree. Diagnosing suboptimal plans requires reading the SQL — often difficult to correlate back to the original SPARQL.

### What changes

Add an `"algebra"` key to the JSON output of `sparql_explain()` containing the optimised SPARQL algebra tree as a nested JSON object. The algebra should be emitted after `sparopt` optimisation (post-simplification, post-join-reordering). Include variable bindings at each node.

### Deliverables

| ID | Deliverable |
|----|-------------|
| EXPLAIN-ALG-01a | `"algebra"` key in `sparql_explain()` output |
| EXPLAIN-ALG-01b | Algebra emitted post-sparopt optimisation |
| EXPLAIN-ALG-01c | Regression test: algebra tree contains expected node types |
| EXPLAIN-ALG-01d | EXPLAIN documentation updated |

## MERGE-HBEAT-01 — Merge Worker Heartbeat

### Current state

The merge background worker runs silently. An operator cannot determine whether the worker is running, how many predicates it has processed, or when the last merge occurred.

### What changes

Add a periodic `LOG`-level heartbeat at `pg_ripple.merge_heartbeat_interval_seconds` (default: 60 s):
```
LOG: pg_ripple merge worker heartbeat: last_merge=2026-05-01T12:34:56Z predicates_merged=42 rows_moved=1234567
```
Also update `_pg_ripple.merge_worker_status(last_heartbeat_at, predicates_merged_total, rows_moved_total)` table.

### Deliverables

| ID | Deliverable |
|----|-------------|
| MERGE-HBEAT-01a | `pg_ripple.merge_heartbeat_interval_seconds` GUC |
| MERGE-HBEAT-01b | Periodic LOG-level heartbeat |
| MERGE-HBEAT-01c | `_pg_ripple.merge_worker_status` table |

## STATS-DOC-01 — graph_stats() Documentation and Scan Limit

### Current state

## REDACT-01 — redacted_error() Uniformity

### Current state

Three HTTP handlers in `pg_ripple_http/src/routing.rs` return raw Rust error strings in the response body instead of going through `redacted_error()`. These may include internal table names, SPI error details, or stack fragments.

### What changes

Audit all `?` and `.map_err()` returns in routing.rs. Any that produce an HTTP response body must use `redacted_error()` to strip internal details and return only the error code.

### Deliverables

| ID | Deliverable |
|----|-------------|
| REDACT-01a | All routing.rs error paths use `redacted_error()` |
| REDACT-01b | Integration test: internal SPI error does not appear in HTTP response body |

## ADMIN-LOCK-01 — Administration Function Lock Documentation

### Current state

`docs/src/reference/administration.md` does not document which PostgreSQL lock levels are acquired by `ripple_vacuum()`, `ripple_reindex()`, and (new in v0.81.0) `recover_stuck_promotions()`. Operators cannot plan maintenance windows appropriately.

### What changes

Add a "Lock levels" section to each function's documentation listing the lock type, object, and typical hold duration.

| ID | Deliverable |
