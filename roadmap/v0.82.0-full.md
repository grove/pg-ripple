## DECODE-BIND-01 — Batch Decode Bind Parameter

### Current state

`src/dictionary/mod.rs` `batch_decode()` emits:
```sql
SELECT id, term FROM _pg_ripple.dictionary WHERE id IN (1234, 5678, …)
```
PostgreSQL generates a new query plan for each unique combination of IDs. Under high-concurrency workloads, this causes plan cache churn.

### What changes

Replace with:
```sql
SELECT id, term FROM _pg_ripple.dictionary WHERE id = ANY($1::bigint[])
```
using `Spi::run_with_args()` with a single `DatumArray<PgBuiltInOids::INT8OID>` parameter.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DECODE-BIND-01a | `ANY($1)` binding in `batch_decode()` |
| DECODE-BIND-01b | Benchmark: plan cache hit rate before/after |

## MERGE-PRED-01 — Merge Worker Predicate ID Cache

### Current state

The merge background worker (`src/storage/merge.rs`) queries `_pg_ripple.predicates` on every merge cycle to discover which predicates have pending delta rows. Under high-ingest workloads, this adds significant overhead to every merge wake-up.

### What changes

Maintain a `HashSet<PredicateId>` in the worker's local state. Refresh it from `_pg_ripple.predicates` only when a `NOTIFY pg_ripple_predicates_changed` message is received (already emitted by the promote path) or on a 60-second fallback timer.

### Deliverables

| ID | Deliverable |
|----|-------------|
| MERGE-PRED-01a | Predicate ID cache in merge worker state |
| MERGE-PRED-01b | Cache invalidated on NOTIFY |
| MERGE-PRED-01c | Benchmark: merge worker CPU before/after |

## STATS-CACHE-01 — Predicate Stats Cache

### Current state

`pg_ripple.graph_stats()` scans every VP table on each call to compute triple counts. Under workloads with hundreds of predicates, this creates significant lock contention.

## PGSS-NORM-01 — pg_stat_statements Normalisation

### Current state

SPARQL queries passed to `pg_stat_statements` include literal values and IRIs verbatim, causing each distinct query to create a separate `pg_stat_statements` entry. A workload of 10,000 distinct SPARQL queries creates 10,000 entries.

### What changes

Before recording a query in `pg_stat_statements` (via `pgss_ProcessUtility`), normalise the SPARQL query text by replacing:
- String literals with `$S`
- IRI literals with `$I`
- Numeric literals with `$N`

Use the existing `sparql_normalise()` helper in `src/sparql/parse.rs`.

### Deliverables

| ID | Deliverable |
|----|-------------|
| PGSS-NORM-01a | Normalisation applied before pg_stat_statements recording |
| PGSS-NORM-01b | Test: 100 queries differing only in literal values produce 1 pg_stat_statements row |

## GUC-BOUNDS-01 — GUC Validator Bounds

### Current state

`vp_promotion_threshold`, `dictionary_cache_size`, and `merge_batch_size` GUCs have no min/max constraints. A user who sets `vp_promotion_threshold = -1` or `dictionary_cache_size = 0` triggers a panic in the promotion or dictionary subsystem.

- `vp_promotion_threshold`: min 100, max 10,000,000
- `dictionary_cache_size`: min 1024, max 1,073,741,824 (1 GiB)
- `merge_batch_size`: min 100, max 100,000,000

### Deliverables

| ID | Deliverable |
|----|-------------|
| GUC-BOUNDS-01a | Bounds added to all three GUCs |
| GUC-BOUNDS-01b | Test: values below min and above max are rejected with clear error |

## LISTEN-LEN-01 — SSE Subscription ID Length Validation

### Current state

`pg_ripple_http/src/routing/mod.rs` `sparql_subscription_sse()` validates `subscription_id` for character set (alphanumeric, hyphen, underscore) but not length. PostgreSQL's `LISTEN "channel"` silently truncates identifier names to 63 bytes. An oversized `subscription_id` will be truncated to 63 chars, which could collide with another legitimately-named subscription channel.

### What changes

Add a length check immediately after the character-set validation:

```rust
if subscription_id.len() > 63 {
    return (
        StatusCode::BAD_REQUEST,
        Json(ErrorResponse { code: "PT400", message: "subscription_id must not exceed 63 characters".into() }),
    ).into_response();
}
```

### Deliverables

| ID | Deliverable |
|----|-------------|
| LISTEN-LEN-01a | Length check added to subscription_id validation |
| LISTEN-LEN-01b | Error response uses JSON `ErrorResponse` type |
| LISTEN-LEN-01c | Test: 64-char subscription_id → HTTP 400 |
| LISTEN-LEN-01d | Test: 63-char subscription_id → HTTP 200 (accepted) |

## PROPPATH-UNBOUNDED-01 — Property-Path All-Nodes Query Size Limit

### Current state

`src/sparql/property_path.rs` `build_all_nodes_sql()` and `build_all_predicates_with_p()` construct a `UNION ALL` subquery with two branches per predicate (one for `s`, one for `o`). On a schema with 1,000 predicates this generates a 2,000-branch UNION ALL. With 10,000 predicates the generated SQL string exceeds PostgreSQL's parser stack limit. No predicate count limit exists.

These functions are invoked for `*` (ZeroOrMorePath) and negated property sets — common in exploratory SPARQL queries.

### What changes

Add GUC `pg_ripple.all_nodes_predicate_limit` (integer, default 500, min 10, max 50000). When the predicate count exceeds the limit, log a WARNING and generate the query using only the top-N predicates ordered by `triple_count DESC`. Document the behaviour and GUC in the configuration reference.

|----|-------------|
| PROPPATH-UNBOUNDED-01a | `pg_ripple.all_nodes_predicate_limit` GUC |
| PROPPATH-UNBOUNDED-01b | WARNING when predicate count exceeds limit |

## VACUUM-DICT-BATCH-01 — Dictionary Vacuum Batched UNION ALL

### Current state

`src/maintenance_api.rs` `vacuum_dictionary()` builds a single SQL string of the form:
```sql
SELECT s FROM _pg_ripple.vp_111 UNION ALL SELECT o FROM _pg_ripple.vp_111
UNION ALL SELECT s FROM _pg_ripple.vp_222 …
```
with two branches per predicate plus the `vp_rare` terms. On an instance with 5,000 predicates this generates a ~500 KB SQL string that PostgreSQL's parser must process as a single query. This can cause `ERROR: stack depth limit exceeded` or planning timeouts.

### What changes

Add GUC `pg_ripple.vacuum_dict_batch_size` (integer, default 200, min 10, max 10000). Process predicates in batches of this size, accumulating live IDs into the temp table incrementally. Each batch runs as an independent SPI call, keeping individual SQL strings below ~20 KB.

### Deliverables

| ID | Deliverable |
|----|-------------|
| VACUUM-DICT-BATCH-01a | `pg_ripple.vacuum_dict_batch_size` GUC |
| VACUUM-DICT-BATCH-01b | Predicate processing loops in batches |
| VACUUM-DICT-BATCH-01c | Test: vacuum_dictionary with 250 predicates at batch size 100 → runs to completion |

## MERGE-LOCK-GUC-01 — Merge Lock Timeout GUC

### Current state

`src/storage/merge.rs` sets `SET LOCAL lock_timeout = '5s'` before acquiring the merge fence lock. This is hardcoded. On a busy production system with long-running read queries, 5 seconds may be too short and merges may be skipped more often than intended. On a fast SSD-based system a 1-second timeout may be preferable.

### What changes

```rust
let timeout_ms = crate::MERGE_LOCK_TIMEOUT_MS.get();
Spi::run_with_args(
    &format!("SET LOCAL lock_timeout = '{timeout_ms}ms'"),
    &[],
)?;
```

## DATALOG-SILENT-01 — Datalog Cleanup Error Visibility

### Current state

`src/datalog/wfs.rs` contains 13 and `src/datalog/seminaive.rs` contains 16 `let _ = Spi::run_with_args(...)` calls for `DROP TABLE IF EXISTS` cleanup of temporary tables. Failures are completely invisible. If cleanup fails (e.g. due to lock conflicts, disk full, or schema corruption), the temp table persists and subsequent Datalog queries may use stale data.

### What changes

Replace all `let _ = Spi::run_with_args(...)` cleanup calls with:

```rust
Spi::run_with_args(&sql, &[])
    .unwrap_or_else(|e| pgrx::log!("datalog cleanup: DROP TABLE failed: {e}"));
```

`pgrx::log!` emits at PostgreSQL's `LOG` severity, which is normally suppressed but visible when `log_min_messages = debug1`. This preserves the "best effort" semantics for cleanup while making failures observable.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DATALOG-SILENT-01a | All 29 `let _ = Spi` cleanup calls in wfs.rs and seminaive.rs replaced |
| DATALOG-SILENT-01b | LOG-level message includes table name and error |

## EMBED-MODEL-01 — Embedding Model GUC Consistency

### Current state

`src/sparql/embedding.rs` has code paths that fall back to the hardcoded string `"text-embedding-3-small"` when the caller passes `None` for the model parameter:

```rust
let model = model_tag.unwrap_or("text-embedding-3-small");
```

The GUC `pg_ripple.embedding_model` exists and is registered, but this fallback ignores it. Users who set `pg_ripple.embedding_model = 'text-embedding-ada-002'` still get `text-embedding-3-small` for some call paths.

```rust
let model = model_tag.unwrap_or_else(|| {
```

Audit all call sites in `embedding.rs` and `src/sparql/embedding_planner.rs` for the same pattern.

## FED-COUNTER-ORDER-01 — Federation Call Counter Ordering

### Current state

`src/sparql/federation.rs` `execute_remote()` increments `FED_CALL_COUNT` before performing the endpoint policy allowlist check. If the endpoint is blocked by policy, the counter is still incremented, producing inflated "calls made" metrics.

### What changes

Move the `FED_CALL_COUNT.fetch_add(1, ...)` to after the endpoint policy check passes. Blocked requests should increment a separate `FED_BLOCKED_COUNT` counter (new) for observability.

### Deliverables

| ID | Deliverable |
|----|-------------|
| FED-COUNTER-ORDER-01a | `FED_CALL_COUNT` incremented only after policy check passes |
| FED-COUNTER-ORDER-01b | New `FED_BLOCKED_COUNT` counter incremented on policy rejection |
| FED-COUNTER-ORDER-01c | Both counters exposed via `pg_ripple.federation_stats()` SQL function |
