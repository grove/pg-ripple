## Background and motivation

### How derived triples work today

pg_ripple already writes derived triples back into VP tables via two mechanisms:

1. **Datalog rules** (`create_datalog_rule()`): a fixed-point recursive
   inference engine that materialises inferred triples with `source = 1`.
   Incremental maintenance is via semi-naive evaluation and Delete-Rederive.
2. **CONSTRUCT views** (`create_construct_view()`, v0.18.0): produces triples
   into an external pg_trickle stream table — *outside* the VP storage layer.

What is missing is a mechanism that combines the expressiveness of SPARQL
CONSTRUCT (arbitrary graph patterns, FILTER, OPTIONAL, UNION, property paths)
with the persistence of Datalog writeback (derived triples land in VP tables
and are immediately queryable from other SPARQL queries and rules).

### Use case: raw-to-canonical model maintenance

The canonical pattern in enterprise knowledge graphs:

```
raw graph          CONSTRUCT rule          canonical graph
─────────────  →  ────────────────────  →  ────────────────
<urn:raw>          raw_to_canonical         <urn:canonical>
```

All downstream consumers query `<urn:canonical>`. The rule is the single
source of truth for how raw data maps to the canonical vocabulary. When
the raw schema evolves, only the rule is updated; consumers are unaffected.

## Data model

### Catalog table

```sql
CREATE TABLE _pg_ripple.construct_rules (
    name            TEXT PRIMARY KEY,
    sparql          TEXT NOT NULL,
    generated_sql   TEXT,              -- delta SQL, set after compilation
    target_graph    TEXT NOT NULL,     -- IRI of the target named graph
    target_graph_id BIGINT NOT NULL,   -- dictionary-encoded target graph ID
    mode            TEXT NOT NULL DEFAULT 'incremental',  -- 'incremental' | 'full'
    source_graphs   TEXT[],            -- IRIs of all graphs read by the pattern
    rule_order      INT,               -- topological sort position in pipeline
    created_at      TIMESTAMPTZ DEFAULT now(),
    last_refreshed  TIMESTAMPTZ
);
```

### Derived-triple provenance

All triples produced by a construct rule are stored with `source = 1` (the
existing inferred-triple tag) and `g = target_graph_id`. In addition, every
derived triple is recorded in a per-rule provenance table so that multiple
rules can safely share a target graph (CWB-11):

```sql
CREATE TABLE _pg_ripple.construct_rule_triples (
    rule_name TEXT    NOT NULL REFERENCES _pg_ripple.construct_rules(name)
                               ON DELETE CASCADE,
    pred_id   BIGINT  NOT NULL,
    s         BIGINT  NOT NULL,
    o         BIGINT  NOT NULL,
    g         BIGINT  NOT NULL,
    PRIMARY KEY (rule_name, pred_id, s, o, g)
);
```

The delete path (CWB-03) uses this table to implement reference-count
semantics: a derived triple is removed from the VP table only when its
provenance row count drops to zero — i.e. no other rule still supports it.
This is the same approach Datalog provenance tracking uses for shared
inferred triples.

## Lifecycle SQL functions

### `create_construct_rule()`

```sql
pg_ripple.create_construct_rule(
    name         TEXT,
    sparql       TEXT,
    target_graph TEXT,
    mode         TEXT DEFAULT 'incremental'
) RETURNS void
```

Steps at registration time:

1. Parse `sparql` with `spargebra`. Reject if not a CONSTRUCT query.
2. Identify all named graphs read by the WHERE pattern (`source_graphs`).
3. **Cycle check**: if `target_graph` ∈ `source_graphs`, raise
   `"construct rule '\{name\}' reads from and writes to the same graph '\{target_graph\}' — cycle not allowed"`.
4. **Pipeline stratification**: if any existing rule writes to a graph that
   this rule reads from, set `rule_order` accordingly. If a mutual dependency
   is detected, raise `"construct rules '\{a\}' and '\{b\}' form a cycle — mutual writeback is not supported"`.
5. Compile the CONSTRUCT WHERE pattern to a SQL SELECT via the existing
   `translate_select` pipeline (integer joins only — all IRIs/literals
   dictionary-encoded at compile time).
6. Generate the delta SQL (see below) and store in `generated_sql`.
7. Dictionary-encode `target_graph` and store `target_graph_id`.
8. Insert into `_pg_ripple.construct_rules`.
9. Run an initial full recompute so the target graph is populated immediately.

### `drop_construct_rule()`

```sql
pg_ripple.drop_construct_rule(name TEXT, retract BOOLEAN DEFAULT TRUE) RETURNS void
```

- If `retract = TRUE` (default): removes all provenance rows for this rule
  from `_pg_ripple.construct_rule_triples`, then deletes from the VP tables
  only those triples that have no remaining provenance row from any other
  rule (reference count drops to zero).
- Removes the catalog row.

### `refresh_construct_rule()`

```sql
pg_ripple.refresh_construct_rule(name TEXT) RETURNS BIGINT
```

Full recompute: truncate all derived triples in `target_graph` (source=1),
re-run the CONSTRUCT query against the current triplestore, insert results.
Returns the number of triples written.

### `list_construct_rules()`

```sql
pg_ripple.list_construct_rules()
RETURNS TABLE(name TEXT, sparql TEXT, target_graph TEXT, mode TEXT,
              source_graphs TEXT[], rule_order INT, last_refreshed TIMESTAMPTZ)
```

### `explain_construct_rule()`

```sql
pg_ripple.explain_construct_rule(name TEXT)
RETURNS TABLE(section TEXT, content TEXT)
```

Returns rows for `delta_insert_sql`, `delta_delete_sql`, `source_graphs`,
`rule_order`, and the CONSTRUCT template triple count.

## Incremental evaluation design

### Insert path (triggered after every INSERT into a source VP table)

The key insight is that the CONSTRUCT WHERE pattern is translated to a SQL
SELECT. The delta insert is computed by running the same SELECT against only
the *newly inserted triples* (the HTAP delta partition), unioning with existing
facts to satisfy joins, then filtering out triples already present in the
target graph:

```sql
INSERT INTO _pg_ripple.vp_{pred_id} (s, o, g, source)
SELECT DISTINCT s_expr, o_expr, $target_graph_id, 1
FROM   (
    -- pattern evaluated with new delta triples visible
    {translated_select_with_delta_filter}
)
WHERE (s_expr, o_expr) NOT IN (
    SELECT s, o FROM _pg_ripple.vp_{pred_id}
    WHERE g = $target_graph_id AND source = 1
)
ON CONFLICT DO NOTHING;
```

The `delta_filter` restricts at least one triple pattern in the WHERE clause
to the HTAP delta partition (the newly-inserted rows), ensuring only triples
that could be affected by the change are re-evaluated.

### Delete path (Delete-Rederive, triggered after DELETE from a source VP table)

When source triples are deleted, the Delete-Rederive algorithm (already
implemented for Datalog in `src/datalog/dred.rs`) is adapted for CONSTRUCT
rules:

1. **Identify candidates**: find all derived triples whose derivation *could*
   have depended on the deleted source triples by running the CONSTRUCT pattern
   against a snapshot that includes the deleted triples but tags them
   as "tentatively deleted".
2. **Rederive**: re-run the CONSTRUCT pattern against the updated triplestore
   (without the deleted source triples).
3. **Retract**: derived triples that appeared in step 1 but not step 2 are
   removed from `_pg_ripple.construct_rule_triples` for this rule. A triple
   is then deleted from the VP table only if no other rule's provenance row
   still covers it (i.e. `COUNT(*) = 0` in `construct_rule_triples` for that
   `(pred_id, s, o, g)` tuple).

### Transaction boundary

Both insert and delete paths are executed synchronously within the same
PostgreSQL transaction as the source write, maintaining serializability.
Rules are executed in `rule_order` to respect pipeline dependencies.

The evaluation is triggered via `AFTER INSERT OR DELETE ON _pg_ripple.vp_{pred_id}`
statement-level triggers, one per source VP table referenced by one or more rules.

## Interaction with existing systems

### HTAP merge

Derived triples with `source = 1` in the target graph participate in the
HTAP merge like any other triples. The merge worker moves them from the
delta partition to the main partition without special handling.

### SHACL validation

SHACL shapes can be applied to the target graph. Because derived triples
are written within the same transaction as the trigger, SHACL constraints
on the canonical graph are evaluated against a fully consistent snapshot.

### Datalog rules

A Datalog rule may read from the target graph of a construct rule, and
vice versa. Pipeline stratification (CWB-06) covers mixed Datalog +
CONSTRUCT rule pipelines: the topological sort is computed over both
rule types together.

### SPARQL UPDATE

`SPARQL INSERT DATA` into a source graph triggers the same delta path as
a direct SQL insert. The trigger fires on the VP table, not at the
SPARQL protocol layer.

## Cycle and stratification algorithm

The dependency graph $G$ has one node per named graph (source or target of
any rule) and one directed edge $(A, B)$ for each rule that reads from
graph $A$ and writes to graph $B$.

- **Self-loop** ($A = B$): rejected at registration time (CWB-05).
- **Cycle of length ≥ 2**: detected by DFS in $G$ at registration time;
  registration fails with an error listing the cycle (CWB-06).
- **DAG**: `rule_order` is the topological sort order of $G$; rules are
  executed in ascending `rule_order` within each transaction.
