# v0.69.0 - Module Architecture Restructuring

> **Full technical details:** [v0.69.0-full.md](v0.69.0-full.md)

**Status: Released ✅** | **Scope: Large**

> v0.69.0 splits the large source modules along narrow architectural contracts so that future contributors can understand, test, and modify one concern without reading 2,000-line files.

---

## What is this?

Several core modules grew across the v0.60–v0.68 sprint to 1,100–2,300 lines and now mix multiple distinct responsibilities. `src/sparql/mod.rs` handles parsing, planning, translation, and execution. `pg_ripple_http/src/main.rs` contains routing, auth, SPI bridging, Arrow IPC encoding, and streaming. `src/construct_rules.rs` combines the rule catalog, incremental delta engine, and the topological sort scheduler. v0.69.0 splits each along the boundaries that the mutation journal, RLS, and Arrow work in v0.67–v0.68 have already established.

## Headline outcomes

- `src/sparql/` split into `parse/`, `plan/`, `translate/` (already partially split), `execute/`, and `cursor.rs`.
- `pg_ripple_http/src/main.rs` split into `routing.rs`, `auth.rs`, `spi_bridge.rs`, `arrow_encode.rs`, and `stream.rs`.
- `src/construct_rules.rs` split into `catalog.rs`, `delta.rs`, `scheduler.rs`, and `retract.rs`.
- `src/storage/mod.rs` refactored to expose a narrower mutation-journal-based API after the journal lands in v0.67.
- All public API functions, GUC registration, and `_PG_init` entry points remain unchanged.
- No pg_regress tests break; existing expected output files do not change.

## Assessment findings covered

From [plans/PLAN_OVERALL_ASSESSMENT_9.md](../plans/PLAN_OVERALL_ASSESSMENT_9.md):

- Action item 21 (L, v0.69.0): Split large modules along mutation, translation, export, and HTTP routing boundaries.

General architecture finding:
- Architecture & Long-term Health: "side effects are attached to API wrappers rather than core architectural contracts" — the refactoring makes the contract structure visible in the file system.
