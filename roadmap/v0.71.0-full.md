## Deliverables

### FLIGHT-STREAM-01 — Arrow Flight streaming verification

**Root cause**: Static analysis in Assessment 10 could not confirm whether `pg_ripple_http/src/arrow_encode.rs` uses `Body::from_stream(...)` over a lazy cursor or builds the entire response buffer in memory.

**Work items**:

1. Inspect `arrow_encode.rs` and confirm (or implement) chunked streaming via `tokio_stream` + `axum::body::Body::from_stream(stream)`.
2. Add `tests/http_integration/arrow_export_large.rs` (or shell equivalent):
   - Inserts 10,000,000 triples into a named graph.
   - Calls `POST /flight/do_get` with a valid signed ticket.
   - Asserts HTTP response is streaming (`Transfer-Encoding: chunked` or `Content-Length` not present until close).
   - Asserts HTTP companion process RSS does not exceed `EXPECTED_MEMORY_BOUND_MB` (e.g., 512 MB) during export.
3. Document the memory bound in `docs/src/reference/arrow-flight.md` (this file was previously a missing evidence citation — create it here).

**Exit criterion**: 10 M-row export completes with RSS < 512 MB and the response body decodes as valid Arrow IPC with correct row count.

### CITUS-INT-01 — Citus RLS propagation integration test

**Root cause**: `feature_status()` at [src/feature_status.rs:247](../src/feature_status.rs#L247) cites `tests/integration/citus_rls_propagation.sh` which does not exist.

**Work items**:

1. Create `tests/integration/citus_rls_propagation.sh`:
   - Starts a Citus cluster via `docker-compose.yml`.
   - Enables Citus sharding: `SELECT pg_ripple.enable_citus_sharding()`.
   - Creates a named graph and inserts enough triples to promote a predicate past the threshold.
   - Grants a non-superuser role access to one graph only via `pg_ripple.grant_graph_access(...)`.
   - Inserts triples in both the allowed and a restricted graph, including triples with the promoted predicate.
   - Queries as the non-superuser role and asserts restricted-graph triples are absent from results.
   - Promotes an additional predicate while the grant is active and asserts the policy applies to the new VP table.
2. Wire the test into `ci.yml` under a `citus-integration` job (informational/non-blocking until Citus CI infrastructure is stable; blocking after v0.73.0).

**Exit criterion**: Script runs to completion without error when a Citus cluster is available; `feature_status()` citation resolves.

### COMPAT-01 — Extension / HTTP companion compatibility

**Root cause**: `pg_ripple_http/Cargo.toml` is independently versioned (currently 0.16.0). There is no documented mapping between extension versions and compatible HTTP companion versions.

**Work items**:

1. Create `docs/src/operations/compatibility.md` with a table mapping pg_ripple extension version ranges to compatible `pg_ripple_http` version ranges and noting the first version of each pair that is tested together.
2. Add a startup check in `pg_ripple_http/src/main.rs`: call `SELECT version FROM pg_ripple.feature_status() LIMIT 1` (or a dedicated `pg_ripple.extension_version()` function) and compare against a compiled-in `COMPATIBLE_EXTENSION_RANGE`; log a warning or refuse to start if out of range.
3. Document the independent versioning decision in AGENTS.md under HTTP companion notes.

### HLL-DOC-01 — HLL accuracy documentation and test

**Root cause**: `COUNT(DISTINCT)` via HLL ([src/sparql/translate/group.rs:240-246](../src/sparql/translate/group.rs#L240-L246)) uses the PostgreSQL `hll` extension default precision (typically `log2m=14`, ~0.02% standard error), but this is not documented or tested.

**Work items**:

1. Create `docs/src/reference/approximate-aggregates.md` documenting:
   - When HLL is used (GUC `pg_ripple.approx_distinct = on` + `hll` extension present).
   - Error bounds at default precision (≈ ±0.02% for cardinalities ≥ 10,000).
   - Fallback behavior (exact `COUNT(DISTINCT)` when HLL is unavailable).
2. Add `tests/pg_regress/sql/hll_accuracy.sql`:
   - Insert 100,000 distinct subjects into a named graph.
   - Run `SELECT COUNT(DISTINCT ?s) WHERE { GRAPH <g> { ?s ?p ?o } }` with `approx_distinct = on`.
   - Assert result is within 1% of 100,000.

### CITUS-BENCH-01 — SERVICE shard annotation benchmark

**Root cause**: The SERVICE shard annotation hook ([src/sparql/translate/graph.rs:364-366](../src/sparql/translate/graph.rs#L364-L366)) is wired but its effectiveness is not validated.

**Work items**:

1. Add `tests/pg_regress/sql/citus_service_pruning.sql`:
   - Calls `EXPLAIN (FORMAT JSON) SELECT ...` on a SPARQL SERVICE query with `citus_service_pruning = off` and `= on`.
   - Asserts the shard count in the explain JSON is lower with pruning on.
2. Document the expected behavior in `docs/src/reference/citus-service-pruning.md`.
