# v0.37.0 — Storage Concurrency Hardening & Error Safety

**Effort estimate: 9–11 person-weeks**

**Goal:** A comprehensive reliability release responding to the first full code audit. Two concurrency bugs that could silently lose deletes or leave predicates in the wrong storage tier are fixed with proper advisory locking. Every place in the code that could crash the database server on an unexpected input is replaced with a typed error message. Configuration parameters now validate their values at set time. A `diagnostic_report()` function gives operators a complete health snapshot.

---

## HTAP Merge Cutover Race Fix

**Concurrent deletes can no longer be lost during the merge window.**

The background merge worker swaps out the main VP partition for a freshly merged one. If a concurrent delete arrives during the brief window between the swap and the tombstone update, the delete was previously silently lost — the deleted triple would reappear after the merge because the tombstone was not in the new main partition. The fix uses a per-predicate PostgreSQL advisory lock: the merge worker and the delete path both acquire this lock, preventing the race. A stress test with 50 concurrent writers and a 1-second merge interval runs for 5 minutes and verifies zero lost deletes.

---

## No More Panics in Extension Code

**Every `unwrap()` and `expect()` call in library code is replaced with a proper error.**

Rust's `unwrap()` and `expect()` operations abort the calling process if an unexpected condition occurs. In a PostgreSQL extension, "aborting the calling process" means crashing the database server. Over 30 such calls were found in non-test extension code. Each one has been replaced with a `Result`-propagating error path that surfaces a typed `PT` error code to the calling SQL statement and rolls back the transaction, rather than bringing down the server. A clippy deny directive enforces this policy on all future code.

---

## GUC Value Validators

**Configuration parameters now reject bad values at set time.**

PostgreSQL allows extensions to register GUC (Grand Unified Configuration) parameters. Without validators, you could set `pg_ripple.shacl_mode = 'typo'` and not discover the error until a SHACL operation was attempted — or worse, it would silently use a default. This release adds validation functions to all enum-valued GUCs (mode switches, strategy selectors) and minimum-value bounds to all integer GUCs. Invalid values are rejected immediately with a clear error.

---

## `diagnostic_report()` Function

**A single function call reports the health of the entire pg_ripple installation.**

The `pg_ripple.diagnostic_report()` function returns a table of key–value pairs covering: the validity of all GUC values, the shared-memory cache hit and miss rates, the number of pending rows in all delta tables (the "merge backlog"), the depth of the SHACL validation queue, the health of registered federation endpoints, and whether the schema version in `_pg_ripple.schema_version` matches the installed extension version. This is designed to be the first command to run when troubleshooting a performance or correctness issue.

---

## Schema Version Table

**The installed version is stamped in a dedicated table for upgrade verification.**

A new `_pg_ripple.schema_version` table records the current version, when it was installed, and what version it was upgraded from. The `diagnostic_report()` function checks this table against the running extension version to detect incomplete upgrades. Every future `ALTER EXTENSION pg_ripple UPDATE` stamps this table automatically.
