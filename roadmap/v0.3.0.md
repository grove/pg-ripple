# v0.3.0 — SPARQL Query Engine (Basic)

**Effort estimate: 4–6 person-weeks**

**Goal:** The first proper query language arrives. This release implements the core of SPARQL — the W3C standard query language for RDF knowledge graphs — covering the patterns most developers reach for first: basic graph matching, filtering, optional data, ordering, pagination, and asking yes/no questions about the data.

---

## Basic Graph Patterns and FILTER

**Write a pattern describing what you are looking for, and get back every match.**

SPARQL's core construct is the basic graph pattern: you describe the shape of triples you want to find, using variables for the parts you do not know. "Find all things where Alice knows them and they have a name" is expressed as a pattern with two triple shapes and a variable for the name. The SPARQL engine translates this into SQL that joins the relevant VP tables together on the integer IDs, so the query runs entirely in terms of compact numbers rather than string comparisons. FILTER expressions let you narrow results further — comparisons, string tests, numeric ranges.

---

## OPTIONAL, GRAPH, LIMIT, OFFSET, ORDER BY, DISTINCT

**All the standard modifiers for practical queries work out of the box.**

Real-world queries need optional data (return the label if it exists, but do not drop the row if it does not), graph-scoped patterns (query only the facts in a specific named graph), sorting, pagination, and deduplication. All of these are implemented in this release. OPTIONAL translates to a LEFT JOIN in the generated SQL. GRAPH translates to a WHERE condition on the graph-ID column. LIMIT and OFFSET pass straight through to PostgreSQL.

---

## Star-Pattern Optimisation

**Queries with the same subject and many predicates get a smarter SQL plan.**

A very common SPARQL pattern is the "star": one subject linked to several different predicates. Naively, the engine would translate each triple pattern into a separate subquery and join them. This release detects star patterns in the algebra and collapses them into a single scan with multiple joins against the same subject ID, eliminating redundant subqueries. For a star with ten predicates this can reduce query planning time by an order of magnitude.

---

## Query Plan Cache

**Frequently repeated queries skip the translation step on subsequent calls.**

SPARQL-to-SQL translation involves parsing, algebra normalisation, and code generation. For applications that run the same query hundreds of times (dashboards, APIs, background jobs), paying that cost on every call is wasteful. This release adds an in-process plan cache keyed on the normalised query algebra. Cache hits skip straight to SQL execution. The cache is bounded in size and is invalidated automatically when the schema changes.

---

## `sparql_explain()` Function

**Inspect the generated SQL for any SPARQL query.**

The `sparql_explain()` function accepts a SPARQL query string and returns the SQL that the engine would generate, without executing it. This is the first diagnostic tool for query troubleshooting — if a query is slower than expected, you can examine the underlying SQL to understand why.

---

## Security: SQL Injection Tests

**The SPARQL translator is verified to never produce injectable SQL.**

Because the translator converts user-supplied SPARQL text into SQL, there is a real risk of SQL injection if any user-controlled value is interpolated unsafely into the generated query. The test suite for this release verifies that all constant values in SPARQL queries are dictionary-encoded to integers before being placed in SQL, so the generated statements contain only integer literals — never raw strings that could contain SQL syntax.
