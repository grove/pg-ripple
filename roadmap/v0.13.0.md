# v0.13.0 — Performance Hardening

**Effort estimate: 4–6 person-weeks**

**Goal:** With a complete SPARQL query engine and storage architecture in place, this release focuses on making pg_ripple fast enough for production deployments. The key improvements are smarter join ordering based on actual data statistics, exploitation of PostgreSQL's parallel query workers, and a comprehensive benchmark suite that catches performance regressions before they ship.

---

## BGP Join Reordering by Selectivity

**Queries are automatically rewritten to join the smallest tables first.**

The basic graph pattern optimizer analyses the size of each VP table involved in a query and reorders the join sequence so that the most selective (smallest) table is evaluated first. For a query joining ten triple patterns, the difference between optimal and worst-case join order can be many orders of magnitude in execution time. The reordering uses PostgreSQL statistics (`pg_class.reltuples`) and bind-variable analysis: a pattern with a bound subject is estimated much more selective than one with all variables.

---

## Parallel Query Execution

**Queries automatically use multiple CPU cores when beneficial.**

PostgreSQL's parallel query infrastructure (parallel sequential scans, parallel hash joins, parallel aggregates) is made available to pg_ripple queries. For large VP table scans and multi-predicate joins over millions of triples, this can deliver near-linear speedups with core count. The parallelism settings follow PostgreSQL's standard `max_parallel_workers_per_gather` configuration.

---

## SHACL-Driven Query Hints

**Data quality constraints inform the query planner.**

When a SHACL shape declares that a predicate has at most one value per subject (`sh:maxCount 1`), the SPARQL query engine can omit the `DISTINCT` operation that would otherwise be needed to prevent duplicate rows from that predicate's VP table. Similarly, when `sh:minCount 1` guarantees a value always exists, an OPTIONAL join on that predicate can be upgraded to an INNER JOIN, which PostgreSQL can plan more efficiently. These hints are applied automatically based on loaded shapes.

## BSBM Benchmark Suite

**Performance is measured against the Berlin SPARQL Benchmark at 10M triples.**

The Berlin SPARQL Benchmark (BSBM) is a widely used standard for comparing RDF store performance. This release introduces automated BSBM runs as part of the CI pipeline, providing a reproducible baseline. The performance targets at this release are: over 100,000 triples per second for bulk loading, and under 10 milliseconds for simple single-predicate lookups on a 10 million triple dataset.

---

## Fuzz Testing Setup

**Automated random-input testing finds crash bugs before users do.**

The `cargo-fuzz` fuzzing infrastructure is integrated into the development workflow. The initial fuzz target covers the SPARQL parser: random byte sequences are fed into the parser and any panic or crash is reported as a failure. This provides continuous coverage of the input-handling code that processes untrusted SPARQL queries from application clients.
