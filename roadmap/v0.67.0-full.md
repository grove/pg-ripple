## Release theme

v0.67.0 has two inter-dependent phases. The first phase resolves the four Critical findings from PLAN_OVERALL_ASSESSMENT_9 — the storage mutation contract, tenant RLS coverage, Arrow Flight security/correctness, and release-truth gate failures — plus the high-priority hardening items that follow. These are not polish items: they are execution-path correctness and security failures that invalidate several v0.65–v0.66 claims. The second phase gathers the production-readiness evidence that makes v1.0.0 credible. Running the evidence sprint before the Critical remediations are closed would produce misleading artifacts.

## Deliverables: Critical and High Finding Remediation

### MJOURNAL-01 — Storage-level graph mutation journal

Introduce a transaction-local graph mutation journal at the storage boundary shared by all write paths:

- `storage::insert_triple_by_ids()`
- `storage::delete_triple_by_ids()`
- `storage::batch_insert_encoded()`
- SPARQL `INSERT DATA`, `DELETE DATA`, `DELETE/INSERT WHERE`
- All bulk loaders (`load_turtle`, `load_ntriples`, `load_rdfxml`, etc.)

The journal accumulates affected `(graph_id, delta)` pairs for the duration of a statement or transaction. A flush hook at statement end calls CONSTRUCT writeback (`on_graph_write` / `on_graph_delete`), provenance capture, CWB metrics, and Citus/RLS side effects from the journal rather than from selected API wrappers.

Requirements:
- Journal is transaction-local; concurrent transactions see independent journals.
- Journal must be zero-overhead when no construct rules are registered for a graph.
- Journal flush must survive a partial transaction rollback without leaving derived graphs inconsistent.

### MJOURNAL-02 — Rewire side effects to mutation journal

After MJOURNAL-01 lands:

- Remove the direct `on_graph_write()` and `on_graph_delete()` calls from `dict_api.rs` wrappers.
- Remove any CWB hook call that is wired to a specific public wrapper rather than the journal.
- Ensure provenance cleanup and CWB metric increments (`increment_cwb_rules_applied`, etc.) are driven by the journal flush, not by individual storage functions.
- Verify `stats::increment_cursor_rows_streamed`, `increment_arrow_batches_sent`, and `increment_arrow_ticket_rejections` — currently `#[allow(dead_code)]` in `stats.rs` — are wired from the journal and HTTP handler respectively.

### MJOURNAL-03 — pg_regress CWB equivalence tests

Add pg_regress test file `tests/pg_regress/sql/cwb_write_path_equivalence.sql` that:

- Registers a CONSTRUCT rule on a named graph.
- Inserts via `pg_ripple.insert_triple()`.
- Inserts via `SPARQL INSERT DATA`.
- Inserts via `SPARQL DELETE/INSERT WHERE`.
- Inserts via `DELETE DATA` then re-inserts.
- Inserts via each bulk loader (`load_ntriples`, `load_turtle`).
- After each insertion, queries the derived graph and asserts it equals the expected triples.
- Deletes the inserted triples via each path and asserts retraction.

Add the corresponding expected output file.

### RLS-01 — Graph RLS on all VP relations

Apply row-level security to VP tables and views at every lifecycle event:

1. **Table creation** (`ensure_vp_table()`): after creating `_delta`, `_main`, `_tombstones`, and the unified read view, call `enable_graph_rls()` for each new relation.
2. **Rare-predicate promotion** (`promote_predicate()`): after moving rows to the HTAP split, apply the current active graph policies to all three new relations.
3. **`grant_graph_access(graph_iri, role)`**: iterate `_pg_ripple.predicates` and apply the new policy to every existing VP relation in addition to `_pg_ripple.vp_rare`.
4. **`revoke_graph_access(graph_iri, role)`**: drop the policy from every VP relation.
5. **Migration scripts**: the next migration file must `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` for any VP tables that exist on upgrade from an earlier version.

### RLS-02 — Promoted-predicate RLS tests

Add `tests/pg_regress/sql/rls_promotion.sql`:

- Create a named graph.
- Grant a role access to that graph.
- Insert triples for a predicate below `vp_promotion_threshold`; verify the role sees only its graph's triples.
- Force promotion (lower the threshold or insert above it).
- After promotion, verify the role still sees only its graph's triples.
- Run an HTAP merge cycle.
- After merge, verify isolation again.
- Revoke access; verify the role sees no rows.

### FLIGHT-SEC-01 — Require signed Arrow tickets

Changes to `flight.rs` and `pg_ripple_http/src/main.rs`:

- `build_signed_ticket()` must return an error (not an `unsigned` token) when `ARROW_FLIGHT_SECRET` is not set and `pg_ripple.arrow_unsigned_tickets_allowed` GUC is `off` (default).
- Add GUC `pg_ripple.arrow_unsigned_tickets_allowed` (bool, default `off`).  When `on`, the existing unsigned path is retained for local development.
- `validate_flight_ticket()` in both the extension and the HTTP service must reject `sig = unsigned` unless the GUC/env flag is explicitly enabled.
- Document the required environment variable in README and operator runbook.

### FLIGHT-SEC-02 — Arrow export with tombstone exclusion and paged cursors

Rewrite `flight_do_get()` in `pg_ripple_http/src/main.rs`:

- Replace the `_main UNION ALL _delta` query with the same `(main EXCEPT tombstones) UNION ALL delta` read semantics used by SPARQL.
- Replace the single `query_spi_rows()` call with a database cursor (SPI `DECLARE … CURSOR` or portal-based paging).
- Stream Arrow record batches using `Body::from_stream`; do not buffer the full result.
- Batch size defaults to 1,000 rows per Arrow batch; configurable via GUC `pg_ripple.arrow_batch_size`.
- Update Arrow metrics: `increment_arrow_batches_sent()` per batch, `increment_arrow_ticket_rejections()` on validation failure.

### FLIGHT-SEC-03 — HTTP Arrow integration tests

Add `tests/http/arrow_flight_test.sh` (or equivalent Rust integration test) covering:

- Request with valid signed ticket → 200 + valid Arrow IPC stream.
- Request with `sig = unsigned` when `arrow_unsigned_tickets_allowed = off` → 401.
- Request with wrong HMAC signature → 401.
- Request with expired ticket → 401 or 403.
- Request for a graph that has tombstoned triples → tombstoned rows absent from response.
- Multi-batch export for a large graph (> batch size) → response is chunked, all rows present.

### GATE-01 — Portable release-truth scripts

Reimplement `scripts/check_api_drift.sh` and `scripts/check_roadmap_evidence.sh` in portable Python 3 (no GNU-only flags):

`check_api_drift.py`:
- Accepts `--version X.Y.Z` as a required argument.
- Extracts `#[pg_extern]` function names from `src/` using Python `re` (no `grep -P`).
- Loads the declared public API from `feature_status()` output or a generated JSON manifest.
- Reports missing, added, and renamed functions; exits non-zero on any discrepancy.
- Exits non-zero and prints an error if it extracts zero function names.

`check_roadmap_evidence.py`:
- Accepts `--version X.Y.Z` as a required argument.
- Parses `CHANGELOG.md` to locate the section for the given version (ignores `[Unreleased]`).
- Verifies that all claimed features in that section have corresponding evidence paths.
- Exits non-zero on extraction failure or missing evidence.

Update `justfile` to call the Python scripts; add a CI step in `release.yml` that runs both checks.

### GATE-02 — CI validation of feature_status() evidence

Add CI job `validate-feature-status` that:

- For every row where `evidence` is not null, verifies the file/path exists in the repository.
- For rows that reference test files, verifies the test appears in the pg_regress test list.
- Exits non-zero on any missing evidence file or test reference.

Remove all evidence paths from `feature_status.rs` that cite absent files. Update statuses from `implemented` to `experimental` or `planned` where evidence is missing or contradicted.

## Conflicts and risks

- The Critical remediation phase (MJOURNAL, RLS, FLIGHT-SEC, GATE) must complete before the production evidence phase begins. Running soak tests or security audits against a system with CF-1 through CF-4 open would produce misleading artifacts.
- The mutation journal (MJOURNAL-01) is an XL item. If the journal introduces measurable overhead, the HTAP merge benchmark baselines may need a separate pre-journal baseline to isolate regression.
- Long soak tests are expensive and may not fit normal PR CI. Run them nightly or before release candidates, but make the artifact required for v1.0.0.
- Third-party audit scheduling can slip. An internal threat model is an acceptable fallback only if findings are tracked and P0/P1 issues close.
- Public benchmarks can reveal regressions. Treat that as a feature: benchmark truth is part of release trust.
- Some Citus evidence requires multi-node infrastructure. Mark unavailable infrastructure explicitly instead of silently skipping.

## Exit criteria

### Critical remediation
- Storage mutation journal is in place; all write paths call it; no direct storage call bypasses CWB.
- Graph RLS is applied to every VP relation at creation and promotion; promoted-predicate RLS tests pass.
- Arrow Flight rejects unsigned tickets by default; tombstoned rows are absent from exports.
- Release-truth scripts are fail-closed and version-argument required; both pass locally on macOS and in CI.
- `feature_status()` evidence paths are all present; CI job validates them.
- SBOM is regenerated on release and version-checked; `sbom.json` agrees with `Cargo.toml`.
- Benchmark workflow does not invoke `.sql` as a shell script; parse failure fails CI.
- `construct_rules.rs` contains no `panic!`; cycle error returns a PostgreSQL recoverable error.

### Production evidence
- 72-hour soak artifact is published; 30-day target plan is documented.
- Security audit or threat model has no open P0/P1 findings.
- Public benchmark baselines and raw artifacts are published.
- Upgrade, dump/restore, and backup/restore acceptance tests pass.
- Release evidence dashboard is mandatory for v1.0.0 release candidates.
- API and docs freeze checklist is complete.
