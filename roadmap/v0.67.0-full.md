## Release theme

v0.67.0 is not a feature sprint. It is the proof package for a production release: stress evidence, security evidence, benchmark evidence, upgrade evidence, and operator evidence. It converts the repaired v0.64-v0.66 roadmap into the artifacts that make v1.0.0 credible.

## Deliverables

### PROD-01 - Soak testing

Run a mixed workload for at least 72 hours, with a documented 30-day continuous-merge target plan.

Workload must include:

- concurrent SPARQL SELECT/CONSTRUCT/ASK/DESCRIBE
- SPARQL Update and direct SQL inserts/deletes
- HTAP merge cycles
- construct-rule incremental maintenance
- Datalog materialization and deletion retraction
- SHACL validation
- HTTP streaming and Arrow export
- Citus sharded query path when infrastructure is available

Artifacts:

- memory usage trend
- merge latency and throughput
- query p50/p95/p99 latency
- error and warning counts
- feature-status snapshots
- crash/restart recovery logs

### PROD-02 - Security audit or threat model closure

Complete one of:

- third-party security audit with tracked findings, or
- internal threat model reviewed by maintainers with all P0/P1 findings closed.

Minimum scope:

- SPI query generation and dynamic SQL boundaries
- Arrow Flight ticket signing and replay protection
- HTTP auth, body-size, CORS, and error redaction
- federation SSRF and CA-bundle behavior
- GitHub Actions and release supply chain
- container hardening
- graph RLS and Citus propagation
- shared memory/cache race conditions

### PROD-03 - Public benchmark baselines

Publish benchmark results and scripts for:

- BSBM
- WatDiv
- LUBM
- W3C/Jena/OWL 2 RL pass rates
- bulk N-Triples/Turtle load
- HTAP merge throughput
- construct-rule incremental maintenance
- Datalog materialization and DRed deletion
- vector hybrid search
- Arrow IPC export
- Citus shard pruning and fan-out streaming

Artifacts must include hardware, PostgreSQL settings, dataset size, command line, raw output, and summary tables.

### PROD-04 - Upgrade and backup acceptance

Add tests for:

- migration chain from supported 0.x versions to current
- `pg_dump`/restore round trip
- physical backup/restore smoke test where practical
- extension update with existing construct rules, provenance, Citus catalogs, feature-status state, and materialized derived triples
- rollback guidance for failed migration attempts

### PROD-05 - Mandatory release evidence dashboard

The evidence dashboard introduced in v0.64.0 becomes required for every release candidate.

Release cannot be tagged unless evidence includes:

- migration chain result
- conformance pass rates
- benchmark deltas
- fuzz status
- action pinning status
- docs/API drift status
- Docker digest and Trivy status
- feature-status snapshot
- soak-test summary

### PROD-06 - API and documentation freeze

Before v1.0.0:

- Public `pg_ripple.*` SQL APIs are inventoried and classified stable/experimental/deprecated.
- `_pg_ripple.*` internals are documented as reserved.
- All docs examples are tested or explicitly marked illustrative.
- Known limitations are visible from README, docs site, and `feature_status()`.
- Changelog dates and version narratives are corrected.

### PROD-07 - Operator runbooks

Add runbooks for:

- construct-rule maintenance failures
- stale feature status or degraded `/ready`
- merge and BRIN summarise failures
- Citus worker lag or RLS propagation failures
- Arrow ticket validation failures
- streaming memory/backpressure issues
- federation endpoint failures
- backup/restore and upgrade recovery

### PROD-08 - Conformance threshold report

Produce a release report that lists:

- required suites and pass thresholds
- informational suites and current pass rates
- unsupported SPARQL/SHACL/OWL features
- accepted deviations with rationale
- tests skipped due to missing optional dependencies

## Conflicts and risks

- Long soak tests are expensive and may not fit normal PR CI. Run them nightly or before release candidates, but make the artifact required for v1.0.0.
- Third-party audit scheduling can slip. An internal threat model is an acceptable fallback only if findings are tracked and P0/P1 issues close.
- Public benchmarks can reveal regressions. Treat that as a feature: benchmark truth is part of release trust.
- Some Citus evidence requires multi-node infrastructure. Mark unavailable infrastructure explicitly instead of silently skipping.

## Exit criteria

- 72-hour soak artifact is published; 30-day target plan is documented.
- Security audit or threat model has no open P0/P1 findings.
- Public benchmark baselines and raw artifacts are published.
- Upgrade, dump/restore, and backup/restore acceptance tests pass.
- Release evidence dashboard is mandatory for v1.0.0 release candidates.
- API and docs freeze checklist is complete.
