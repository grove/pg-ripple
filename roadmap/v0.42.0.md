# v0.42.0 — Parallel Merge, Cost-Based Federation & Live CDC

**Effort estimate: 9–11 person-weeks**

**Goal:** Three infrastructure capabilities that collectively close the gap between pg_ripple and production data platform requirements. Parallel merge workers prevent write stalls on datasets with hundreds of predicates. Cost-based federation selects which remote SPARQL endpoints can actually answer each part of a query, drastically reducing wasted remote calls. Live CDC subscriptions let applications receive pushed notifications when specific triple patterns change.

---

## Multi-Worker Parallel HTAP Merge

**Datasets with hundreds of predicates merge all partitions simultaneously.**

The HTAP background merge worker previously ran a single goroutine that merged one predicate's delta partition at a time. For a dataset with 200 predicates, a write-heavy workload could leave some predicates with a large delta backlog while the worker was busy merging others. This release introduces a configurable pool of up to 16 merge workers (`pg_ripple.merge_workers` GUC, default 1). Each predicate is assigned to one worker using consistent hashing, and idle workers steal work from busy workers. The result is that all predicates merge at roughly the same rate regardless of how many there are.

---

## VoID Statistics and Cost-Based Source Selection

**Federation SERVICE queries no longer broadcast to endpoints that lack the relevant data.**

Currently, when a SPARQL query contains a `SERVICE` clause pointing to multiple registered endpoints, the federation layer sends the query to all of them and discards empty results. For large federated setups, this wastes latency and endpoint capacity. This release adds a VoID (Vocabulary of Interlinked Datasets) statistics catalog. When federation endpoints are registered, their VoID description is fetched and stored in `_pg_ripple.void_stats`. The federation planner uses these statistics to predict which endpoints hold each predicate and skips endpoints that definitely cannot contribute to the result. The FedX source-selection algorithm is the basis for this optimisation.

---

## Parallel SERVICE Execution with Streaming

**Multiple federation endpoints are queried concurrently; results stream back as they arrive.**

Where multiple endpoints might contribute results to a single SERVICE clause, this release queries all candidate endpoints in parallel (up to `pg_ripple.federation_parallel_calls` concurrent requests) and streams results back as each endpoint responds. The first results are returned to the client before the last endpoint has finished. Combined with the SSRF-prevention IP/CIDR allowlist (which existed since v0.16.0 but is now enforced more consistently), federation is both faster and safer.

---

## Live RDF CDC Subscriptions

**Applications receive a push notification whenever specific triples change.**

`pg_ripple.create_subscription(name, filter_sparql, filter_shape)` creates a live subscription. Any INSERT or DELETE of triples matching the filter SPARQL pattern or failing the filter SHACL shape triggers an event. The `pg_ripple_http` companion service exposes subscriptions as WebSocket connections: clients that open `GET /ws/subscriptions/{name}` receive a JSON notification for each matching change, including the changed triples and the operation type. The subscription catalog is stored in `_pg_ripple.subscriptions`. Subscriptions survive server restart.
