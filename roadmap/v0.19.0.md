# v0.19.0 — Federation Performance

**Effort estimate: 4–5 person-weeks**

**Goal:** The federation feature introduced in v0.16.0 works correctly, but it is not yet optimised for production use at scale. This release makes federated queries significantly faster and more robust through connection pooling to remote endpoints, result caching, smarter variable projection, and adaptive handling of partial or slow responses.

---

## Connection Pooling for Remote SERVICE Calls

**Reuse HTTP connections to the same remote endpoint across multiple queries.**

Establishing an HTTP connection has a measurable cost: DNS resolution, TCP handshake, potentially TLS negotiation. For an application that runs many federated queries against the same remote SPARQL endpoint, paying that cost on every query adds up quickly. This release introduces a connection pool per registered endpoint, so the HTTP connection is established once and reused for subsequent queries. The pool size and idle timeout are configurable.

---

## Federation Result Caching with TTL

**Repeated identical SERVICE calls return cached results instead of hitting the remote.**

For remote data that changes infrequently (ontologies, reference datasets, regulatory lookups), there is no need to re-fetch the same result on every query. A configurable time-to-live cache stores the results of SERVICE calls keyed on the endpoint URL, the SPARQL sub-query, and the current variable bindings. Cache entries expire after a configurable TTL (default: 60 seconds). The cache is particularly valuable for reference data endpoints that might have strict rate limits.

---

## Variable Projection Reduces Data Transferred

**Only the variables the query actually uses are requested from the remote endpoint.**

A naive SERVICE implementation fetches all result variables from the remote endpoint and discards the ones not needed by the outer query. This wastes both network bandwidth and remote server processing. The projection optimisation analyses which variables from the SERVICE result are actually used by the rest of the query and adds a `SELECT ?var1 ?var2` projection to the sub-query sent to the remote endpoint, so only the needed columns travel across the network.

---

## Batch Multiple SERVICE Calls to the Same Endpoint

**When the same endpoint appears multiple times, calls are merged into one.**

A query with three separate SERVICE clauses pointing to the same endpoint can be rewritten to make a single HTTP request that fetches all three patterns in one SPARQL query, rather than three separate round-trips. The results are then split and distributed to each SERVICE clause's variable bindings. This reduces latency, server load, and the number of HTTP connections needed.

---

## Adaptive Timeout and Partial Result Handling

**The engine learns from experience how long each endpoint typically takes.**

Each registered endpoint's response-time history is tracked. If an endpoint consistently responds in 50 milliseconds, the timeout for that endpoint is calibrated accordingly. For endpoints that return partial results before timing out, the engine can extract whatever complete result rows arrived before the cutoff rather than discarding everything. This makes queries resilient to occasionally slow remote services without sacrificing correctness for complete results.
