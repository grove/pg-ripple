## Schema Optimization: Storage, Performance, and Scalability

### DICT-01 — `dictionary.hash BYTEA` → `(hash_hi BIGINT, hash_lo BIGINT)` (perf)

**Problem**: The XXH3-128 hash is stored as `BYTEA`, which incurs a 4-byte varlena header and heap detoast on every lookup. Since every SPARQL query touches the dictionary for every unique term, this overhead compounds across billions of lookups.

**Deliverable**:
1. Replace `hash BYTEA NOT NULL` with `hash_hi BIGINT NOT NULL, hash_lo BIGINT NOT NULL` in `_pg_ripple.dictionary` and `_pg_ripple.dictionary_hot`.
2. Replace the unique index on `hash` with `UNIQUE (hash_hi, hash_lo)`.
3. Update the Rust `dictionary::encode()` path to split the u128 into (hi, lo) before inserting.
4. Update all lookup queries to filter on `(hash_hi, hash_lo)` pair.

**Affected files**: `src/dictionary/mod.rs`, `src/dictionary/encode.rs`, migration SQL.

### DICT-02 — Vertical-partition nullable `dictionary` columns (storage)

**Problem**: `_pg_ripple.dictionary` stores `datatype TEXT`, `lang TEXT` (only set for literals), and `qt_s BIGINT`, `qt_p BIGINT`, `qt_o BIGINT` (only set for RDF-star quoted triples) on every row. For a system where the vast majority of dictionary entries are plain IRIs, these 5 mostly-NULL columns add dead weight to every heap tuple and every index scan. PostgreSQL still allocates the null bitmap and padding for nullable columns even when NULL.

**Deliverable**:
1. Create `_pg_ripple.dictionary_literals (id BIGINT PRIMARY KEY REFERENCES dictionary(id), datatype TEXT, lang TEXT)`.
2. Create `_pg_ripple.dictionary_quoted (id BIGINT PRIMARY KEY REFERENCES dictionary(id), qt_s BIGINT NOT NULL, qt_p BIGINT NOT NULL, qt_o BIGINT NOT NULL)`.
3. Drop `datatype`, `lang`, `qt_s`, `qt_p`, `qt_o` from the main `dictionary` table.
4. Update all Rust paths that write/read these columns to use the satellite tables.
5. Add indexes on `dictionary_quoted(qt_s)`, `(qt_p)`, `(qt_o)` for RDF-star subject/object lookups.

**Affected files**: `src/dictionary/mod.rs`, `src/sparql/decode.rs`, migration SQL.

### DICT-03 — `dictionary.access_count` → separate table or sampled update (perf)

**Problem**: `access_count BIGINT NOT NULL DEFAULT 0` on `_pg_ripple.dictionary` causes every dictionary read to generate a write, bloating WAL and causing heap page splits. This converts a read-heavy workload into a write-heavy one on the hottest table in the system.

**Deliverable**:
1. Move `access_count` to `_pg_ripple.dictionary_access_counts (id BIGINT PRIMARY KEY, access_count BIGINT NOT NULL DEFAULT 0)`.
2. Alternatively: only update the count on 1-in-N reads (GUC `pg_ripple.access_count_sample_rate`, default 1/100) to amortize the write cost.
3. Mark `dictionary_access_counts` as `UNLOGGED` if exact durability is not required.

**Affected files**: `src/dictionary/mod.rs`, migration SQL.

### REDUNDANT-01 — Drop `extvp_tables.pred1_iri TEXT` and `pred2_iri TEXT` (storage)

**Problem**: `_pg_ripple.extvp_tables` stores both `pred1_iri TEXT`, `pred2_iri TEXT` AND `pred1_id BIGINT`, `pred2_id BIGINT`. The IRI strings are fully derivable via `SELECT value FROM _pg_ripple.dictionary WHERE id = pred1_id`. The string copies serve no indexed purpose.

**Deliverable**:
1. Drop `pred1_iri TEXT` and `pred2_iri TEXT` from `_pg_ripple.extvp_tables`.
2. Update any display paths to decode via the dictionary on read.

**Affected files**: migration SQL, any Rust code reading `pred1_iri`/`pred2_iri` for display.

### ENUM-01 — `graph_access.permission TEXT` in composite PK → `SMALLINT` (perf)

**Problem**: `permission TEXT NOT NULL CHECK (permission IN ('read', 'write', 'admin'))` is part of the composite PK `(role_name, graph_id, permission)`. Every RLS check joins this table. Storing and comparing a string in a primary key used on every query costs more than a 2-byte integer.

**Deliverable**:
1. Replace `permission TEXT` with `permission SMALLINT NOT NULL CHECK (permission IN (1, 2, 3))` with constants `1=read, 2=write, 3=admin`.
2. Add a view `pg_ripple.graph_access_decoded` that exposes human-readable names.
3. Update all Rust code in `src/admin/` and RLS paths.

**Affected files**: `src/admin/`, migration SQL.

### ENUM-02 — `federation_endpoints.complexity TEXT` → `SMALLINT` (perf)

**Problem**: `complexity TEXT NOT NULL DEFAULT 'normal' CHECK (complexity IN ('fast', 'normal', 'slow'))` is read by the federation planner on every multi-endpoint query to order execution. Three-value enum stored as TEXT.

**Deliverable**:
1. Replace with `complexity SMALLINT NOT NULL DEFAULT 2 CHECK (complexity IN (1, 2, 3))` (`1=fast`, `2=normal`, `3=slow`).
2. Update federation planner in `src/sparql/execute.rs`.

**Affected files**: `src/sparql/execute.rs`, migration SQL.

### JSON-01 — `endpoint_stats.predicate_stats_json TEXT` → `JSONB` (perf + scalability)

**Problem**: VoID predicate statistics used by the cost-based federation planner are stored as `TEXT NOT NULL DEFAULT '{}'`. Every planner access requires a `json_parse()` round-trip. `JSONB` is pre-parsed, binary-stored, and supports GIN indexing and `@>` containment queries natively.

**Deliverable**:
1. Change `predicate_stats_json TEXT` to `predicate_stats JSONB NOT NULL DEFAULT '{}'`.
2. Add a `GIN` index on `predicate_stats` for predicate-level containment queries.
3. Update all Rust reads/writes.

**Affected files**: `src/sparql/execute.rs` (federation planner), migration SQL.

### HASH-01 — `rag_cache` hash columns TEXT → `BYTEA(16)` (storage)

**Problem**: `rag_cache` primary key is `(question_hash TEXT, k INT, schema_digest TEXT)`. Both hash columns store hex-encoded fingerprints (~32 chars each). Storing raw bytes halves the key size and corresponding index leaf size.

**Deliverable**:
1. Change `question_hash TEXT` → `BYTEA NOT NULL` and `schema_digest TEXT` → `BYTEA NOT NULL`.
2. Update Rust RAG cache lookup to encode hashes as bytes.
3. If `result TEXT` stores structured JSON output, change to `result JSONB`.

**Affected files**: `src/sparql/` (RAG cache path), migration SQL.

**Affected files**: `src/shacl/`, migration SQL.

### IRI-02 — `shacl_dag_monitors.shape_iri TEXT PRIMARY KEY` → `shape_id BIGINT` (consistency)

**Problem**: Same pattern as IRI-01. `shacl_dag_monitors` tracks which shapes have been compiled into pg_trickle stream tables, keyed by raw IRI string.

**Deliverable**:

**Affected files**: `src/shacl/`, migration SQL.

### IRI-03 — `prov_catalog.activity_iri TEXT` → `activity_id BIGINT` (consistency)

**Problem**: `_pg_ripple.prov_catalog.activity_iri TEXT` stores a PROV-O activity IRI as raw string. This IRI is inserted into the dictionary during bulk load provenance recording. The table is inconsistent with the rest of the system.

**Deliverable**:
1. Add `activity_id BIGINT` column populated by `dictionary::encode(activity_iri, KIND_IRI)`.
2. Keep `activity_iri TEXT` as a human-readable display column.

**Affected files**: `src/export/` (prov recording path), migration SQL.

### PART-01 — `federation_health` range-partition on `probed_at` (scalability)

**Problem**: `federation_health` is written after every `SERVICE` call and grows unboundedly. `is_endpoint_healthy()` only needs recent rows (last N hours). Without partitioning, the index and heap bloat indefinitely.

**Deliverable**:
1. Convert `federation_health` to a range-partitioned table on `probed_at` (weekly or monthly partitions).
2. Add a background worker job (or pg_cron entry) to detach and drop partitions older than the retention window (`pg_ripple.federation_health_retention`, default `7 days`).

**Affected files**: migration SQL, `src/storage/` (merge worker or new cleanup function).

### PART-02 — `audit_log` range-partition on `ts` (scalability)

**Problem**: `audit_log` is append-only and grows unboundedly for high-volume UPDATE/DELETE workloads. Old audit rows are rarely queried but consume the same index space as recent rows.

**Deliverable**:
1. Convert `audit_log` to a range-partitioned table on `ts` (monthly partitions).
2. Expose `pg_ripple.archive_audit_log(before TIMESTAMPTZ)` to detach old partitions.

**Affected files**: migration SQL.

### PART-03 — `statement_id_timeline` → range-mapping table (storage, scalability)

**Problem**: `statement_id_timeline` stores one row per statement ID, mapping `sid → inserted_at`. At billions of triples this table itself becomes enormous. Since SIDs are monotonically increasing and timestamps are tightly correlated (SIDs from a single load batch have nearly identical timestamps), the per-row granularity is wasteful.

**Deliverable**:
1. Replace `statement_id_timeline (sid BIGINT PK, inserted_at TIMESTAMPTZ)` with `statement_id_timeline (sid_min BIGINT, sid_max BIGINT, ts_min TIMESTAMPTZ, ts_max TIMESTAMPTZ, PRIMARY KEY (sid_min))` — one row per bulk-load batch.
2. Update `point_in_time()` query to use range containment: `WHERE $ts BETWEEN ts_min AND ts_max` or binary search on `sid_min`.
3. This mirrors the existing `_pg_ripple.statements` catalog pattern.

**Affected files**: `src/sparql/` (temporal query path), migration SQL.

### PART-04 — `rule_firing_log` range-partition on `fired_at` (scalability)

**Problem**: `rule_firing_log` is written during every Datalog inference run. `explain_inference()` only walks recent derivation chains. Without partitioning, rows accumulate across all inference runs ever executed.

**Deliverable**:
1. Convert `rule_firing_log` to a range-partitioned table on `fired_at` (weekly partitions).
2. Auto-drop partitions outside the `explain_inference()` lookback window (`pg_ripple.inference_log_retention`, default `30 days`).

**Affected files**: migration SQL.

### UNLOGGED-01–03 — Mark transient queue tables `UNLOGGED` (perf)

**Problem**: Three tables are work queues where data loss on crash is acceptable (the producer re-populates on restart) but WAL is currently written for every insert/delete:
- `validation_queue` (async SHACL validator work items)
- `embedding_queue` (entities awaiting embedding)
- `federation_health` (best-effort probe log)

**Deliverable**:
1. `ALTER TABLE _pg_ripple.validation_queue SET UNLOGGED`.
2. `ALTER TABLE _pg_ripple.embedding_queue SET UNLOGGED`.
3. `ALTER TABLE _pg_ripple.federation_health SET UNLOGGED` (also covered by PART-01).
4. Verify crash-recovery code re-enqueues appropriately in each case.

**Affected files**: migration SQL, startup/recovery paths for each subsystem.

### IDX-01 — Add `statements(predicate_id)` index (perf)

**Problem**: `_pg_ripple.statements (sid_min, sid_max, predicate_id, table_oid)` with `PRIMARY KEY (sid_min)` has no index on `predicate_id`. Finding all SID ranges for a given predicate (used by temporal queries and the merge worker) requires a full sequential scan.

**Deliverable**:
1. `CREATE INDEX idx_statements_predicate ON _pg_ripple.statements (predicate_id)`.

**Affected files**: migration SQL.

### TOAST-01 — `lz4` compression on large TEXT columns (storage)

**Problem**: `dictionary.value` is already configured for `lz4` compression. These high-cardinality or large TEXT columns are not:
- `rules.rule_text` — complex Datalog programs can be kilobytes
- `audit_log.query` — full SPARQL UPDATE text per event
- `replication_status.batch_data` — N-Triples batch payloads (can be megabytes)
- `sparql_views.generated_sql`, `construct_views.generated_sql`, `datalog_views.generated_sql`, `describe_views.generated_sql`, `ask_views.generated_sql`
- `framing_views.generated_construct`

**Deliverable**:
1. `ALTER TABLE _pg_ripple.rules ALTER COLUMN rule_text SET COMPRESSION lz4`.
2. Same for each column listed above.
3. Wrap in the same `EXCEPTION WHEN OTHERS` guard used for `dictionary.value` to handle builds without lz4.

**Affected files**: migration SQL.

### FILL-01 — `FILLFACTOR` tuning for hot-update tables (perf)

**Problem**: Tables with frequently-updated metric or counter columns default to `FILLFACTOR=100`, which prevents HOT (Heap-Only Tuple) updates and causes index bloat when counters are updated.

**Deliverable**:
1. `ALTER TABLE _pg_ripple.construct_rules SET (fillfactor = 70)` — `successful_run_count`, `failed_run_count`, `derived_triple_count`, `last_incremental_run`, `last_error` updated on every rule run.
2. `ALTER TABLE _pg_ripple.predicates SET (fillfactor = 70)` — `triple_count` and `brin_summarize_failures` updated on every insert/merge.
3. `ALTER TABLE _pg_ripple.dictionary SET (fillfactor = 80)` — `access_count` updated on reads (if not moved per DICT-03).
4. `ALTER TABLE _pg_ripple.federation_endpoints SET (fillfactor = 90)` — `enabled` toggled occasionally.

**Affected files**: migration SQL.
