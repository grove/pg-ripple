## Worst-Case Optimal Joins via Leapfrog Triejoin

**Cyclic graph patterns that previously timed out now run in milliseconds.**

Standard database join algorithms (hash join, merge join, nested loop) perform poorly on cyclic query patterns — queries where three or more triple patterns form a loop of shared variables. For example, "find all triangles in a social graph" (Alice knows Bob, Bob knows Carol, Carol knows Alice) can produce enormous intermediate results even when the final answer is small. The Leapfrog Triejoin algorithm (Ngo et al., 2012) avoids these intermediate blowups by intersecting sorted lists simultaneously across all join partners rather than joining pairs one by one. The result is a provably optimal algorithm for any join pattern, cyclic or not.

## Automatic Detection of Cyclic Patterns

**The SPARQL engine automatically uses the Leapfrog algorithm when it would help.**

The SPARQL-to-SQL translator detects cyclic join patterns at query translation time — any SELECT with three or more triple patterns where the variable bindings form a cycle. For detected cycles, the translation routes execution through the Leapfrog scan node rather than standard PostgreSQL hash joins. The `pg_ripple.wcoj_enabled` GUC allows the optimisation to be turned off for debugging. A benchmark running the triangle-detection query on a 1M-edge social graph shows 10× to over 100× speedup compared to the standard planner.

## Lattice-Based Datalog

**Rules can now propagate typed values that combine according to custom algebraic rules.**

Standard Datalog can only derive boolean facts (something is true or it isn't). Lattice-based Datalog extends this to typed lattice values: the "join" of two values is defined by the lattice's algebra rather than by boolean OR. The built-in lattice types include `MinLattice` (combine by taking the minimum — useful for trust propagation where trust is the weakest link in a chain), `MaxLattice` (combine by maximum), `SetLattice` (combine by union), and `IntervalLattice` (combine by interval hull). You can also define custom lattice types by providing a PostgreSQL aggregate function as the join operation.

---

## Trust Propagation Example

**"My trust in X through Y is the minimum of my trust in Y and Y's trust in X."**

The classic lattice Datalog use case is trust propagation in a network: if I trust Alice 0.9 and Alice trusts Bob 0.7, my effective trust in Bob through Alice is the minimum of those scores (0.7 — the weakest link). A single lattice Datalog rule expresses this and converges to a fixed point that propagates trust scores through the entire network. This pattern applies equally to supply chain reliability, system availability in dependency graphs, and confidence scores in entity resolution.
