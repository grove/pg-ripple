# v0.64.0 - Release Truth and Safety Freeze

> **Full technical details:** [v0.64.0-full.md](v0.64.0-full.md)

**Status: Planned** | **Scope: Large**

> v0.64.0 turns the PLAN_OVERALL_ASSESSMENT_8 findings into an explicit release-truth gate. It does not add headline query features; it makes shipped capabilities honest, observable, and safe to release by fixing supply-chain pinning, release workflow integrity, feature-status reporting, and documentation drift.

---

## What is this?

The v0.62.0 and v0.63.0 releases introduced ambitious claims around Arrow Flight, WCOJ, SHACL-SPARQL rules, Citus pushdown, and CONSTRUCT writeback. The assessment found that several of those surfaces are real but partial, stubbed, or not wired into the normal execution path. v0.64.0 creates the guardrails that prevent that drift from recurring.

## Headline outcomes

- Every roadmap and changelog item has a machine-checkable implementation status.
- `pg_ripple.feature_status()` and `pg_ripple_http /ready` expose partial, degraded, unavailable, and CI-gated feature states.
- GitHub Actions are pinned to immutable commit SHAs and a CI lint rejects mutable action refs.
- Docker releases fail on build/push errors and scan the immutable digest that was just published.
- README, docs, changelog, and implementation-plan version claims are corrected for v0.63.0 reality.

## Why this comes before more features

Operators can tolerate known limitations. They cannot safely tolerate limitations that are documented as complete features. This release makes pg_ripple's public contract match the running code before the project proceeds to deeper correctness and scalability work.

## Assessment findings covered

- C3: GitHub Actions SHA pinning is claimed but not implemented.
- H4: SHACL-SPARQL rules are documented as supported but are not executed.
- H5: Docker build/push can fail while release workflow continues.
- SC3: changelog dates, README version, and release narratives drift.
- SC4/H1: WCOJ docs describe Leapfrog Triejoin and nonexistent plan nodes.
- A1/A3/A4: release claims need acceptance gates, degradation semantics, and no dead/unwired helpers presented as complete capabilities.
- Documentation and developer-experience gaps: README current-version drift, pgrx 0.17 references, docs/API signature drift, `just assess-release`, docs example tests, release evidence dashboard foundation.
