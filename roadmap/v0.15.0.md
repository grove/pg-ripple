# v0.15.0 — SPARQL Protocol (HTTP Endpoint)

**Effort estimate: 4–6 person-weeks**

**Goal:** pg_ripple has been accessible only through PostgreSQL connections. This release launches the `pg_ripple_http` companion service — a standalone HTTP server that exposes pg_ripple's capabilities through the W3C SPARQL 1.1 Protocol. Any tool, library, or application that speaks standard SPARQL HTTP can now query your knowledge graph without a PostgreSQL driver.

---

## W3C SPARQL 1.1 Protocol Endpoint

**Standard `GET /sparql` and `POST /sparql` endpoints serve any SPARQL client.**

The W3C SPARQL 1.1 Protocol defines how SPARQL clients should communicate with a triplestore over HTTP. The `pg_ripple_http` service implements both the GET and POST variants: GET for short queries embedded in the URL query string, POST for longer queries in the request body or as URL-encoded form parameters. Any tool that works with standard SPARQL endpoints — Apache Jena, RDF4J, SPARQLWrapper for Python, any triplestore explorer — works out of the box with pg_ripple_http.

---

## Content Negotiation

**The server returns results in the format the client requests.**

Different clients prefer different result formats: curl users might want CSV, a JavaScript frontend might want JSON, a reasoning tool might want Turtle or JSON-LD. The HTTP service implements standard content negotiation via the Accept header: `application/sparql-results+json` for SELECT/ASK results in JSON format, `application/sparql-results+xml` for XML format, `text/csv` for CSV, `text/turtle` for CONSTRUCT/DESCRIBE results in Turtle, and `application/ld+json` for JSON-LD.

---

## Connection Pooling

**A pool of PostgreSQL connections is shared across all HTTP clients.**

Each HTTP request needs a PostgreSQL connection to execute its SPARQL query. Rather than opening a fresh connection for every request (which has significant overhead), `pg_ripple_http` maintains a pool of connections to the PostgreSQL backend. The pool size is configurable and adapts to load. This makes the HTTP service suitable for API workloads with many concurrent clients.

---

## Authentication

**HTTP Basic and Bearer token authentication protect the endpoint.**

The service supports both HTTP Basic authentication (username and password in the Authorization header) and Bearer token authentication (a pre-shared token for application-to-application use cases). Unauthenticated requests are rejected with a 401 response. The credentials are validated against a PostgreSQL role or a configured token — no additional authentication service is required.

---

## Prometheus Metrics and Docker Integration

**Monitor the HTTP service with a standard metrics endpoint.**

A `/metrics` endpoint exposes query counts, latency histograms, connection pool utilisation, and error rates in Prometheus format. These metrics integrate directly with Grafana dashboards and alerting. The service is also packaged as a Docker image so it can be deployed alongside a PostgreSQL container using Docker Compose or Kubernetes, with configuration supplied through environment variables.
