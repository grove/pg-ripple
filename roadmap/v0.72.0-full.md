## Release theme

v0.72.0 is an architecture depth and protocol hardening release. The headlining item is XACT-01: registering a proper PostgreSQL transaction/subtransaction callback so the mutation journal does not fire CWB for triples whose storage writes were subsequently rolled back. The remaining items harden conformance, observability, and protocol correctness.

## Deliverables

### XACT-01 — Mutation journal SAVEPOINT safety

**Root cause**: [src/storage/mutation_journal.rs](../src/storage/mutation_journal.rs) uses `thread_local! RefCell<Vec<JournalEntry>>` cleared by `flush()`. If a subtransaction is rolled back, already-flushed CWB invocations have written derived triples against now-absent source rows.

**Fix**:

1. In `_PG_init` (or `mutation_journal::init()` called from `_PG_init`), register:
   ```rust
   pgrx::pg_sys::RegisterSubXactCallback(Some(sub_xact_callback), std::ptr::null_mut());
   pgrx::pg_sys::RegisterXactCallback(Some(xact_callback), std::ptr::null_mut());
   ```
2. `sub_xact_callback`: on `PG_SUBXACT_EVENT_ABORT_SUB`, clear the thread-local journal.
3. `xact_callback`: on `XACT_EVENT_PRE_COMMIT`, call `mutation_journal::flush()` (the deferred flush introduced in v0.70.0 FLUSH-01); on `XACT_EVENT_ABORT`, clear the journal.
4. Remove the `RegisterXactCallback` workaround from v0.70.0 FLUSH-01 if it uses a simpler mechanism; replace with the canonical callback approach.

**Test**: Add `tests/pg_regress/sql/cwb_savepoint_rollback.sql`:
- Register a CONSTRUCT rule.
- `BEGIN; SAVEPOINT s1; INSERT triple into source graph; ROLLBACK TO SAVEPOINT s1; COMMIT;`
- Assert derived target graph is empty (no phantom derived triples from the rolled-back source).

### FLIGHT-NONCE-01 — Arrow Flight replay protection

**Root cause**: Assessment 10 MF-19 noted that a signed ticket can be replayed within its expiry window.

**Fix**:

1. In `pg_ripple_http/src/arrow_encode.rs`, maintain a `DashMap<String, Instant>` (nonce → accepted-at) keyed by the ticket `nonce` field.
2. On ticket validation, check if `nonce` is in the map. If present and within the expiry window, reject with HTTP 401 and `arrow_ticket_rejections` metric increment.
3. Evict entries where `accepted-at + expiry_secs` has elapsed (lazy eviction on insertion).
4. Cap map size at `max_nonce_cache` entries (configurable, default 10,000).

**Test**: Add HTTP integration test that presents the same ticket twice within the expiry window and asserts the second response is 401.
