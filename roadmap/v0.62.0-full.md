# v0.62.0 — Query Frontier — Full Details

**Theme:** Cypher/GQL transpiler, Arrow Flight, WCOJ planner integration, visual graph explorer.
**Scope:** Very Large  
**Prerequisite:** v0.61.0

---

## Why this release

v0.62.0 opens pg_ripple to the much larger property-graph community (Cypher /
GQL), provides a high-throughput bulk-export path for data-pipeline consumers
(Arrow Flight), and finally wires the existing WCOJ infrastructure into the
live SPARQL planner — delivering the 10–100× cyclic-join speedup that has been
latent since v0.36.0.

---

## Deliverables

### Cypher / openCypher / GQL read-only transpiler

- **[6.1] `pg_ripple.cypher(query TEXT)`** — New SRF executing an openCypher
  read-only query (`MATCH … WHERE … RETURN`). Implementation path:
  1. Parse Cypher AST using the `cypher-parser` or `gql-parser` crate.
  2. Translate `MATCH` node/relationship patterns to SPARQL BGPs:
     - Cypher labels → `rdf:type` triple patterns
     - Relationship types → predicate triple patterns
     - Properties → SPARQL FILTER expressions
  3. Hand off the resulting SPARQL algebra to the existing
     `src/sparql/translate/` pipeline.
  4. Map result columns back from integer IDs to string values.
  - New module `src/cypher/` with `parser.rs`, `translator.rs`.
  - New SQL function `pg_ripple.cypher(query TEXT) → SETOF RECORD`.
  - New pg_regress test `tests/pg_regress/sql/cypher_match.sql` proving
    shape parity with `sparql_basic.sql` for equivalent queries.
  - `WRITE` operations (`CREATE`, `SET`, `MERGE`, `DELETE`) are **out of scope**
    for v0.62.0 (deferred to v1.1.0).

### Apache Arrow Flight bulk export

- **[6.4 / L-6.1] `pg_ripple.export_arrow_flight(graph_iri TEXT)`** — New SQL
  function returning a Flight ticket (opaque BYTEA) that a client can redeem
  against the `pg_ripple_http` Arrow Flight endpoint (`/flight`). The endpoint
  streams VP rows as Arrow record batches at wire speed.
  - New crate `pg_ripple_flight` (thin wrapper around the existing HTTP runtime
    + `arrow2` crate).
  - New route `POST /flight/do_get` in `pg_ripple_http`.
  - Benchmark: assert ≥ 500k triples/second throughput on localhost.

### WCOJ planner integration

- **[A7-3 / B7-1 / E7-1] Wire `analyse_bgp` and `apply_wcoj_hints` into the
  live SPARQL → SQL translation path** — The WCOJ infrastructure in
  `src/sparql/wcoj.rs` has been dormant since v0.36.0. This release:
  1. Calls `analyse_bgp()` from the BGP-translation step in
     `src/sparql/translate/bgp.rs` for every multi-triple BGP.
  2. When `WcojAnalysis::is_cyclic` is true, injects the WCOJ schedule hints
     via `apply_wcoj_hints()` and emits the Leapfrog-Triejoin SQL preamble
     via `wcoj_session_preamble()`.
  3. Falls back to hash-join when `is_cyclic` is false (no change to existing
     behaviour for acyclic BGPs).
  - Remove all `#[allow(dead_code)]` annotations from `src/sparql/wcoj.rs`.
  - Add WatDiv cyclic-pattern regression test to verify speedup ≥ 5× vs.
    the hash-join baseline.
  - Apply the same wiring to the Datalog compiler in
    `src/datalog/compiler.rs` for cyclic rule bodies.

### Visual graph explorer

- **[6.9] `pg_ripple_http` `/explorer` route** — Embed a single-page application
  (sigma.js or d3-force; static assets via `rust-embed`) at `/explorer` that:
  - Accepts a SPARQL CONSTRUCT query via a text box.
  - Renders the resulting triples as a force-directed graph.
  - Allows clicking a node to expand its neighbourhood.
  - No authentication required for read access to the same graphs the API
    token can already query.

## Exit criteria

- `pg_ripple.cypher('MATCH (n:Person) RETURN n.name')` returns the same rows
  as the equivalent SPARQL query; verified by `cypher_match.sql` pg_regress.
- Arrow Flight export benchmarks ≥ 500k triples/second on localhost.
- WatDiv cyclic pattern queries show ≥ 5× improvement vs. hash-join baseline
  in the benchmark workflow.
- `/explorer` loads and renders a CONSTRUCT query result in a browser.
- No `#[allow(dead_code)]` annotations remaining in `src/sparql/wcoj.rs`.
- Migration script `sql/pg_ripple--0.61.0--0.62.0.sql` present.
- `pg_ripple.control` updated to `default_version = '0.62.0'`.
