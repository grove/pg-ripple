# v0.62.0 — Query Frontier — Full Details

**Theme:** Arrow Flight bulk export, WCOJ planner integration, visual graph explorer, Citus scalability (property-path push-down, vp_rare archival, tiered dictionary, distributed inference, live rebalance, multi-hop pruning).
**Scope:** Large  
**Prerequisite:** v0.61.0

---

## Why this release

v0.62.0 provides a high-throughput bulk-export path for data-pipeline consumers
(Arrow Flight), and finally wires the existing WCOJ infrastructure into the
live SPARQL planner — delivering the 10–100× cyclic-join speedup that has been
latent since v0.36.0. Cypher/GQL support (read-only transpiler and write operations)
is delivered in v1.1.0.

---

## Deliverables

### Apache Arrow Flight bulk export

- **[6.4 / L-6.1] `pg_ripple.export_arrow_flight(graph_iri TEXT)`** — New SQL
  function returning a Flight ticket (opaque BYTEA) that a client can redeem
  against the `pg_ripple_http` Arrow Flight endpoint (`/flight`). The endpoint
  streams VP rows as Arrow record batches at wire speed.
  - New crate `pg_ripple_flight` (thin wrapper around the existing HTTP runtime
    + `arrow2` crate).
  - New route `POST /flight/do_get` in `pg_ripple_http`.
  - Benchmark: assert ≥ 500k triples/second throughput on localhost.

### WCOJ planner integration

- **[A7-3 / B7-1 / E7-1] Wire `analyse_bgp` and `apply_wcoj_hints` into the
  live SPARQL → SQL translation path** — The WCOJ infrastructure in
  `src/sparql/wcoj.rs` has been dormant since v0.36.0. This release:
  1. Calls `analyse_bgp()` from the BGP-translation step in
     `src/sparql/translate/bgp.rs` for every multi-triple BGP.
  2. When `WcojAnalysis::is_cyclic` is true, injects the WCOJ schedule hints
     via `apply_wcoj_hints()` and emits the Leapfrog-Triejoin SQL preamble
     via `wcoj_session_preamble()`.
  3. Falls back to hash-join when `is_cyclic` is false (no change to existing
     behaviour for acyclic BGPs).
  - Remove all `#[allow(dead_code)]` annotations from `src/sparql/wcoj.rs`.
  - Add WatDiv cyclic-pattern regression test to verify speedup ≥ 5× vs.
    the hash-join baseline.
  - Apply the same wiring to the Datalog compiler in
    `src/datalog/compiler.rs` for cyclic rule bodies.

### Visual graph explorer

- **[6.9] `pg_ripple_http` `/explorer` route** — Embed a single-page application
  (sigma.js or d3-force; static assets via `rust-embed`) at `/explorer` that:
  - Accepts a SPARQL CONSTRUCT query via a text box.
  - Renders the resulting triples as a force-directed graph.
  - Allows clicking a node to expand its neighbourhood.
  - No authentication required for read access to the same graphs the API
    token can already query.

### Citus scalability

- **[CITUS-24] Property-path worker push-down** — In `src/sparql/translate/property_path.rs`,
  detect when a `+` / `*` / `{n,m}` property path has a bound start subject
  that shard-prunes to a single worker. Wrap the entire `WITH RECURSIVE` CTE
  in a call to Citus `run_command_on_workers(worker, sql)` so the recursion
  executes locally on that worker. Only the final result set (typically small)
  travels back to the coordinator. Falls back to coordinator execution when
  the start node is unbound or pruning fails. Extend `explain_sparql(citus := true)`
  to report `property_path_worker_pushdown: true/false`. Add pg_regress test
  `tests/pg_regress/sql/citus_property_path_pushdown.sql` (Citus-conditional).

- **[CITUS-25] `vp_rare` cold-entry archival** — New SQL function
  `pg_ripple.vacuum_vp_rare() RETURNS TABLE(predicate_id BIGINT, rows_removed BIGINT)`.
  Scans `_pg_ripple.vp_rare` for predicate IDs where `triple_count = 0` in
  `_pg_ripple.predicates` (or where the predicate no longer appears in
  `_pg_ripple.predicates` at all). For each such predicate, deletes all rows
  from `vp_rare` with that predicate and returns the count. Integrated into
  the existing `pg_ripple.vacuum()` flow when sharding is enabled. Adds a column
  `vp_rare_rows` to `pg_ripple.merge_status()` showing current `vp_rare` size
  per worker.

- **[CITUS-26] Tiered dictionary** — New GUC `pg_ripple.dictionary_tier_threshold`
  (integer, default `-1` = disabled). When set to N > 0, terms with
  `access_count < N` (tracked by a new `access_count BIGINT DEFAULT 0` column
  added to `_pg_ripple.dictionary`) are eligible for the cold tier: they are
  removed from the in-memory LRU and looked up via a worker-local
  `_pg_ripple.dictionary_local` table (non-reference; one copy per worker,
  populated lazily). Warm/hot terms (access_count ≥ N) remain in the reference
  table on all workers as today. Schema change: `ALTER TABLE _pg_ripple.dictionary
  ADD COLUMN access_count BIGINT NOT NULL DEFAULT 0`. Migration script updates
  all existing rows to `access_count = 0`. The tiered path is only active when
  `citus_sharding_enabled = on`.

- **[CITUS-27] Datalog inference worker dispatch** — In `src/datalog/compiler.rs`,
  when `citus_sharding_enabled = on`, wrap each stratum-iteration
  `INSERT … SELECT` in `SELECT run_command_on_all_nodes($sql)` so each worker
  runs the join over its local VP shards in parallel. The coordinator polls
  `SUM(rows_inserted)` across workers after each iteration to determine fixpoint
  termination. Add GUC `pg_ripple.datalog_citus_dispatch` (bool, default `off`;
  requires `citus_sharding_enabled = on`). Add pg_regress test
  `tests/pg_regress/sql/datalog_citus_dispatch.sql` asserting correctness
  (same inferred triple count as single-node path).

- **[CITUS-28] Live shard rebalance** — New SQL function
  `pg_ripple.citus_live_rebalance() RETURNS SETOF rebalance_progress_row`.
  Uses Citus copy-based shard migration (`citus_move_shard_placement` with
  `shard_transfer_mode := 'force_logical'`) so the source shard remains
  writable during the copy phase. Only a brief `AccessShareLock` is taken
  during the final cutover swap. The merge-fence advisory lock is **not**
  acquired, allowing bulk loads to continue throughout. Emits
  `pg_ripple.live_rebalance_start` / `pg_ripple.live_rebalance_end` NOTIFY
  signals. Documents operator guidance on when to prefer live vs. blocking
  rebalance in `docs/src/operations/citus-sharding.md`.

- **[CITUS-29] Multi-hop shard-pruning carry-forward** — Extend
  `src/citus.rs` with a `ShardPruneSet` type (a sorted `Vec<i64>` of subject IDs)
  and a `prune_hop(current_subjects: &ShardPruneSet, vp_table: &str) -> ShardPruneSet`
  function. In the property-path translator, after the first hop resolves a set
  of intermediate subject IDs, pass those IDs as a `WHERE s = ANY($1::BIGINT[])`
  bind parameter on the next hop's VP table scan, progressively restricting
  the fan-out as the path narrows. Activates only when the carry-forward set
  has ≤ `pg_ripple.citus_prune_carry_max` entries (new GUC, default `1000`);
  degrades gracefully to full fan-out above that threshold.

## Exit criteria

- Arrow Flight export benchmarks ≥ 500k triples/second on localhost.
- WatDiv cyclic pattern queries show ≥ 5× improvement vs. hash-join baseline
  in the benchmark workflow.
- `/explorer` loads and renders a CONSTRUCT query result in a browser.
- No `#[allow(dead_code)]` annotations remaining in `src/sparql/wcoj.rs`.
- Migration script `sql/pg_ripple--0.61.0--0.62.0.sql` present.
- `pg_ripple.control` updated to `default_version = '0.62.0'`.
- `citus_property_path_pushdown.sql` pg_regress test passes (or is correctly skipped).
- `datalog_citus_dispatch.sql` pg_regress test confirms identical inference results vs. single-node.
- `pg_ripple.vacuum_vp_rare()` documented and covered by a pg_regress fixture.
- `pg_ripple.citus_live_rebalance()` documented in `docs/src/operations/citus-sharding.md`.
