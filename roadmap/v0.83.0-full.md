## PROPTEST-02 — Proptest Reference-Implementation Comparison

### Current state

`tests/proptest/` contains property tests that check internal invariants (no panic, result is non-empty, etc.) but do not compare output against a reference implementation. A systematic bug in the SPARQL→SQL translation would pass all current property tests.

### What changes

Add cross-implementation comparison to two proptest suites:

**SPARQL SELECT proptest**: Generate random triple datasets and random SELECT patterns. Execute via `sparql_query()` and via `oxigraph` (in-memory RDF engine). Assert that result sets are equivalent (modulo ordering).

**Datalog proptest**: Generate random Datalog programs with no aggregation or negation. Execute via `datalog_query()` and via a simple Rust Datalog interpreter. Assert that derived fact sets are equivalent.

Both comparisons use `proptest::prop_assume!` to skip programs that are too complex for the reference implementation.

### Deliverables

| ID | Deliverable |
|----|-------------|
| PROPTEST-02a | SPARQL SELECT proptest vs oxigraph |
| PROPTEST-02b | Datalog proptest vs reference interpreter |
| PROPTEST-02c | Both added to CI; run on `cargo test --features proptest` |

## CDC-ASYNC-01 — CDC Test Synchronisation Barriers

### Current state

`tests/cdc/` integration tests use `std::thread::sleep(Duration::from_secs(2))` to wait for CDC events to propagate. These tests are flaky on slow CI machines and add unnecessary latency on fast machines.

### What changes

Replace sleep-based synchronisation with `LISTEN`/`NOTIFY` barriers:
1. The CDC worker sends `NOTIFY pg_ripple_cdc_batch_applied, 'slot_name'` after each batch commit (already emitted as of v0.74.0 for monitoring; this changes the payload to include slot name).
2. Tests use `pg_notify_await(channel, timeout_ms)` (a new test helper) to block until the notification arrives.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CDC-ASYNC-01a | `pg_ripple_cdc_batch_applied` NOTIFY with slot-name payload |
| CDC-ASYNC-01b | `pg_notify_await()` test helper function |
| CDC-ASYNC-01c | All CDC integration tests migrated from sleep to NOTIFY |

## API-RENAME-01 — json_ld_load Deprecation

### Current state

The JSON-LD bulk loader is exposed as `pg_ripple.json_ld_load(url TEXT)`. This does not follow the `verb_noun` naming pattern of other load functions (`import_turtle_url`, `load_rdf_xml`).

### What changes

1. Add new function `pg_ripple.load_jsonld(url TEXT, graph_uri TEXT DEFAULT NULL)`.
2. Reimplement `json_ld_load` as a wrapper that emits `NOTICE 'json_ld_load is deprecated; use load_jsonld'` and delegates to `load_jsonld`.

## API-GRAPH-COL-01 — RETURNS TABLE Graph Column

### Current state

VP-row–returning functions (e.g. `triple_match()`, `triple_scan()`) define `RETURNS TABLE (s BIGINT, p BIGINT, o BIGINT)`. The graph `g` column is absent, requiring callers to perform a secondary JOIN to determine named-graph membership.

### What changes

Add `g BIGINT` as the fourth column to all `RETURNS TABLE` definitions for VP-row functions. Existing callers using positional column references require a migration notice; callers using named columns are unaffected.

## P-05-EVAL — Shared-Memory Dictionary Cache Investigation

### Current state

The dictionary decode LRU cache (`src/dictionary/mod.rs`) is stored in process-local (per-backend) memory. Under a 200-connection pool, each connection maintains its own LRU independently. A shared-memory LRU would reduce total memory usage and improve cache hit rates.

### What changes

Investigation spike only:
1. Prototype a `PgSharedMem`-backed LRU using `pgrx::PgSharedMem` with a fixed-size hash table.
2. Measure: latency overhead of shared-memory pointer indirection vs process-local cache.
3. Measure: total memory usage at 200 connections with process-local vs shared.
4. Produce a decision: implement or close with rationale.

### Deliverables

| ID | Deliverable |
|----|-------------|
| P-05-EVAL-a | Prototype or design doc |
| P-05-EVAL-b | Benchmark numbers (latency + memory at 200 connections) |
| P-05-EVAL-c | Decision recorded in `plans/` or implemented |
