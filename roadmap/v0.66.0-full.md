## Release theme

v0.66.0 is the execution-reality milestone. It closes the gap between high-level performance/scalability claims and normal code paths by making streaming actually stream, Arrow actually use Arrow IPC, Citus optimizations actually influence query/storage execution, and WCOJ status explicit. Any advanced path that remains a hint or helper is labeled as such in docs, `feature_status()`, and `sparql_explain`.

## Deliverables

### STREAM-01 - True SPARQL cursor streaming

Replace cursor functions that call `sparql::sparql(query)` and collect all rows with a paged SPI cursor or set-returning iterator.

Requirements:

- Memory use is proportional to `cursor_page_size`, not full result size.
- `sparql_cursor`, `sparql_cursor_turtle`, and `sparql_cursor_jsonld` stream pages.
- `sparql_max_rows` and overflow behavior still apply.
- Cancellation closes the underlying cursor cleanly.

Verification:

- pgbench or integration test streams a 10M-row result with peak memory below `2 * cursor_page_size * row_width` plus fixed overhead.
- Existing cursor API tests continue to pass.

### STREAM-02 - HTTP streaming with backpressure

Connect `pg_ripple_http` streaming responses to the true cursor path.

Requirements:

- Backpressure from the HTTP client pauses database fetches.
- Dropped clients cancel database cursors.
- Metrics record rows streamed, bytes sent, pages fetched, cancellations, and peak buffered rows.

### FLIGHT-01 - Signed Arrow Flight tickets

Replace unsigned JSON tickets with signed, expiring tickets.

Required fields:

- graph IRI and graph id
- issued-at and expiry
- audience or endpoint binding
- nonce or replay guard
- requested export format/version
- HMAC signature using configured server secret

Verification:

- Tampered, expired, wrong-audience, and replayed tickets are rejected.
- Ticket validation failures expose metrics and safe error messages.

### FLIGHT-02 - Real Arrow IPC streaming

Implement Arrow IPC streaming in `pg_ripple_http /flight/do_get`.

Requirements:

- Enumerate `_pg_ripple.vp_rare` and every promoted VP predicate for the selected graph.
- Support dictionary-decoded columns and optional encoded ids.
- Batch rows with configurable batch size.
- Apply backpressure and cancellation.
- Benchmark against COPY and JSON-LD export.

Verification:

- Integration test decodes the response as Arrow IPC.
- Parity test compares Arrow export triples with SPARQL CONSTRUCT/export output.
- Throughput benchmark records rows/sec in release evidence.

### WCOJ-01 - Explainable scoped cyclic-join mode

Keep the current cyclic BGP planner-hint implementation unless a real triejoin executor is built, but make the distinction explicit.

Requirements:

- `sparql_explain` reports `cyclic_bgp_detected`, `wcoj_mode`, `planner_settings`, and fallback reason.
- Planner settings use transaction-scoped `SET LOCAL` or fragment-local SQL where possible.
- Docs no longer instruct users to look for nonexistent leapfrog plan nodes.
- If a true executor is implemented, add algorithm-specific tests and benchmarks; otherwise use `planner_hint` status.

Verification:

- Tests assert explain output, not just result equivalence.
- Benchmark gate measures cyclic-pattern speedup without claiming worst-case optimality unless true.

### CITUS-01 - SERVICE result shard pruning integration

Wire SERVICE result subject bindings into local BGP translation.

Requirements:

- Encode SERVICE subject IRIs to dictionary ids.
- Determine shard ids through existing pruning helpers.
- Push shard restrictions into subsequent local VP joins.
- Expose fallback reason when result set exceeds `citus_prune_carry_max`.

Verification:

- Citus-conditional test proves fewer workers queried for bounded SERVICE results.
- `sparql_explain(distributed := true)` shows pruned shard ids.

### CITUS-02 - HLL approximate distinct pushdown

When `pg_ripple.approx_distinct = on` and `hll` is installed, translate SPARQL `COUNT(DISTINCT ?x)` to worker-mergeable HLL SQL.

Requirements:

- Opt-in only.
- Error margin documented.
- Exact count remains default.
- Fallback visible when `hll` is missing.

Verification:

- Tests with and without `hll` installed.
- Explain output reports `approx_distinct: hll` or fallback.

### CITUS-03 - Non-blocking VP promotion or honest downgrade

Either implement the shadow-table promotion pattern or explicitly mark current promotion as synchronous in docs and feature status.

Implementation path if delivered:

- Add `promotion_state` to `_pg_ripple.predicates`.
- Route new writes to the dedicated table while old `vp_rare` rows migrate in batches.
- Atomically mark promotion complete and remove migrated rare rows.
- Expose promotion state in status/metrics.

Verification:

- Concurrent write/read test during promotion.
- Citus DDL lock duration is bounded and measured.

### CITUS-04 - Per-worker BRIN summarise after merge

Call `run_command_on_shards` after HTAP merge for distributed VP main tables, falling back to local `brin_summarize_new_values` when Citus is disabled.

Verification:

- Citus test asserts worker shard BRIN summaries exist after merge.
- Metrics track summarise failures by predicate and worker.

### CITUS-05 - Per-worker RLS propagation test

Add `tests/integration/citus_rls_propagation.sh` with one coordinator and two workers.

Sequence:

1. `grant_graph(g, role_a)`.
2. Query graph `g` directly on each worker as `role_a`.
3. `revoke_graph(g, role_a)`.
4. Query each worker again and assert zero rows.

The test must prove propagation is synchronous or report the feature as eventually consistent.

### CITUS-06 - Multi-hop pruning carry-forward integration

Wire `ShardPruneSet` and `prune_hop()` into property-path or BGP translation, or downgrade the feature to helper-only.

Verification:

- No dead-code annotations remain for shipped capability paths.
- Explain output shows carry-forward set size and fallback reason.

### OBS-01 - Streaming and distributed observability

Add metrics and explain fields for:

- rows materialized vs streamed
- bytes sent
- cursor pages fetched
- Arrow batches sent
- ticket validation failures
- coordinator memory estimate
- Citus shard pruning reason
- worker SQL fragments
- approximate vs exact aggregate mode
- BRIN summarise status by worker

## Conflicts and risks

- True streaming may require deeper changes to SPARQL execution than the cursor module alone.
- Arrow Flight signing needs key-management guidance. Avoid hard-coded secrets and require operator configuration.
- Citus tests are heavier than pg_regress and may need separate CI infrastructure.
- Implementing a true WCOJ executor is optional; do not block this release on it if planner-hint truth labeling is sufficient.
- Non-blocking promotion may require schema changes and careful migration planning.

## Exit criteria

- `sparql_cursor` memory is bounded by page size in a large-result test.
- `/flight/do_get` returns decodable Arrow IPC batches and rejects invalid tickets.
- Citus SERVICE pruning, HLL distinct, BRIN summarise, RLS propagation, and multi-hop pruning are either integrated and tested or explicitly marked non-implemented in `feature_status()` and docs.
- WCOJ explain output accurately reports planner-hint or executor mode.
- Streaming/export/distributed metrics are visible through SQL and HTTP metrics surfaces.
