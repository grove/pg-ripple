## Release theme

v0.68.0 closes the gap between the distributed and streaming *capability* delivered in v0.62–v0.66 and the *contract* those versions claimed. Several features were implemented as helpers or labelled planned in `feature_status.rs`: HLL COUNT(DISTINCT), Citus SERVICE pruning, nonblocking VP promotion, and true CONSTRUCT streaming. This version makes them production-complete or formally labels them as experimental with honest documentation. It also closes the continuous fuzzing gap that has existed since the 12 fuzz targets were registered.

## Deliverables

### STREAM-01 — Streaming CONSTRUCT Turtle/JSON-LD iterators

Replace the current `cursor.rs` CONSTRUCT helpers that call `sparql_construct()` and collect full result vectors.

New design:
- Open a named portal for the underlying triple-pattern query.
- Yield triples from the portal in configurable page-size batches.
- Serialize each batch to Turtle or JSON-LD incrementally; do not buffer the full document.
- Expose via `pg_ripple.sparql_construct_cursor(query, page_size)` returning `SETOF text` for streaming Turtle chunks, and a corresponding JSON-LD variant.
- Update `feature_status()` to change `construct_turtle_streaming` and `construct_jsonld_streaming` status from the current partial claim to `experimental` or `implemented` with accurate evidence paths.

**If full streaming is not achievable before v0.68.0 release**: rename the helper functions to `sparql_construct_batch` and mark them non-cursor in documentation and `feature_status()`. This avoids the overclaim and unblocks users who need accurate capability discovery.

### CITUS-HLL-01 — Citus HLL aggregate translation

Wire the existing `citus_hll_available()` helper in `citus.rs` into the aggregate translation path in `src/sparql/translate/group.rs`:

- When `pg_ripple.approx_distinct = on` (new GUC, default `off`) and `citus_hll_available()` returns true, generate `hll_add_agg(hll_hash_bigint(x))::hll` instead of exact `COUNT(DISTINCT x)`.
- When the HLL extension is absent or `approx_distinct = off`, fall back to exact `COUNT(DISTINCT x)`.
- Add pg_regress test `tests/pg_regress/sql/citus_hll_translation.sql` that verifies the generated SQL shape (via `EXPLAIN`) and the aggregate result.
- Update `feature_status()` to change `citus_hll_count_distinct` from `planned` to `experimental`.

### CITUS-SVC-01 — Citus SERVICE + multihop pruning in SPARQL translator

**Preconditions**: explain output and correctness tests must exist before this item is implemented.

Wire Citus SERVICE endpoint pruning into the SPARQL federation translator:

- After cost-based SERVICE endpoint selection, check whether the endpoint is a Citus worker. If so, add shard-constraint annotations to the subquery to prune irrelevant shards.
- For multihop join chains where consecutive SERVICE calls access the same Citus cluster, coalesce the calls into a single distributed query when possible.
- Gate the rewrite behind a `pg_ripple.citus_service_pruning = on` GUC (default `on` when Citus is detected, `off` otherwise).
- Add pg_regress test that runs `EXPLAIN` on a federated query and asserts pruned shard count < total shard count.
- Update `feature_status()` to change `citus_service_pruning` from `planned` to `experimental`.

**If Citus multi-node infrastructure is unavailable**: implement the rewrite with mock/stub Citus responses in tests and document the infrastructure requirement.

### PROMO-01 — Nonblocking VP promotion with shadow tables

Replace the synchronous lock-and-promote path in `src/storage/mod.rs` with a shadow-table pattern:

#### Phase 1 — Shadow copy (background)
- When rare-predicate count crosses `vp_promotion_threshold`, mark the predicate as `promoting` in `_pg_ripple.predicates`.
- Background worker creates `_pg_ripple.vp_{id}_shadow_delta`, `_shadow_main`, `_shadow_tombstones` with identical schema and RLS policies.
- Copies data from `_pg_ripple.vp_rare WHERE p = predicate_id` into the shadow tables in batches.
- New writes during the copy period go to both `_pg_ripple.vp_rare` and the shadow tables (dual-write).

#### Phase 2 — Atomic rename (brief lock)
- Once the copy is complete, acquire a short advisory lock.
- Rename shadow tables to their production names (`vp_{id}_delta`, etc.).
- Delete the predicate's rows from `_pg_ripple.vp_rare`.
- Update `_pg_ripple.predicates.status` from `promoting` to `promoted`.
- Release the lock.

#### Phase 3 — Crash recovery
- On startup, scan `_pg_ripple.predicates` for any row with `status = promoting`.
- If shadow tables exist and the rename is incomplete, restart from Phase 1.
- If shadow tables do not exist, reset status to `rare` and retry from threshold check.

#### RLS propagation
- At the start of Phase 1, apply current active graph policies to the shadow tables.
- At Phase 2 rename, verify RLS policies match `_pg_ripple.vp_rare` policies for the predicate.

#### Tests
- Add pg_regress test `tests/pg_regress/sql/vp_promotion_nonblocking.sql`:
  - Verify no table lock appears in `pg_locks` during background copy phase.
  - Verify query results are correct before, during, and after promotion.
  - Verify RLS isolation is maintained throughout.
  - Verify crash recovery restarts correctly (simulated via deliberate abandon of a promoting state).

### FUZZ-01 — Scheduled fuzz workflow for all 12 targets

Add `.github/workflows/fuzz.yml`:

- **Short schedule** (nightly, `schedule: cron`): runs each of the 12 fuzz targets for 60 seconds with `cargo fuzz run {target} -- -max_total_time=60`.
- **Long manual run** (`workflow_dispatch` with a `duration` input): runs each target for a configurable duration (default 3600 seconds).
- Steps must fail if `cargo fuzz` exits non-zero for any target.
- Artifact: upload corpus and crash files from `fuzz/corpus/` and `fuzz/artifacts/` on each run.
- Update README fuzz section to reference the CI schedule.
- Update `feature_status()` `continuous_fuzzing` entry from `partial` to `experimental` with the new workflow as evidence.

## Conflicts and risks

- Nonblocking VP promotion (PROMO-01) is XL scope. A parallel dual-write design must not allow the background copy to fall behind under high-throughput insert workloads. Add a backpressure mechanism or accept that promotion blocks inserts above a configurable lag threshold.
- Citus SERVICE pruning (CITUS-SVC-01) requires multi-node infrastructure for full integration testing. Mark the feature `experimental` and include explicit infrastructure notes in documentation rather than silently skipping tests.
- CONSTRUCT streaming (STREAM-01) changes the public API: callers that previously received a single text result now receive a `SETOF text`. Existing application code that joins or casts the result may need to be updated.

## Exit criteria

- `feature_status()` shows `construct_turtle_streaming` and `construct_jsonld_streaming` as `experimental` with accurate implementation evidence.
- `citus_hll_count_distinct` and `citus_service_pruning` are no longer `planned`; both have pg_regress or explain-based tests.
- Nonblocking VP promotion passes dual-write correctness, RLS isolation, and crash recovery tests.
- Nightly fuzz workflow runs all 12 targets and uploads artifacts; no target exits non-zero on the initial corpus.
- README fuzz section references the CI schedule.
