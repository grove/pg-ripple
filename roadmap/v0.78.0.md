# v0.78.0 - Bidirectional Integration Operations

> **Full technical details:** [v0.78.0-full.md](v0.78.0-full.md)

**Status: Planned** | **Scope: Medium-Large**

> v0.78.0 closes the operational gaps identified in the v0.77.0 review. v0.77.0 delivered the data semantics and pg-trickle transport contract (late-binding rewrite, sparse-CAS, normalize-aware conflicts, non-destructive losers, linkback rendezvous, outbox/inbox delivery). v0.78.0 turns that into an operable product by adding write-side outbox policies, per-subscription side-band auth and frame-level redaction, explicit new-events-only schema-evolution policy, property-based and chaos tests, a reconciliation toolkit, and a consolidated operations surface.

---

## What is this?

v0.77.0 makes pg_ripple correct for bidirectional integration. v0.78.0 makes it operable.

The review of v0.77.0 surfaced eight categories of gap that real production deployments hit but the data-semantics layer cannot solve on its own:

1. **Backpressure and outbox management** — slow consumers leave outbox tables filling unboundedly; pg-trickle handles delivery-side retry/DLQ but write-side policies (max depth, age expiry, overflow handling) must live with the bridge writer.
2. **Schema evolution policies** — the wire format `version` field is defined, but there is no policy for what happens when a subscription's `frame` predicates change, when an `iri_pattern` changes, or when `exclude_graphs` changes while already-rendered events are queued in pg-trickle's outbox.
3. **Per-subscription auth and RBAC** — `pg_ripple_http` has bearer tokens but no per-subscription scoping for the side-band façade (`/linkback`, `/divergence`, `/abandon`); a CRM-relay token can today call linkback on ERP events.
4. **Frame-level redaction** — the bridge writer emits frame predicates verbatim; PII / secret-bearing predicates have no first-class suppression mechanism.
5. **Audit log** — no record of who called the side-band APIs (`record_linkback`, `reconciliation_*`, `requeue_dead_letter`) on which event when.
6. **Property-based and chaos tests** — v0.77.0 ships per-feature unit tests and one end-to-end fixture; convergence under random insert/update/delete sequences and crash-mid-flush recovery are not exercised.
7. **Reconciliation toolkit** — receivers are told to "queue for reconciliation" on CAS divergence with no guidance on what the queue is or how to drain it; no in-database primitives exist for enqueue / lease / resolve.
8. **Consolidated operations surface** — knobs are scattered across multiple GUCs; there is no single dashboard view for outbox depth, conflict rate, linkback latency, and rewrite-miss volume.

v0.78.0 adds eleven deliverables (BIDIOPS-QUEUE-01 through BIDIOPS-DOC-01) addressing these without changing the v0.77.0 wire format or outbox/inbox contract. Existing relays continue to work; new capability is opt-in.

## Headline outcomes

- **Outbox depth limits and dead-letter** — `create_subscription(..., max_queue_depth => N, dead_letter_after => INTERVAL, overflow_policy => 'pause'|'drop_oldest'|'drop_newest')`. The bridge writer enforces these write-side policies; pg-trickle's delivery-side DLQ is unchanged. Source mutations never block on outbox depth.
- **Subscription pause / resume (delegated to pg-trickle)** — pg-trickle already ships `pg_trickle.pause_subscription` / `pg_trickle.resume_subscription` (and `pause_all_subscriptions`); pg_ripple defers to those rather than wrapping them. The bridge writer keeps appending events while paused; pg-trickle stops draining; `bidi_status()` exposes `pg_trickle_paused` and `pause_reason` by joining `pg_trickle.subscriptions`; `bidi_health()` reports `paused`. When `overflow_policy = 'pause'` and depth hits `max_queue_depth`, additional events route to dead-letter on the same code path.
- **Schema-evolution policies** — explicit `new_events_only` policy on what to do when:
  - The `frame` changes (already-rendered outbox rows drain unchanged; new events use the new frame).
  - The `iri_pattern` for a graph in `rewrite_target_graph` changes (queued outbox rows keep the original pattern's IRIs; new events use the new pattern; no retroactive rewrite).
  - `exclude_graphs` changes (applies only to events emitted after the change; queued outbox rows drain unchanged).
  - All three are documented in `_pg_ripple.subscription_schema_changes` for audit.
- **Per-subscription auth + scope tokens** — `register_subscription_token(subscription, token, scopes => ARRAY['linkback','divergence','abandon'])`. `pg_ripple_http` enforces per-route scope checks on the side-band façade; the optional `outbox_read` scope grants standard Postgres SELECT on the subscription's outbox table for direct readers. Bearer tokens are SHA-256 hashed at rest (`_pg_ripple.subscription_tokens`); operator UI lists active tokens with last-use timestamp and rotation guidance.
- **Frame-level redaction at write time** — frames support `"@redact": true` per predicate. The bridge writer renders each event into a redacted outbox (`{"@redacted": true}` placeholder) and, when configured, into a parallel unredacted outbox (cleartext). Per-subscription tokens (combined with standard Postgres GRANTs) gate which outbox each consumer can read. CAS still works because both sides redact identically.
- **Audit log for side-band calls** — `_pg_ripple.event_audit (event_id, subscription_name, action, actor_token_hash, observed_at, http_remote_addr)` records every `record_linkback` / `reconciliation_*` / `abandon_linkback` / `requeue_dead_letter` / admin action with the requesting token's hash and (for HTTP callers) the source address. Outbox-row consumption by sinks is audited by pg-trickle (out of scope here). Configurable retention via `pg_ripple.audit_retention` GUC (default `90 days`); a background worker prunes older rows.
- **Property-based test harness** — `proptest` strategies for (subject, predicate, value, source-graph, timestamp) tuples; convergence assertions on the resolved projection after random sequences of insert/update/delete from N sources. Asserts: no triple lost, conflict policies applied deterministically, `latest_wins` respects timestamp order regardless of arrival order, and `record_linkback` followed by buffered-flush produces the same final state as direct rewriting.
- **Chaos test harness** — Jepsen-style controlled fault injection: relay crashes after pg-trickle delivery but before linkback; relay crashes between linkback and next outbox-row consumption; pg_ripple restarts mid-`conflict_winners` recompute; pg-trickle restarts mid-delivery. Asserts: no duplicate target creates, no lost CDC events, outbox eventually drains, dead-letter contains exactly the events that hit policy limits.
- **Reconciliation toolkit** — `_pg_ripple.reconciliation_queue` plus three SQL helpers: `reconciliation_enqueue(event_id, divergence_summary)`, `reconciliation_next(subscription_name)` (lease + skip-locked), `reconciliation_resolve(event_id, action)` with actions `accept_external` / `force_internal` / `merge_via_owl_sameAs` / `dead_letter`. The toolkit is the canonical in-database implementation; relay clients in any language call the SQL/HTTP API.
- **Consolidated operations surface** — `pg_ripple.bidi_status()` returns one row per subscription with: outbox depth, oldest-event age, pg-trickle pause/delivery status, dead-letter count, conflict-rejection rate, linkback-pending count and oldest-pending age, recent rewrite-miss count. Companion `pg_ripple.bidi_health()` returns a single overall status (`healthy` / `degraded` / `paused` / `failing`) with a list of triggering conditions.
- **Migration script** — `sql/pg_ripple--0.77.0--0.78.0.sql` creates the new catalog tables (`event_dead_letters`, `subscription_tokens`, `subscription_schema_changes`, `event_audit`, `reconciliation_queue`) and registers the new function signatures so v0.77.0 deployments upgrade in place.
- **Documentation** — operations runbook covering queue draining, token rotation, redaction policy, schema-evolution rollouts, reconciliation playbook, and chaos-test fixtures. Plus a "Production-readiness checklist" page.
- **Draft RDF Bidirectional Integration Profile v1 (BIDI-SPEC-01)** — an informational draft at `docs/spec/rdf-bidi-integration-v1.md` extracting the store-agnostic parts of v0.77.0 + v0.78.0. It is non-blocking for v0.78.0; candidate conformance scenarios are cross-referenced to tests where available, with gaps filed as follow-up spec work.

## Assessment findings covered

This release closes the operational gaps from the v0.77.0 review:

- ✓ Queue management with dead-letter and quotas (BIDIOPS-QUEUE-01, BIDIOPS-PAUSE-01)
- ✓ Schema-evolution explicit policies (BIDIOPS-EVOLVE-01)
- ✓ Per-subscription auth + frame-level redaction (BIDIOPS-AUTH-01, BIDIOPS-REDACT-01)
- ✓ Audit log (BIDIOPS-AUDIT-01)
- ✓ Property-based + chaos tests (BIDIOPS-PROPTEST-01, BIDIOPS-CHAOS-01)
- ✓ Reconciliation toolkit, not just docs (BIDIOPS-RECON-01)
- ✓ Consolidated operations surface (BIDIOPS-DASH-01)
- ✓ Draft vendor-neutral RDF Bidirectional Integration Profile spec extracted (BIDI-SPEC-01) — captures the design for broader RDF ecosystem review without blocking the implementation release

Out of scope, deferred to later releases:

- Multi-region / multi-master pg_ripple instances (later release).
- Dynamic queued-event re-framing / retroactive policy hot-reload (later release).
- Visual operator UI beyond the SQL functions (out of scope; embedded in pg-trickle's UI work).
