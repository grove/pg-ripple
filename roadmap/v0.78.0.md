# v0.78.0 - Bidirectional Integration Operations

> **Full technical details:** [v0.78.0-full.md](v0.78.0-full.md)

**Status: Planned** | **Scope: Medium-Large**

> v0.78.0 closes the operational gaps identified in the v0.74.0 review. v0.74.0 delivered world-class data semantics (late-binding rewrite, sparse-CAS, normalize-aware conflicts, non-destructive losers, linkback rendezvous). v0.78.0 turns that into a complete product by adding queue management with dead-letter, per-subscription auth and frame-level redaction, explicit schema-evolution policies, property-based and chaos tests, a reconciliation toolkit, and consolidated operations surface.

---

## What is this?

v0.74.0 made pg_ripple correct for bidirectional integration. v0.78.0 makes it operable.

The review of v0.74.0 surfaced eight categories of gap that real production deployments hit but the data-semantics layer cannot solve on its own:

1. **Backpressure and queue management** — slow consumers leave millions of events queued; nothing dead-letters; no per-subscription quotas; the inbound write path could in principle block on outbound queue depth.
2. **Schema evolution policies** — the wire format `version` field is defined, but there is no policy for what happens when a subscription's `frame` predicates change mid-flight, when an `iri_pattern` changes after events have been emitted, or when `exclude_graphs` changes while events are queued.
3. **Per-subscription auth and RBAC** — `pg_ripple_http` has bearer tokens but no per-subscription scoping; the CRM relay can today read the ERP queue.
4. **Frame-level redaction** — frames emit fields verbatim; PII / secret-bearing predicates have no first-class suppression mechanism.
5. **Audit log** — no record of who pulled / acked / nacked / linkbacked which event when.
6. **Property-based and chaos tests** — v0.74.0 ships per-feature unit tests and one end-to-end fixture; convergence under random insert/update/delete sequences and crash-mid-flush recovery are not exercised.
7. **Reconciliation toolkit** — receivers are told to "queue for reconciliation" on CAS divergence with no guidance on what the queue is or how to drain it. The reference Python relay implements one strategy.
8. **Consolidated operations surface** — knobs are scattered across multiple GUCs; there is no single dashboard view for queue depth, conflict rate, linkback latency, and rewrite-miss volume.

v0.78.0 adds eleven deliverables (BIDIOPS-QUEUE-01 through BIDIOPS-DOC-01) addressing these without changing the v0.74.0 wire format or SQL contract. Existing relays continue to work; new capability is opt-in.

## Headline outcomes

- **Queue depth limits and dead-letter** — `create_subscription(..., max_queue_depth => N, dead_letter_after => INTERVAL)`. When depth exceeds `max_queue_depth`, the inbound write path emits a `WARNING` and the subscription is marked `paused`; new events for that subscription are dropped with a counter increment. Events older than `dead_letter_after` are moved to `_pg_ripple.event_dead_letters` with the original payload, ack/nack history, and reason. Inbound writes never block on outbound queue depth.
- **Subscription pause / resume** — `pg_ripple.pause_subscription(name, reason)` and `pg_ripple.resume_subscription(name)` for graceful operator control during target-system outages. Paused subscriptions still record events into `event_queue` until `max_queue_depth` is hit, at which point new events route to `event_dead_letters` with `reason = 'paused_overflow'`.
- **Schema-evolution policies** — explicit per-subscription policy on what to do when:
  - The `frame` changes (default: re-frame queued events using the new frame; alternative: drain at old frame then switch).
  - The `iri_pattern` for a graph in `rewrite_target_graph` changes (default: queued events keep the original pattern's IRIs; new events use the new pattern; no retroactive rewrite).
  - `exclude_graphs` changes (default: applies only to events emitted after the change; queued events drain unchanged).
  - All three are documented in `_pg_ripple.subscription_schema_changes` for audit.
- **Per-subscription auth + scope tokens** — `register_subscription_token(subscription, token, scopes => ARRAY['pull','ack','linkback'])`. `pg_ripple_http` enforces per-route scope checks. Bearer tokens are SHA-256 hashed at rest (`_pg_ripple.subscription_tokens`); operator UI lists active tokens with last-use timestamp and rotation guidance.
- **Frame-level redaction** — frames support `"@redact": true` per predicate; redacted predicates are emitted as `{"@redacted": true}` in `after`/`base`. CAS still works because both sides redact identically. Receivers that need the cleartext use a separate "elevated" subscription with a token that has `unredacted` scope.
- **Audit log** — `_pg_ripple.event_audit (event_id, subscription_name, action, actor_token_hash, observed_at, http_remote_addr)` records every `next_event` / `ack_event` / `nack_event` / `record_linkback` / `abandon_linkback` call with the requesting token's hash and (for HTTP callers) the source address. Configurable retention via `pg_ripple.audit_retention` GUC (default `90 days`); a background worker prunes older rows.
- **Property-based test harness** — `proptest` strategies for (subject, predicate, value, source-graph, timestamp) tuples; convergence assertions on the resolved projection after random sequences of insert/update/delete from N sources. Asserts: no triple lost, conflict policies applied deterministically, `latest_wins` respects timestamp order regardless of arrival order, and `record_linkback` followed by ack-buffered-flush produces the same final state as direct rewriting.
- **Chaos test harness** — Jepsen-style controlled fault injection: relay crashes mid-`record_linkback`, mid-flush, mid-ack; pg_ripple restarts mid-`conflict_winners` recompute; HTTP companion crashes between pull and ack. Asserts: no duplicate target creates, no lost CDC events, queue eventually drains, dead-letter contains exactly the events that hit policy limits.
- **Reconciliation toolkit** — `_pg_ripple.reconciliation_queue` plus three SQL helpers: `reconciliation_enqueue(event_id, divergence_summary)`, `reconciliation_next(subscription_name)` (lease + skip-locked, mirroring `next_event`), `reconciliation_resolve(event_id, action)` with actions `accept_external` / `force_internal` / `merge_via_owl_sameAs` / `dead_letter`. Reference Python relay extended with the divergence dispatch table; the toolkit is the canonical implementation, not just docs.
- **Consolidated operations surface** — `pg_ripple.bidi_status()` returns one row per subscription with: queue depth, oldest-event age, dead-letter count, conflict-rejection rate, linkback-pending count and oldest-pending age, recent rewrite-miss count, last-pull and last-ack timestamps. Companion `pg_ripple.bidi_health()` returns a single overall status (`healthy` / `degraded` / `paused` / `failing`) with a list of triggering conditions.
- **Migration script** — `sql/pg_ripple--0.74.0--0.75.0.sql` creates the new catalog tables (`event_dead_letters`, `subscription_tokens`, `subscription_schema_changes`, `event_audit`, `reconciliation_queue`) and registers the new function signatures so v0.74.0 deployments upgrade in place.
- **Documentation** — operations runbook covering queue draining, token rotation, redaction policy, schema-evolution rollouts, reconciliation playbook, and chaos-test fixtures. Plus a "Production-readiness checklist" page.
- **RDF Bidirectional Integration Profile v1 (BIDI-SPEC-01)** — a vendor-neutral specification at `docs/spec/rdf-bidi-integration-v1.md` extracting the ≈70 % of v0.74.0 + v0.78.0 that is store-agnostic (conflict policies, late-binding rewrite, sparse-`base` CAS, linkback, redaction, reconciliation, schema evolution, audit, ops surface) into a W3C-style spec with normative MUST/SHOULD/MAY language, two conformance levels (Bidi-1.0 and Bidi-1.0-Ops), and a 15-scenario conformance corpus. pg_ripple becomes the reference implementation; other triplestores (Jena, Stardog, Oxigraph, GraphDB) can implement the same protocol. Co-located with pg_ripple for v1; candidate for W3C Community Group donation post-pg_ripple-v1.0.

## Assessment findings covered

This release closes the operational gaps from the v0.74.0 review:

- ✓ Queue management with dead-letter and quotas (BIDIOPS-QUEUE-01, BIDIOPS-PAUSE-01)
- ✓ Schema-evolution explicit policies (BIDIOPS-EVOLVE-01)
- ✓ Per-subscription auth + frame-level redaction (BIDIOPS-AUTH-01, BIDIOPS-REDACT-01)
- ✓ Audit log (BIDIOPS-AUDIT-01)
- ✓ Property-based + chaos tests (BIDIOPS-PROPTEST-01, BIDIOPS-CHAOS-01)
- ✓ Reconciliation toolkit, not just docs (BIDIOPS-RECON-01)
- ✓ Consolidated operations surface (BIDIOPS-DASH-01)
- ✓ Vendor-neutral RDF Bidirectional Integration Profile spec extracted (BIDI-SPEC-01) — turns the design into a contribution to the broader RDF ecosystem rather than a Postgres-only feature

Out of scope, deferred to later releases:

- Multi-region / multi-master pg_ripple instances (v0.76.0+).
- Dynamic policy hot-reload without subscription restart (v0.76.0+).
- Visual operator UI beyond the SQL functions (out of scope; embedded in pg-trickle's UI work).
