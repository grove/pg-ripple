# v0.11.0 — Incremental SPARQL/Datalog Views

**Effort estimate: 5–7 person-weeks**

**Goal:** Instead of running a SPARQL query every time you need fresh results, this release lets you register a query as a live, automatically-maintained view. When underlying data changes, only the affected rows in the view are updated — not the whole query rerun. This capability requires the optional pg-trickle extension.

---

## SPARQL Queries as Live Stream Tables

**Register a SPARQL SELECT as a live view that updates incrementally.**

The `create_sparql_view()` function takes a SPARQL SELECT query and a view name, and creates a PostgreSQL table that always holds the current results of that query. When triples are inserted or deleted, pg-trickle's change-propagation infrastructure computes the delta (which rows to add, which to remove from the view) and applies it — without re-running the full query. For dashboards, APIs, or downstream materialized views, this means fresh results with a fraction of the computational cost.

---

## Datalog Views with Goal Patterns

**Datalog-derived facts can also be maintained incrementally as views.**

The same incremental maintenance mechanism applies to Datalog inferences. A `create_datalog_view()` call registers a Datalog goal pattern (for example, "all entities reachable from concept X via the `rdfs:subClassOf` hierarchy") as a live view. When new base facts arrive, only the newly affected derivations are computed and added to the view — the rest is left unchanged.

---

## ExtVP Semi-Join Precomputed Tables

**Commonly joined predicate pairs are precomputed for faster SPARQL pattern matching.**

For SPARQL patterns that join two predicates on a shared variable very frequently (for example, all patterns of the form `?x foaf:knows ?y . ?y rdf:type foaf:Person`), an ExtVP (Extended Vertical Partitioning) precomputed table stores the result of that specific join in advance. This converts a runtime join into a single table scan, which can be dramatically faster for star-shaped queries on large graphs.

---

## Auto-Refresh on Data Change

**Views stay current automatically without any polling or manual refresh.**

Unlike PostgreSQL's MATERIALIZED VIEW (which requires explicit `REFRESH MATERIALIZED VIEW` calls), pg_ripple's incremental views are maintained by pg-trickle's event stream. Each time the CDC channel emits a change notification, the view maintenance logic runs and propagates only the effect of that change. From the application's perspective, the view table always reflects the current state of the graph.

---

## Dependency on pg-trickle

**This feature requires the pg-trickle extension to be installed.**

All incremental view functionality relies on pg-trickle's relay transport and event-propagation infrastructure. If pg-trickle is not installed, the `create_sparql_view` and `create_datalog_view` functions return a clear error message rather than failing silently. The rest of pg_ripple — SPARQL queries, SHACL validation, Datalog inference — works fully without pg-trickle.
