# v0.8.0 — SHACL Advanced

**Effort estimate: 4–6 person-weeks**

**Goal:** The core SHACL constraints from v0.7.0 handle simple property-level rules. This release adds the more expressive logical and structural constraints that real-world schemas require, and introduces an asynchronous validation pipeline that enforces constraints in the background without slowing down write-intensive workloads.

---

## Complex Logical Constraints

**Constraints can now combine conditions with AND, OR, and NOT.**

Many data quality rules are inherently conjunctive or disjunctive: "a person must have either an email or a phone number", "a product must not be both archived and available". SHACL's `sh:or`, `sh:and`, and `sh:not` allow exactly this. This release compiles all three into SQL expressions that the validator evaluates efficiently. The `sh:qualifiedValueShape` constraint (exactly N values matching a nested sub-shape) is also implemented.

---

## Class and Node Constraints

**Shapes can require that values belong to specific RDF classes.**

The `sh:class` constraint requires that a property's values are instances of a given RDF class, according to the graph's own type statements. The `sh:node` constraint delegates validation of a value to another named shape. These two constraints together enable schema hierarchies where each layer can reference and build on others — the same composability that object-oriented type systems provide, but for graph data.

---

## Asynchronous Validation Pipeline

**Validation can run in the background, decoupled from the insert path.**

High-throughput write workloads cannot afford the latency of synchronous constraint checking on every insert. The `async` validation mode sends violations to a background validation worker instead. The worker processes each write against the loaded shapes and routes any violations to a configurable dead-letter queue table — a separate PostgreSQL table where violations accumulate for review and remediation. Applications see fast inserts and can check the queue periodically or on a schedule.

---

## Dead-Letter Queue for Violations

**Violations are captured in a table for later inspection and replay.**

When asynchronous validation is enabled, every detected violation is written to `_pg_ripple.validation_results` with the focus node, the violated constraint, the offending value, and a timestamp. Operations teams can query this table, build dashboards, trigger alerts, or feed violations into a data-quality remediation workflow. Once a violation is resolved (the offending data corrected), the row can be deleted from the queue.
