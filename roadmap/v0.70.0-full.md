## Deliverables

### BULK-01 — Bulk load → mutation journal

**Root cause**: `storage::batch_insert_encoded` at [src/storage/mod.rs:618–690](../src/storage/mod.rs#L618-L690) contains zero `mutation_journal::*` calls. `bulk_load.rs` `flush_batch` calls `batch_insert_encoded` at [src/bulk_load.rs:125](../src/bulk_load.rs#L125) — and similar lines for every loader variant.

**Fix**:

1. Add `mutation_journal::record_write(g_id)` inside the VP-insert loop in `batch_insert_encoded` (the `g_id` is the third element of each `(s, o, g)` tuple). Guard with `if !crate::construct_rules::has_no_rules()` (reuse the existing fast-path macro).
2. Add `storage::mutation_journal::flush()` at the end of each `load_*` function in `bulk_load.rs` (after all batches have been flushed and promotion has been called).

**Test**: Extend `tests/pg_regress/sql/cwb_write_path_equivalence.sql` with a new section that calls `load_ntriples(...)` as the write path and asserts the derived target graph is non-empty immediately after the load (without calling `refresh_construct_rule`).

**Exit criterion**: `cwb_write_path_equivalence.sql` expected output shows non-zero derived triple count for the bulk-load arm.

### FLUSH-01 — Statement-boundary mutation journal flush

**Root cause**: Both `insert_triple_by_ids` ([src/storage/mod.rs:1692-1693](../src/storage/mod.rs#L1692-L1693)) and `delete_triple_by_ids` ([src/storage/mod.rs:1791-1792](../src/storage/mod.rs#L1791-L1792)) call `flush()` immediately after `record_*`. A 100-quad `INSERT DATA` block fires the full CWB pipeline 100 times.

**Fix**:

1. In `storage::mutation_journal`, add a `flush_deferred` flag (thread-local bool) that `record_write`/`record_delete` set. Remove the explicit `flush()` calls from `insert_triple_by_ids` and `delete_triple_by_ids`.
2. Register a `RegisterXactCallback` in `_PG_init` (or lazily on first `record_write`) that calls `mutation_journal::flush()` on `XACT_EVENT_PRE_COMMIT`.
3. Ensure `dict_api.rs` `insert_triple` wrapper (which also called `flush()` per call) similarly defers.
4. Keep explicit `flush()` call at end of each `load_*` function (added in BULK-01) because bulk loads may not always complete inside a transaction that reaches pre-commit before needing the derived triples.

**Test**: Add a pg_regress test that inserts 50 triples via a single SPARQL `INSERT DATA` block with a registered CONSTRUCT rule and asserts that `on_graph_write` was called exactly once (verifiable via the derived-triple count appearing in a single batch, not 50 individual batches producing 50× overhead).

**Exit criterion**: SPARQL `INSERT DATA { ... }` with N triples fires CWB exactly once regardless of N.

### SHACL-DOC-01 — SHACL documentation truth

**Root cause**: [docs/src/features/shacl-sparql-rules.md](../docs/src/features/shacl-sparql-rules.md) claims `sh:SPARQLRule` is supported; [src/shacl/af_rules.rs:171-175](../src/shacl/af_rules.rs#L171-L175) warns and skips.

**Fix**: Rewrite the documentation page to state:
- `sh:TripleRule` is fully supported and compiled to Datalog rules.
- `sh:SPARQLConstraint` (validation) is supported.
- `sh:SPARQLRule` (inference) is **not yet supported** in v0.70.0. When detected, the extension emits a PostgreSQL `WARNING` (PT481) and skips the rule. Full implementation is targeted for v1.0.0.

- "Known limitations in v0.67.0" → "Known limitations in v0.69.0" with content updated to reflect v0.68.0 and v0.69.0 changes (CONSTRUCT cursor streaming is real, module restructuring done, etc.).

Add `scripts/check_readme_version.sh`:
```bash
#!/usr/bin/env bash
set -euo pipefail
CARGO_VER=$(grep '^version' Cargo.toml | head -1 | sed 's/.*"\(.*\)".*/\1/')
README_VER=$(grep -oP '(?<=What works today \(v)\d+\.\d+\.\d+' README.md || echo "NOT_FOUND")
if [[ "$CARGO_VER" != "$README_VER" ]]; then
  echo "ERROR: README version ($README_VER) does not match Cargo.toml ($CARGO_VER)"
  exit 1
fi
echo "README version check passed: $README_VER"
```

### RLS-SQL-01 — Role name quoting
