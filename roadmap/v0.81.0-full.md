## Background

After v0.80.0 closes all Critical and High must-fix findings, v0.81.0 addresses the thirteen correctness findings (C-04 through C-16) and seven concurrency findings (CC-01 through CC-07) from Assessment 12. These represent data-integrity risks and race conditions that may manifest under production load or unusual transaction patterns.

## MERGE-SID-01 — HTAP Merge Non-Deterministic SID Fix

### Current state

The HTAP merge CTE in `src/storage/promote.rs` uses `DISTINCT ON (s, o, g)` without an `ORDER BY` clause. PostgreSQL's `DISTINCT ON` without ORDER BY selects an arbitrary row from each group, meaning that when `(s, o, g)` appears in both `main` and `delta`, either SID may be preserved in the merged output. The result is non-deterministic statement identifiers after merge, which breaks provenance queries and CONSTRUCT writeback that key on SIDs.

### What changes

Prepend `ORDER BY s, o, g, i ASC` to each arm of the merge CTE before the `DISTINCT ON` step, guaranteeing that the lowest (oldest) SID is always preserved.

### Deliverables

| ID | Deliverable |
|----|-------------|
| MERGE-SID-01a | `ORDER BY … i ASC` added to merge CTE |
| MERGE-SID-01b | Regression test: same triple in main + delta → SID of main-partition copy is preserved |

## PROMO-ATOMIC-01 — Promotion Status Update Atomicity

### Current state

`src/storage/promote.rs` — after creating the new VP table and copying rows — updates `_pg_ripple.predicates SET table_oid = …` in a separate statement. If the server crashes between the copy and the status update, the VP table exists but the catalog still points to the old table. Recovery falls back to a full re-promotion on next access, which can lose or double triples.

### What changes

Combine the VP table creation, row copy, and predicates status update into a single CTE using PostgreSQL's `WITH … INSERT … RETURNING`:

```sql
WITH new_table AS (
    CREATE TABLE _pg_ripple.vp_{id} (…) -- via dynamic DDL wrapped in a DO block
), copy AS (
    INSERT INTO _pg_ripple.vp_{id} SELECT … FROM _pg_ripple.vp_rare WHERE p = $1
)
UPDATE _pg_ripple.predicates SET table_oid = … WHERE id = $1
```

Because DDL cannot appear inside a CTE, the approach uses an advisory lock + explicit savepoint: (a) acquire lock, (b) check catalog, (c) create table, (d) copy rows, (e) update catalog — all inside a single subtransaction with `ON CONFLICT DO NOTHING` on the catalog update.

### Deliverables

| ID | Deliverable |
|----|-------------|
| PROMO-ATOMIC-01a | Promotion wrapped in subtransaction with savepoint |
| PROMO-ATOMIC-01b | Catalog update uses `ON CONFLICT DO NOTHING` |
| PROMO-ATOMIC-01c | Regression test: simulated crash between copy and catalog update |

## DICT-RACE-01 — Dictionary Encode Race Handling

### Current state

`src/dictionary/mod.rs` calls `RETURNING id` after inserting a new dictionary entry. If a concurrent session deletes the entry (or a hash collision causes a rollback), `RETURNING` yields 0 rows. The current code silently falls back to a second `SELECT`, which may return a wrong ID.

### What changes

After a 0-row `RETURNING` result, emit a `pg_warning!` and retry the full encode() path once. If the second attempt also fails, return `Err(PgError::new("dictionary encode conflict"))`. Remove the silent SELECT fallback.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DICT-RACE-01a | 0-row RETURNING treated as error, not silent fallback |
| DICT-RACE-01b | Single retry with warning on transient conflict |

## SHACL-TXN-01 — SHACL Shape Store Transaction

### Current state

`src/shacl/mod.rs` writes shape-store entries and DDL `CONSTRAINT` hints in two separate SPI calls. A failure in the DDL hint insertion leaves the shape store in an inconsistent state (shapes registered but hints absent).

### What changes

Wrap both writes in an explicit subtransaction using `Spi::connect(|client| { client.execute_savepoint(…) })`. On any error in the DDL step, roll back to the savepoint and return an error to the caller.

### Deliverables

| ID | Deliverable |
|----|-------------|
| SHACL-TXN-01a | Shape store + DDL hints wrapped in subtransaction |
| SHACL-TXN-01b | Regression test: DDL failure → shape store unchanged |

## FED-URL-01 — Federation URL Normalisation

### Current state

`src/sparql/federation.rs` compares federation endpoint URLs against the GUC allowlist using a string equality check. URLs that differ only in scheme capitalisation (`HTTP://` vs `http://`) or trailing slash bypass the allowlist.

### What changes

Normalise both the incoming URL and each allowlist entry to `scheme.to_lowercase() + "://" + host.to_lowercase() + path.trim_end_matches('/')` before comparison.

### Deliverables

| ID | Deliverable |
|----|-------------|
| FED-URL-01a | URL normalisation before allowlist comparison |
| FED-URL-01b | Tests: uppercase scheme, trailing slash, mixed case host |

## FED-TRUNC-01 — Federation JSON Truncation Warning

### Current state

When a federation response exceeds `federation_result_max_bytes`, the JSON is silently truncated mid-stream. The truncated JSON fails to parse, and the error is swallowed. The caller receives a partial (possibly empty) result set with no indication of truncation.

### What changes

Replace silent truncation with:
1. A `pg_warning!("federation response truncated at {} bytes from {}", limit, url)`.
2. Partial materialisation: parse the truncated response for complete JSON objects only and return the successfully parsed subset.
3. An `is_truncated` flag in the federation result that causes the query engine to annotate results with a `WARNING` notice.

### Deliverables

| ID | Deliverable |
|----|-------------|
| FED-TRUNC-01a | WARNING emitted on truncation |
| FED-TRUNC-01b | Partial JSON parse for complete objects |
| FED-TRUNC-01c | Regression test: response exactly at limit, at limit+1, at 2× limit |

## OPT-INNER-01 — Multi-Predicate OPTIONAL→INNER JOIN Promotion

### Current state

`src/sparql/plan.rs` promotes `OPTIONAL { ?s :p ?o }` to an INNER JOIN only for single-predicate patterns. Multi-predicate optional blocks (e.g. `OPTIONAL { ?s :p1 ?o1 . ?s :p2 ?o2 }`) are always emitted as LEFT JOINs even when `?o1` and `?o2` appear in mandatory SELECT columns.

### What changes

Extend the OPTIONAL→INNER JOIN analysis to check all variables in the block. If every variable bound by the OPTIONAL block is referenced in a mandatory (non-OPTIONAL, non-FILTER) position in the outer query, promote the entire block to INNER JOIN.

### Deliverables

| ID | Deliverable |
|----|-------------|
| OPT-INNER-01a | Multi-predicate OPTIONAL promotion analysis |
| OPT-INNER-01b | Regression tests: 2-predicate optional fully promoted, partially promoted, not promoted |

## DL-AGG-01 — Datalog Aggregation in Recursive Head Guard

### Current state

`src/datalog/stratifier.rs` does not check for aggregation functions in the head position of recursive rules. A rule like `count(X, N) :- count(X, M), triple(X, P, O), N is M + 1` is accepted and produces incorrect results (infinite or undefined count).

### What changes

Add a check in the rule head parser: if the rule is recursive (head predicate appears in a body predicate) and the head contains an aggregation function, return `Err(DatalogError::AggregationInRecursiveHead)` with a descriptive message.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DL-AGG-01a | Guard in stratifier rejects aggregation in recursive heads |
| DL-AGG-01b | Regression test: attempted recursive aggregation produces clear error |

## DL-PAR-01 and DL-PAR-02 — Datalog Parallel SCC Correctness

### Current state

`src/datalog/parallel.rs` partitions strata into parallel execution groups. Two bugs exist:
1. (C-12) The intra-stratum cycle check uses stratum-level reachability, not rule-level reachability. Two rules in the same stratum that are mutually recursive are placed in different parallel groups.
2. (CC-05) The SCC detection in the parallel group partitioner uses stratum order rather than topological rule order, producing non-terminating evaluation for certain recursive rule sets.

### What changes

1. Replace the stratum-level cycle check with a rule-level cycle check using Tarjan's SCC algorithm on the rule dependency graph.
2. Ensure parallel group assignment respects topological order: a rule may not be placed in a group that executes before all its dependencies have been evaluated.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DL-PAR-01a | Rule-level SCC analysis replaces stratum-level |
| DL-PAR-02a | Topological order enforced in group assignment |
| DL-PAR-01b | Regression: mutually recursive rules within a stratum are serialised |
| DL-PAR-02b | Regression: rule with inter-stratum dependency is not evaluated early |

## BN-SCOPE-01 — Blank-Node Variable Scoping

### Current state

`src/sparql/plan.rs` emits blank-node variable names (e.g. `_:b0`) directly into the SQL query without scoping them to the enclosing query block. In subqueries, the same blank-node name in an outer and inner scope resolves to the same dictionary ID, causing incorrect JOIN behaviour.

### What changes

Prefix each blank-node variable with a query-scope UUID generated at plan-compile time:
```sql
-- Before: encode('_:b0', 'blank') AS b0
-- After:  encode('_:7f3a2c_b0', 'blank') AS b0
```
The UUID prefix is generated per top-level query parse, ensuring blank nodes across different calls to `sparql_query()` do not alias.

### Deliverables

| ID | Deliverable |
|----|-------------|
| BN-SCOPE-01a | Query-scoped UUID prefix for blank-node variable names |
| BN-SCOPE-01b | Regression: outer-query blank node does not alias subquery blank node |

## FED-CACHE-01 — Federation Cache Key Normalisation

### Current state

The federation plan cache in `src/sparql/federation.rs` uses the raw SPARQL query string as the cache key. Queries that differ only in whitespace or variable naming generate separate cache entries, leading to cache thrashing under workloads that produce minor syntactic variations.

### What changes

Normalise the cache key by: (a) collapsing all whitespace runs to a single space, (b) lowercasing keywords, (c) canonicalising variable names using a stable rename map. Reuse the existing query normalisation logic from `src/sparql/parse.rs`.

### Deliverables

| ID | Deliverable |
|----|-------------|
| FED-CACHE-01a | Cache key normalisation applied before lookup |
| FED-CACHE-01b | Test: whitespace-variant queries share a cache entry |

## DICT-STRICT-01 — Strict Dictionary GUC

### Current state

When `decode()` receives an ID that has no entry in `_pg_ripple.dictionary`, it silently returns a placeholder string (`"_unknown_<id>"`). In strict data-quality contexts, this masks missing dictionary entries (e.g. caused by incomplete bulk loads).

### What changes

Add GUC `pg_ripple.strict_dictionary` (boolean, default `false`). When enabled, `decode()` raises a `pg_error!` for missing IDs instead of returning the placeholder string. Document the GUC in `docs/src/reference/configuration.md`.

### Deliverables

## DICT-SUBXACT-01 — Dictionary Decode Cache SubXact Invalidation

### Current state

The dictionary decode LRU cache in `src/dictionary/mod.rs` is not invalidated on subtransaction abort. If a subtransaction encodes a new IRI and then aborts, the LRU cache retains the (now-invalid) `id → string` mapping. A subsequent decode of the same ID in the parent transaction returns the stale value.

### What changes

Register a `SubXactCallback` in `_PG_init` that calls `dictionary::invalidate_decode_cache()` on `SUBXACT_EVENT_ABORT_SUB`. This matches the pattern used for the transaction-level `SubXact` callback registered in `src/lib.rs` (v0.72.0 CC-01 fix).

### Deliverables

| ID | Deliverable |
|----|-------------|
| DICT-SUBXACT-01a | `SubXactCallback` registered in `_PG_init` |
| DICT-SUBXACT-01b | Cache invalidation on subtransaction abort |
| DICT-SUBXACT-01c | Regression test: encode in savepoint, rollback savepoint, decode returns error (strict mode) |

## CDC-SLOT-01 — CDC Replication Slot Cleanup Worker

### Current state

`src/cdc.rs` creates PostgreSQL replication slots for CDC subscriptions. If a CDC session terminates abnormally (e.g. client disconnect, server restart), the slot is left open and continues consuming WAL. There is no automated cleanup.

### What changes

Add a background worker `cdc_slot_cleanup` that:
1. Wakes on `pg_ripple.cdc_slot_idle_timeout_seconds` interval (default: 3600 s).
2. Queries `pg_replication_slots` for slots created by pg_ripple that have not advanced their `confirmed_flush_lsn` within the timeout.
3. Calls `pg_drop_replication_slot()` on each idle slot.
4. Logs each dropped slot at `LOG` level.

### Deliverables

| ID | Deliverable |
|----|-------------|
| CDC-SLOT-01a | `cdc_slot_cleanup` background worker |
| CDC-SLOT-01b | `pg_ripple.cdc_slot_idle_timeout_seconds` GUC |
| CDC-SLOT-01c | Integration test: orphaned slot is dropped after timeout |

## PROMO-LOCK-01 — Per-Predicate Promotion Advisory Locks

### Current state

`src/storage/promote.rs` acquires a table-level `ShareUpdateExclusiveLock` on `_pg_ripple.predicates` during VP table promotion. This serialises all concurrent promotion attempts across all predicates.

### What changes

Replace the table lock with per-predicate advisory locks using `pg_advisory_xact_lock(predicate_id)`. Two promotions for different predicates can now proceed concurrently, improving bulk-ingest throughput.

## MERGE-FENCE-01 — Merge Fence Lock Scope Reduction

### Current state

`src/storage/merge.rs` holds a `ShareUpdateExclusiveLock` for the entire duration of the BRIN reindex + main-partition swap. The lock blocks all concurrent autovacuum on the VP table.

### What changes

Restructure the merge into two phases:
1. **Build phase** (no fence lock): write the new `vp_{id}_main_new` partition and BRIN index.
2. **Swap phase** (short `ExclusiveLock`): rename `main_new` to `main` and drop tombstones. Lock held for `< 50 ms` on typical datasets.

### Deliverables

| ID | Deliverable |
|----|-------------|
| MERGE-FENCE-01a | Two-phase merge with short swap lock |
| MERGE-FENCE-01b | Benchmark: autovacuum no longer blocked during merge |

## CDC-LSN-01 — CDC LSN Watermark Table

### Current state

There is no persistent record of which WAL LSN has been fully applied for each CDC subscription. After a restart, the CDC worker cannot determine whether to replay from the last known LSN or from the beginning of the slot.

### What changes

Add table `_pg_ripple.cdc_lsn_watermark(slot_name TEXT PRIMARY KEY, last_lsn PG_LSN, updated_at TIMESTAMPTZ)`. Update it transactionally after each CDC batch commit. The CDC worker reads this on startup to resume from the correct LSN.

## PROMO-STUCK-01 — Stuck Promotion Recovery Function

### Current state

A VP table promotion that crashes mid-flight leaves the `_pg_ripple.predicates` table with a `promoting = true` flag. There is no automatic recovery; the state is resolved only by restarting the PostgreSQL server.

### What changes

Add `pg_ripple.recover_stuck_promotions()` SQL function that:
1. Identifies predicates with `promoting = true` where the promoting backend PID is no longer active.
2. Drops the partially-created VP table if it exists.
3. Resets `promoting = false` in `_pg_ripple.predicates`.
4. Returns the list of recovered predicate IDs.

Call this function from `_PG_init` on extension load (after acquiring the extension lock).

### Deliverables

| ID | Deliverable |
|----|-------------|
| PROMO-STUCK-01a | `pg_ripple.recover_stuck_promotions()` SQL function |
| PROMO-STUCK-01b | Called automatically on extension load |
| PROMO-STUCK-01c | Regression test: simulated mid-flight crash recovered on reload |

## FILTER-STRICT-01 — Strict SPARQL Filters GUC

### Current state

When the SPARQL query engine encounters an unknown built-in function (e.g. `FILTER(ex:myFunction(?x))`) it evaluates the expression to `UNDEF` and continues. This silently drops filter conditions that the user may expect to be enforced.

## REPL-UNWRAP-01 — Replication Module Panic Elimination

### Current state

`src/replication.rs` contains 7 `.unwrap()` calls and 3 `.expect()` calls on fallible operations (SPI results, slot creation, PG lock acquisition). Under concurrent usage, any of these can trigger a Rust panic, which PostgreSQL surfaces as a `FATAL` error rather than a recoverable `ERROR`.

### What changes

Replace each `.unwrap()`/`.expect()` with:
- `?` operator for functions that return `Result`
- `pg_error!("replication: …")` for recoverable conditions
- A `// SAFETY:` comment if the unwrap is genuinely infallible (e.g. constant initialisation)

### Deliverables

| ID | Deliverable |
|----|-------------|
| REPL-UNWRAP-01a | All `.unwrap()` / `.expect()` in `src/replication.rs` replaced |
| REPL-UNWRAP-01b | Clippy `unwrap_used` lint enabled for `src/replication.rs` |
