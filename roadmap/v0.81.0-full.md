These represent data-integrity risks and race conditions that may manifest under production load or unusual transaction patterns.

## MERGE-SID-01 — HTAP Merge Non-Deterministic SID Fix

### Current state

The HTAP merge CTE in `src/storage/promote.rs` uses `DISTINCT ON (s, o, g)` without an `ORDER BY` clause. PostgreSQL's `DISTINCT ON` without ORDER BY selects an arbitrary row from each group, meaning that when `(s, o, g)` appears in both `main` and `delta`, either SID may be preserved in the merged output. The result is non-deterministic statement identifiers after merge, which breaks provenance queries and CONSTRUCT writeback that key on SIDs.

### What changes

Prepend `ORDER BY s, o, g, i ASC` to each arm of the merge CTE before the `DISTINCT ON` step, guaranteeing that the lowest (oldest) SID is always preserved.

### Deliverables

| ID | Deliverable |
|----|-------------|
| MERGE-SID-01a | `ORDER BY … i ASC` added to merge CTE |
| MERGE-SID-01b | Regression test: same triple in main + delta → SID of main-partition copy is preserved |

## PROMO-ATOMIC-01 — Promotion Status Update Atomicity

### Current state

`src/storage/promote.rs` — after creating the new VP table and copying rows — updates `_pg_ripple.predicates SET table_oid = …` in a separate statement. If the server crashes between the copy and the status update, the VP table exists but the catalog still points to the old table. Recovery falls back to a full re-promotion on next access, which can lose or double triples.

### What changes

Combine the VP table creation, row copy, and predicates status update into a single CTE using PostgreSQL's `WITH … INSERT … RETURNING`:

```sql
WITH new_table AS (
    CREATE TABLE _pg_ripple.vp_{id} (…) -- via dynamic DDL wrapped in a DO block
), copy AS (
    INSERT INTO _pg_ripple.vp_{id} SELECT … FROM _pg_ripple.vp_rare WHERE p = $1
)
UPDATE _pg_ripple.predicates SET table_oid = … WHERE id = $1
```

Because DDL cannot appear inside a CTE, the approach uses an advisory lock + explicit savepoint: (a) acquire lock, (b) check catalog, (c) create table, (d) copy rows, (e) update catalog — all inside a single subtransaction with `ON CONFLICT DO NOTHING` on the catalog update.

### Deliverables

| ID | Deliverable |
|----|-------------|
| PROMO-ATOMIC-01a | Promotion wrapped in subtransaction with savepoint |
| PROMO-ATOMIC-01b | Catalog update uses `ON CONFLICT DO NOTHING` |
| PROMO-ATOMIC-01c | Regression test: simulated crash between copy and catalog update |

## DICT-RACE-01 — Dictionary Encode Race Handling

### Current state

`src/dictionary/mod.rs` calls `RETURNING id` after inserting a new dictionary entry. If a concurrent session deletes the entry (or a hash collision causes a rollback), `RETURNING` yields 0 rows. The current code silently falls back to a second `SELECT`, which may return a wrong ID.

### What changes

After a 0-row `RETURNING` result, emit a `pg_warning!` and retry the full encode() path once. If the second attempt also fails, return `Err(PgError::new("dictionary encode conflict"))`. Remove the silent SELECT fallback.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DICT-RACE-01a | 0-row RETURNING treated as error, not silent fallback |
| DICT-RACE-01b | Single retry with warning on transient conflict |

## FED-TRUNC-01 — Federation JSON Truncation Warning

### Current state

When a federation response exceeds `federation_result_max_bytes`, the JSON is silently truncated mid-stream. The truncated JSON fails to parse, and the error is swallowed. The caller receives a partial (possibly empty) result set with no indication of truncation.

### What changes

Replace silent truncation with:
1. A `pg_warning!("federation response truncated at {} bytes from {}", limit, url)`.
2. Partial materialisation: parse the truncated response for complete JSON objects only and return the successfully parsed subset.
3. An `is_truncated` flag in the federation result that causes the query engine to annotate results with a `WARNING` notice.

### Deliverables

| ID | Deliverable |
|----|-------------|
| FED-TRUNC-01a | WARNING emitted on truncation |
| FED-TRUNC-01b | Partial JSON parse for complete objects |
| FED-TRUNC-01c | Regression test: response exactly at limit, at limit+1, at 2× limit |

## OPT-INNER-01 — Multi-Predicate OPTIONAL→INNER JOIN Promotion

### Current state

## DL-PAR-01 and DL-PAR-02 — Datalog Parallel SCC Correctness

### Current state

`src/datalog/parallel.rs` partitions strata into parallel execution groups. Two bugs exist:
1. (C-12) The intra-stratum cycle check uses stratum-level reachability, not rule-level reachability. Two rules in the same stratum that are mutually recursive are placed in different parallel groups.
2. (CC-05) The SCC detection in the parallel group partitioner uses stratum order rather than topological rule order, producing non-terminating evaluation for certain recursive rule sets.

### What changes

1. Replace the stratum-level cycle check with a rule-level cycle check using Tarjan's SCC algorithm on the rule dependency graph.
2. Ensure parallel group assignment respects topological order: a rule may not be placed in a group that executes before all its dependencies have been evaluated.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DL-PAR-01a | Rule-level SCC analysis replaces stratum-level |
| DL-PAR-02a | Topological order enforced in group assignment |
| DL-PAR-01b | Regression: mutually recursive rules within a stratum are serialised |
| DL-PAR-02b | Regression: rule with inter-stratum dependency is not evaluated early |

## BN-SCOPE-01 — Blank-Node Variable Scoping

### Current state

`src/sparql/plan.rs` emits blank-node variable names (e.g. `_:b0`) directly into the SQL query without scoping them to the enclosing query block. In subqueries, the same blank-node name in an outer and inner scope resolves to the same dictionary ID, causing incorrect JOIN behaviour.

### What changes

Prefix each blank-node variable with a query-scope UUID generated at plan-compile time:
```sql
-- Before: encode('_:b0', 'blank') AS b0
-- After:  encode('_:7f3a2c_b0', 'blank') AS b0
```
The UUID prefix is generated per top-level query parse, ensuring blank nodes across different calls to `sparql_query()` do not alias.

### Deliverables

| ID | Deliverable |
|----|-------------|
| BN-SCOPE-01a | Query-scoped UUID prefix for blank-node variable names |
| BN-SCOPE-01b | Regression: outer-query blank node does not alias subquery blank node |

## DICT-SUBXACT-01 — Dictionary Decode Cache SubXact Invalidation

### Current state

The dictionary decode LRU cache in `src/dictionary/mod.rs` is not invalidated on subtransaction abort. If a subtransaction encodes a new IRI and then aborts, the LRU cache retains the (now-invalid) `id → string` mapping. | DICT-SUBXACT-01a | `SubXactCallback` registered in `_PG_init` |

## CDC-SLOT-01 — CDC Replication Slot Cleanup Worker

### Current state

`src/cdc.rs` creates PostgreSQL replication slots for CDC subscriptions. If a CDC session terminates abnormally (e.g. client disconnect, server restart), the slot is left open and continues consuming WAL. There is no automated cleanup.

### What changes

Add a background worker `cdc_slot_cleanup` that:
1. Wakes on `pg_ripple.cdc_slot_idle_timeout_seconds` interval (default: 3600 s).

3. Calls `pg_drop_replication_slot()` on each idle slot.
4. Logs each dropped slot at `LOG` level.

## MERGE-FENCE-01 — Merge Fence Lock Scope Reduction

### Current state

`src/storage/merge.rs` holds a `ShareUpdateExclusiveLock` for the entire duration of the BRIN reindex + main-partition swap. The lock blocks all concurrent autovacuum on the VP table.

### What changes

Restructure the merge into two phases:
1. **Build phase** (no fence lock): write the new `vp_{id}_main_new` partition and BRIN index.
2. **Swap phase** (short `ExclusiveLock`): rename `main_new` to `main` and drop tombstones. Lock held for `< 50 ms` on typical datasets.

### Deliverables

| ID | Deliverable |
|----|-------------|
| MERGE-FENCE-01a | Two-phase merge with short swap lock |
| MERGE-FENCE-01b | Benchmark: autovacuum no longer blocked during merge |

## PGFINI-01 — Extension Unload Callback Cleanup

### Current state

`_PG_init` (in `src/lib.rs`) registers multiple PostgreSQL callbacks:
- A `SubXactCallback` for dictionary cache invalidation
- An `ExecutorEndHook` for merge worker notification
- A transaction-level callback for mutation journal flush

None of these are unregistered when the extension is dropped (`DROP EXTENSION pg_ripple`). If the extension is subsequently recreated in the same backend session (e.g. in test scripts), the old callbacks remain in PostgreSQL's hook lists pointing to unloaded code, causing undefined behaviour or crashes.

### What changes

Add a `_PG_fini` function that unregisters all registered callbacks:

```rust
#[allow(non_snake_case)]
#[pg_guard]
pub extern "C-unwind" fn _PG_fini() {
    crate::storage::unregister_executor_end_hook();
    crate::storage::mutation_journal::unregister_xact_callback();
    crate::dictionary::unregister_sub_xact_callback();
}
```

The `pgrx` framework calls `_PG_fini` before unloading the shared library. Each unregister function wraps the corresponding PostgreSQL `Unregister*` C call.

### Deliverables

| ID | Deliverable |
|----|-------------|
| PGFINI-01a | `_PG_fini` function implemented and registered |
| PGFINI-01b | Each registered callback has a corresponding unregister path |
| PGFINI-01c | Regression test: `DROP EXTENSION pg_ripple; CREATE EXTENSION pg_ripple;` in same session succeeds without crash |

## PLAN-CACHE-GUC-02 — Plan Cache Key GUC Completeness

### Current state

`src/sparql/plan_cache.rs` `cache_key()` includes only `MAX_PATH_DEPTH` and `BGP_REORDER` in the cache key. The following GUCs also influence SQL generation but are absent:

- `NORMALIZE_IRIS` — when enabled, IRIs are NFC-normalised before dictionary lookup, producing different IDs for the same visual IRI
- `WCOJ_ENABLED` — controls whether WCOJ planner hints are applied  
- `WCOJ_MIN_TABLES` — threshold for WCOJ activation
- `INFERENCE_MODE` — determines which inference rules are applied in the SELECT path

Extend `cache_key()` to include the four additional GUC values:

```rust
    let max_depth  = crate::MAX_PATH_DEPTH.get();
    let bgp_reorder = crate::BGP_REORDER.get();
    let normalize_iris = crate::NORMALIZE_IRIS.get();
    let wcoj_enabled   = crate::WCOJ_ENABLED.get();
    let wcoj_min       = crate::WCOJ_MIN_TABLES.get();
    let inference      = crate::INFERENCE_MODE.get().unwrap_or("off");
    // … hash including all six values …
}
```

| ID | Deliverable |
|----|-------------|
| PLAN-CACHE-GUC-02a | `NORMALIZE_IRIS`, `WCOJ_ENABLED`, `WCOJ_MIN_TABLES`, `INFERENCE_MODE` added to cache key |

## RETRACT-PARAM-01 — Retraction DELETE Parameterisation

### Current state

`src/construct_rules/retract.rs` line ~94 constructs the DELETE statement for flat VP tables using string interpolation:

```rust
let sql = format!(
    "DELETE FROM _pg_ripple.vp_{pred_id} \
     WHERE s = {s} AND o = {o} AND g = {g} AND source = 1"
);
Spi::run(&sql).unwrap_or_else(|e| pgrx::warning!("retract VP: {e}"));
```

Although `s`, `o`, `g` are `i64` values (internal dictionary IDs, not user input), this violates the codebase's architectural rule: "No dynamic SQL string concatenation for values — always use parameterised queries." The inconsistency with the `vp_rare` path immediately below (which correctly uses `Spi::run_with_args`) is confusing and could be a source of regressions if the types change.

### What changes

Replace with the parameterised form:

```rust
Spi::run_with_args(
    "DELETE FROM _pg_ripple.vp_{pred_id_literal} \
     WHERE s = $1 AND o = $2 AND g = $3 AND source = 1",
    &[
        DatumWithOid::from(s),
        DatumWithOid::from(o),
        DatumWithOid::from(g),
    ],
)
.unwrap_or_else(|e| pgrx::warning!("retract VP: {e}"));
```

Note: the table name is formed from the integer `pred_id` (safe) but the `WHERE` clause values must use bind parameters.

### Deliverables

| ID | Deliverable |
|----|-------------|
| RETRACT-PARAM-01a | Flat-VP DELETE uses `Spi::run_with_args()` |
| RETRACT-PARAM-01b | Clippy `use_debug` lint passes on retract.rs |

## SCHEDULER-ERR-01 — CONSTRUCT Rule Scheduler Error Propagation

### Current state

`src/construct_rules/scheduler.rs` topological sort calls `pgrx::error!()` when an internal invariant is violated (missing `in_degree` entry). This immediately aborts the current PostgreSQL transaction with a FATAL-style error. It is not possible for the caller to catch and handle the condition, add context, or attempt recovery.

### What changes

Convert the function to return `Result<Vec<String>, String>` and replace `pgrx::error!()` with `return Err(format!(...))`. The caller in `delta.rs` propagates the error to the SQL function boundary and converts it to a `pgrx::error!()` with added context (rule name, graph IRI).

### Deliverables

| ID | Deliverable |
|----|-------------|

## DRED-FIXPOINT-01 — DRed Full Fixpoint Re-derivation

### Current state

`src/datalog/dred.rs` re-derivation runs only one "seed pass" after a triple is deleted. The code comment says:

> "DRed only does the seed pass (not the full fixpoint) for DRed — this is sufficient because the deleted triple can only break derivations that directly reference it (depth-1)."

This assumption is **incorrect for rules with shared body predicates across strata**. Consider:
- Rule 1: `C(x) :- A(x, y)` 
- Rule 2: `D(x) :- C(x), B(x, z)`

If triple `A(1, 2)` is deleted, Rule 1 retracts `C(1)`. But Rule 2 also derived `D(1)` from `C(1)`, and the single seed pass does not retract `D(1)`.

### What changes

Replace the single seed pass with a stratified fixpoint loop:
1. Remove deleted facts from the affected stratum.
2. Re-run the full seminaive fixpoint for that stratum and all higher strata.
3. Compare derived facts before and after; retract anything that no longer holds.

This is the textbook DRed algorithm. The performance cost is acceptable since retraction is an infrequent operation.

### Deliverables

| ID | Deliverable |
|----|-------------|
| DRED-FIXPOINT-01a | Full stratified fixpoint in DRed re-derivation |
| DRED-FIXPOINT-01b | Regression: 3-hop chain — remove base triple → all derived facts retracted |
| DRED-FIXPOINT-01c | Regression: multi-stratum rule set with shared body predicates |

## RETRACT-HTAP-01 — Retraction HTAP-State Consistency

### Current state

`src/construct_rules/retract.rs` has two code paths:

1. HTAP path (when `is_htap(pred_id)`): insert into tombstones table ✓
2. Flat path (else): `DELETE FROM _pg_ripple.vp_{pred_id}` — direct DELETE

The flat path executes even when HTAP mode is enabled. If the row is in the `_main` partition (not `_delta`), a direct DELETE on the flat `vp_{pred_id}` table name — which in HTAP mode may be an alias or view — will either fail or silently not find the row. The promoted triple is then orphaned in `_main` without a tombstone.

### What changes

Replace the `is_htap()` check condition. In HTAP mode, **all** tables must use the tombstone path. The flat-VP direct DELETE should only execute when HTAP is confirmed disabled and the table is a genuine flat table.

Add a test that:
1. Inserts a triple → merge fires → triple moves to `_main`.
2. Retracts the triple via CONSTRUCT rule deletion.
3. Asserts the triple no longer appears in SPARQL queries.

### Deliverables

| ID | Deliverable |
|----|-------------|
| RETRACT-HTAP-01a | Tombstone path used for all promoted-VP retractions |
| RETRACT-HTAP-01b | Direct DELETE only for flat (non-HTAP) VP tables |
| RETRACT-HTAP-01c | Regression: retract triple that has been merged to `_main` |
