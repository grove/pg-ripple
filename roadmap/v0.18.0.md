# v0.18.0 — SPARQL CONSTRUCT/DESCRIBE/ASK Views

**Effort estimate: 4–6 person-weeks**

**Goal:** Just as v0.11.0 made SELECT queries into live views, this release extends the same incremental maintenance capability to the other SPARQL query forms. CONSTRUCT and DESCRIBE queries can be registered as views that materialise derived triples and keep them current as the base graph changes. ASK queries become boolean live views that flip when a pattern becomes satisfiable or stops being satisfiable — perfect for monitoring and alerting. This feature requires pg-trickle.

---

## CONSTRUCT Views as Live Triple Stores

**Register a CONSTRUCT query as a live view; derived triples stay current automatically.**

A CONSTRUCT query builds a new RDF graph from an existing one by applying a pattern-and-template pair: find all entities matching the pattern, and create new triples from the template. Registering such a query as a view with `create_construct_view()` means the derived triples live in a dedicated named graph that is maintained incrementally. Add a base triple that matches the CONSTRUCT pattern, and the corresponding derived triple appears in the view graph within milliseconds.

---

## DESCRIBE Views

**Materialise descriptions of entities and keep them fresh.**

A DESCRIBE query returns all known facts about one or more entities (their "concise bounded description"). A DESCRIBE view materialises this snapshot for a configured set of entities or entity types. This is useful when you need a pre-built per-entity document that can be served from a cache or an API without re-querying the full graph on every request.

---

## ASK Views as Boolean Monitors

**A live boolean flag that flips when a condition becomes true or false.**

An ASK query returns true if at least one triple matching its pattern exists in the graph. An ASK view maintains a boolean table that reflects the current answer to that question in real time. When data is inserted that makes the pattern satisfiable, the view flips to true; when the last matching triple is deleted, it flips back to false. This is the right building block for constraint monitors ("is there any patient who has been waiting more than 24 hours?"), dashboard indicators, and alerts.

---

## Dependency on pg-trickle

**Incremental view maintenance runs on pg-trickle's event stream.**

All three view types use pg-trickle's change-propagation engine to compute and apply the minimal delta when base data changes. Without pg-trickle, the `create_construct_view`, `create_describe_view`, and `create_ask_view` functions return a `PT800` error explaining what is needed. The views can be manually refreshed using `refresh_view()` even without pg-trickle, but incremental maintenance is not available.
