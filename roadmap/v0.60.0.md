# v0.60.0 — Production Hardening Sprint

> **Full technical details:** [v0.60.0-full.md](v0.60.0-full.md)

**Status: Planned** | **Scope: Large**

> Before pg_ripple can be called production-ready, a handful of sharp edges
> need to be ground down: one storage-layer race condition that can briefly
> make a table invisible, several gaps in automated security testing, and a
> few small SPARQL features that users have already started asking for.
> v0.60.0 is entirely about closing those gaps — no new features, just
> making what already exists rock-solid.

---

## What is this?

Every software project has a moment where it stops adding features and
spends a release making sure everything already built is safe, correct, and
testable. v0.60.0 is that release for pg_ripple. It addresses every outstanding
concern raised in the v0.59.0 deep-analysis review before the project moves
forward to v1.0.0.

---

## Fixing the storage race condition

When pg_ripple's background merge worker compacts data from its fast-write
"delta" store into the optimised "main" store, it briefly drops and recreates
an internal database view. During that brief window, a query arriving at
exactly the wrong millisecond would see an error saying the table doesn't
exist.

v0.60.0 replaces this sequence with an atomic rename-swap: the new table is
built in its entirety, then swapped into place in a single locked operation.
The view definition never changes — only the table behind it is swapped.
There is no window where the table is absent.

To prove this works under real concurrency, a new chaos test hammers the
database with continuous queries while the merge worker churns. It must
record zero errors over a 60-second run before the release ships.

## CI supply-chain hardening

The automated build and test pipeline (GitHub Actions) runs dozens of
external tools — compilers, linters, container scanners — using third-party
scripts. If any of those scripts were silently updated by their authors, a
malicious change could slip into a build undetected.

v0.60.0 pins every external tool in the pipeline to an immutable content
digest (a SHA hash) rather than a floating version tag. Dependabot is
configured to open a pull request whenever an upstream tool releases a new
version, keeping the pins current without sacrificing auditability.

Additionally:
- A lint step is added to CI that actively searches the extension's SQL
  definitions for the `SECURITY DEFINER` keyword. This is a PostgreSQL
  privilege-escalation mechanism; the extension should only use it in
  exactly two places. If a code change accidentally adds it elsewhere,
  CI fails immediately.
- The release workflow runs a container vulnerability scan (Trivy) against
  the published Docker image and fails if it finds any high or critical CVE.

## Automated crash-finding (fuzzing)

Fuzz testing is a technique where a program is fed millions of randomly
mutated inputs to see if any of them cause a crash or incorrect behaviour.
pg_ripple has had a fuzzing harness since v0.47.0 covering the SPARQL parser
and federation result decoder. Three new harnesses are added:

- **GeoSPARQL WKT geometry parser** — feeds random geographic coordinate
  strings to check the parser never panics on malformed input.
- **R2RML mapping documents** — feeds random mapping files to the R2RML
  importer.
- **LLM prompt builder** — feeds random input to the AI prompt sanitizer
  and asserts that no prompt-injection marker (a string that could trick a
  language model into ignoring its instructions) survives the sanitization
  step.

## Geographic distance queries (`geof:distance`)

v0.55.0 added the GeoSPARQL function library for spatial queries. One
commonly requested function was missing: `geof:distance(?a, ?b, <units>)`
— the straight-line distance between two geographic points, in a chosen
unit (metres, kilometres, etc.).

v0.60.0 adds it, backed by PostGIS `ST_Distance`. You can now write SPARQL
queries like "find all museums within 500 metres of this point" directly.

## Kubernetes readiness probe (`/ready`)

When pg_ripple_http (the companion HTTP service) is deployed on Kubernetes,
the cluster needs two separate health signals:

- **Liveness** (`/health`): is the process alive? Kubernetes restarts it if not.
- **Readiness** (`/ready`): is the process ready to accept traffic? Kubernetes
  removes it from the load-balancer until it is.

Previously both were served by `/health`. v0.60.0 adds a dedicated `/ready`
endpoint that returns a 503 error until the service has successfully connected
to PostgreSQL at least once. This prevents Kubernetes from routing traffic to
a pod that is still starting up.

## Backup and restore test

A standard PostgreSQL `pg_dump` / `pg_restore` cycle must work correctly
for pg_ripple databases. This sounds obvious, but has never been verified
automatically in CI. v0.60.0 adds an integration test that:

1. Populates a pg_ripple instance with triples, inferred facts, and SHACL
   validation results.
2. Dumps it with `pg_dump`.
3. Restores to a fresh instance.
4. Asserts that all triple counts, query results, and validation hashes
   match exactly.

## Scope

v0.60.0 is a large release measured by the number of files touched, but it
does not change the user-visible query language or storage format. Every
change is either a bug fix, a test addition, or a safety gate. Upgrading
from v0.59.0 requires no changes to queries, schemas, or application code.
