# v0.33.0 — Documentation Site & Content Overhaul

**Effort estimate: 8–12 person-weeks**

**Goal:** pg_ripple has reached feature maturity across 32 releases. The documentation has kept pace in breadth but not in usability — it grew organically alongside the codebase rather than being designed for the people who use it. This release delivers a complete documentation overhaul: new information architecture, eight feature-deep-dive chapters, a full operations guide, a complete function reference, and a CI harness that keeps every code example honest by running it against a live instance.

---

## Problem-Centric Information Architecture

**The documentation is reorganised around what users need to do, not how the code is structured.**

The new structure serves five distinct user archetypes: the Data Engineer loading and transforming knowledge graphs, the Application Developer writing SPARQL queries and using the HTTP API, the Knowledge Architect designing ontologies and validation rules, the Operations Engineer deploying and monitoring, and the AI/ML Engineer building RAG pipelines and hybrid search. Each archetype has a clear entry point and a guided path through the relevant sections.

---

## Eight Feature Deep-Dive Chapters

**Each major feature gets a complete chapter covering what, how, patterns, pitfalls, and next steps.**

Eight comprehensive chapters cover: Storing Knowledge, Loading Data, Querying with SPARQL, Validating Data Quality, Reasoning and Inference, Exporting and Sharing (including GraphRAG), AI Retrieval and Graph RAG, and APIs and Integration. Each chapter follows a consistent seven-part structure: What and Why, How It Works, Worked Examples, Common Patterns, Performance and Trade-offs, Debugging, and Next Steps. Every code example uses a shared bibliographic fixture dataset so examples across chapters are consistent and comparable.

---

## Complete Operations Guide

**Operators get ten pages covering every aspect of running pg_ripple in production.**

The operations section covers deployment models, all configuration parameters by subsystem, monitoring and observability, performance tuning for different workload profiles, backup and disaster recovery, safe upgrade procedures, scaling guidance, troubleshooting with fifteen or more runbook entries, and security. These pages are written for database administrators who may not be familiar with RDF or SPARQL — the goal is that an experienced PostgreSQL DBA can deploy and manage pg_ripple without needing to understand the knowledge graph concepts.

---

## SQL Function Reference

**Every public function is documented with signature, parameters, and a working example.**

The SQL function reference lists all pg_ripple functions grouped by use case (Loading, Querying, Validating, Reasoning, Exporting, Administration). Each entry has the complete function signature, a description of every parameter, and one working example with expected output. The reference is generated from the source code and checked by CI to stay in sync with every code change.

---

## CI Documentation Test Harness

**Every code example in the documentation is verified to run correctly on every PR.**

A CI job spins up a pg_ripple instance, extracts all fenced SQL code blocks from the documentation, runs them in document order, and compares the output against expected-output comments embedded in the document. Any documentation change that introduces a broken example fails CI before the change can merge. This is the only way to prevent documentation drift — the problem where examples stop working as the code evolves but no one notices for months.
