# v0.58.0 — Temporal Queries, SPARQL-DL & Horizontal Sharding

**Effort estimate: 6–9 person-weeks**

**Goal:** Complete the pre-v1.0.0 feature roadmap by delivering temporal RDF queries, SPARQL-DL, Citus-based horizontal sharding, and PROV-O graph provenance. After this release, the codebase transitions to v1.0.0 production hardening.

## Temporal RDF Queries

**Query data as it existed at any point in time.**
RDF stores often accumulate historical assertions that are later retracted. `pg_ripple.point_in_time(ts TIMESTAMPTZ)` returns a query context in which SPARQL patterns are evaluated against only those triples whose statement ID was inserted before `ts`. Under the hood this leverages PostgreSQL 18's system-time period tracking on the `i` (statement-ID) column without requiring a physical snapshot or a second copy of the data.

A companion `valid_from` / `valid_to` extension integrates RDF application-level validity intervals: any triple can carry reified `schema:validFrom` / `schema:validThrough` metadata that is automatically pushed down as a range filter on VP table scans.

## SPARQL-DL

**Evaluate description-logic class and property queries inside SPARQL.**
SPARQL-DL is a subset of OWL-API's DL query language that can be expressed inside SPARQL via `owl:equivalentClass`, `owl:subClassOf`, and other property axiom patterns. v0.58.0 adds a `SPARQL-DL` mode that re-routes axiom-level BGPs to the reasoner rather than the VP table scanner, enabling queries of the form "give me all subclasses of Drug that are not also Enzyme."

## Citus Horizontal Sharding

**Scale beyond one server's I/O.**
When pg_ripple is installed inside a Citus cluster, VP tables are hash-partitioned across Citus workers using the triple's `s` (subject hash) as the distribution column. The planner emits parallel `SELECT … FROM workers` fan-out queries, automatically using Citus's `run_command_on_shards`. The `merge_worker` coordinates cross-shard merges. This requires Citus 12 and the `pg_columnar` AME already available in v0.57.0.

## PROV-O Graph Provenance

**Automatically record the origin of every inferred and imported triple.**
When `pg_ripple.prov_enabled = on`, every triple inserted by `load_turtle()`, `load_rdf_xml()`, or a Datalog inference stratum records a corresponding `prov:wasDerivedFrom` / `prov:wasGeneratedBy` assertion in a dedicated `_pg_ripple.prov` named graph. Consumers can query provenance with standard SPARQL:

```sparql
SELECT ?triple ?activity WHERE {
  GRAPH <_pg_ripple:prov> {
    ?triple prov:wasGeneratedBy ?activity .
  }
}
```
