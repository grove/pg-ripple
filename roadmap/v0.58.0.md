# v0.58.0 — Temporal Queries, SPARQL-DL & Horizontal Sharding

**Effort estimate: 6–9 person-weeks**

**Goal:** Complete the pre-v1.0.0 feature roadmap by delivering temporal RDF queries, SPARQL-DL, Citus-based horizontal sharding, and PROV-O graph provenance. After this release, the codebase transitions to v1.0.0 production hardening.

## Temporal RDF Queries

**Query data as it existed at any point in time.**
RDF stores often accumulate historical assertions that are later retracted. `pg_ripple.point_in_time(ts TIMESTAMPTZ)` returns a query context in which SPARQL patterns are evaluated against only those triples whose statement ID was inserted before `ts`. Under the hood this leverages PostgreSQL 18's system-time period tracking on the `i` (statement-ID) column without requiring a physical snapshot or a second copy of the data.

A companion `valid_from` / `valid_to` extension integrates RDF application-level validity intervals: any triple can carry reified `schema:validFrom` / `schema:validThrough` metadata that is automatically pushed down as a range filter on VP table scans.

## SPARQL-DL

**Evaluate description-logic class and property queries inside SPARQL.**
SPARQL-DL is a subset of OWL-API's DL query language that can be expressed inside SPARQL via `owl:equivalentClass`, `owl:subClassOf`, and other property axiom patterns. v0.58.0 adds a `SPARQL-DL` mode that re-routes axiom-level BGPs to the reasoner rather than the VP table scanner, enabling queries of the form "give me all subclasses of Drug that are not also Enzyme."

## Citus Horizontal Sharding

**Scale beyond one server's I/O.**
When pg_ripple is installed inside a Citus cluster, VP tables are hash-partitioned across Citus workers using the triple's `s` (subject hash) as the distribution column. The planner emits parallel `SELECT … FROM workers` fan-out queries, automatically using Citus's `run_command_on_shards`. The `merge_worker` coordinates cross-shard merges. This requires Citus 12 and the `pg_columnar` AME already available in v0.57.0.

**pg-trickle alignment.** After a thorough cross-project review against pg-trickle's detailed v0.32.0 and v0.33.0 specs, this release ships thirteen items to make Citus sharding fully compatible with pg-trickle's Citus CDC work:

- VP delta tables are created with `REPLICA IDENTITY FULL` so pg-trickle's per-worker logical replication slots receive complete row images.
- The `merge_worker` emits `pg_notify` fencing signals (`pg_ripple.merge_start` / `pg_ripple.merge_end`) around the main-partition swap, fired on the coordinator only (not inside worker lambdas), with crash-safe `Drop`-guard delivery and a configurable `pg_ripple.merge_fence_timeout_ms` timeout that lets pg-trickle recover autonomously if the notification never arrives.
- The merge swap is changed from `DROP TABLE + CREATE TABLE AS SELECT` to `TRUNCATE + INSERT` to minimize DDL events in the Citus worker WAL streams that pg-trickle's WAL decoder polls.
- `promote_vp_table()` publishes a `pg_ripple.vp_promoted` notification including `shard_count` and `shard_table_prefix` so pg-trickle can enumerate shard tables and create per-worker slots without querying `pg_dist_shard`.
- A new `pg_ripple.citus_trickle_compat` GUC enforces the co-location contract: pg-trickle stream tables that join VP data must use `s BIGINT` as their distribution column. Documentation covers the distribution column override required in pg-trickle v0.33.0's `CREATE STREAM TABLE` syntax (since pg-trickle defaults to `row_id`).
- Row identity for pg-trickle's `DELETE + INSERT` pattern on VP-derived distributed stream tables is documented as `(s, predicate_hash, g)`.
- `_pg_ripple.dictionary` and `_pg_ripple.predicates` are promoted to Citus **reference tables** so VP table joins never require coordinator fan-out.
- `_pg_ripple.vp_rare` is a reference table (keyed by `p`, not `s`; cannot be co-located with VP tables).
- The `statement_id_seq` sequence switches to a bulk-allocation strategy when sharding is on, avoiding per-row `nextval()` on worker nodes.
- A new `pg_ripple.enable_citus_sharding()` function retrofits an existing installation by converting all VP tables and promoting catalog tables to reference tables — idempotent, safe to run more than once.
- `vp_{id}_tombstones` tables are co-located with `vp_{id}_main` by distributing on `s` — without this, the HTAP query path `(main EXCEPT tombstones) UNION ALL delta` becomes a coordinator fan-out, defeating sharding.
- The Temporal RDF trigger (`statement_id_timeline`) is replaced with coordinator-side timeline recording when Citus is enabled, since worker-side triggers cannot efficiently write to a coordinator-local table.
- The TRUNCATE + INSERT merge blocking window (AccessExclusiveLock per shard during the INSERT phase) is documented with scheduling guidance for operators.
- Worker restarts during merge are handled gracefully — per-shard try/catch with retry-on-next-cycle, exposed via `pg_ripple.merge_status()`.
- A joint Compose integration test validates the full stack — 50 k triple load, CDC delivery, a complete merge cycle, worker restart, and pg-trickle frontier recovery — with zero slot-loss errors.

**Before coding starts:** the [full technical spec](v0.58.0-full.md) includes an eight-item coordination checklist (C-1 through C-8). Code inspection of `pg-trickle1` on 2026-04-25 resolved three items immediately: **C-6** (WAL decoder uses `test_decoding` text protocol — DDL rows from merge are silently skipped, no panic) and **C-8** (tombstones already have `s BIGINT NOT NULL` since v0.6.0, no migration needed). **C-7 requires a code change in pg-trickle**: `pgtrickle.create_stream_table()` has no `output_distribution_column` parameter — this must be added to pg-trickle v0.33.0 before VP-sourced distributed stream tables can be co-located with VP tables. Items C-1 through C-5 remain for joint agreement.

## PROV-O Graph Provenance

**Automatically record the origin of every inferred and imported triple.**
When `pg_ripple.prov_enabled = on`, every triple inserted by `load_turtle()`, `load_rdf_xml()`, or a Datalog inference stratum records a corresponding `prov:wasDerivedFrom` / `prov:wasGeneratedBy` assertion in a dedicated `_pg_ripple.prov` named graph. Consumers can query provenance with standard SPARQL:

```sparql
SELECT ?triple ?activity WHERE {
  GRAPH <_pg_ripple:prov> {
    ?triple prov:wasGeneratedBy ?activity .
  }
}
```
