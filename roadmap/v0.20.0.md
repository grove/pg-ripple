# v0.20.0 — W3C Conformance & Stability Foundation

**Effort estimate: 6–8 person-weeks**

**Goal:** Before adding more features, this release pauses to measure and validate what has already been built. The official W3C test suites for SPARQL 1.1 and SHACL Core are integrated into CI, crash recovery is tested systematically, memory leaks are detected, the first formal security review is performed, and a baseline BSBM benchmark at 100 million triples is established.

---

## W3C Conformance Test Suites

**Official W3C tests for SPARQL 1.1 and SHACL Core run in CI and must pass at 95%.**

The W3C publishes official test suites for both SPARQL 1.1 (about 3,000 tests covering query evaluation, update, syntax, and federation) and SHACL Core (covering all constraint types). Until this release, pg_ripple's conformance was assessed by writing custom tests. This release integrates the official W3C test runner directly into the CI pipeline. A pass rate target of 95% is set for SPARQL 1.1 and SHACL Core. Tests that pg_ripple cannot pass are recorded in a known-failures manifest with explanations.

---

## Crash Recovery Testing

**The system must survive being killed in the middle of any operation.**

A database that can corrupt data when the server crashes is not safe for production. This release introduces automated crash recovery tests that simulate the worst case: the PostgreSQL process is killed with SIGKILL during merge operations, bulk loads, and SHACL validation. After each kill, the database is restarted and checked for consistency — no partial merges visible, no orphaned tables, no phantom triples. This establishes the recovery baseline that all future storage changes must maintain.

---

## Memory Leak Detection

**Long-running processes must not accumulate memory over time.**

The background merge worker and the query execution path are both long-running. Even a tiny memory leak that grows with every merge cycle or every query execution will eventually exhaust available memory. Automated leak detection using Valgrind or Rust's built-in leak sanitiser is added to the CI pipeline, with a zero-tolerance policy: no new leaks can be introduced.

---

## Security Review Phase 1

**A systematic review of the most critical attack surfaces.**

The first formal security review covers three areas: SPI injection (verifying that no user-controlled data is ever unsafely concatenated into SQL executed through the PostgreSQL SPI), shared memory safety (verifying that concurrent access to shared-memory structures cannot cause data races or undefined behaviour), and SSRF (verifying that federation endpoints cannot be used to reach internal network addresses). Any findings are fixed before this release ships.

---

## BSBM at 100M Triples Baseline

**Performance numbers at realistic production scale are established.**

The Berlin SPARQL Benchmark at 100 million triples is run and the results are published as the official performance baseline. This baseline serves as the reference for all future performance comparisons: any release that regresses below it by more than 10% triggers an investigation before shipping. The numbers cover bulk load throughput, SPARQL query latency at various complexities, and merge throughput under concurrent write load.
