# v0.41.0 — Full W3C SPARQL 1.1 Test Suite

**Effort estimate: 5–7 person-weeks**

**Goal:** Every significant SPARQL engine bug found to date — including the OPTIONAL-inside-GRAPH failure fixed in v0.40.0 — was caught by manual testing or user reports rather than by a systematic test suite. This release fixes that: the complete official W3C SPARQL 1.1 test suite (~3,000 tests) runs in parallel in under 2 minutes in CI, and a 180-test smoke subset runs on every pull request in under 30 seconds.

---

## The W3C SPARQL 1.1 Test Harness

**The entire official test suite runs automatically on every release build.**

The W3C publishes a comprehensive test suite covering all 13 sub-suites of SPARQL 1.1: basic patterns, optional, aggregates, property paths, federation, update, syntax, and more. Each test specifies input data, a SPARQL query or update, and expected results. The harness in this release parses the W3C test manifests, loads each test's RDF fixtures into an isolated pg_ripple named graph, executes the query or update, and compares results using proper RDF graph equivalence (not just row counting). Blank nodes are compared by canonical traversal order, not by internal ID.

---

## Parallelised for Speed

**8 parallel workers run the full 3,000-test suite in under 2 minutes.**

Each test is independent: it loads its own isolated data and queries its own graph namespace. The test runner dispatches tests across 8 parallel workers using a lock-free work queue. Each worker manages its own pg_ripple graph namespace prefix to prevent cross-test pollution. The full suite — 3,000+ tests — completes in under 2 minutes on an 8-core CI runner, making it practical to include as a required check on every release branch.

---

## Smoke Subset for Pull Request Checks

**180 curated tests run on every pull request in under 30 seconds.**

The 180-test smoke subset covers the three categories most likely to catch SQL-generation bugs: `optional` (80 tests), `aggregates` (60 tests), and `grouping` (40 tests). These run as a required CI check on every pull request and every push to the main branch. A failure in the smoke subset blocks merge. The full suite runs on release branches and periodically on main, with results reported as a non-blocking annotation.

---

## Known-Failures Manifest

**Tests that currently fail are tracked with documented reasons, not silently skipped.**

Any W3C test that pg_ripple does not yet pass is listed in a `known_failures.txt` manifest with a one-line explanation (for example, "OPTIONAL inside GRAPH — fixed in v0.40.0"). Tests in the manifest are reported as "expected failure" (XFAIL) rather than "failure" (FAIL). If an XFAIL test unexpectedly passes, it is flagged as XPASS — a signal that the fix is ready to be removed from the manifest. The smoke subset has zero XFAIL entries at release.

---

## Conformance Badges and Per-Category Reporting

**The README shows the current W3C conformance percentage, updated automatically.**

A CI job uploads a `report.json` artifact containing per-category pass, fail, skip, and timeout counts after every full suite run. The README badge links to the latest report, giving users a clear picture of where pg_ripple stands relative to the full SPARQL 1.1 standard.
