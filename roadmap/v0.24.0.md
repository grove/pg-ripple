# v0.24.0 — Semi-naive Datalog & Performance Hardening

**Effort estimate: 6–8 person-weeks**

**Goal:** The Datalog inference engine gets a major performance upgrade through semi-naive evaluation — a well-known algorithm that avoids redundant re-checking of facts that did not change between iterations. Four missing OWL reasoning rules are added, SPARQL result decoding is batched, and the VP storage layout is tuned based on real workload analysis.

---

## Semi-naive Datalog Evaluation

**The inference engine only processes new facts each iteration, not all facts.**

The naive approach to Datalog fixpoint computation checks every rule against the entire set of known facts on every iteration. For a knowledge graph with a million triples where only ten are new in each round, this means rechecking 999,990 facts that haven't changed — pure wasted work. Semi-naive evaluation maintains a "delta" table of facts that are new since the last iteration, and processes only those. For large ontologies with many rules and long derivation chains, this delivers 5× to 100× faster inference. This improvement was dependent on the stratification correctness fix in v0.23.0 and lands here once that foundation is solid.

---

## Four Additional OWL RL Rules

**More OWL 2 RL reasoning rules are now supported.**

The OWL 2 RL profile specifies dozens of reasoning rules for class hierarchies, property chains, and identity semantics. Four rules that were previously missing are added: full transitive closure of `rdfs:subClassOf` (previously only one level was handled), `owl:allValuesFrom` chaining, inverse-functional property inference (if two entities share the same value for an inverse-functional property, they are the same entity), and sub-property chaining. Combined with the semi-naive algorithm, these rules can be evaluated efficiently even over large ontologies.

## Batch SPARQL Result Decoding

**All term IDs in a result set are decoded in a single database round-trip.**

When SPARQL returns a result set, each cell contains a dictionary ID that must be decoded back to an IRI or literal string for the application. Previously this was done one ID at a time: each unique term required a database lookup. For a result set with 1,000 rows and 5 columns, that could mean thousands of round-trips. This release collects all distinct IDs across the entire result set and resolves them in a single batch query, reducing decoding overhead from O(N) database calls to exactly one.

---

## Property Path Depth Limit

**A configurable depth limit prevents runaway recursion on cyclic graphs.**

Property path operators like `rdfs:subClassOf*` traverse relationships recursively. In a graph with cycles — which are common in real ontologies — an unbounded traversal would loop forever if not for PostgreSQL's cycle-detection clause. This release adds a `pg_ripple.property_path_max_depth` GUC (default 64) that limits the recursion depth. When a query hits the depth limit, a `WARNING` is emitted noting that some paths may have been truncated, but the query continues and returns the paths it did find. This prevents runaway queries without hiding the fact that truncation occurred.

---

## BRIN Index Moved to Statement ID Column

**The read-optimised main partition gets a smarter index for sequential access patterns.**

The BRIN (Block Range INdex) index on the main VP partition was previously on the subject column, which has near-random distribution — making BRIN essentially useless for subject-based lookups. The statement ID column (`i`), by contrast, is monotonically increasing with insertion order, giving BRIN near-perfect correlation. Queries that scan the main partition in insertion order (bulk export, incremental maintenance) now benefit from effective BRIN filtering.
