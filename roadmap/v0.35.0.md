# v0.35.0 — Parallel Stratum Evaluation

**Effort estimate: 5–7 person-weeks**

**Goal:** The Datalog engine evaluates one rule at a time within each stratum. This release allows rules that derive independent predicates — predicates with no shared derivation dependencies — to run concurrently using PostgreSQL's background worker infrastructure. For OWL RL reasoning, which has roughly ten independent rule groups in its first stratum, this enables near-linear speedup with the number of available workers.

---

## Independent Rules Run Concurrently

**Rules that cannot interfere with each other are dispatched to parallel workers.**

Within a single Datalog stratum, rules that derive different predicates and share no derived body predicates with each other are fully independent: their SQL INSERT statements write to different VP tables and can run simultaneously without any coordination overhead. This release identifies those independent groups at rule-compilation time and dispatches each group to a separate PostgreSQL background worker. The main process waits at a barrier until all workers complete, then begins the next semi-naive iteration.

---

## Configurable Worker Count

**The number of parallel workers scales from 1 (sequential) to 16.**

The `pg_ripple.datalog_parallel_workers` GUC controls how many background workers are used. The default of 4 is appropriate for most hardware; larger values suit machines with many CPU cores and large rule sets. Setting the value to 1 disables parallelism entirely, which is useful for debugging rule interactions. The system automatically limits parallelism when the estimated data volume is small (controlled by `pg_ripple.datalog_parallel_threshold`) to avoid the overhead of worker dispatch for trivially small strata.

---

## OWL RL Example

**For a 1M-triple dataset, OWL RL inference completes in 40% of the previous time with 4 workers.**

The OWL RL rule set has approximately ten independent rule groups in its first stratum (subclass, subproperty, domain, range, inverse properties, etc.). With 4 parallel workers, the wall-clock time for full OWL RL closure on a 1 million triple dataset drops to roughly 40% of the sequential time. For larger datasets the improvement is proportionally similar: the bottleneck shifts from CPU computation to storage I/O at very high triple counts, but the speedup remains substantial in the typical range.

---

## SPARQL Queries See Fresh Derived Triples Sooner

**Parallel evaluation reduces the staleness window after bulk data loads.**

When a large bulk load triggers a full inference run, the time between load completion and the moment when derived triples are visible to SPARQL queries (the "staleness window") is directly proportional to inference time. By cutting inference time with parallelism, this release reduces how long SPARQL queries might observe a partially-inferred knowledge graph. Applications that load data and then immediately query it benefit directly from the shorter wait.
