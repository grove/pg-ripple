### BIDIOPS-QUEUE-01 — Event queue depth limits and dead-letter

**Root cause**: v0.77.0 ships `_pg_ripple.event_queue` but with no upper bound and no dead-letter destination. A relay that goes down for hours leaves the queue growing unbounded; if the inbound write path holds even a brief lock against the queue table, ingest throughput collapses; expired events with no consumer accumulate forever. World-class CDC systems (Debezium → Kafka, AWS DMS → Kinesis) have these primitives at the transport layer; pg_ripple needs them inside the database since the queue *is* a database table.

**API additions** to `create_subscription`:

```sql
pg_ripple.create_subscription(
    name                  TEXT,
    target_graph          TEXT     DEFAULT NULL,
    frame                 JSONB,
    filter_sparql         TEXT     DEFAULT NULL,
    exclude_graphs        TEXT[]   DEFAULT NULL,
    propagation_depth     SMALLINT DEFAULT 1,
    rewrite_target_graph  TEXT     DEFAULT NULL,
    on_missing_rewrite    TEXT     DEFAULT 'emit_canonical',
    emit_base             BOOLEAN  DEFAULT TRUE,
    transaction_grouping  TEXT     DEFAULT 'subject',
    -- v0.78.0 additions:
    max_queue_depth       BIGINT   DEFAULT 1000000,    -- soft cap
    dead_letter_after     INTERVAL DEFAULT '7 days',   -- events older than this DLQ
    overflow_policy       TEXT     DEFAULT 'pause'     -- 'pause' | 'drop_oldest' | 'drop_newest'
) RETURNS VOID
```

**Catalog tables**:

```sql
-- Events that exceeded max_queue_depth, hit dead_letter_after, or were
-- nacked beyond the configured retry limit.
CREATE TABLE _pg_ripple.event_dead_letters (
    event_id          UUID    PRIMARY KEY,
    subscription_name TEXT    NOT NULL,
    payload           JSONB   NOT NULL,
    emitted_at        TIMESTAMPTZ NOT NULL,
    dead_lettered_at  TIMESTAMPTZ DEFAULT now(),
    reason            TEXT    NOT NULL,         -- 'queue_overflow' | 'expired' | 'paused_overflow' | 'nack_limit'
    nack_history      JSONB,                    -- array of {at, reason} entries
    last_attempt_at   TIMESTAMPTZ
);
CREATE INDEX ON _pg_ripple.event_dead_letters (subscription_name, dead_lettered_at);
```

**Semantics**:

1. **Inbound writes never block on outbound queue depth.** The emit path checks queue depth via a non-blocking estimate (`pg_class.reltuples` per subscription, refreshed by the existing autovacuum stats). Exact-count check happens lazily in the background-worker sweep.
2. **`overflow_policy = 'pause'`** (default): when the soft cap is hit, the subscription is marked `paused = true` in `_pg_ripple.subscriptions`. New events are still appended (so no in-flight transaction fails for queue reasons), but emit emits a `WARNING` once per minute summarising the depth and pause state. Background sweep moves overflow events to `event_dead_letters` with `reason = 'paused_overflow'` until depth falls below `max_queue_depth * 0.9`.
3. **`overflow_policy = 'drop_oldest'`**: background sweep moves the oldest events (by `emitted_at`) to dead-letter as soon as depth exceeds the cap.
4. **`overflow_policy = 'drop_newest'`**: background sweep refuses to enqueue new events; the inbound write succeeds but the new event is appended directly to dead-letter with `reason = 'queue_overflow'`.
5. **Aged-out dead-lettering**: a background worker runs once per minute and moves any `event_queue` row with `emitted_at < now() - dead_letter_after` to `event_dead_letters` with `reason = 'expired'`.
6. **Nack-limit dead-lettering**: events nacked more than `max_nacks` times (GUC `pg_ripple.event_max_nacks`, default 10) move to dead-letter with `reason = 'nack_limit'` and the full nack history.

**Operator API**:

```sql
-- List dead-lettered events for a subscription (paginated).
pg_ripple.list_dead_letters(
    subscription_name TEXT,
    since             TIMESTAMPTZ DEFAULT NULL,
    limit_n           INT         DEFAULT 100
) RETURNS TABLE (event_id UUID, payload JSONB, reason TEXT, dead_lettered_at TIMESTAMPTZ);

-- Re-enqueue a dead-lettered event (e.g. after fixing the relay or target system).
-- Inserts back into event_queue with a fresh emitted_at and clears any leases;
-- preserves event_id for traceability.
pg_ripple.requeue_dead_letter(event_id UUID) RETURNS VOID;

-- Permanently drop a dead-lettered event after operator review.
pg_ripple.drop_dead_letter(event_id UUID) RETURNS VOID;
```

**HTTP additions**:

| Method | Path | Purpose |
|---|---|---|
| `GET`  | `/subscriptions/{name}/dead-letters?since=&limit=` | List dead-lettered events. |
| `POST` | `/dead-letters/{event_id}/requeue` | Re-enqueue. |
| `DELETE` | `/dead-letters/{event_id}` | Permanently drop. |

**Tests**:
- Cap at 100 events, push 200; verify subscription pauses, the first 100 stay in `event_queue`, the next 100 land in `event_dead_letters` with `reason = 'paused_overflow'`. Resume; verify queue drains.
- `overflow_policy = 'drop_oldest'`: cap at 100, push 200; verify the queue holds the latest 100 and dead-letter holds the oldest 100.
- `overflow_policy = 'drop_newest'`: same setup; verify the queue holds the oldest 100 and dead-letter holds the newest 100.
- `dead_letter_after = '5 seconds'`: emit one event, do not consume; verify it moves to dead-letter after 5 seconds with `reason = 'expired'`.
- Nack-limit: nack the same event 10 times; verify it lands in dead-letter with full `nack_history`.
- `requeue_dead_letter`: requeue an expired event; verify it appears in `next_event` with the original `event_id`.
- Inbound-write isolation: simulate a subscription with depth 10× the soft cap; verify ingest throughput against an unrelated graph is within 5 % of baseline (gated under BIDIOPS-PERF-01).
- HTTP round-trip: list / requeue / drop dead-letters via REST.

### BIDIOPS-PAUSE-01 — Operator pause / resume

**Root cause**: Target systems go down for maintenance windows. Operators need a graceful "stop emitting outbound events for the next 30 minutes" without altering the subscription definition or risking event loss.

**API**:

```sql
pg_ripple.pause_subscription(name TEXT, reason TEXT DEFAULT NULL) RETURNS VOID;
pg_ripple.resume_subscription(name TEXT) RETURNS VOID;
```

**Semantics**:

1. `pause_subscription` sets `paused = true` and `pause_reason = ...` in `_pg_ripple.subscriptions`.
2. `next_event` returns NULL for paused subscriptions (no events leased, no rows touched).
3. New events continue to enqueue into `event_queue` until `max_queue_depth` is hit (then BIDIOPS-QUEUE-01 overflow policy kicks in).
4. `resume_subscription` clears `paused`, emits an `INFO` log, and the subscription is immediately consumable.
5. The pause status is reflected in `bidi_status()` (BIDIOPS-DASH-01) and `bidi_health()` reports `paused` if any subscription is paused.

**Tests**:
- Pause; verify `next_event` returns NULL while events accumulate. Resume; verify all queued events are consumable.
- Pause with `reason = 'crm maintenance window'`; verify `bidi_status()` exposes the reason.
- Idempotent pause/resume: calling either twice is a no-op.

### BIDIOPS-EVOLVE-01 — Schema-evolution policies

**Root cause**: v0.77.0 freezes the wire `version` field but does not say what happens when in-flight subscriptions change shape mid-flight. Three concrete cases:

- The subscription's `frame` is updated to add or remove a predicate (e.g. CRM starts tracking `ex:vipStatus`).
- The `iri_pattern` for a graph in `rewrite_target_graph` is updated (e.g. CRM moves from `https://crm.example.com/contacts/{id}` to `https://crm.example.com/v2/contacts/{id}`).
- `exclude_graphs` is updated (e.g. a new partner graph is added).

Without explicit policy, every operator builds their own ad-hoc story. Subtle data-loss bugs follow.

**API**: each `create_subscription` and corresponding `alter_subscription` accepts:

```sql
pg_ripple.alter_subscription(
    name                       TEXT,
    -- any field from create_subscription, plus:
    frame_change_policy        TEXT DEFAULT 'reframe_queued',  -- 'reframe_queued' | 'drain_then_switch'
    pattern_change_policy      TEXT DEFAULT 'no_retroactive',  -- 'no_retroactive' | 'reframe_queued'
    exclude_change_policy      TEXT DEFAULT 'apply_to_new'     -- 'apply_to_new' | 'apply_to_queued'
) RETURNS VOID;
```

**Frame change policies**:

- `reframe_queued` (default): when the frame is altered, all queued events for this subscription are reframed using the new frame at next-pull time. Events currently leased finish under the old frame (for safety: the receiver may already be applying them).
- `drain_then_switch`: the subscription is implicitly paused; new emit goes to a parallel queue under the new frame; once the old queue drains, the parallel queue takes over.

**Pattern change policies**:

- `no_retroactive` (default): queued events keep the original pattern's IRIs; new events use the new pattern. Matches v0.77.0 BIDI-REF-01's "no retroactive rewrite" rule.
- `reframe_queued`: queued events are re-rewritten under the new pattern at next-pull time. Useful when the old pattern was wrong and any in-flight events need correcting.

**Exclude change policies**:

- `apply_to_new` (default): the new `exclude_graphs` list applies only to events emitted after the alter. Queued events drain unchanged. This is correct when adding a new excluded graph (we don't want to discard already-emitted legitimate events).
- `apply_to_queued`: re-apply the new exclude list to queued events at next-pull time. Filtered-out events move to dead-letter with `reason = 'exclude_filter_retroactive'`.

**Audit table**:

```sql
CREATE TABLE _pg_ripple.subscription_schema_changes (
    subscription_name TEXT        NOT NULL,
    changed_at        TIMESTAMPTZ DEFAULT now(),
    changed_by        TEXT,                   -- session_user
    field             TEXT        NOT NULL,   -- 'frame' | 'iri_pattern' | 'exclude_graphs' | ...
    old_value         JSONB,
    new_value         JSONB,
    policy_applied    TEXT,
    affected_event_count BIGINT
);
CREATE INDEX ON _pg_ripple.subscription_schema_changes (subscription_name, changed_at);
```

**Tests**:
- Frame change with `reframe_queued`: queue 5 events under frame v1, alter frame to add a predicate, pull; verify the new predicate appears in `after`/`base` for the queued events.
- Frame change with `drain_then_switch`: same setup; verify subscription pauses, queued events drain under v1, new events use v2.
- `iri_pattern` change with `no_retroactive`: queue 5 events under pattern v1, alter pattern, pull; verify queued events still carry pattern-v1 IRIs and a row is recorded in `subscription_schema_changes`.
- `iri_pattern` change with `reframe_queued`: same setup; verify queued events carry pattern-v2 IRIs.
- `exclude_graphs` change with `apply_to_queued`: queue 5 events from graph A, add A to `exclude_graphs`; verify queued events move to dead-letter.
- All three changes record one row per change in `subscription_schema_changes`.

### BIDIOPS-AUTH-01 — Per-subscription auth and scope tokens

**Root cause**: `pg_ripple_http` today supports bearer tokens for the entire HTTP surface. There is no way to say "this token can only consume the `crm_relay` subscription" or "this token can pull but not linkback". In any deployment with more than one relay, the absence of per-subscription scoping is a meaningful security gap.

**Catalog table**:

```sql
CREATE TABLE _pg_ripple.subscription_tokens (
    token_hash        BYTEA       PRIMARY KEY,             -- SHA-256 of the raw token
    subscription_name TEXT        NOT NULL,
    scopes            TEXT[]      NOT NULL,                -- subset of {pull, ack, nack, linkback, unredacted}
    label             TEXT,                                -- operator-friendly name
    created_at        TIMESTAMPTZ DEFAULT now(),
    last_used_at      TIMESTAMPTZ,
    revoked_at        TIMESTAMPTZ
);
CREATE INDEX ON _pg_ripple.subscription_tokens (subscription_name) WHERE revoked_at IS NULL;
```

**API**:

```sql
-- Returns the raw token (shown ONCE; subsequent calls return only metadata).
-- Token format: 'pgrt_' || base64url(32 random bytes).
pg_ripple.register_subscription_token(
    subscription_name TEXT,
    scopes            TEXT[]  DEFAULT ARRAY['pull','ack','linkback'],
    label             TEXT    DEFAULT NULL
) RETURNS TEXT;

pg_ripple.revoke_subscription_token(token_hash BYTEA) RETURNS VOID;

pg_ripple.list_subscription_tokens(subscription_name TEXT)
RETURNS TABLE (token_hash BYTEA, scopes TEXT[], label TEXT,
               created_at TIMESTAMPTZ, last_used_at TIMESTAMPTZ, revoked_at TIMESTAMPTZ);

-- Operator escape hatch: rotate by registering a new token, distributing it,
-- and revoking the old one once last_used_at is older than the deployment's
-- token-cache TTL.
```

**HTTP enforcement**:

- All `/subscriptions/{name}/*`, `/events/{event_id}/*`, and `/dead-letters/*` endpoints require a bearer token whose `subscription_name` matches the path's subscription and whose `scopes` includes the action being performed:
  - `GET /subscriptions/{name}/events` requires `pull`.
  - `POST /events/{event_id}/ack` requires `ack`. The handler verifies the event belongs to the token's subscription before accepting.
  - `POST /events/{event_id}/nack` requires `nack` (or `ack` if `nack` not granted; nack is a strict subset of ack semantically).
  - `POST /events/{event_id}/linkback` requires `linkback`.
- Mismatch → 403 with `{"error": "scope required: linkback"}`.
- Per-process LRU cache of `(token_hash → token row)` with 60-second TTL keeps verification fast; revocation visibility is bounded by the TTL (documented).
- A global admin token is still accepted for operator endpoints (`pause_subscription`, `register_subscription_token`); admin tokens are stored separately in `_pg_ripple.admin_tokens` and never carry per-subscription scope.

**Tests**:
- Register a token with `scopes => ARRAY['pull']`; verify pull works, ack returns 403.
- Token for subscription A used against subscription B → 403.
- Revoke a token; verify subsequent calls fail (allowing for the 60-second cache TTL).
- Token cache: hammer the same token 10k times; verify only one DB lookup per TTL window.
- Rotation: register new, dispatch, verify both work concurrently, revoke old, verify last_used_at on old is up-to-date.

### BIDIOPS-REDACT-01 — Frame-level redaction

**Root cause**: Outbound events emit frame predicates verbatim. PII (`ex:taxId`, `ex:phone`, `ex:passportNumber`), secrets (`ex:apiKey`), and HIPAA-bearing predicates have no first-class suppression mechanism. Operators today must omit them from the subscription frame entirely — but then receivers that *should* see the cleartext (a separate trusted relay) can't.

**API**: extend the JSON-LD frame syntax with `"@redact": true`:

```json
{
  "@context":  { "ex": "https://example.com/ns#" },
  "@type":     "ex:Contact",
  "ex:name":   {},
  "ex:phone":  { "@redact": true },
  "ex:email":  {},
  "ex:taxId":  { "@redact": true }
}
```

**Semantics**:

1. Predicates marked `"@redact": true` are emitted in `after` and `base` as `{"@redacted": true}` (literal object, not a string).
2. CAS still works: receivers compare `actual[predicate] == base[predicate]` with redacted-on-both-sides yielding equality; the receiver knows it cannot independently verify a redacted CAS and either trusts pg_ripple's claim or escalates via the reconciliation toolkit.
3. **Unredacted scope token**: a subscription token with `scopes => ARRAY['pull','ack','linkback','unredacted']` receives the cleartext values. The same `event_queue` row is rendered differently per consumer at pull time based on the requesting token's scopes.
4. The frame's redaction declaration is the *only* source of truth — there is no separate "predicates allowed in unredacted mode" list. If a frame declares a predicate redacted, only `unredacted`-scoped tokens see it.
5. Internal `_pg_ripple.event_queue.payload` always stores the unredacted shape; redaction happens at pull time so that elevating a token does not require re-emitting events.

**Audit interaction**: every pull by an `unredacted`-scoped token records `action = 'pull_unredacted'` in `_pg_ripple.event_audit`, distinct from regular pulls.

**Tests**:
- Frame with `ex:phone` redacted; pull with non-unredacted token; verify `after["ex:phone"] == {"@redacted": true}`. Pull the same event with an unredacted token; verify cleartext.
- CAS with redacted predicate: receiver applies `base["ex:phone"] == {"@redacted": true}`; verify the receiver has no way to compare against actual cleartext but can still acknowledge.
- Frame change to add `@redact` to a predicate after events are queued: under `frame_change_policy = 'reframe_queued'`, queued events render with the new redaction at pull time.
- `event_audit` records `pull_unredacted` distinct from `pull` for the same event.

### BIDIOPS-AUDIT-01 — Audit log

**Root cause**: For compliance and forensics, every outbound event consumption needs to be recorded with who, what, and when.

**Catalog table**:

```sql
CREATE TABLE _pg_ripple.event_audit (
    audit_id          BIGSERIAL   PRIMARY KEY,
    event_id          UUID        NOT NULL,
    subscription_name TEXT        NOT NULL,
    action            TEXT        NOT NULL,    -- 'pull' | 'pull_unredacted' | 'ack' | 'nack' | 'linkback' | 'abandon_linkback' | 'requeue_dead_letter' | 'drop_dead_letter'
    actor_token_hash  BYTEA,                   -- NULL for SQL-direct callers; populated for HTTP callers
    actor_session     TEXT,                    -- session_user when called via SQL
    http_remote_addr  INET,                    -- populated by pg_ripple_http
    observed_at       TIMESTAMPTZ DEFAULT now(),
    extra             JSONB                    -- nack reason, linkback target_id, dead-letter requeue reason, etc.
);
CREATE INDEX ON _pg_ripple.event_audit (event_id);
CREATE INDEX ON _pg_ripple.event_audit (subscription_name, observed_at);
CREATE INDEX ON _pg_ripple.event_audit (observed_at);  -- for retention sweep
```

**Retention**: GUC `pg_ripple.audit_retention` (default `90 days`). A background worker prunes rows older than the retention window once per hour. Setting the GUC to `0` disables retention pruning (manual archival required).

**SQL-direct caller attribution**: when a user calls `next_event` directly from psql, the audit row carries `actor_session = session_user` and `actor_token_hash = NULL`. HTTP-driven calls populate `actor_token_hash` (the SHA-256 of the bearer token used) and `http_remote_addr`.

**Tests**:
- Pull / ack / nack / linkback an event; verify one audit row per call with the correct fields.
- HTTP pull populates `actor_token_hash` and `http_remote_addr`; SQL pull populates `actor_session` only.
- Retention sweep: set retention to 1 second; insert a row, wait 2 seconds, run the worker; verify the row is gone.
- Dead-letter requeue is audited.

### BIDIOPS-PROPTEST-01 — Property-based convergence harness

**Root cause**: v0.77.0 ships per-feature unit tests and one CRM⇄ERP fixture. There is no test that random sequences of inserts, updates, and deletes from N concurrent sources converge to a deterministic resolved projection. World-class CDC systems (Materialize, FoundationDB) have property-based test suites of this shape.

**Implementation**: Rust `proptest` crate. Strategies generate:

- Random source-graph IRIs from a small fixed set (`{<urn:source:s1>, ..., <urn:source:s4>}`).
- Random subjects from a small fixed set (`{ex:e1, ..., ex:e10}`) — keeping cardinality small ensures conflicts actually happen.
- Random predicates from a fixed schema with mixed `sh:maxCount 1` / unbounded.
- Random literal values (strings, integers, dates).
- Random source timestamps within a fixed window.
- Random operations: `insert(graph, subject, predicate, value, timestamp)`, `update(...)`, `delete(graph, subject, predicate)`, `delete_subject(graph, subject)`.

**Properties asserted**:

1. **Determinism**: applying the same sequence twice (with `latest_wins` everywhere) yields the same `_pg_ripple.conflict_winners`.
2. **Order-independence under `latest_wins`**: shuffling a sequence preserves the resolved projection (because comparison is on source timestamps, not arrival order).
3. **No-loss invariant**: every triple ever inserted is recoverable via a graph-scoped query of its source graph (losing assertions are non-destructive).
4. **`source_priority` correctness**: with `source_priority [s1, s2]`, the resolved value for a `(subject, predicate)` slot is always from the highest-priority source that has a value, regardless of timestamp.
5. **Linkback round-trip**: a sequence of `INSERT → relay-create → record_linkback → ack → subsequent UPDATE` produces the same final state as `INSERT → direct rewrite of subject IRI → UPDATE`.
6. **Convergence under retries**: any subset of operations replayed (idempotent CDC) yields the same final state as a single application.

**Setup**: each property test starts a fresh pg_ripple instance via `cargo pgrx test`, applies the random sequence via SPI, and asserts properties via SPARQL queries.

**Tests**: 100k+ random sequences per CI run, with shrinking on failure to find a minimal counterexample.

### BIDIOPS-CHAOS-01 — Jepsen-style fault injection

**Root cause**: v0.77.0 documents crash-recovery semantics for `record_linkback` and the subscription buffer, but no test actually crashes pg_ripple or the relay mid-flow.

**Implementation**: a Rust harness using `tokio` to drive concurrent inbound writes, outbound consumption, linkbacks, and acks while injecting controlled faults:

- **Crash 1**: kill the relay process between `next_event` and `ack_event`.
- **Crash 2**: kill the relay process between `record_linkback` and `ack_event`.
- **Crash 3**: kill pg_ripple (SIGKILL) mid-`conflict_winners` recompute.
- **Crash 4**: kill pg_ripple mid-`record_linkback` flush (between writing `owl:sameAs` and emitting buffered events).
- **Crash 5**: kill the HTTP companion mid-pull; verify the lease expires and the event is redelivered.
- **Network**: insert a 10s sleep between HTTP pull and ack; verify the lease behaviour and that re-pulls are blocked while leased.

**Properties asserted after each fault**:

1. No duplicate target creates: every `record_linkback(target_id => ...)` for the same hub subject either succeeds idempotently or surfaces the existing `owl:sameAs`.
2. No lost CDC events: every inbound write is reflected in either `event_queue`, `event_dead_letters`, or as already-acked.
3. Queue eventually drains: with the relay restored and ack flowing, queue depth converges to zero.
4. Dead-letter contents are exactly the events that hit a documented policy limit; no event is in dead-letter without a corresponding `reason`.
5. `subscription_buffer` rows are flushed in `sequence` order even if `record_linkback` is retried after a crash.
6. Audit log entries are not duplicated by retries: a successful `record_linkback` after a crash recovers and emits exactly one audit row.

**Setup**: docker-compose with pg_ripple, `pg_ripple_http`, and a fake target system; a controller process that orchestrates the faults and asserts the properties.

### BIDIOPS-RECON-01 — Reconciliation toolkit

**Root cause**: v0.77.0's receiver guidance ("on CAS divergence: re-apply / escalate / force / dead-letter") is prose only — there are no in-database primitives. Operators building bespoke relays reimplement the queue, the dispatch table, and the operator UI from scratch every time.

**Catalog table**:

```sql
CREATE TABLE _pg_ripple.reconciliation_queue (
    reconciliation_id BIGSERIAL   PRIMARY KEY,
    event_id          UUID        NOT NULL,
    subscription_name TEXT        NOT NULL,
    enqueued_at       TIMESTAMPTZ DEFAULT now(),
    leased_until      TIMESTAMPTZ,
    leased_by         TEXT,
    divergence_summary JSONB      NOT NULL,                -- {"predicate_iri": {"actual": ..., "base": ..., "after": ...}}
    resolved_at       TIMESTAMPTZ,
    resolution        TEXT,                                -- 'accept_external' | 'force_internal' | 'merge_via_owl_sameAs' | 'dead_letter'
    resolved_by       TEXT,
    resolution_note   TEXT
);
CREATE INDEX ON _pg_ripple.reconciliation_queue (subscription_name, leased_until, enqueued_at)
    WHERE resolved_at IS NULL;
```

**API**:

```sql
-- Called by the relay client when CAS fails and divergence is detected
-- (some actual values differ from base in ways inconsistent with after).
pg_ripple.reconciliation_enqueue(
    event_id            UUID,
    divergence_summary  JSONB
) RETURNS BIGINT;  -- reconciliation_id

-- Operator pulls the next unresolved divergence (lease + skip-locked, mirroring next_event).
pg_ripple.reconciliation_next(subscription_name TEXT)
RETURNS TABLE (reconciliation_id BIGINT, event_id UUID, payload JSONB,
               divergence_summary JSONB, enqueued_at TIMESTAMPTZ);

-- Operator resolves the divergence with one of four actions.
pg_ripple.reconciliation_resolve(
    reconciliation_id BIGINT,
    action            TEXT,             -- 'accept_external' | 'force_internal' | 'merge_via_owl_sameAs' | 'dead_letter'
    note              TEXT DEFAULT NULL
) RETURNS VOID;
```

**Resolution semantics**:

- **`accept_external`**: ingest the external system's actual values into the source graph as if they had arrived via `ingest_json`; emit a NOTICE; ack the original event as no-op. The external value becomes the resolved-projection winner if conflict policies say so.
- **`force_internal`**: re-emit the same event as a "force overwrite" (ignoring `base`); the relay applies it unconditionally on next pull.
- **`merge_via_owl_sameAs`**: assert `owl:sameAs` between the two divergent values' subjects (used when divergence reveals a duplicate that should have been merged earlier); ack.
- **`dead_letter`**: move the event to `event_dead_letters` with `reason = 'reconciliation_dead_letter'` and the divergence summary in `extra`.

**HTTP endpoints**:

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/events/{event_id}/divergence` | Enqueue (relay → pg_ripple). |
| `GET`  | `/subscriptions/{name}/divergences?limit=N` | Pull (operator). |
| `POST` | `/divergences/{reconciliation_id}/resolve` | Resolve. |

**Tests**:
- Receiver simulation: change `actual["ex:phone"]` between two events; second event's CAS fails; verify `reconciliation_enqueue` is called with the right divergence summary.
- All four resolution actions exercised end-to-end with verified post-state.
- Concurrent operators: two `reconciliation_next` calls return different rows.

### BIDIOPS-DASH-01 — Consolidated operations surface

**Root cause**: v0.77.0 has `graph_stats()` per graph but no per-subscription health view. Operators monitoring N relays today need to query 5 different tables.

**API**:

```sql
-- One row per subscription.
pg_ripple.bidi_status() RETURNS TABLE (
    subscription_name        TEXT,
    paused                   BOOLEAN,
    pause_reason             TEXT,
    queue_depth              BIGINT,
    queue_oldest_age         INTERVAL,
    dead_letter_count        BIGINT,
    conflict_rejection_rate  DOUBLE PRECISION,    -- per minute, last 5 min
    pending_linkback_count   BIGINT,
    pending_linkback_oldest_age INTERVAL,
    rewrite_miss_count_24h   BIGINT,
    last_pull_at             TIMESTAMPTZ,
    last_ack_at              TIMESTAMPTZ,
    reconciliation_open      BIGINT
);

-- Single overall status with triggering conditions.
pg_ripple.bidi_health() RETURNS TABLE (
    status     TEXT,                 -- 'healthy' | 'degraded' | 'paused' | 'failing'
    reasons    TEXT[],               -- e.g. ['crm_relay queue depth 850000', 'erp_relay paused 30 min']
    checked_at TIMESTAMPTZ
);
```

**Health rules**:

- `paused`: any subscription is paused.
- `degraded`: any subscription queue depth > 80 % of `max_queue_depth`, OR any pending linkback older than `linkback_timeout * 0.5`, OR open reconciliation count > 0.
- `failing`: any subscription queue depth at `max_queue_depth`, OR any dead-letter rate > 1/minute, OR any `pending_linkback_oldest_age > linkback_timeout`.
- `healthy`: none of the above.

**HTTP**:

- `GET /bidi/status` — returns `bidi_status()` as JSON.
- `GET /bidi/health` — returns `bidi_health()`. HTTP status code mirrors: `healthy` → 200, `degraded`/`paused` → 200 with body, `failing` → 503.

**Prometheus metrics**: `pg_ripple_http` exposes per-subscription gauges (`pgripple_queue_depth`, `pgripple_dead_letter_count`, `pgripple_pending_linkback_count`, `pgripple_conflict_rejection_rate`) so existing observability stacks pick them up automatically.

**Tests**:
- After ingest with conflicts, verify `conflict_rejection_rate` is non-zero.
- After pausing a subscription, verify `bidi_health()` returns `paused`.
- After hitting `max_queue_depth`, verify `failing`.
- HTTP `/bidi/health` returns 503 when failing.

### BIDI-SPEC-01 — RDF Bidirectional Integration Profile (vendor-neutral spec)

**Root cause**: Roughly 70 % of v0.77.0 + v0.78.0 — conflict policies, late-binding rewrite via `owl:sameAs`, sparse-`base` CAS with framing symmetry, target-assigned-ID linkback, flat wire format with `Link: rel="describedby"`, frame-level redaction, four-action reconciliation, schema-evolution policies, audit, scope tokens, convergence properties — is not Postgres-specific. It is a missing layer in the RDF ecosystem: every triplestore has its own CDC story; none has a coherent *bidirectional integration* story. Without a vendor-neutral specification, pg_ripple's design value gets read as "another Postgres-extension feature" rather than as defining the protocol.

**Deliverable**: `docs/spec/rdf-bidi-integration-v1.md` — a standalone, vendor-neutral specification under W3C-style normative language (MUST / SHOULD / MAY). Structure:

1. **Motivation** — the eight integration concerns existing CDC mechanisms do not address.
2. **Conformance levels** — Bidi-1.0 (data + protocol) and Bidi-1.0-Ops (additionally schema evolution, auth, redaction, audit, ops surface).
3. **Data model** — sources as named graphs (`urn:source:` convention), per-source `prov:generatedAtTime` annotations, four conflict-resolution policies, normalize-aware echo suppression, `owl:sameAs` equivalence.
4. **Wire format** — flat JSON event with `version` discriminator, JSON-LD `after`/`base`, framing-symmetry rule for byte-string CAS, HTTP `Link: rel="describedby"` schema advertisement, object-level grouping.
5. **Receiver protocol** — `next_event` / `ack_event` / `nack_event` semantics, sparse-CAS algorithm, linkback for target-assigned IDs, subscription buffering during pending linkback, late-binding IRI rewrite, loop prevention via `exclude_graphs` + `propagation_depth`.
6. **Reconciliation** — `reconciliation_enqueue` / `reconciliation_next` and the four resolution actions (`accept_external` / `force_internal` / `merge_via_owl_sameAs` / `dead_letter`).
7. **Schema evolution** — frame, IRI-pattern, exclude-graphs change policies.
8. **Auth and redaction** — per-subscription scope tokens (SHA-256 hashed at rest), frame-level `@redact: true`, `unredacted` scope.
9. **Audit** — required fields and retention.
10. **Operations surface** — required per-subscription status fields and overall health vocabulary.
11. **Conformance test corpus** — 15 black-box scenarios; Bidi-1.0 requires 1\u20138, Bidi-1.0-Ops additionally 9\u201315.
12. **Non-normative reference implementation** — explicit pointer to pg_ripple with a list of pg_ripple-specific concerns explicitly *out of scope* for the spec (HTAP storage layout, queue substrate, HTTP companion, migration mechanism, Prometheus metric naming).
13. **Versioning** — MAJOR.MINOR semantics; receivers MAY refuse unknown majors, MUST ignore unknown minors.
14. **Security considerations** — token hashing, scope enforcement, HTTPS requirement, audit append-only-ness, elevated authorization for `force_internal`.
15. **References** — RDF 1.1, RDF-star, SPARQL 1.1, SHACL, OWL 2 RL, JSON-LD 1.1 + Framing, PROV-O, CloudEvents 1.0, pg_ripple roadmap.

**Why this lives in the pg_ripple repo for v1**: the spec is born from pg_ripple's implementation experience; co-locating ensures the reference implementation and the specification cannot drift. Once stable (post-v1.0 of pg_ripple), the spec is a candidate for donation to a W3C Community Group.

**Mapping document**: extend `docs/src/operations/pg-trickle-relay.md` with a "pg_ripple ↔ Bidi Profile mapping" section that maps each spec section to its pg_ripple realization (e.g. spec \u00a75.1 `next_event` \u2192 pg_ripple `pg_ripple.next_event(subscription_name TEXT)`; spec \u00a76 `reconciliation_enqueue` \u2192 pg_ripple `pg_ripple.reconciliation_enqueue(event_id UUID, divergence_summary JSONB)`).

**Conformance evidence**: identify which existing pg_regress tests / fixtures already cover each of the 15 conformance scenarios; file follow-up issues for any gaps. The 15 scenarios are derived from the fixture under `tests/fixtures/bidi/crm_erp_contacts/` plus the property-based and chaos harnesses (BIDIOPS-PROPTEST-01, BIDIOPS-CHAOS-01); coverage should be near-total at ship time with a documented gap list.

**Tests**:
- Markdown lint passes on the spec file.
- Every normative term (MUST / SHOULD / MAY / REQUIRED) is uppercase per RFC 2119.
- Every spec-section reference in `docs/src/operations/pg-trickle-relay.md`'s mapping table resolves to an existing section in `rdf-bidi-integration-v1.md`.
- A short `tests/spec_conformance_check.sh` script greps the spec for "MUST" clauses and reports the pg_regress test (or a TODO) covering each.

**Out of scope for v1 of the spec** (deferred to v2):

- Federated multi-store bidi (one bidi network spanning multiple Bidi-conformant stores). v1 is two-party (one Bidi store + N external systems).
- A formal model-theoretic semantics for `latest_wins` under arbitrary clock skew.
- Wire encryption beyond "use HTTPS"; v1 punts on JWT / mTLS specifics.
- A normative JSON Schema document for the wire format (planned for v1.1; v1 sketches it informally).
