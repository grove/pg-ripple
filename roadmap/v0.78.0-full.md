# v0.78.0 Full Technical Details - Bidirectional Integration Operations

> v0.78.0 closes the operational gaps identified in the v0.77.0 review. The data semantics from v0.77.0 are unchanged; this release adds the management plane that production deployments need: queue limits and dead-letter, per-subscription auth, frame-level redaction, schema-evolution policies, audit, property-based and chaos tests, a reconciliation toolkit, and a consolidated operations surface.

### BIDIOPS-QUEUE-01 — Outbox depth limits and dead-letter

**Root cause**: v0.77.0's BIDI-OUTBOX-01 writes events into pg-trickle outbox tables. pg-trickle owns delivery state and retry, but pg_ripple owns event payload creation and the subscription's write-side retention policy. A relay that goes down for hours can let the outbox grow unbounded — source-mutation throughput collapses if the bridge-writer transaction starts contending for space, and expired events with no consumer accumulate forever. pg-trickle's own dead-letter mechanism handles delivery-side failures (sink unreachable, decode error); pg_ripple still needs *write-side* policies to bound outbox growth and to express "events older than X are uninteresting".

**API additions** to `create_subscription`:

```sql
pg_ripple.create_subscription(
    name                  TEXT,
    frame                 JSONB,
    target_graph          TEXT     DEFAULT NULL,
    filter_sparql         TEXT     DEFAULT NULL,
    exclude_graphs        TEXT[]   DEFAULT NULL,
    propagation_depth     SMALLINT DEFAULT 1,
    rewrite_target_graph  TEXT     DEFAULT NULL,
    on_missing_rewrite    TEXT     DEFAULT 'emit_canonical',
    emit_base             BOOLEAN  DEFAULT TRUE,
    transaction_grouping  TEXT     DEFAULT 'subject',
    -- v0.78.0 additions:
    max_queue_depth       BIGINT   DEFAULT 1000000,    -- soft cap
    dead_letter_after     INTERVAL DEFAULT '7 days',   -- events older than this DLQ
    overflow_policy       TEXT     DEFAULT 'pause'     -- 'pause' | 'drop_oldest' | 'drop_newest'
) RETURNS VOID
```

**Catalog tables**:

```sql
-- Events that exceeded max_queue_depth, hit dead_letter_after, or were
-- explicitly dead-lettered by semantic reconciliation.
CREATE TABLE _pg_ripple.event_dead_letters (
    event_id          UUID        NOT NULL,
    subscription_name TEXT        NOT NULL,
    outbox_table      TEXT        NOT NULL,
    outbox_variant    TEXT        DEFAULT 'default', -- 'default' | 'redacted' | 'unredacted' | custom
    s                 BIGINT,                     -- distribution key for safe requeue on Citus
    payload           JSONB       NOT NULL,
    emitted_at        TIMESTAMPTZ NOT NULL,
    dead_lettered_at  TIMESTAMPTZ DEFAULT now(),
    reason            TEXT        NOT NULL,        -- 'queue_overflow' | 'expired' | 'paused_overflow' | 'reconciliation_dead_letter'
    extra             JSONB,
    last_attempt_at   TIMESTAMPTZ,
    PRIMARY KEY (subscription_name, outbox_table, event_id)
);
CREATE INDEX ON _pg_ripple.event_dead_letters (subscription_name, dead_lettered_at);
```

**Semantics**:

1. **Source mutations never block on outbox depth.** The bridge writer checks the outbox depth via a non-blocking estimate (`pg_class.reltuples` per outbox table, refreshed by autovacuum stats). Exact-count check happens lazily in the background-worker sweep.
2. **`overflow_policy = 'pause'`** (default): when the soft cap is hit, the bridge writer emits a `WARNING` once per minute summarising depth and signals the operator to pause the matching pg-trickle subscription (or the operator may have done so already; cf. BIDIOPS-PAUSE-01). New events continue to be appended to the outbox so no source mutation fails for queue reasons. A background sweep may move only rows that pg-trickle exposes as safe-to-dead-letter through a documented pg-trickle view/API; pg_ripple must not infer lease state from private relay columns. Safe overflow rows move from the outbox to `event_dead_letters` with `reason = 'paused_overflow'` until depth falls below `max_queue_depth * 0.9`. The pause state itself lives in pg-trickle's catalog — pg_ripple does not maintain a redundant flag.
3. **`overflow_policy = 'drop_oldest'`**: background sweep moves the oldest outbox rows (by `emitted_at`) to dead-letter as soon as depth exceeds the cap.
4. **`overflow_policy = 'drop_newest'`**: background sweep refuses to enqueue new events; the source mutation succeeds but the new event is appended directly to dead-letter with `reason = 'queue_overflow'`.
5. **Aged-out dead-lettering**: a background worker runs once per minute and moves any pg-trickle-safe outbox row with `emitted_at < now() - dead_letter_after` to `event_dead_letters` with `reason = 'expired'`. This is a *write-side* age policy; it is independent of pg-trickle's *delivery-side* retry/DLQ policy.
6. **pg-trickle's delivery-side dead-letter** (sink failures, decode errors, retry/lease exhaustion) is unchanged from v0.52.0 / pg-trickle's roadmap. pg_ripple's `event_dead_letters` table is for *outbox-side* and semantic policy only; pg-trickle has its own DLQ for *transport-side* failures. Operators see both via BIDIOPS-DASH-01.

**Operator API**:

```sql
-- List dead-lettered events for a subscription (paginated).
pg_ripple.list_dead_letters(
    subscription_name TEXT,
    outbox_table      TEXT        DEFAULT NULL,
    since             TIMESTAMPTZ DEFAULT NULL,
    limit_n           INT         DEFAULT 100
) RETURNS TABLE (event_id UUID, outbox_table TEXT, outbox_variant TEXT, payload JSONB, reason TEXT, dead_lettered_at TIMESTAMPTZ);

-- Re-enqueue a dead-lettered event (e.g. after fixing the relay or target system).
-- Inserts back into the subscription's outbox table with a fresh emitted_at;
-- preserves event_id for traceability.
pg_ripple.requeue_dead_letter(subscription_name TEXT, outbox_table TEXT, event_id UUID) RETURNS VOID;

-- Permanently drop a dead-lettered event after operator review.
pg_ripple.drop_dead_letter(subscription_name TEXT, outbox_table TEXT, event_id UUID) RETURNS VOID;
```

**HTTP additions**:

| Method | Path | Purpose |
|---|---|---|
| `GET`  | `/subscriptions/{name}/dead-letters?since=&limit=` | List dead-lettered events. |
| `POST` | `/subscriptions/{name}/dead-letters/{outbox_table}/{event_id}/requeue` | Re-enqueue. |
| `DELETE` | `/subscriptions/{name}/dead-letters/{outbox_table}/{event_id}` | Permanently drop. |

**Tests**:
- Cap at 100 events, push 200; verify the bridge writer emits the overflow warning, the first 100 stay in the outbox table, and rows beyond the cap land in `event_dead_letters` with `reason = 'paused_overflow'` without flipping a pg_ripple-side pause flag. Pause/resume delivery via pg-trickle and verify the outbox drains.
- `overflow_policy = 'drop_oldest'`: cap at 100, push 200; verify the queue holds the latest 100 and dead-letter holds the oldest 100.
- `overflow_policy = 'drop_newest'`: same setup; verify the queue holds the oldest 100 and dead-letter holds the newest 100.
- `dead_letter_after = '5 seconds'`: emit one event, do not consume; verify it moves to dead-letter after 5 seconds with `reason = 'expired'`.
- `requeue_dead_letter`: requeue an expired event; verify it appears as a new outbox row with the original `event_id` preserved and pg-trickle picks it up.
- Inbound-write isolation: simulate a subscription with depth 10× the soft cap; verify ingest throughput against an unrelated graph is within 5 % of baseline (gated under BIDIOPS-PERF-01).
- HTTP round-trip: list / requeue / drop dead-letters via REST.

### BIDIOPS-PAUSE-01 — Operator pause / resume (delegated to pg-trickle)

**Root cause**: Target systems go down for maintenance windows. Operators need a graceful "stop delivering outbound events for the next 30 minutes" without altering the subscription definition or risking event loss.

**Design decision**: pg-trickle already ships subscription pause/resume primitives (`pg_trickle.pause_subscription(name)` / `pg_trickle.resume_subscription(name)` / `pg_trickle.pause_all_subscriptions()`; see also the Citus rebalance example in `examples/citus_rebalance_with_trickle.sql`). pg_ripple defers to those rather than introducing parallel `pg_ripple.pause_subscription` / `pg_ripple.resume_subscription` SQL functions. The bridge writer keeps appending events to the outbox during a pause; pg-trickle stops draining them. When `overflow_policy = 'pause'` (BIDIOPS-QUEUE-01) and `max_queue_depth` is reached, additional events route to dead-letter — same code path as any other capacity event.

**What pg_ripple provides**:

1. `bidi_status()` (BIDIOPS-DASH-01) reports the pg-trickle pause state of each subscription's outbox by joining against `pg_trickle.subscriptions` and exposing a `pg_trickle_paused BOOLEAN` column plus the pg-trickle-supplied `pause_reason TEXT`.
2. `bidi_health()` returns `paused` if any subscription is reported paused by pg-trickle.
3. The audit log (BIDIOPS-AUDIT-01) records pause/resume calls observed via pg-trickle's audit table when present (`_pg_ripple.event_audit` does not duplicate pg-trickle's own audit; the join is documented).
4. The production checklist (BIDIOPS-DOC-01) tells operators to use `pg_trickle.pause_subscription` (not a pg_ripple wrapper) and links to pg-trickle's docs.

**Tests**:
- Pause via `pg_trickle.pause_subscription(<bidi outbox name>)`; verify the bridge writer continues to append outbox rows (assert `outbox_depth` grows in `bidi_status`); verify `bidi_status.pg_trickle_paused` is `true`; resume; verify pg-trickle drains the backlog.
- Pause + `overflow_policy = 'pause'` + sustained source mutations: verify rows accumulate up to `max_queue_depth`, then route to dead-letter.
- `bidi_health()` returns `paused` while any subscription is paused.

### BIDIOPS-EVOLVE-01 — Schema-evolution policies

**Root cause**: v0.77.0 freezes the wire `version` field but does not say what happens when in-flight subscriptions change shape mid-flight. Three concrete cases:

- The subscription's `frame` is updated to add or remove a predicate (e.g. CRM starts tracking `ex:vipStatus`).
- The `iri_template` / `iri_match_pattern` for a graph in `rewrite_target_graph` is updated (e.g. CRM moves from `https://crm.example.com/contacts/{id}` to `https://crm.example.com/v2/contacts/{id}`).
- `exclude_graphs` is updated (e.g. a new partner graph is added).

Without explicit policy, every operator builds their own ad-hoc story. Subtle data-loss bugs follow.

**API**: each `create_subscription` and corresponding `alter_subscription` accepts:

```sql
pg_ripple.alter_subscription(
    name                       TEXT,
    -- any field from create_subscription, plus:
    frame_change_policy        TEXT DEFAULT 'new_events_only', -- only supported v0.78 policy
    iri_change_policy          TEXT DEFAULT 'new_events_only', -- queued rows keep rendered IRIs
    exclude_change_policy      TEXT DEFAULT 'new_events_only'  -- queued rows drain unchanged
) RETURNS VOID;
```

**Frame change policies**:

- `new_events_only` (default and only v0.78 policy): queued pg-trickle outbox rows are already rendered and drain unchanged; events emitted after the `alter_subscription` transaction use the new frame. This aligns with the write-time rendering model and avoids mutating rows that pg-trickle may already be delivering.
- Deferred policies (`reframe_queued`, `drain_then_switch`) are explicitly out of scope for v0.78. They require an unrendered staging queue and a pg_ripple-owned render step, which would reintroduce the pull-time rendering layer this roadmap intentionally removed.

**IRI template/match-pattern change policies**:

- `new_events_only` (default and only v0.78 policy): queued events keep the original template/match-pattern IRIs; new events use the new template/match-pattern. Matches v0.77.0 BIDI-REF-01's "no retroactive rewrite" rule.
- Correcting a bad template or match pattern for already-rendered rows is an operator procedure: pause pg-trickle delivery, drop/requeue affected rows if appropriate, and document the action in `subscription_schema_changes`.

**Exclude change policies**:

- `new_events_only` (default and only v0.78 policy): the new `exclude_graphs` list applies only to events emitted after the alter. Queued events drain unchanged. This is correct when adding a new excluded graph (we don't want to discard already-emitted legitimate events that may already be in pg-trickle delivery).

**Audit table**:

```sql
CREATE TABLE _pg_ripple.subscription_schema_changes (
    subscription_name TEXT        NOT NULL,
    changed_at        TIMESTAMPTZ DEFAULT now(),
    changed_by        TEXT,                   -- session_user
    field             TEXT        NOT NULL,   -- 'frame' | 'iri_template' | 'iri_match_pattern' | 'exclude_graphs' | ...
    old_value         JSONB,
    new_value         JSONB,
    policy_applied    TEXT,
    affected_event_count BIGINT
);
CREATE INDEX ON _pg_ripple.subscription_schema_changes (subscription_name, changed_at);
```

**Tests**:
- Frame change: queue 5 events under frame v1, alter frame to add a predicate, emit one new event, and verify queued rows drain with frame v1 while the new row uses frame v2.
- `iri_template` / `iri_match_pattern` change: queue 5 events under template/pattern v1, alter it, emit one new event, and verify queued rows keep v1 IRIs while the new row uses v2.
- `exclude_graphs` change: queue 5 events from graph A, add A to `exclude_graphs`, emit one new event from graph A, and verify queued rows drain unchanged while the new event is suppressed.
- All three changes record one row per change in `subscription_schema_changes`.

### BIDIOPS-AUTH-01 — Per-subscription auth and scope tokens

**Root cause**: `pg_ripple_http` today supports bearer tokens for the entire HTTP surface. There is no way to say "this token can only act on the `crm_relay` subscription" or "this token can report divergence but not link back target IDs". In any deployment with more than one relay, the absence of per-subscription scoping is a meaningful security gap.

**Catalog table**:

```sql
CREATE TABLE _pg_ripple.subscription_tokens (
    token_hash        BYTEA       PRIMARY KEY,             -- SHA-256 of the raw token
    subscription_name TEXT        NOT NULL,
    scopes            TEXT[]      NOT NULL,                -- subset of {linkback, divergence, abandon, outbox_read, dead_letter_admin}
    label             TEXT,                                -- operator-friendly name
    created_at        TIMESTAMPTZ DEFAULT now(),
    last_used_at      TIMESTAMPTZ,
    revoked_at        TIMESTAMPTZ
);
CREATE INDEX ON _pg_ripple.subscription_tokens (subscription_name) WHERE revoked_at IS NULL;
```

**API**:

```sql
-- Returns the raw token (shown ONCE; subsequent calls return only metadata).
-- Token format: 'pgrt_' || base64url(32 random bytes).
pg_ripple.register_subscription_token(
    subscription_name TEXT,
    scopes            TEXT[]  DEFAULT ARRAY['linkback','divergence','abandon'],
    label             TEXT    DEFAULT NULL
) RETURNS TEXT;

pg_ripple.revoke_subscription_token(token_hash BYTEA) RETURNS VOID;

pg_ripple.list_subscription_tokens(subscription_name TEXT)
RETURNS TABLE (token_hash BYTEA, scopes TEXT[], label TEXT,
               created_at TIMESTAMPTZ, last_used_at TIMESTAMPTZ, revoked_at TIMESTAMPTZ);

-- Operator escape hatch: rotate by registering a new token, distributing it,
-- and revoking the old one once last_used_at is older than the deployment's
-- token-cache TTL.
```

**HTTP enforcement**:

- All `/subscriptions/{name}/events/{event_id}/*`, `/subscriptions/{name}/divergences/*`, and `/subscriptions/{name}/dead-letters/*` endpoints exposed by `pg_ripple_http` (the BIDI-INBOX-01 façade) require a bearer token whose `subscription_name` matches `{name}` and whose `scopes` includes the action being performed. The handler also verifies that `event_id` belongs to `{name}` before accepting:
    - `POST /subscriptions/{name}/events/{event_id}/linkback`  requires `linkback`.
    - `POST /subscriptions/{name}/events/{event_id}/divergence` requires `divergence`.
    - `POST /subscriptions/{name}/events/{event_id}/abandon` requires `abandon`.
  - `GET /dead-letters/...` and management actions require `dead_letter_admin`.
  - The optional `outbox_read` scope grants standard Postgres SELECT on the subscription's outbox table for relays that consume directly via psql/JDBC instead of pg-trickle.
- Mismatch → 403 with `{"error": "scope required: linkback"}`.
- Per-process LRU cache of `(token_hash → token row)` with 60-second TTL keeps verification fast; revocation visibility is bounded by the TTL (documented).
- A global admin token is still accepted for operator endpoints (`register_subscription_token`, dead-letter management); admin tokens are stored separately in `_pg_ripple.admin_tokens` and never carry per-subscription scope.

**Tests**:
- Register a token with `scopes => ARRAY['linkback']`; verify linkback works, divergence returns 403.
- Token for subscription A used against subscription B → 403.
- Revoke a token; verify subsequent calls fail (allowing for the 60-second cache TTL).
- Token cache: hammer the same token 10k times; verify only one DB lookup per TTL window.
- Rotation: register new, dispatch, verify both work concurrently, revoke old, verify last_used_at on old is up-to-date.

### BIDIOPS-REDACT-01 — Frame-level redaction

**Root cause**: Outbound events emit frame predicates verbatim. PII (`ex:taxId`, `ex:phone`, `ex:passportNumber`), secrets (`ex:apiKey`), and HIPAA-bearing predicates have no first-class suppression mechanism. Operators today must omit them from the subscription frame entirely — but then receivers that *should* see the cleartext (a separate trusted relay) can't.

**API**: extend the JSON-LD frame syntax with `"@redact": true`:

```json
{
  "@context":  { "ex": "https://example.com/ns#" },
  "@type":     "ex:Contact",
  "ex:name":   {},
  "ex:phone":  { "@redact": true },
  "ex:email":  {},
  "ex:taxId":  { "@redact": true }
}
```

**Semantics**:

1. Predicates marked `"@redact": true` are emitted in `after` and `base` as `{"@redacted": true}` (literal object, not a string).
2. Redacted predicates are not independently CAS-verifiable by a receiver that only sees the redacted outbox. Receivers may treat `{"@redacted": true}` on both sides as "pg_ripple says unchanged" for trusted pipelines, but any predicate that must be strictly CAS-checked by the target system must be delivered through an unredacted outbox or omitted from `base`.
3. **Unredacted vs redacted outbox tables**: redaction happens at *bridge-writer time*, not at pull time. A subscription that needs both audiences declares two outboxes: `crm_relay_outbox` (redacted) and `crm_relay_outbox_unredacted` (cleartext). The bridge writer renders the same source mutation into both, applying frame-level redaction to the first. Per-subscription auth tokens (BIDIOPS-AUTH-01) gate which outbox each consumer can read via standard Postgres GRANTs (the HTTP façade enforces the same scope check before returning side-band data; pg-trickle relay credentials govern sink delivery).
4. The frame's redaction declaration is the *only* source of truth — there is no separate "predicates allowed in unredacted mode" list. If a frame declares a predicate redacted, only consumers with access to the unredacted outbox see cleartext.
5. The frame's redaction declaration is the *only* source of truth for which predicates can be redacted. Internally, redacted and unredacted outbox tables are separate; the bridge writer renders each event twice (once per outbox) when both are configured. This is simpler and more auditable than runtime per-token rendering and aligns with standard Postgres GRANT-based table access control.

**Audit interaction**: when optional bridge-writer emit auditing is enabled and both a redacted and an unredacted outbox are configured for the same subscription, `event_audit` records `action = 'emit_unredacted'` on bridge-writer rows that go to the unredacted outbox, distinct from regular `'emit'` rows. Operators can audit which bridge-writer instances rendered cleartext without making per-row emit audit part of the required hot path.

**Tests**:
- Frame with `ex:phone` redacted; verify the redacted outbox row has `after["ex:phone"] == {"@redacted": true}` and the unredacted outbox row has cleartext.
- CAS with redacted predicate: verify the redacted receiver cannot compare against actual cleartext and either treats the redacted value as trusted pg_ripple state or escalates; strict receiver-side CAS is covered by the unredacted outbox.
- Frame change to add `@redact` to a predicate after events are queued: queued rows retain their original redaction state; rows emitted after the frame change use the new redaction declaration.
- With optional emit auditing enabled, audit records `emit_unredacted` distinct from `emit` for the same event when both outboxes are configured.

### BIDIOPS-AUDIT-01 — Audit log

**Root cause**: For compliance and forensics, every side-band mutation and privileged operator action needs to be recorded with who, what, and when. Outbox consumption itself is pg-trickle's responsibility and is audited by pg-trickle, not duplicated here.

**Catalog table**:

```sql
CREATE TABLE _pg_ripple.event_audit (
    audit_id          BIGSERIAL   PRIMARY KEY,
    event_id          UUID,
    subscription_name TEXT,
    resource_type     TEXT        NOT NULL,    -- 'event' | 'subscription' | 'token' | 'dead_letter' | 'reconciliation' | 'system'
    resource_id       TEXT,
    action            TEXT        NOT NULL,    -- 'linkback' | 'divergence' | 'abandon_linkback' | 'requeue_dead_letter' | 'drop_dead_letter' | optional 'emit'/'emit_unredacted'
    actor_token_hash  BYTEA,                   -- NULL for SQL-direct callers; populated for HTTP callers
    actor_session     TEXT,                    -- session_user when called via SQL
    http_remote_addr  INET,                    -- populated by pg_ripple_http
    observed_at       TIMESTAMPTZ DEFAULT now(),
    extra             JSONB                    -- linkback target_id, divergence summary, dead-letter requeue reason, etc.
);
CREATE INDEX ON _pg_ripple.event_audit (event_id) WHERE event_id IS NOT NULL;
CREATE INDEX ON _pg_ripple.event_audit (subscription_name, observed_at) WHERE subscription_name IS NOT NULL;
CREATE INDEX ON _pg_ripple.event_audit (resource_type, resource_id, observed_at);
CREATE INDEX ON _pg_ripple.event_audit (observed_at);  -- for retention sweep
```

**Retention**: GUC `pg_ripple.audit_retention` (default `90 days`). A background worker prunes rows older than the retention window once per hour. Setting the GUC to `0` disables retention pruning (manual archival required).

**SQL-direct caller attribution**: when a user calls a side-band helper (`record_linkback`, `reconciliation_*`, `requeue_dead_letter`) directly from psql, the audit row carries `actor_session = session_user` and `actor_token_hash = NULL`. HTTP-driven calls (via the BIDI-INBOX-01 façade in `pg_ripple_http`) populate `actor_token_hash` (the SHA-256 of the bearer token used) and `http_remote_addr`. Outbox row consumption itself is audited by pg-trickle (sink delivery) and is out of scope for `event_audit`; pg_ripple's audit covers only the side-band APIs that mutate hub state.

**Tests**:
- Linkback / divergence / abandon a pending event; verify one audit row per call with the correct fields. Optional bridge-writer `emit` audit can be enabled for regulated deployments but is not part of the required hot path.
- HTTP side-band calls populate `actor_token_hash` and `http_remote_addr`; SQL calls populate `actor_session` only.
- Retention sweep: set retention to 1 second; insert a row, wait 2 seconds, run the worker; verify the row is gone.
- Dead-letter requeue is audited.

### BIDIOPS-PROPTEST-01 — Property-based convergence harness

**Root cause**: v0.77.0 ships per-feature unit tests and one CRM⇄ERP fixture. There is no test that random sequences of inserts, updates, and deletes from N concurrent sources converge to a deterministic resolved projection. World-class CDC systems (Materialize, FoundationDB) have property-based test suites of this shape.

**Implementation**: Rust `proptest` crate. Strategies generate:

- Random source-graph IRIs from a small fixed set (`{<urn:source:s1>, ..., <urn:source:s4>}`).
- Random subjects from a small fixed set (`{ex:e1, ..., ex:e10}`) — keeping cardinality small ensures conflicts actually happen.
- Random predicates from a fixed schema with mixed `sh:maxCount 1` / unbounded.
- Random literal values (strings, integers, dates).
- Random source timestamps within a fixed window.
- Random operations: `insert(graph, subject, predicate, value, timestamp)`, `update(...)`, `delete(graph, subject, predicate)`, `delete_subject(graph, subject)`.

**Properties asserted**:

1. **Determinism**: applying the same sequence twice (with `latest_wins` everywhere) yields the same `_pg_ripple.conflict_winners`.
2. **Order-independence under `latest_wins`**: shuffling a sequence preserves the resolved projection (because comparison is on source timestamps, not arrival order).
3. **No-loss invariant**: every triple ever inserted is recoverable via a graph-scoped query of its source graph (losing assertions are non-destructive).
4. **`source_priority` correctness**: with `source_priority [s1, s2]`, the resolved value for a `(subject, predicate)` slot is always from the highest-priority source that has a value, regardless of timestamp.
5. **Linkback round-trip**: a sequence of `INSERT → relay-create → record_linkback → pg-trickle delivery completion → subsequent UPDATE` produces the same final state as `INSERT → direct rewrite of subject IRI → UPDATE`.
6. **Convergence under retries**: any subset of operations replayed (idempotent CDC) yields the same final state as a single application.

**Setup**: each property test starts a fresh pg_ripple instance via `cargo pgrx test`, applies the random sequence via SPI, and asserts properties via SPARQL queries.

**Tests**: required CI runs a bounded, deterministic seed set (for example 1,000 random sequences with shrinking on failure). Nightly/stress CI runs the full 100k+ sequence corpus and records the seed for any failing case.

### BIDIOPS-CHAOS-01 — Jepsen-style fault injection

**Root cause**: v0.77.0 documents crash-recovery semantics for `record_linkback` and the subscription buffer, but no test actually crashes pg_ripple or the relay mid-flow.

**Implementation**: a Rust harness using `tokio` to drive concurrent inbound writes, pg-trickle outbound delivery, linkbacks, and side-band callbacks while injecting controlled faults:

- **Crash 1**: kill the relay process after pg-trickle has delivered an outbox row but before the relay calls `record_linkback` via the inbox. Verify the delivery is replayed (pg-trickle redelivery) and `record_linkback` lands exactly once when the relay restarts.
- **Crash 2**: kill the relay process between `record_linkback` and the next outbox-row consumption. Verify the buffered events flush to the outbox on restart and the relay sees them in order.
- **Crash 3**: kill pg_ripple (SIGKILL) mid-`conflict_winners` recompute.
- **Crash 4**: kill pg_ripple mid-`record_linkback` flush (between writing `owl:sameAs` and emitting buffered events).
- **Crash 5**: kill pg-trickle's relay mid-delivery; verify pg-trickle's retry/DLQ policy handles redelivery and pg_ripple's outbox/inbox invariants remain intact.
- **Network**: inject a 10s sink timeout in pg-trickle; verify pg-trickle owns retry/lease behaviour and pg_ripple reports the delivery-side backlog via `bidi_status()` without attempting its own ack/nack.

**Properties asserted after each fault**:

1. No duplicate target creates: every `record_linkback(target_id => ...)` for the same hub subject either succeeds idempotently or surfaces the existing `owl:sameAs`.
2. No lost CDC events: every source mutation produces a row in either the subscription's outbox table, `event_dead_letters`, or pg-trickle's delivery-side DLQ. Combined coverage is total; intersection is empty.
3. Queue eventually drains: with the relay restored and pg-trickle delivery healthy, queue depth converges to zero.
4. Dead-letter contents are exactly the events that hit a documented policy limit; no event is in dead-letter without a corresponding `reason`.
5. `subscription_buffer` rows are flushed in `sequence` order even if `record_linkback` is retried after a crash.
6. Audit log entries are not duplicated by retries: a successful `record_linkback` after a crash recovers and emits exactly one audit row.

**Setup**: docker-compose with pg_ripple, `pg_ripple_http`, and a fake target system; a controller process that orchestrates the faults and asserts the properties. Required CI runs a smoke subset with deterministic fault points; nightly/stress CI runs the full process-kill and network-timeout matrix.

### BIDIOPS-RECON-01 — Reconciliation toolkit

**Root cause**: v0.77.0's receiver guidance ("on CAS divergence: re-apply / escalate / force / dead-letter") is prose only — there are no in-database primitives. Operators building bespoke relays reimplement the queue, the dispatch table, and the operator UI from scratch every time.

**Catalog table**:

```sql
CREATE TABLE _pg_ripple.reconciliation_queue (
    reconciliation_id BIGSERIAL   PRIMARY KEY,
    event_id          UUID        NOT NULL,
    subscription_name TEXT        NOT NULL,
    enqueued_at       TIMESTAMPTZ DEFAULT now(),
    leased_until      TIMESTAMPTZ,
    leased_by         TEXT,
    divergence_summary JSONB      NOT NULL,                -- {"predicate_iri": {"actual": ..., "base": ..., "after": ...}}
    resolved_at       TIMESTAMPTZ,
    resolution        TEXT,                                -- 'accept_external' | 'force_internal' | 'merge_via_owl_sameAs' | 'dead_letter'
    resolved_by       TEXT,
    resolution_note   TEXT
);
CREATE INDEX ON _pg_ripple.reconciliation_queue (subscription_name, leased_until, enqueued_at)
    WHERE resolved_at IS NULL;
```

**API**:

```sql
-- Called by the relay client when CAS fails and divergence is detected
-- (some actual values differ from base in ways inconsistent with after).
pg_ripple.reconciliation_enqueue(
    event_id            UUID,
    divergence_summary  JSONB
) RETURNS BIGINT;  -- reconciliation_id

-- Operator pulls the next unresolved divergence (lease + skip-locked).
pg_ripple.reconciliation_next(subscription_name TEXT)
RETURNS TABLE (reconciliation_id BIGINT, event_id UUID, payload JSONB,
               divergence_summary JSONB, enqueued_at TIMESTAMPTZ);

-- Operator resolves the divergence with one of four actions.
pg_ripple.reconciliation_resolve(
    reconciliation_id BIGINT,
    action            TEXT,             -- 'accept_external' | 'force_internal' | 'merge_via_owl_sameAs' | 'dead_letter'
    note              TEXT DEFAULT NULL
) RETURNS VOID;
```

**Resolution semantics**:

- **`accept_external`**: ingest the external system's actual values into the source graph as if they had arrived via `ingest_json`; emit a NOTICE; mark the reconciliation entry resolved as no-op for the original event. The external value becomes the resolved-projection winner if conflict policies say so.
- **`force_internal`**: re-emit the same event as a "force overwrite" (ignoring `base`); the relay applies it unconditionally on the next pg-trickle delivery.
- **`merge_via_owl_sameAs`**: assert `owl:sameAs` between the two divergent values' subjects (used when divergence reveals a duplicate that should have been merged earlier); mark the reconciliation entry resolved.
- **`dead_letter`**: move the event to `event_dead_letters` with `reason = 'reconciliation_dead_letter'` and the divergence summary in `extra`.

**HTTP endpoints**:

| Method | Path | Purpose |
|---|---|---|
| `POST` | `/events/{event_id}/divergence` | Enqueue (relay → pg_ripple). |
| `GET`  | `/subscriptions/{name}/divergences?limit=N` | Pull (operator). |
| `POST` | `/divergences/{reconciliation_id}/resolve` | Resolve. |

**Tests**:
- Receiver simulation: change `actual["ex:phone"]` between two events; second event's CAS fails; verify `reconciliation_enqueue` is called with the right divergence summary.
- All four resolution actions exercised end-to-end with verified post-state.
- Concurrent operators: two `reconciliation_next` calls return different rows.

### BIDIOPS-DASH-01 — Consolidated operations surface

**Root cause**: v0.77.0 has `graph_stats()` per graph but no per-subscription health view. Operators monitoring N relays today need to query 5 different tables.

**API**:

```sql
-- One row per subscription. The pause columns are looked up from pg_trickle.subscriptions
-- via LEFT JOIN; pg_ripple does not maintain a redundant pause flag.
pg_ripple.bidi_status() RETURNS TABLE (
    subscription_name        TEXT,
    pg_trickle_paused        BOOLEAN,             -- from pg_trickle.subscriptions
    pg_trickle_pause_reason  TEXT,                -- from pg_trickle.subscriptions
    outbox_depth             BIGINT,
    outbox_oldest_age        INTERVAL,
    dead_letter_count        BIGINT,
    conflict_rejection_rate  DOUBLE PRECISION,    -- per minute, last 5 min
    pending_linkback_count   BIGINT,
    pending_linkback_oldest_age INTERVAL,
    rewrite_miss_count_24h   BIGINT,
    last_emit_at             TIMESTAMPTZ,
    pg_trickle_last_delivery_at TIMESTAMPTZ,
    pg_trickle_last_error    TEXT,
    pg_trickle_retry_count   BIGINT,
    pg_trickle_delivery_dlq_count BIGINT,
    reconciliation_open      BIGINT
);

-- Single overall status with triggering conditions.
pg_ripple.bidi_health() RETURNS TABLE (
    status     TEXT,                 -- 'healthy' | 'degraded' | 'paused' | 'failing'
    reasons    TEXT[],               -- e.g. ['crm_relay queue depth 850000', 'erp_relay paused 30 min']
    checked_at TIMESTAMPTZ
);
```

**Health rules**:

- `paused`: any subscription is reported paused by `pg_trickle.subscriptions`.
- `degraded`: any subscription outbox depth > 80 % of `max_queue_depth`, OR any pending linkback older than `linkback_timeout * 0.5`, OR open reconciliation count > 0.
- `failing`: any subscription outbox depth at `max_queue_depth`, OR any dead-letter rate > 1/minute, OR any `pending_linkback_oldest_age > linkback_timeout`.
- `healthy`: none of the above.

**HTTP**:

- `GET /bidi/status` — returns `bidi_status()` as JSON.
- `GET /bidi/health` — returns `bidi_health()`. HTTP status code mirrors: `healthy` → 200, `degraded`/`paused` → 200 with body, `failing` → 503.

**Prometheus metrics**: `pg_ripple_http` exposes per-subscription gauges (`pgripple_outbox_depth`, `pgripple_dead_letter_count`, `pgripple_pending_linkback_count`, `pgripple_conflict_rejection_rate`) so existing observability stacks pick them up automatically. Delivery-side metrics (rows delivered, sink errors) come from pg-trickle's own metrics endpoint.

**Tests**:
- After ingest with conflicts, verify `conflict_rejection_rate` is non-zero.
- After pausing a subscription, verify `bidi_health()` returns `paused`.
- After hitting `max_queue_depth`, verify `failing`.
- HTTP `/bidi/health` returns 503 when failing.

### BIDI-SPEC-01 — RDF Bidirectional Integration Profile (vendor-neutral spec)

**Root cause**: Roughly 70 % of v0.77.0 + v0.78.0 — conflict policies, late-binding rewrite via `owl:sameAs`, sparse-`base` CAS with framing symmetry, target-assigned-ID linkback, flat wire format with `Link: rel="describedby"`, frame-level redaction, four-action reconciliation, schema-evolution policies, audit, scope tokens, convergence properties — is not Postgres-specific. It is a missing layer in the RDF ecosystem: every triplestore has its own CDC story; none has a coherent *bidirectional integration* story. Without a vendor-neutral specification, pg_ripple's design value gets read as "another Postgres-extension feature" rather than as defining the protocol.

**Deliverable**: `docs/spec/rdf-bidi-integration-v1.md` — a standalone, vendor-neutral **draft** specification. It is intentionally non-blocking for v0.78.0: the implementation ships against the pg_ripple roadmap and pg-trickle contract first, and the draft captures the store-agnostic parts for review. Any remaining mismatch between the draft and the implementation is filed as a spec issue, not a release blocker. Structure:

1. **Motivation** — the eight integration concerns existing CDC mechanisms do not address.
2. **Candidate conformance levels** — Bidi-1.0 (data + protocol) and Bidi-1.0-Ops (additionally schema evolution, auth, redaction, audit, ops surface); non-normative until the draft stabilizes.
3. **Data model** — sources as named graphs (`urn:source:` convention), per-source `prov:generatedAtTime` annotations, four conflict-resolution policies, normalize-aware echo suppression, `owl:sameAs` equivalence.
4. **Wire format** — flat JSON event with `version` discriminator, JSON-LD `after`/`base`, framing-symmetry rule for byte-string CAS, HTTP `Link: rel="describedby"` schema advertisement, object-level grouping.
5. **Receiver protocol** — outbox-row payload as canonical wire format; sparse-CAS algorithm; linkback for target-assigned IDs; subscription buffering during pending linkback (initial unresolved INSERT is visible; follow-up events are held outside the outbox until linkback lands); late-binding IRI rewrite; loop prevention via `exclude_graphs` + `propagation_depth`. The spec is transport-agnostic: it MUST NOT mandate pg-trickle, but recommends Postgres-native outbox/inbox tables as one conformant transport.
6. **Reconciliation** — `reconciliation_enqueue` / `reconciliation_next` and the four resolution actions (`accept_external` / `force_internal` / `merge_via_owl_sameAs` / `dead_letter`).
7. **Schema evolution** — frame, IRI-pattern, exclude-graphs change policies.
8. **Auth and redaction** — per-subscription scope tokens (SHA-256 hashed at rest), frame-level `@redact: true`, `unredacted` scope.
9. **Audit** — required fields and retention.
10. **Operations surface** — required per-subscription status fields and overall health vocabulary.
11. **Candidate conformance test corpus** — 15 black-box scenarios; cross-referenced to pg_ripple tests where they already exist, with gaps documented rather than blocking the release.
12. **Non-normative reference implementation** — explicit pointer to pg_ripple with a list of pg_ripple-specific concerns explicitly *out of scope* for the spec (HTAP storage layout, queue substrate, HTTP companion, migration mechanism, Prometheus metric naming).
13. **Versioning** — MAJOR.MINOR semantics; receivers MAY refuse unknown majors, MUST ignore unknown minors.
14. **Security considerations** — token hashing, scope enforcement, HTTPS requirement, audit append-only-ness, elevated authorization for `force_internal`.
15. **References** — RDF 1.1, RDF-star, SPARQL 1.1, SHACL, OWL 2 RL, JSON-LD 1.1 + Framing, PROV-O, CloudEvents 1.0, pg_ripple roadmap.

**Why this lives in the pg_ripple repo for v1**: the spec is born from pg_ripple's implementation experience; co-locating ensures the reference implementation and the specification cannot drift. Once stable (post-v1.0 of pg_ripple), the spec is a candidate for donation to a W3C Community Group.

**Mapping document**: extend `docs/src/operations/pg-trickle-relay.md` with a "pg_ripple ↔ Bidi Profile mapping" section that maps each spec section to its pg_ripple realization (e.g. spec receiver delivery → pg-trickle outbox/relay semantics; spec linkback → pg_ripple `pg_ripple.record_linkback(event_id UUID, ...)`; spec reconciliation → pg_ripple `pg_ripple.reconciliation_enqueue(event_id UUID, divergence_summary JSONB)`). The mapping must explicitly state that pg_ripple does **not** expose `next_event` / `ack_event` / `nack_event`; those are transport responsibilities delegated to pg-trickle or to a direct-table polling adapter.

**Conformance evidence**: identify which existing pg_regress tests / fixtures already cover each of the 15 candidate scenarios; file follow-up issues for any gaps. The 15 scenarios are derived from the fixture under `tests/fixtures/bidi/crm_erp_contacts/` plus the property-based and chaos harnesses (BIDIOPS-PROPTEST-01, BIDIOPS-CHAOS-01); coverage is informational for v0.78.0.

**Tests**:
- Markdown lint passes on the spec file.
- Every normative term (MUST / SHOULD / MAY / REQUIRED) is uppercase per RFC 2119.
- Every spec-section reference in `docs/src/operations/pg-trickle-relay.md`'s mapping table resolves to an existing section in `rdf-bidi-integration-v1.md`.
- A short `tests/spec_conformance_check.sh` script lists draft requirement clauses and reports the pg_regress test (or a TODO) covering each; TODOs are allowed in v0.78.0.

**Out of scope for v1 of the spec** (deferred to v2):

- Federated multi-store bidi (one bidi network spanning multiple Bidi-conformant stores). v1 is two-party (one Bidi store + N external systems).
- A formal model-theoretic semantics for `latest_wins` under arbitrary clock skew.
- Wire encryption beyond "use HTTPS"; v1 punts on JWT / mTLS specifics.
- A normative JSON Schema document for the wire format (planned for v1.1; v1 sketches it informally).
