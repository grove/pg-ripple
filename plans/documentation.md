# pg_ripple — Documentation Implementation Plan

> **Date**: 2026-04-15
> **Scope**: Documentation site structure, tooling, and milestone-by-milestone delivery plan
> **Inspired by**: pg_trickle (mdbook + GitHub Pages)

## Overview

pg_ripple's documentation will be a static site generated by [mdBook](https://rust-lang.github.io/mdBook/) and published to GitHub Pages. The site is the primary destination for users, operators, and contributors. It lives in a `docs/` directory at the root of the repository and is published automatically on every push to `main`.

The documentation is organized into three top-level books or sections:

| Section | Audience | Purpose |
|---|---|---|
| **User Guide** | Users and operators | Task-oriented: install, query, configure, scale |
| **Reference** | Users, contributors | Look-up: error codes, FAQ, changelog, runbook |
| **Research** | Contributors, curious readers | Architecture rationale, prior art, design decisions |

## Tooling Setup

### mdBook

```toml
# docs/book.toml
[book]
title = "pg_ripple"
authors = ["pg_ripple contributors"]
src = "src"
language = "en"

[build]
build-dir = "book"

[output.html]
default-theme = "navy"
preferred-dark-theme = "navy"
git-repository-url = "https://github.com/grove/pg-ripple"
edit-url-template = "https://github.com/grove/pg-ripple/edit/main/docs/src/{path}"
additional-css = ["theme/custom.css"]

[output.html.search]
enable = true
limit-results = 20
use-boolean-and = true
```

Optional preprocessors to add over time:
- `mdbook-mermaid` — for architecture diagrams (add at v0.6.0 when HTAP diagrams are useful)
- `mdbook-linkcheck` — CI validation that all internal and external links are live (add from day one)

### GitHub Actions — Publish to GitHub Pages

```yaml
# .github/workflows/docs.yml
name: Publish documentation
on:
  push:
    branches: [main]
    paths: ['docs/**', 'CHANGELOG.md', 'ROADMAP.md', 'RELEASE.md']
  workflow_dispatch:
permissions:
  contents: read
  pages: write
  id-token: write
jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install mdBook
        run: cargo install mdbook --locked
      - name: Sync mirrored files
        run: |
          cp CHANGELOG.md docs/src/reference/changelog.md
          cp ROADMAP.md docs/src/reference/roadmap.md
          cp RELEASE.md docs/src/reference/release-process.md
          # Research mirrors
          cp plans/ecosystem/prior_art.md docs/src/research/prior-art.md
          # (add more as needed)
      - name: Build
        run: mdbook build docs
      - name: Deploy
        uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: docs/book
```

The "sync mirrored files" step copies content from `plans/` and the project root into the docs tree at build time. This avoids duplication in the repository while keeping the docs site coherent.

## Cross-Cutting Conventions

### Writing Style

- **Lead with what the user can do**, not how the implementation works. "Load a Turtle file in one call" before any mention of VP tables.
- **One working example per function** in the SQL Reference — copy-paste ready, not illustrative pseudocode.
- Short paragraphs; headers every 3–4 paragraphs.
- Technical implementation detail in a collapsible `<details>` block or a clearly marked "Internal / Advanced" subsection.
- Use the same term formatting as the rest of the project: IRIs in `<angle brackets>`, literals in `"double quotes"^^xsd:type`, blank nodes as `_:label`.

### SUMMARY.md Maintenance

`SUMMARY.md` is the mdBook table of contents and must always reflect the current state of the docs. Every new page created at a milestone must be added to `SUMMARY.md` in the same commit. Stub pages (placeholder `# Page title\n\n> Coming soon.`) are preferred over missing entries.

### Mirrored Files

Files mirrored from `plans/` or the project root are **copied at CI build time** — they are not symlinked in the repository. This keeps `git status` clean and avoids mdBook misreading relative links in the source. The CI step that copies them is listed in the `docs.yml` workflow definition above and must be extended each time a new mirror is added.

### Versioned Callouts

When a feature was added in a specific version, add a callout at the top of the relevant section:

```markdown
> **Available since v0.4.0**
```

This helps users on older versions understand which pages are relevant to them.

### Link to ROADMAP

Every SQL Reference page should include a one-line note linking to the roadmap milestone where the feature was introduced. This creates a bidirectional trail: the roadmap links to code, the docs link back to the roadmap.

## Open Questions

1. **Research section**: should the `plans/` documents be lightly edited for the docs audience, or published verbatim? The implementation plan documents contain Rust-level detail that may not serve the general docs reader. Suggest: publish verbatim for now, add a short introduction paragraph per document, and refine later.

2. **SQL Reference depth**: the reference currently groups by function. An alternative is to group by use case (e.g. "Loading Data", "Querying Data", "Managing Shapes"). Worth reconsidering once the reference reaches ~20+ functions and becomes hard to navigate by function name alone.

3. **Versioned docs**: does each released version need a snapshot of the docs (e.g. `docs.pg-ripple.io/v0.3.0/`)? mdBook supports this via the `[output.html.redirect]` table but requires a more complex CI workflow. Defer until 1.0.0 when the API stabilizes enough to make older docs worth preserving.

4. **Playground hosting**: the playground page today points to a local Docker command. If containerized cloud playgrounds (e.g. Killercoda, Play with Docker) become viable, the playground page should link to a hosted sandbox. Flag for review at v0.5.0 when SPARQL is mature.
